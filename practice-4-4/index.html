4.4 · Logic Programming — Coding Practice

◷ approx. 11 min read · 31 worked examples

These problems build a feel for declarative logic programming by working on the Python matching model the section constructs: facts stored as tuples and nested lists, queries posed as patterns with ?-variables, and the unify/match machinery that searches for substitutions. You will write flat and recursive matchers, derive compound and recursive relations like child, ancestor, and pedigree, and run a relational append in both directions.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Match one fact

A logic query is a pattern matched against a stored fact. Write unify_pair(pattern, fact) that matches two equal-length tuples. Walk them in parallel: a token starting with ? is a variable that binds to the matching value (and must not contradict an earlier binding of the same variable), and any other token must equal its counterpart exactly. Return a dict of variable bindings on success, or None if the lengths differ or any non-variable token fails to match.

Examples

Input:  unify_pair(("parent", "abraham", "?child"), ("parent", "abraham", "barack"))
Output: {'?child': 'barack'}
Explanation: "parent" and "abraham" match exactly; ?child binds to "barack".

Input:  unify_pair(("parent", "abraham", "?child"), ("parent", "delano", "herbert"))
Output: None
Explanation: "abraham" does not equal "delano", so the match fails.

Input:  unify_pair(("parent", "?p", "?p"), ("parent", "abraham", "barack"))
Output: None
Explanation: ?p binds to "abraham", then clashes with "barack" at its second use.

Constraints

  • pattern and fact are tuples of strings; variables are tokens beginning with ?.
  • Do not import anything; build the bindings dict yourself.

Solutions

Setup

These problems share the recursive list matcher match from the 4.4-B Step-Through Checkpoint, the parent facts from the teaching documents, and the hierarchical dog records. Define them once before running the solutions:

1. Match one fact

Approach — flat unification. This is the unify matcher from the 4.4-A Step-Through Checkpoint: walk pattern and fact in parallel, bind each ?-token, and demand exact equality everywhere else. A variable that reappears must keep its first value, which is why (parent ?p ?p) rejects a fact with two different objects.

def unify_pair(pattern, fact):
    if len(pattern) != len(fact):
        return None
    result = {}
    for token, value in zip(pattern, fact):
        if token.startswith("?"):
            if token in result and result[token] != value:
                return None
            result[token] = value
        elif token != value:
            return None
    return result

Why it works. Each position either records a binding or insists on an exact match, and a contradicting rebinding immediately returns None, exactly the matching rule the query interpreter uses for one pattern against one fact.

Complexity. Time O(n), Space O(n), for n the tuple length (the bindings dict).

assert unify_pair(("parent", "abraham", "?child"),
                  ("parent", "abraham", "barack")) == {"?child": "barack"}
assert unify_pair(("parent", "abraham", "?child"),
                  ("parent", "delano", "herbert")) is None
assert unify_pair(("parent", "?p", "?p"),
                  ("parent", "abraham", "barack")) is None
assert unify_pair(("parent", "?p", "?c"),
                  ("parent", "fillmore", "grover")) == {"?p": "fillmore",
                                                        "?c": "grover"}
assert unify_pair(("a", "b"), ("a", "b", "c")) is None

2. Run a one-variable query

Approach — slide the stencil. A query is a stencil slid over every fact; the matches that survive report what the variable must be. Try unify_pair against each fact, and collect the variable's value when the match succeeds, keeping only distinct values in first-seen order.

def query_var(facts, pattern, var):
    results = []
    for fact in facts:
        bindings = unify_pair(pattern, fact)
        if bindings is not None and var in bindings:
            value = bindings[var]
            if value not in results:
                results.append(value)
    return results

Why it works. Each successful unification yields the substitution for that fact; gathering the variable's value across all matches is precisely what (query (parent abraham ?child)) reports.

Complexity. Time O(f⋅n), Space O(f), for f facts of length n.

assert query_var(PARENTS, ("parent", "abraham", "?child"), "?child") == \
    ["barack", "clinton"]
assert query_var(PARENTS, ("parent", "fillmore", "?child"), "?child") == \
    ["abraham", "delano", "grover"]
assert query_var(PARENTS, ("parent", "?p", "fillmore"), "?p") == \
    ["eisenhower"]
assert query_var(PARENTS, ("parent", "millard", "?child"), "?child") == []

3. Children of a dog

Approach — compound fact child. The rule (child ?c ?p) (parent ?p ?c) says the children of a dog are the third slot of every parent fact whose second slot is that dog. A single scan collects them.

def children_of(parents, name):
    children = []
    for relation, parent, child in parents:
        if parent == name:
            children.append(child)
    return children

Why it works. Fixing the parent slot to name and reading off the child slot is exactly the derivation the child rule performs against the parent facts.

Complexity. Time O(f), Space O(k), for f facts and k children found.

assert children_of(PARENTS, "fillmore") == ["abraham", "delano", "grover"]
assert children_of(PARENTS, "abraham") == ["barack", "clinton"]
assert children_of(PARENTS, "delano") == ["herbert"]
assert children_of(PARENTS, "herbert") == []

4. Negation as failure

Approach — proof, then negate. not does not search for a counterexample; it tries to prove the inner relation and succeeds only when no proof exists. So scan the facts: the moment one matches, the inner relation is proved and not fails; if none matches, not succeeds.

def not_holds(facts, pattern):
    for fact in facts:
        if unify_pair(pattern, fact) is not None:
            return False
    return True

Why it works. A pattern with a variable still counts as proved if any fact unifies with it, which is why (not (parent abraham ?who)) fails — ?who binds to barack, so the inner relation succeeds and its negation cannot.

Complexity. Time O(f⋅n), Space O(n), for f facts of length n.

assert not_holds(PARENTS, ("parent", "abraham", "barack")) is False
assert not_holds(PARENTS, ("parent", "abraham", "?who")) is False
assert not_holds(PARENTS, ("parent", "barack", "?who")) is True
assert not_holds(PARENTS, ("parent", "grover", "herbert")) is True

5. Match a nested record

Approach — recursive list matching. The match helper descends into nested sub-lists, binding a ?-variable to a single symbol or to a whole sub-list depending on where it sits, and a trailing dot variable to the rest. The solution just supplies the empty starting environment.

def match_record(pattern, data):
    return match(pattern, data, {})

Why it works. match already encodes the structural rules — descend into equal-length lists, let a variable swallow a whole sub-list, let a dot variable capture the tail — so the wrapper only kicks off the search with no bindings yet.

Complexity. Time O(s), Space O(s), for s the total number of nodes in the structure (recursion plus environment copies).

assert match_record(["dog", ["name", "clinton"], ["color", "?color"]],
                    ["dog", ["name", "clinton"], ["color", "white"]]) == \
    {"?color": "white"}
assert match_record(["dog", ["name", "clinton"], "?info"],
                    ["dog", ["name", "clinton"], ["color", "white"]]) == \
    {"?info": ["color", "white"]}
assert match_record(["?first", ".", "?rest"], ["a", "b", "c"]) == \
    {"?first": "a", "?rest": ["b", "c"]}
assert match_record(["dog", ["name", "barack"], ["color", "?color"]],
                    ["dog", ["name", "clinton"], ["color", "white"]]) is None

6. All ancestors

Approach — recursive fact. Mirror the two ancestor facts: the base case is a direct parent, and the recursive case takes one parent step then asks the ancestor question again on the shorter chain. For each fact whose child is name, add the parent and then its ancestors, skipping any name already seen.

def ancestors(parents, name):
    result = []
    for relation, parent, child in parents:
        if child == name:
            if parent not in result:
                result.append(parent)
            for older in ancestors(parents, parent):
                if older not in result:
                    result.append(older)
    return result

Why it works. A direct parent is added by the base step; the recursive call climbs one rung at a time, so the result lists each ancestor one rung higher than the last — exactly the chain (ancestor ?a name) proves.

Complexity. Time O(f⋅d), Space O(d), for f facts and chain depth d.

assert ancestors(PARENTS, "herbert") == ["delano", "fillmore", "eisenhower"]
assert ancestors(PARENTS, "barack") == ["abraham", "fillmore", "eisenhower"]
assert ancestors(PARENTS, "fillmore") == ["eisenhower"]
assert ancestors(PARENTS, "eisenhower") == []

7. Common ancestors

Approach — shared-variable compound query. A repeated variable across subexpressions must take one shared value, so (ancestor ?a barack) (ancestor ?a herbert) keeps only the dogs in both ancestor lists. Compute each list, then keep the members of one that also appear in the other.

def common_ancestors(parents, name_a, name_b):
    ancestors_a = ancestors(parents, name_a)
    ancestors_b = ancestors(parents, name_b)
    shared = []
    for who in ancestors_a:
        if who in ancestors_b:
            shared.append(who)
    return shared

Why it works. Intersecting the two ancestor lists enforces the shared-variable constraint of the compound query: a dog qualifies only when it is simultaneously an ancestor of both.

Complexity. Time O(a⋅b), Space O(a), for ancestor-list sizes a and b.

assert common_ancestors(PARENTS, "barack", "herbert") == \
    ["fillmore", "eisenhower"]
assert common_ancestors(PARENTS, "barack", "clinton") == \
    ["abraham", "fillmore", "eisenhower"]
assert common_ancestors(PARENTS, "barack", "grover") == \
    ["fillmore", "eisenhower"]
assert common_ancestors(PARENTS, "herbert", "abraham") == ["fillmore",
                                                           "eisenhower"]

8. Dogs of a color

Approach — query inside a hierarchical fact. Fix the color field and leave the name open: the pattern ["dog", ["name", "?name"], ["color", color]] matches a record only when its color equals color, and match then binds ?name to that record's name.

def dogs_by_color(dogs, color):
    pattern = ["dog", ["name", "?name"], ["color", color]]
    names = []
    for record in dogs:
        env = match(pattern, record, {})
        if env is not None:
            names.append(env["?name"])
    return names

Why it works. match succeeds only when every fixed field lines up, so the color slot acts as a filter while ?name reports the matching dog — the hierarchical-query pattern from the docs.

Complexity. Time O(d⋅s), Space O(k), for d records of size s and k matches.

assert dogs_by_color(DOGS, "white") == ["abraham", "clinton", "delano"]
assert dogs_by_color(DOGS, "tan") == ["barack", "eisenhower", "grover"]
assert dogs_by_color(DOGS, "brown") == ["fillmore", "herbert"]
assert dogs_by_color(DOGS, "spotted") == []

9. Every way to split a list

Approach — append run backward. The relation (append-to-form ?left ?right whole) enumerates every split of whole. The base case (append-to-form () ?x ?x) gives the empty-left split; the recursive fact prepends the head onto the left of each split of the rest. Mirror that exactly in Python recursion.

def splits(whole):
    if whole == []:
        return [([], whole[:])]
    first = whole[0]
    rest = whole[1:]
    result = [([], whole[:])]
    for left, right in splits(rest):
        result.append(([first] + left, right))
    return result

Why it works. The leading ([], whole) is the base-case split; every other split carries first straight through to the front of a left list, recursing on the shorter rest, which is precisely how the recursive append fact builds the result.

Complexity. Time O(n2), Space O(n2), for length n (each of n+1 splits copies up to n elements).

assert splits(["a", "b", "c"]) == [
    ([], ["a", "b", "c"]),
    (["a"], ["b", "c"]),
    (["a", "b"], ["c"]),
    (["a", "b", "c"], []),
]
assert splits([]) == [([], [])]
assert splits(["x"]) == [([], ["x"]), (["x"], [])]
assert len(splits(["a", "b", "c", "d", "e"])) == 6
assert splits(["a", "b", "c", "d", "e"])[2] == (["a", "b"], ["c", "d", "e"])

10. Build a pedigree

Approach — recursive dotted-list fact. The base-case pedigree is just [name]; the recursive pedigree fact begins ?child ?parent . ?rest, prepending the dog onto its parent's pedigree. Following each parent link, prepend name to every lineage the parent has.

def pedigree(parents, name):
    lineages = [[name]]
    for relation, parent, child in parents:
        if child == name:
            for tail in pedigree(parents, parent):
                lineages.append([name] + tail)
    return lineages

Why it works. The first lineage [name] is the base case; each longer one prepends name onto a lineage of a parent, so the chain extends one ancestor at a time exactly as the recursive pedigree fact does.

Complexity. Time O(f⋅d), Space O(d2), for f facts and chain depth d.

assert pedigree(PARENTS, "barack") == [
    ["barack"],
    ["barack", "abraham"],
    ["barack", "abraham", "fillmore"],
    ["barack", "abraham", "fillmore", "eisenhower"],
]
assert pedigree(PARENTS, "herbert") == [
    ["herbert"],
    ["herbert", "delano"],
    ["herbert", "delano", "fillmore"],
    ["herbert", "delano", "fillmore", "eisenhower"],
]
assert pedigree(PARENTS, "eisenhower") == [["eisenhower"]]
assert pedigree(PARENTS, "grover") == [
    ["grover"],
    ["grover", "fillmore"],
    ["grover", "fillmore", "eisenhower"],
]

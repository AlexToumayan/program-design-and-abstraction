Chapter 4/4.5 Unification

4.5.5 Search and Unique Variable Names

◷ approx. 15 min read · 4 worked examples · best read in two passes

By now you have met unification: the operation that pattern matches two expressions and, when it succeeds, records the variable bindings that made them match. Unification answers one small question at a time: can this expression and that expression be made equal?

That is not yet enough to answer a query. A query usually has several clauses, the database has many facts and rules, and rules can chain into other rules. Something has to decide which fact to try, in what order, and what to do when a rule has its own hypotheses that must be proved first. That something is search.

By the end of this lesson you will be able to do five things: say in one sentence what the job of search is, recognize the base case that ends a successful proof, read the official search procedure layer by layer (negated clauses, trying every fact, proving a rule's hypotheses), explain why a generator with yield is the right shape for it, and explain why the interpreter must rename variables before it unifies. The last point is subtle and is where beginners most often go wrong, so we give it its own section.

The Difference Between Unification and Search

Imagine you are solving a maze with a ring of keys. Unification is the act of taking one key and checking whether it fits the door in front of you: a yes-or-no test on a single door. Search is everything else: deciding which hallway to walk down, which door to try next, backing up when a hallway dead-ends, and remembering which keys you have already used so a turn you took earlier still holds as you go deeper.

The source draws the same line. To establish a query from the facts already in the system, the query interpreter performs a search over the space of all possible facts, and unification is the primitive operation that pattern matches two expressions inside that search. One is the strategy; the other is the single step the strategy keeps invoking.

unification: can this query clause match this fact's conclusion right now?
search:      which facts and rules do we try, in what order, and what carries forward?

Hold onto a second idea from the previous lesson, because search is built to satisfy it. Each query asserts that there is some assignment to its variables such that all of its sub-expressions simultaneously follow from the facts of the system. The role of the query interpreter is to verify that this is so. Search is the procedure that does the verifying.

The Job of search

The source states the design directly: the recursive search function implements the search procedure for the logic language. It is recursive because proving one clause can require proving a rule, and proving that rule's hypotheses is itself a search.

search takes three inputs. Reading them as the columns of a single line:

search( clauses, env, depth )
├─ search ..... the recursive procedure that establishes clauses
├─ clauses .... the clauses still left to prove, as a Scheme list
├─ env ........ the unifier so far: the bindings already committed
├─ depth ...... how many rules have been chained to get here
└─ produces ... a stream of environments, one per successful proof

The docstring states its contract: it searches for an application of rules to establish all the clauses, non-destructively extending the unifier env, and it limits the search to the nested application of depth rules. Non-destructively matters: each path through the search gets its own copy of the bindings, so a guess that fails on one path cannot corrupt another.

A word on depth. The docstring is the only place the source explains it: search is limited to the nested application of depth rules. The practical reason is that a recursive rule, such as a rule for ancestor that refers to ancestor again, could otherwise chain forever. Capping the depth gives the interpreter a stopping point. The source does not editorialize beyond the docstring, so treat the runaway-recursion rationale as the standard reason for a depth limit rather than a quoted claim.

The Base Case: No Clauses Left

Every recursion needs a stopping condition that signals success. For search it is the plainest possible one: if there are no clauses left to prove, every part of the query has already been established, so the bindings you are holding are a complete answer.

Walk the base-case line token by token:

if clauses is nil: yield env
├─ if .......... begin the success test
├─ clauses ..... the clauses still left to prove
├─ is nil ...... there are none; the list is empty
├─ yield ....... hand back a result without ending the procedure
├─ env ......... the current environment: the answer found so far
└─ result ...... emit env as one successful proof, then keep searching

The choice of yield rather than return is deliberate and is the key to how the whole interpreter behaves. yield makes search a generator: it can produce one successful environment, pause exactly where it is, and later resume to look for more answers. A query like "who are the common ancestors" may have several answers, and a generator lets search surface them one at a time instead of computing all of them up front or stopping after the first.

The Official search Structure

Here is the complete procedure from the source. It is dense on purpose; we will read it in three layers right after.

# (1)
>>> def search(clauses, env, depth):
...     """Search for an application of rules to establish all the CLAUSES,
...     non-destructively extending the unifier ENV. Limit the search to
...     the nested application of DEPTH rules."""
...     if clauses is nil:
...         yield env
...     elif DEPTH_LIMIT is None or depth <= DEPTH_LIMIT:
...         if clauses.first.first in ('not', '~'):
...             clause = ground(clauses.first.second, env)
...             try:
...                 next(search(clause, glob, 0))
...             except StopIteration:
...                 env_head = Frame(env)
...                 for result in search(clauses.second, env_head, depth+1):
...                     yield result
...         else:
...             for fact in facts:
...                 fact = rename_variables(fact, get_unique_id())
...                 env_head = Frame(env)
...                 if unify(fact.first, clauses.first, env_head):
...                     for env_rule in search(fact.second, env_head, depth+1):
...                         for result in search(clauses.second, env_rule, depth+1):
...                             yield result

The outer elif is the depth guard: if there is no limit, or we have not yet hit it, keep going. Everything below splits into two cases, and the very first decision search makes is whether the clause it is looking at is negated.

Layer 1: Negated Clauses

The first clause of the remaining query might be a negation, written with not or ~. Search detects this by looking at the head of the head of the clause list:

if clauses.first.first in ('not', '~'):
├─ clauses ......... the clauses still to prove
├─ .first .......... the first clause in that list
├─ .first .......... the operator at the front of that clause
├─ in .............. test membership
├─ ('not', '~') .... the two ways a negation can be written
└─ branch taken .... this clause is a negation; handle it specially

Negation here means negation as failure: a negated clause succeeds precisely when its inside cannot be proved. So rather than trying to unify the first clause with a fact, search tries to prove the positive version, and reads success or failure from whether that inner search produces anything. The source says it directly: in the special case where the first clause is negated, rather than trying to unify the first clause of the query with a fact, we check that there is no such unification possible through a recursive call to search; if this recursive call yields nothing, we continue the search process with the rest of the clauses.

Here ground simply fills in any variables in the inner clause that already have bindings in env, producing a concrete expression to test; the helper itself is defined in the accompanying logic example. In code that check is a try/except around pulling the very first result out of the inner search:

next(search(clause, glob, 0))
├─ search .......... a fresh proof attempt on the inner clause
├─ clause .......... the grounded inside of the negation
├─ glob ............ the shared global environment that holds the database's own facts (defined with the rest of the helpers in the source), used here because the negation is checked against the database as a whole rather than the current partial bindings
├─ 0 ............... start its depth counter over
├─ next(...) ....... demand the first result from that generator
└─ outcome ......... a result means the inside is provable; raising StopIteration means it is not

If next succeeds, the inside was provable, so the negation fails and this path is abandoned. If next raises StopIteration, the inner search produced nothing, the negation holds, and search proceeds with the remaining clauses on a fresh frame:

# (2)
>>> try:
...     next(search(clause, glob, 0))
... except StopIteration:
...     env_head = Frame(env)
...     for result in search(clauses.second, env_head, depth+1):
...         yield result

Read negation as failure carefully: it does not mean "false everywhere in the universe." It means "cannot be proved from the facts currently in this database." That is a real and important difference, and it is the second item in the traps section below.

Layer 2: Trying Every Fact

If the clause is not negated, search has to find a fact or rule whose conclusion matches it. It does the obvious, exhaustive thing: it loops over every fact in the database.

for fact in facts:
├─ for ......... begin a loop
├─ fact ........ the loop variable: one fact or rule at a time
├─ in .......... iterate over
├─ facts ....... the whole database of facts and rules
└─ each pass ... attempt to use this one fact to satisfy the clause

Inside the loop, before anything else, it renames the fact's variables (Layer 3 below explains why this step is not optional) and then opens a fresh frame for this single attempt:

env_head = Frame(env)
├─ Frame ....... make a new environment layer
├─ env ......... built on top of the bindings committed so far
├─ env_head ... the name for this attempt's private scratch space
└─ effect ...... a sandbox so a failed guess cannot corrupt other paths

Then it attempts the match. The conclusion of the fact is its head; the clause we are trying to prove is the head of the remaining clauses:

unify(fact.first, clauses.first, env_head)
├─ unify ......... try to make two expressions equal
├─ fact.first .... the conclusion (head) of this fact or rule
├─ clauses.first . the current query clause we are trying to prove
├─ env_head ...... the frame that will hold any new bindings
└─ result ........ True if they match, recording bindings in env_head; otherwise no match

If unification is successful, then the clause matches the conclusion of the current rule. That is the green light to move into the rule's hypotheses, which is Layer 3.

Layer 3: Proving the Rule's Hypotheses

A fact may be a bare fact with no conditions, or it may be a rule: a conclusion that holds only if some hypotheses hold. Recall the recursive ancestor rule from the proofs lesson:

(fact (ancestor ?a ?y) (parent ?a ?z) (ancestor ?z ?y))

In the stored structure, fact.first is the conclusion and fact.second is the list of hypotheses. For this rule:

(ancestor ?a ?y)

is the conclusion, and

(parent ?a ?z)
(ancestor ?z ?y)

are the hypotheses. Once the conclusion has matched the clause, the conclusion is only actually established if those hypotheses can be established too. The following for statement attempts to establish the hypotheses of the rule, so that the conclusion can be established:

for env_rule in search(fact.second, env_head, depth+1):
├─ search ......... recursively prove the hypotheses
├─ fact.second .... the list of hypotheses of this rule
├─ env_head ....... starting from the bindings that made the head match
├─ depth+1 ........ one rule deeper, counting toward the limit
├─ env_rule ....... an environment in which the hypotheses all hold
└─ each pass ...... one full way the hypotheses can be satisfied

Each env_rule is a complete set of bindings under which the matched rule's conclusion genuinely holds. But the conclusion was only the first clause of the original query. The rest of the query still has to be proved, now carrying those bindings forward:

for result in search(clauses.second, env_rule, depth+1):
├─ search ......... recursively prove the rest of the query
├─ clauses.second . the remaining query clauses after the first
├─ env_rule ....... bindings carried forward from proving the first clause
├─ depth+1 ........ stay at the deeper level
├─ result ......... an environment satisfying the whole remaining query
└─ yield result ... emit it as one complete answer, then keep searching

This forward flow of bindings is exactly why the clauses of a query can cooperate. A variable bound while proving the first clause is already bound when the second clause is attempted, so a shared variable forces the clauses to agree. Each successful result is yielded, the generator pauses, and on the next request the loops resume to look for more answers.

Why Variable Names Must Be Renamed

Here is the subtle point, and it is worth slowing down for. Unification assumes that no variable is shared between the two expressions it compares. But the facts and queries of the logic language reuse variable names constantly. Two completely unrelated rules might both be written with ?x. Those two ?x symbols are local to their own rules; they are not meant to be the same logical variable. If unification treated them as one, proving one rule could silently force a value onto a variable in another rule that has nothing to do with it.

The source states the problem and the fix together: unification assumes that no variable is shared among both e and f; however, we often reuse variable names in the facts and queries of the logic language; we would not like to confuse an ?x in one fact with an ?x in another, because these variables are unrelated; to ensure that names are not confused, before a fact is passed into unify, its variable names are replaced by unique names using rename_variables by appending a unique integer for the fact.

That is the line you already saw at the top of the fact loop:

fact = rename_variables(fact, get_unique_id())
├─ get_unique_id() ... produce a fresh integer not used before
├─ rename_variables .. rewrite this fact with that integer stamped on every variable
├─ fact .............. rebind the loop variable to the renamed copy
└─ effect ............ this attempt's variables can never collide with another's

The renaming procedure itself is a straightforward recursion over the structure of an expression. A variable gets the integer appended; a pair recurses into both halves; anything else is left untouched:

# (3)
>>> def rename_variables(expr, n):
...     """Rename all variables in EXPR with an identifier N."""
...     if isvar(expr):
...         return expr + '_' + str(n)
...     elif scheme_pairp(expr):
...         return Pair(rename_variables(expr.first, n),
...                     rename_variables(expr.second, n))
...     else:
...         return expr

A single line of it walks like this:

return expr + '_' + str(n)
├─ expr ........ the variable symbol, e.g. ?x
├─ + '_' ....... an underscore separator
├─ + str(n) .... the unique integer rendered as text
└─ result ...... a fresh, collision-proof name such as ?x_12

So a ?x stamped with id 12 becomes ?x_12, and a ?x from a different rule stamped with id 13 becomes ?x_13. They can never again be mistaken for the same variable.

A Step-By-Step Trace: Renaming a Rule

Before reading the official rename_variables, it helps to watch what it produces. Take the ancestor rule and rename it with the id 7. The procedure recurses through every position in the nested structure, replacing each variable and leaving each plain symbol alone.

Starting expression:

(fact (ancestor ?a ?y) (parent ?a ?z) (ancestor ?z ?y))

What rename_variables decides at each kind of token:

Token encountered isvar? scheme_pairp? What is returned
fact no no fact unchanged
ancestor no no ancestor unchanged
parent no no parent unchanged
?a yes — ?a_7
?y yes — ?y_7
?z yes — ?z_7
any pair, e.g. (ancestor ?a ?y) no yes recurse into both halves, rebuild the pair

Result after renaming with id 7:

(fact (ancestor ?a_7 ?y_7) (parent ?a_7 ?z_7) (ancestor ?z_7 ?y_7))

Notice that every ?a became ?a_7, every ?y became ?y_7, and so on: within this one copy the variables stay linked to each other, which is correct, but they are now sealed off from any other rule's variables. Run the same rule through again with a different id and you get a second, independent copy. That is precisely what lets the same recursive rule be used twice in one proof without its two uses contaminating each other.

The Whole Picture in Order

The source describes the algorithm in prose and code, not as a numbered list. Here is the order of operations restated so you can hold it in your head, but read it as a summary of the code above, not as text quoted from the source:

1. No clauses left?          yield the current environment; this proof is done.
2. Past the depth limit?     stop; do not search deeper.
3. First clause negated?     prove the inside; if that fails, continue with the rest.
4. Otherwise, for each fact: rename its variables to be unique,
                             open a fresh frame,
                             unify the fact's conclusion with the clause.
5. If that unifies:          recursively prove the fact's hypotheses,
                             then recursively prove the remaining query clauses,
                             yielding every environment that satisfies the whole query.

That loop, with unification as its single repeated step, is the search through the space of possible proofs. The remaining details, including the user interface to the logic language and the definition of various helper functions, appear in the logic example accompanying the source.

⚠Common Beginner Traps

  • Thinking search finds only one answer. Because it is a generator built on yield, it can produce many successful environments, one at a time, on demand.
  • Reading negation as "false in the whole universe." Negation as failure means only "cannot be proved from the facts currently in the database." Add a fact and a previously failing negation can start succeeding.
  • Forgetting that fact.second holds a rule's hypotheses while fact.first holds its conclusion. Search proves the conclusion only by first establishing the hypotheses.
  • Not seeing why a fresh Frame(env) is opened for each fact. It sandboxes one attempt so a guess that fails cannot leak bindings into another search path; this is what "non-destructively extending the unifier" means.
  • Skipping the renaming step in your mental model. Without rename_variables, an ?x in one rule and an unrelated ?x in another would be forced to share a binding, and proofs that should succeed would fail (or worse, succeed wrongly).
  • Assuming clauses.first.first is a variable. It is the operator at the front of the first clause; search inspects it only to spot not or ~.

Show the answers
  1. It yields the current environment env when clauses is nil, that is, when no clauses remain to prove. Every part of the query has been established, so the bindings in env are a complete answer.
  2. yield makes search a generator, so it can surface one successful environment, pause, and resume later to find more. A query can have several answers, and a generator delivers them one at a time instead of all at once or only the first.
  3. It tries to prove the positive inside of the negation through a recursive call to search. If that inner search yields nothing (it raises StopIteration), the negation succeeds and search continues with the remaining clauses. If the inner search produces a result, the negation fails.
  4. It opens a fresh environment layer for that single attempt, built on the bindings committed so far. This sandboxes the attempt so a failed unification cannot corrupt the bindings used on other search paths; it is what makes the extension of the unifier non-destructive.
  5. fact.first is the conclusion (head) of the fact or rule; fact.second is the list of its hypotheses. Search unifies the clause with fact.first, and if that succeeds it recursively proves fact.second so the conclusion is actually established.
  6. (sibling ?x_4 ?y_4). The plain symbol sibling is neither a variable nor a pair, so it comes back untouched, while each variable gets the id 4 stamped on it as ?x_4 and ?y_4.

§Summary

Search is the strategy that decides how the primitive operation, unification, gets applied to prove a query. The recursive search(clauses, env, depth) procedure stops with success when no clauses remain, yielding its environment; it guards against runaway recursion with a depth limit; it handles a negated first clause by negation as failure (the negation holds only if its inside cannot be proved); and otherwise it loops over every fact, renames the fact's variables to keep them unique, opens a fresh frame, and unifies the fact's conclusion with the clause. On a match it recursively establishes the rule's hypotheses and then the rest of the query, carrying bindings forward so the clauses must agree, and it yields every environment that satisfies the whole query. Renaming variables before unification is the quiet but essential step that keeps an ?x in one rule from being confused with an unrelated ?x in another. Together, unification and search are the core of the logic query interpreter.

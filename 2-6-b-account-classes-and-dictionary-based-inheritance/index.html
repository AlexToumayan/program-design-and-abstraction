Chapter 2/2.6 Implementing Classes and Objects

2.6 Account Classes and Dictionary-Based Inheritance

◷ approx. 15 min read · 15 worked examples · best read in two passes

The previous packet built a small object system out of nothing but functions and dictionaries. It gave you three tools: make_instance, make_class, and the helpers that bind methods and run constructors. That packet stayed abstract on purpose, so the machinery was easy to see.

This packet puts the machinery to work. By the end you will be able to do four things: rebuild a real bank-account class using only dictionaries, create instances and call their methods through get and set messages, explain why setting an attribute on an instance does not disturb the class, and build a subclass whose missing names fall back to its base class. Every example you trace here has an exact counterpart in Python's built-in class statement, so the payoff is that dot notation stops being magic.

We are not proposing that you write programs this way. Real code uses class. The point is to see that classes, instances, methods, constructors, and inheritance are not new primitives baked into the language. They are arrangements of ideas you already command:

functions + dictionaries + local state + lookup rules

An Analogy: A Company Directory

Imagine a company with a printed policy binder and a stack of individual employee folders.

The policy binder holds rules that apply to everyone: the standard vacation allowance, the default job title for a new hire. The binder is shared. There is one copy.

Each employee folder holds facts about one person: their name, their current salary. When you need to know an employee's vacation allowance, you first flip through their personal folder. If the folder says nothing about vacation, you walk to the shelf and consult the shared binder.

That two-step lookup is the whole idea. The folder is an instance, and it stores instance attributes. The binder is the class, and it stores class attributes. Asking for an attribute means "check my own folder first, and only fall back to the binder if my folder is silent." If a particular employee negotiates extra vacation, that note goes in their personal folder. It does not rewrite the shared binder, so everyone else is unaffected. That is exactly how an instance attribute shadows a class attribute, and it is the behavior we are about to reproduce with dictionaries.

A subclass adds a second binder on a higher shelf. A checking-account folder consults the checking-account binder first, and if that binder is silent, it consults the general account binder above it. That chain of binders is inheritance.

The Object System Kernel

This document is standalone, so we begin with the complete object-system kernel from the previous packet. Reproduce it exactly; every line matters later.

This kernel gives us exactly three behaviors, and the rest of the packet is just these three behaviors playing out on a concrete example.

Behavior Where it appears in (1)
Instance attribute storage attributes = {} inside make_instance
Method binding bind_method(value, instance)
Inheritance lookup base_class['get'](name) inside make_class

Rebuilding the Account Class

In ordinary Python, the account class from the data-and-state chapter looks like this. This is the destination we are reconstructing.

# (2)
class Account:
    interest = 0.02

    def __init__(self, account_holder):
        self.holder = account_holder
        self.balance = 0

In our dictionary-based object system, we do not write a class statement. Instead we write a function whose local frame holds the class attributes and methods, and which ends by handing that frame to make_class.

# (3)
def make_account_class():
    """Return the Account class, which has deposit and withdraw methods."""
    interest = 0.02

    def __init__(self, account_holder):
        self['set']('holder', account_holder)
        self['set']('balance', 0)

    def deposit(self, amount):
        """Increase the account balance by amount and return the new balance."""
        new_balance = self['get']('balance') + amount
        self['set']('balance', new_balance)
        return self['get']('balance')

    def withdraw(self, amount):
        """Decrease the account balance by amount and return the new balance."""
        balance = self['get']('balance')
        if amount > balance:
            return 'Insufficient funds'
        self['set']('balance', balance - amount)
        return self['get']('balance')

    return make_class(locals())

This looks strange at first, because self is no longer an object you reach into with a dot. In this system self is an instance dispatch dictionary, so you talk to it by sending it 'get' and 'set' messages.

Here is the translation between the two worlds. Reading an attribute:

self.balance              becomes      self['get']('balance')

Writing an attribute:

self.balance = 0          becomes      self['set']('balance', 0)

Once you internalize that one substitution, the body of (3) reads exactly like the ordinary Account class in (2).

The Local Frame Becomes a Class Dictionary

The final line in (3) is the hinge of the whole technique:

return make_class(locals())

This is the one genuinely new tool in this packet. locals() is a built-in Python function you have not needed before, and it returns a dictionary containing the name-value bindings in the current local frame. Read that line one token at a time.

return make_class(locals())
├─ return ......... hand a value back to the caller of make_account_class
├─ make_class ..... the kernel function that builds a class dispatch dictionary
├─ ( .............. begin the arguments to make_class
├─ locals() ....... call locals, capturing this frame's current name bindings
├─ ) .............. end the call to make_class
└─ evaluates to   a class dispatch dictionary with get, set, and new

Right before that line runs, the local frame for make_account_class holds four names: the class attribute and the three methods just defined.

make_account_class frame
  interest  -> 0.02
  __init__  -> function
  deposit   -> function
  withdraw  -> function

So locals() produces a dictionary with exactly those bindings:

{
  'interest': 0.02,
  '__init__': <function __init__>,
  'deposit': <function deposit>,
  'withdraw': <function withdraw>
}

Then make_class wraps that dictionary inside a class dispatch dictionary that responds to messages:

Account class dispatch dictionary
  'get' -> class lookup function
  'set' -> class assignment function
  'new' -> instance creation function

The original four bindings are now hidden inside the closure of the class's lookup function. You no longer see interest sitting at the top level; you ask for it by message.

A Follow-the-Line Look at the Constructor

Look at the first assignment inside the constructor in (3):

self['set']('holder', account_holder)

In ordinary Python this would be self.holder = account_holder. Here it is a message send. Break the line into its tokens.

self['set']('holder', account_holder)
├─ self ............. the instance dispatch dictionary being initialized
├─ [ ............... begin selecting a value out of that dictionary
├─ 'set' ........... the key whose value is the instance's setter function
├─ ] ............... end the selection, yielding the setter function
├─ ( ............... call the setter function
├─ 'holder' ........ the attribute name to store on this instance
├─ , ............... separate the two arguments
├─ account_holder .. the value to store under that name
├─ ) ............... end the call
└─ stores          holder -> account_holder in this instance's attributes

The constructor runs two such lines, so it stores two instance attributes:

holder  -> account_holder
balance -> 0

Notice what it does not store. The class attribute interest is never written onto the instance. It stays in the class dictionary, and an instance only sees it through fallback lookup, unless that instance later shadows it.

Creating the Account Class

Now we actually build the class dictionary by calling the maker.

# (4)
Account = make_account_class()

After (4) runs, the global frame binds Account to a dispatch dictionary:

Account -> {
  'get': class lookup function,
  'set': class assignment function,
  'new': instance creation function
}

The account-specific attributes are not visible at the top level of this dictionary. They are sealed inside the closure of the class's get_value function. That is precisely why we retrieve them by sending a message rather than by indexing:

Account['get']('interest')

Creating an Account Instance

The official example creates an account by sending the 'new' message to the class.

# (5)
kirk_account = Account['new']('Kirk')

This is the dictionary-object version of the ordinary call Account('Kirk'). Read the line as tokens.

kirk_account = Account['new']('Kirk')
├─ kirk_account .... the name that will hold the finished instance
├─ = ............... bind that name to the value computed on the right
├─ Account ......... the class dispatch dictionary
├─ ['new'] ......... select the class's instance-creation function
├─ ( ............... call that function
├─ 'Kirk' .......... the account-holder argument passed along to __init__
├─ ) ............... end the call
└─ binds          kirk_account to a fresh, initialized instance dictionary

What does sending 'new' actually set in motion? Trace the chain through the kernel from (1).

Account['new']('Kirk')
  calls new('Kirk')
  calls init_instance(Account, 'Kirk')
  calls make_instance(Account)
  returns a fresh instance dispatch dictionary
  looks up Account['get']('__init__')
  calls __init__(instance, 'Kirk')
  stores holder -> 'Kirk'
  stores balance -> 0
  returns the initialized instance

The string 'Kirk' is not stored by new itself. It rides along as account_holder, and __init__ is the code that finally files it under the instance attribute name 'holder'.

Retrieving Attributes and Calling Methods

Now send messages to the finished instance. This is the official transcript.

# (6)
kirk_account['get']('holder')
kirk_account['get']('interest')
kirk_account['get']('deposit')(20)
kirk_account['get']('withdraw')(5)

Running those four lines in the interpreter produces:

'Kirk'
0.02
20
15

Every line uses the one lookup rule built into make_instance: check the instance's own attributes first, and fall back to the class only if the instance is silent.

Expression What happens
kirk_account['get']('holder') Finds 'holder' in the instance's own attributes
kirk_account['get']('interest') Not found locally, so the instance asks Account
kirk_account['get']('deposit') Found in Account, then bound to kirk_account
kirk_account['get']('withdraw') Found in Account, then bound to kirk_account

The last two lines are method calls, and a method call here happens in two visible stages. Look at the structure of one of them.

kirk_account['get']('deposit')(20)
├─ kirk_account['get']('deposit') .. retrieve deposit and bind it to kirk_account
├─ ( ............................... call the bound method
├─ 20 .............................. the one visible argument: the amount
├─ ) ............................... end the call
└─ evaluates to                    20, the new balance

The first stage returns a bound method: a function that already has kirk_account waiting inside it as self. The second stage calls that bound method with the single visible argument 20. The instance is the hidden argument; you never type it.

A Full Trace of deposit(20)

It is worth slowing the method call all the way down, because this is where the purpose of bound methods becomes concrete. The call:

kirk_account['get']('deposit')(20)

unfolds step by step:

1.  Ask kirk_account for 'deposit'.
2.  'deposit' is not in kirk_account's own attributes.
3.  The instance asks Account for 'deposit'.
4.  Account returns the deposit function.
5.  bind_method wraps deposit, with kirk_account waiting inside as self.
6.  Call the bound method with amount -> 20.
7.  The bound method calls deposit(kirk_account, 20).
8.  deposit asks kirk_account for 'balance'.
9.  The current balance is 0.
10. deposit computes 0 + 20.
11. deposit stores kirk_account's 'balance' as 20.
12. deposit returns the updated balance, 20.

The whole point: you wrote one argument, 20, but the method body received two, because binding silently slipped kirk_account in front as self.

Instance Attributes Can Shadow Class Attributes

In Python's built-in system, assigning an attribute on an instance does not change the matching attribute on its class. Our implementation has the same behavior, and the official transcript demonstrates it.

# (7)
kirk_account['set']('interest', 0.04)
Account['get']('interest')
kirk_account['get']('interest')

After the assignment, the two reads disagree:

0.02
0.04

Why do they disagree? Because the setter writes into the receiver's own attribute dictionary and nowhere else. The line:

kirk_account['set']('interest', 0.04)

stores 'interest' in the instance's private attributes. It is emphatically not the same as:

Account['set']('interest', 0.04)

So the class still holds its original value:

interest -> 0.02

while the instance now holds its own:

interest -> 0.04

On every later lookup, the instance value wins, because instance lookup checks local attributes before falling back to the class. This is the negotiated-vacation note from our analogy: it goes in the personal folder and leaves the shared binder untouched.

Rebuilding Inheritance

Now we build a subclass, CheckingAccount, a checking account that charges a small fee on each withdrawal. In ordinary Python from the data-and-state chapter it looked like this:

# (8)
class CheckingAccount(Account):
    interest = 0.01
    withdraw_charge = 1

    def withdraw(self, amount):
        return Account.withdraw(self, amount + self.withdraw_charge)

The dictionary-based version follows the same design: redefine the names that change, and reuse everything else from the base class.

# (9)
def make_checking_account_class():
    """Return the CheckingAccount class, which imposes a 1-unit withdrawal fee."""
    interest = 0.01
    withdraw_fee = 1

    def withdraw(self, amount):
        fee = self['get']('withdraw_fee')
        return Account['get']('withdraw')(self, amount + fee)

    return make_class(locals(), Account)

The final line is where inheritance is wired up. Read it as tokens.

return make_class(locals(), Account)
├─ return ......... hand the finished subclass back to the caller
├─ make_class ..... the kernel function that builds a class dispatch dictionary
├─ ( .............. begin the arguments
├─ locals() ....... this subclass's own attributes: interest, withdraw_fee, withdraw
├─ , .............. separate the two arguments
├─ Account ........ the base class to consult when a name is missing here
├─ ) .............. end the call
└─ evaluates to   a subclass dictionary whose fallback parent is Account

The first argument is the subclass's own bindings:

interest     -> 0.01
withdraw_fee -> 1
withdraw     -> specialized withdraw function

The second argument is the base class, Account. So whenever CheckingAccount is asked for a name it does not have, it forwards the question to Account.

A Follow-the-Line Look at Calling the Base Class

The specialized withdraw ends with a deliberately tricky line:

return Account['get']('withdraw')(self, amount + fee)

It calls the base class's withdraw by hand. Break it into tokens.

return Account['get']('withdraw')(self, amount + fee)
├─ return ........... hand the resulting balance back to the caller
├─ Account .......... the base class dispatch dictionary
├─ ['get'] .......... select the base class's lookup function
├─ ('withdraw') ..... ask the base class for its own withdraw function
├─ ( ................ call that base-class function directly
├─ self ............. pass the instance explicitly, since this is not a bound call
├─ , ................ separate the arguments
├─ amount + fee ..... the larger amount: the request plus the checking fee
├─ ) ................ end the call
└─ evaluates to     the base class's new balance after the fee-inflated withdrawal

Two details earn the trouble. First, we pass self explicitly. We retrieved withdraw from Account rather than from the instance, so it never got bound, which means it still expects the instance as its first argument. Second, and more important, we ask Account for withdraw rather than asking self. If this method called self['get']('withdraw'), lookup would find the checking-account override again, and the method would call itself forever. Asking the base class directly is how a subclass reaches the original method it is overriding.

Creating and Using the Subclass

Now build the subclass and open one checking account.

# (10)
CheckingAccount = make_checking_account_class()
jack_acct = CheckingAccount['new']('Spock')

The official transcript checks the changed interest, the inherited deposit, and the overriding withdraw.

# (11)
jack_acct['get']('interest')
jack_acct['get']('deposit')(20)
jack_acct['get']('withdraw')(5)

The results are:

0.01
20
14

The first two are easy: interest is found right on CheckingAccount, and deposit is inherited from Account. The third is the interesting one. Why is the final balance 14 rather than 15? Trace the arithmetic.

Start balance:                       0
Deposit 20:                          0 + 20 = 20
Withdraw request:                    5
Checking fee added by the override:  1
Amount passed to Account.withdraw:   5 + 1 = 6
Final balance:                       20 - 6 = 14

The override inflates the requested amount by the one-unit fee before handing it to the inherited withdrawal logic, so a 5-unit request actually removes 6.

What Is Inherited

The CheckingAccount class defines only the names it changes:

interest
withdraw_fee
withdraw

It deliberately does not define these:

__init__
deposit

So when those names are needed, lookup climbs the chain of binders. When CheckingAccount['new']('Spock') needs __init__:

CheckingAccount
  does not have '__init__'
Account
  has '__init__'

When jack_acct['get']('deposit') needs deposit:

jack_acct local attributes
  no 'deposit'
CheckingAccount
  no 'deposit'
Account
  has 'deposit'
bind deposit to jack_acct

When jack_acct['get']('withdraw') needs withdraw:

jack_acct local attributes
  no 'withdraw'
CheckingAccount
  has 'withdraw'
bind CheckingAccount.withdraw to jack_acct

That is inheritance, expressed entirely as dictionary lookup with a fallback parent.

A Warm-Up You Can Verify by Hand

Before leaning on the full kernel, it helps to see the shadow-and-fallback rule in miniature, using nothing but two plain dictionaries and a tiny lookup function. This is not the official example; it is a bridge to make the rule obvious, and you can run it in your head.

Trace it by hand. The call look_up('vacation') finds 'vacation' in folder, so it returns 25, the personal override. The call look_up('title') does not find 'title' in folder, so it falls back to shared_binder and returns 'Staff'. The values are:

25
'Staff'

This is the entire instance-then-class lookup rule, stripped of method binding and constructors. The official kernel adds those features back, but the heart of it is just "check my own dictionary, then fall back."

Built-In Python Uses a Similar Storage Idea

Python's built-in object system is far more sophisticated than our small one. It supports metaclasses, descriptors, static methods, class methods, multiple inheritance, introspection, and a long list of special protocols. Python also distinguishes certain special methods that interact with built-in functions, so that those functions behave correctly across many different types. We are not reproducing any of that.

But the core storage idea is the same. In Python, an ordinary instance keeps its own attributes in a dictionary named __dict__, much like the attributes dictionary inside our make_instance.

The last line produces a dictionary shaped like this:

{'holder': 'Kirk', 'balance': 0}

The class attribute interest is not inside simple.__dict__. It lives on the class, and the instance only sees it through fallback, exactly as in our system. Now watch the built-in shadowing behavior directly.

# (14)
simple.interest
simple.interest = 0.04
simple.__dict__
SimpleAccount.interest

The two readings that matter are:

simple.interest        -> 0.04
SimpleAccount.interest -> 0.02

After the assignment, simple.__dict__ has gained its own 'interest' entry, and the instance reading returns 0.04 while the class still holds 0.02. This is the same outcome you saw in code (7). The takeaway is that __dict__ is not a mysterious internal feature once you have built the storage-plus-lookup pattern yourself:

instance attributes can shadow class attributes

✓Check Your Understanding

Use these checks to make sure the account system is a machine you can trace, not a spell you have memorized.

  1. When kirk_account['get']('interest') runs before any local interest is set, where is interest found?
  2. After kirk_account['set']('interest', 0.04), why does Account['get']('interest') still produce 0.02?
  3. Why does CheckingAccount.withdraw call Account['get']('withdraw')(self, amount + fee) instead of self['get']('withdraw')(amount + fee)?
  4. In the checking-account example, why does jack_acct['get']('deposit')(20) work even though make_checking_account_class never defines deposit?
  5. Open a fresh checking account, deposit 30, then withdraw 10: acct = CheckingAccount['new']('Uhura'), acct['get']('deposit')(30), acct['get']('withdraw')(10). What final balance does the last call return?
Show the answers
  1. It is found on the Account class through fallback lookup, because the instance has no local interest yet.
  2. The set message sent to kirk_account stores a local instance attribute, which shadows the class attribute instead of editing it; the class value is untouched.
  3. self['get']('withdraw') would find the checking-account override again and recurse forever. Asking Account directly retrieves the original base-class method, and self is passed explicitly because that retrieved method is unbound.
  4. Missing names in CheckingAccount fall back to Account, so deposit is inherited and then bound to jack_acct.
  5. 19. The deposit raises the balance to 30, and the checking override inflates the 10-unit request by the 1-unit fee before delegating to the inherited Account withdraw, so it removes 11, leaving 30 - 11 = 19.

§Summary

This packet turned the object-system kernel into a working program. make_account_class collects class attributes and methods in a local frame and hands that frame to make_class, producing a class dictionary that answers get, set, and new. Sending Account['new']('Kirk') builds an instance, runs __init__ to file away holder and balance, and returns a dispatch dictionary. Reading an attribute checks the instance first and falls back to the class; calling a method retrieves it, binds the instance in as self, and runs it with your one visible argument as the second. Assigning an attribute on an instance shadows the class value without disturbing it. CheckingAccount shows inheritance: it overrides interest and withdraw, reuses deposit and __init__ by fallback, and reaches its parent's original withdraw by asking Account directly. Because Python's real objects store their own attributes in __dict__ and resolve names with the same instance-then-class rule, the built-in class statement is now far less mysterious: objects and classes are organized around attribute storage plus lookup rules.

This checkpoint is longer than usual, but self-contained: the kernel, then the account class, then the calls that change the balance. Watching it draws the environment diagram that exposes inheritance as plain closures: kirk_account and Account are dispatch dictionaries on the heap, the instance's get_value closes over its private attributes while the class's get_value closes over the snapshot locals() handed to make_class, and each bind_method call produces a short-lived method frame that captures kirk_account as the self slipped in before your one visible argument. # (15)
def make_instance(cls):
    def get_value(name):
        if name in attributes:
            return attributes[name]
        else:
            value = cls['get'](name)
            return bind_method(value, instance)
    def set_value(name, value):
        attributes[name] = value
    attributes = {}
    instance = {'get': get_value, 'set': set_value}
    return instance

def bind_method(value, instance):
    if callable(value):
        def method(*args):
            return value(instance, *args)
        return method
    else:
        return value

def make_class(attributes, base_class=None):
    def get_value(name):
        if name in attributes:
            return attributes[name]
        elif base_class is not None:
            return base_class['get'](name)
    def set_value(name, value):
        attributes[name] = value
    def new(*args):
        return init_instance(cls, *args)
    cls = {'get': get_value, 'set': set_value, 'new': new}
    return cls

def init_instance(cls, *args):
    instance = make_instance(cls)
    init = cls['get']('__init__')
    if init:
        init(instance, *args)
    return instance

def make_account_class():
    interest = 0.02
    def __init__(self, account_holder):
        self['set']('holder', account_holder)
        self['set']('balance', 0)
    def deposit(self, amount):
        new_balance = self['get']('balance') + amount
        self['set']('balance', new_balance)
        return self['get']('balance')
    def withdraw(self, amount):
        balance = self['get']('balance')
        if amount > balance:
            return 'Insufficient funds'
        self['set']('balance', balance - amount)
        return self['get']('balance')
    return make_class(locals())

Account = make_account_class()
kirk_account = Account['new']('Kirk')
holder = kirk_account['get']('holder')
starting_interest = kirk_account['get']('interest')
after_deposit = kirk_account['get']('deposit')(20)
after_withdrawal = kirk_account['get']('withdraw')(5)

Step through it one line at a time and check each prediction as the frames appear. The __init__ writes land in the attributes dictionary inside the make_instance frame, not in the global frame; that closure holds the instance's state. bind_method runs three times, for the interest, deposit, and withdraw lookups: it hands 0.02 back unchanged because a number is not callable, but wraps the two functions with kirk_account waiting inside as self. The after_deposit line opens roughly eight frames, which you can reconstruct one at a time: the instance get_value lookup for 'deposit', the class get_value lookup it falls back to, the bind_method call, the method wrapper that bound call enters, the deposit call itself, and then the three balance operations inside deposit — the get that reads balance, the set that stores the new balance, and the final get that returns it. The deposit body receives two arguments though you typed one. The global frame finishes with holder bound to 'Kirk', starting_interest to 0.02, after_deposit to 20, and after_withdrawal to 15.

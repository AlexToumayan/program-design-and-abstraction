Chapter 3/3.5 Interpreters for Languages with Abstraction

3.5.1 Scheme Interpreter Structure

◷ approx. 18 min read · 3 worked examples · best read in two passes

In Section 3.4 you built an interpreter for the Calculator language. It could read an expression, parse it into data, evaluate it, and loop. That was enough to compute arithmetic, but Calculator could not name anything. It could work out (+ 2 (* 3 4)), yet it had no way to remember a result, no way to define a procedure, and no way to grow a vocabulary.

Scheme adds exactly that missing power. By the end of this lesson you will be able to do four things. You will explain why a Scheme interpreter needs an environment that the Calculator interpreter did not. You will read the official scheme_eval function and name, for any Scheme expression, which of its four cases applies. You will describe how a procedure gets applied to its arguments, and why primitive procedures and user-defined procedures are handled differently. And you will explain why the evaluator and the applicator are mutually recursive — why neither can do its job without calling the other.

You do not need to have written an interpreter before. You do need the habit this course keeps returning to: when something looks dense, slow down and separate it into the smallest meaningful pieces.

From a Calculator to a Language You Can Teach

Picture a basic desk calculator. You press buttons, it gives answers. It is fast and exact, but it is frozen. There is no button labeled square, and there is no way for you to add one. Every session starts from the same fixed set of operations.

Now picture a programmable calculator. You can teach it a new operation once and reuse it forever:

(define (square x) (* x x))

After that single definition, the machine understands a button it did not have a moment ago:

(square 5)

That is the whole jump from Calculator to Scheme. The interpreter is no longer only carrying out built-in arithmetic. It now has to manage names, remember user-defined procedures, and obey special rules such as define, lambda, and if that do not behave like ordinary arithmetic combinations.

So the interpreter's job description grows. The Calculator pipeline turned text into expression data into a value. The Scheme pipeline adds one phrase:

text  ->  expression data  ->  meaning in an environment

The new phrase is "in an environment." A Scheme expression can mention a name like x or square, and a name has no meaning on its own. It only means something once the interpreter knows what it is bound to. The environment is what supplies that knowledge.

The Same Loop, a Richer Eval

A Scheme interpreter reuses the broad shape of the Calculator interpreter. A parser turns text into an expression, and an evaluator works out what that expression means. The familiar loop is unchanged:

Read  ->  Eval  ->  Print  ->  Loop

What changes is the Eval step. Before it can compute anything, the evaluator has to decide what kind of expression it is looking at, because different kinds are handled by completely different rules:

number or string  ->  already a value
symbol            ->  look up its value in the environment
special form      ->  use a special evaluation rule
call expression   ->  evaluate operator and operands, then apply a procedure

This split is why a Scheme interpreter is built from two cooperating functions rather than one. One function evaluates expressions; the other applies procedures to arguments.

  • scheme_eval evaluates an expression in an environment.
  • scheme_apply applies a procedure to already-evaluated arguments.

These are the Scheme counterparts of calc_eval and calc_apply from the Calculator interpreter, but now they have to support abstraction: names, user-defined procedures, and the environment model. Almost everything that makes Scheme harder than Calculator lives in special forms, user-defined procedures, and the environment model of computation.

A note on language before we go further. The interpreter you are studying is written in Python. The expressions it consumes — the programs a user types — are written in Scheme. Keep those two layers separate in your mind. When you see a Python function like scheme_eval, that is the machinery. When you see (square 5), that is a Scheme program the machinery is chewing on.

Parsing: The Reader Turns Scheme Text Into Data

Before evaluation can begin, the reader has to turn the characters the user typed into structured data. The Calculator reader already handled Scheme-style parentheses, so it is nearly enough to parse any valid Scheme expression. Scheme adds two pieces of notation that need a little extra care.

The first is quotation, written with a single quote mark '. The second is the dotted list, written with a ., which lets you build a pair whose second element is not another list.

The official source gives this parsing example. The function read_line takes a string of Scheme text and returns the parsed data structure:

>>> read_line("(car '(1 . 2))")
Pair('car', Pair(Pair('quote', Pair(Pair(1, 2), nil)), nil))

That output looks intimidating, so take the input apart first. The Scheme the user wrote was:

(car '(1 . 2))

This is a call to car with a single operand, and that operand is a quoted dotted pair. Read it in two pieces.

The quote mark is shorthand. Writing 'X is the same as writing (quote X), so the reader does not keep the ' character around. It expands the shorthand into a tree whose first element is the symbol quote. That expansion is exactly why a quote symbol shows up in the parsed output even though you never typed the word.

The dotted pair is the inner data being quoted:

(1 . 2)

An ordinary Scheme list ends in nil. A dotted pair does not. (1 . 2) is a single pair whose first is 1 and whose second is 2, with no trailing nil. That is why the parsed form contains Pair(1, 2) directly rather than a list-shaped chain.

Putting it together, the parsed data says: call car on the quoted data Pair(1, 2). This is the first key idea of the section. After reading, the user's Scheme program is no longer text. It is Python data — Pair objects, symbols, and numbers — that the interpreter can inspect piece by piece.

The Evaluator: scheme_eval

Once an expression has been parsed into data, it is handed to scheme_eval. Its interface is two arguments:

scheme_eval(expr, env)
  • expr is the Scheme expression, represented as Python data.
  • env is the current environment, where any names in expr can be looked up.

Here is the official evaluator. It is written in Python, because it is part of the interpreter's machinery:

# (1)
def scheme_eval(expr, env):
    """Evaluate Scheme expression expr in environment env."""
    if scheme_symbolp(expr):
        return env[expr]
    elif scheme_atomp(expr):
        return expr
    first, rest = expr.first, expr.second
    if first == "lambda":
        return do_lambda_form(rest, env)
    elif first == "define":
        do_define_form(rest, env)
        return None
    else:
        procedure = scheme_eval(first, env)
        args = rest.map(lambda operand: scheme_eval(operand, env))
        return scheme_apply(procedure, args, env)

This function is dense because it folds all four expression kinds into one body. The rest of this lesson pulls those four cases apart, one at a time, in the order they appear.

Case 1: A Symbol Is Looked Up

If the expression is a symbol, the interpreter treats it as a name and asks the environment for its value:

if scheme_symbolp(expr):
    return env[expr]

The helper scheme_symbolp answers the yes-or-no question "is this expression a symbol?" If the answer is yes, env[expr] performs the lookup. Suppose the user evaluates the bare expression:

x

Reading env[x] as an anatomy of one line makes the lookup explicit:

env[x]
├─ env ......... the current environment
├─ [ ........... begin a name lookup
├─ x ........... the name being looked up
├─ ] ........... end the lookup
└─ evaluates to  the value x is bound to in env

If x is bound to 10, evaluating x returns 10. The crucial point is that the symbol is not the value. The symbol x is only a label. The environment is what supplies the value the label points to.

Case 2: An Atom Evaluates to Itself

An atom is an expression with no internal subexpressions to evaluate — there is nothing inside it to take apart. Numbers, strings, and booleans are atoms:

3
"hello"
#t

The evaluator handles them in a single line:

elif scheme_atomp(expr):
    return expr

The helper scheme_atomp checks whether the expression is a self-evaluating atom. If so, scheme_eval returns it unchanged. A number like 3 needs no lookup and no further work, because it already is a value. This is the base case of evaluation: the place where the work stops instead of recursing further.

Tracing 3 through the evaluator is short, but it shows the shape:

scheme_eval(3, env)
├─ scheme_eval ... the evaluator
├─ ( ............. begins the inputs
├─ 3 ............. the atom being evaluated
├─ , ............. separates the inputs
├─ env ........... the current environment
├─ ) ............. ends the inputs
└─ evaluates to .. 3

Case 3: Special Forms Have Their Own Rules

If the expression is not an atom, the evaluator pulls it apart into its first element and the rest:

first, rest = expr.first, expr.second

Most combinations evaluate every operand and then apply a procedure. Special forms are the exception. They are written like combinations — parentheses, a first element, more elements after it — but they deliberately do not follow the ordinary call rule. The evaluator recognizes them by checking the first element against the names of the special forms it knows.

Consider define:

(define x 10)

If this were an ordinary call, the evaluator would try to evaluate x as an operand. But x here is not a value to compute; it is the name being defined. Looking it up would either fail or find the wrong thing. So define needs its own rule.

The same is true of lambda:

(lambda (x) (* x x))

An ordinary call would evaluate the body (* x x) immediately. But lambda is not supposed to run its body. It is supposed to build a procedure that will run the body later, when it is eventually called. So lambda needs its own rule too.

That is why the evaluator checks first before doing anything else:

if first == "lambda":
    return do_lambda_form(rest, env)
elif first == "define":
    do_define_form(rest, env)
    return None

When first is "lambda", the helper do_lambda_form builds a procedure from the parameters and body in rest, and that procedure is returned. When first is "define", the helper do_define_form carries out the binding, and scheme_eval returns None, because a definition performs an action rather than producing a value to print. The Scheme conditional form if is another special form handled the same way: it is recognized by its first element rather than evaluated like a call. In short, the first item in a combination tells the evaluator whether it is staring at a special rule or an ordinary call.

Case 4: A Call Expression Uses Eval and Apply

If first is none of the special-form keywords, the expression is an ordinary call expression, and the else branch handles it:

else:
    procedure = scheme_eval(first, env)
    args = rest.map(lambda operand: scheme_eval(operand, env))
    return scheme_apply(procedure, args, env)

This branch does three things, in order:

  1. Evaluate the operator expression first to get the procedure.
  2. Evaluate every operand in rest to get the argument values. The rest.map(...) call evaluates each operand with scheme_eval and collects the results.
  3. Hand the procedure and the argument values to scheme_apply.

Take a concrete call:

(square (+ 2 3))

The operator is the symbol:

square

The single operand is itself a call expression:

(+ 2 3)

Walking the else branch over this call, token by token, gives:

scheme_apply(procedure, args, env)
├─ procedure ... scheme_eval(square, env): looks up square, a procedure
├─ args ........ scheme_eval((+ 2 3), env): evaluates the operand to 5
├─ env ......... the environment passed along to the application
└─ evaluates to  square applied to the argument list (5)

So the evaluator first looks up square, then evaluates (+ 2 3) down to 5, then asks scheme_apply to apply the square procedure to the argument list containing 5. Notice that step 1 and step 2 are themselves calls to scheme_eval. Evaluating a call expression means evaluating its parts first.

Procedure Application: scheme_apply

scheme_apply takes over once the evaluator has produced a procedure and a list of argument values. Its job is to actually carry out the call. Scheme has two broad kinds of procedure, and scheme_apply treats them differently.

  • A PrimitiveProcedure represents a procedure built into the interpreter and implemented in Python. The built-in primitives +, *, car, and cdr (the pair accessors you have seen in Scheme) are like this.
  • A LambdaProcedure represents a user-defined procedure, created by Scheme code with lambda or with the procedure form of define.

A PrimitiveProcedure holds an attribute, fn, that is bound to a Python function. Applying the primitive means calling that Python function on the argument values. Some primitives may or may not require access to the current environment, but the essential idea is simple: a primitive procedure is carried out directly by Python code. When you see these class names in the Scheme project, read them as implementation labels. PrimitiveProcedure is shorthand for "this Scheme procedure is performed by a Python function."

A LambdaProcedure is richer, because it represents Scheme code rather than Python code. It stores three things:

  • its formal parameters (the parameter names),
  • its body (a Scheme expression), and
  • its parent environment (the environment in which it was created).

Applying a LambdaProcedure is the heart of the environment model. A new frame is added to the environment, the formal parameters are bound to the argument values in that frame, and the body is evaluated with scheme_eval in the resulting environment. That last step is the important one: applying a user-defined procedure ends in a call back to scheme_eval.

This is also what the procedure form of define builds. Writing

(define (square x) (* x x))

is shorthand. It creates a LambdaProcedure with the formal parameter x, the body (* x x), and the current environment as its parent, then binds that procedure to the name square in the current frame. After that, square is a name in the environment that points to a LambdaProcedure, exactly like any other binding.

One Complete Pass Through a User-Defined Call

The single most important new idea in this section is what happens when you apply a procedure you defined yourself. Slow it all the way down.

Assume the environment already contains the definition:

(define (square x) (* x x))

Now the user evaluates:

(square 5)

Here is the full machine trace, with every step the interpreter takes:

1.  scheme_eval sees an ordinary call expression.
2.  It evaluates the operator symbol square.
3.  The environment lookup returns a LambdaProcedure.
4.  It evaluates the operand 5.
5.  The operand is self-evaluating, so the argument value is 5.
6.  scheme_apply applies the LambdaProcedure to the argument value 5.
7.  Applying the LambdaProcedure creates a new frame.
8.  The new frame binds x -> 5.
9.  The interpreter evaluates the body (* x x) in that new frame.
10. Looking up x returns 5, so (* x x) becomes (* 5 5).
11. The primitive * returns 25.

The detail worth pinning down is timing. The body (* x x) is not evaluated when square is defined. At definition time there is no value for x yet. The body waits, stored inside the LambdaProcedure, until the procedure is actually called. Only then does a new frame give x the value 5, and only then does the body run.

Eval and Apply Are Mutually Recursive

Step back from the trace and look at the overall shape. The evaluator and the applicator keep calling each other:

scheme_eval sees a call expression
  scheme_eval evaluates the operator
  scheme_eval evaluates the operands
  scheme_apply applies the procedure
    if it is a user-defined procedure, scheme_eval evaluates its body

This is mutual recursion. Evaluation requires application whenever it meets a call expression. Application requires evaluation twice over: once to turn operand expressions into argument values, and again to evaluate the body of a user-defined procedure. Neither function can do the whole job alone. scheme_eval understands the forms of expressions; scheme_apply understands what it means to call a procedure; user-defined procedures are the hinge that connects them into a cycle.

A recursion this tight would never stop without base cases, and the evaluator has several:

  • a self-evaluating atom evaluates to itself, with no further calls,
  • a symbol lookup returns a value already stored in the environment,
  • a special form may finish without making an ordinary call at all, and
  • a primitive procedure is applied directly by Python, ending the cycle.

Every Scheme evaluation eventually bottoms out in one of these. That eval/apply cycle, grounded by these base cases, is the core engine of language interpretation.

A Small Python Model of Eval and Apply

The official scheme_eval is part of a full interpreter, with Pair objects, a reader, and an environment class behind it. To watch the control flow on its own, here is a stripped-down model. It uses ordinary Python lists to stand in for Scheme calls, so there is nothing to parse and the recursion is easy to follow. This is not a full Scheme interpreter; it is a teaching skeleton of the same two functions.

Expected output:

14
10

Each of the four cases from the official evaluator is visible here in miniature. An int is self-evaluating. A str is a symbol, looked up in env. define is a special form that binds a name and returns None. Anything else is a call: evaluate the operator, evaluate the operands, then apply. The whole movement is one short sentence:

expression  ->  evaluate parts  ->  apply procedure  ->  return value

⚠Common Beginner Traps

Trap 1: Thinking a parsed expression is still text. After the reader runs, the interpreter holds structured data — Pair objects, symbols, numbers — not a string. It never goes back to guessing meaning from characters.

Trap 2: Evaluating every combination the same way. Special forms such as define, lambda, and if look like calls but are not. The evaluator checks the first element precisely so it can give them their own rules instead of evaluating their parts as operands.

Trap 3: Confusing a symbol with its value. The symbol x is a name. Its value lives in the environment. Evaluating a symbol always means a lookup, never reading the name as data.

Trap 4: Applying before evaluating operands. For an ordinary call, every operand is fully evaluated first, and only then is the procedure applied. The argument values, not the argument expressions, are what get handed to the procedure.

Trap 5: Forgetting that user-defined procedures restart evaluation. Applying a LambdaProcedure evaluates its body, which is another call to scheme_eval. Application does not end the recursion for user-defined procedures; it feeds back into it.

§Summary

A Scheme interpreter extends the Calculator interpreter by supporting abstraction, and that extra power lives in special forms, user-defined procedures, and the environment model. The reader turns Scheme text into Python data, expanding quotation shorthand and handling dotted pairs along the way. The evaluator, scheme_eval, receives an expression and an environment and sorts the expression into one of four cases: a symbol is looked up, a self-evaluating atom is returned as is, a special form such as lambda or define follows its own rule, and an ordinary call expression evaluates its operator and operands and then applies a procedure. Application, scheme_apply, runs primitive procedures directly in Python and runs user-defined LambdaProcedure objects by building a new frame, binding parameters, and evaluating the body — which sends control back into scheme_eval. That eval/apply cycle, stopped by a handful of base cases, is the engine at the center of language interpretation.

▶Step-Through Checkpoint

This is the small eval/apply model again, trimmed to a single program so the recursion is easy to watch. Stepping through this Python model draws the environment diagram of the interpreter calling itself: each scheme_eval frame holds its own expr, but they all share the one env dictionary that maps "+" and "*" to their functions. The outer frame stays open, with arguments only half-filled, while the nested scheme_eval frame for ["*", 3, 4] runs. Between them, a scheme_apply frame opens to call the looked-up function on its finished argument values. Before you step through the block below, write down three predictions: how many scheme_eval frames will open over the whole run, what the outer frame's arguments list holds at the moment the frame for ["*", 3, 4] opens, and which of the two scheme_apply frames opens first.

Step through it one line at a time and check each prediction as the frames appear. Seven scheme_eval frames open in all: one for the whole program, one each for "+", 2, and ["*", 3, 4], and three more inside that last call. The outer frame waits with operator bound to the + function and arguments holding only 2 until the nested call returns 12. scheme_apply runs first for *, and only then for +, which turns 2 and 12 into the 14 bound to result. That pause-and-resume is mutual recursion made visible.

Chapter 4/4.2 Implicit Sequences

4.2.6-4.2.8 Iterable and Iterator Interfaces

◷ approx. 17 min read · 14 worked examples · best read in two passes

By the end of this lesson you will be able to build your own objects that work with a for loop, just like a list does. You already know that for works on lists, strings, and dictionaries. Now you will learn the two-method contract that makes that possible, so you can put your own classes on the other side of for.

There are exactly two special methods to understand: __next__, which hands back one value at a time, and __iter__, which hands back something that has a __next__. Most confusion in this topic comes from blurring two roles that sound alike: the iterator (the thing being consumed) and the iterable (the thing that can produce a fresh iterator). Keeping those two apart is the whole game, and we will keep returning to that distinction.

A Ticket Dispenser and a Roll of Tickets

Picture a deli counter. On the wall is a dispenser that gives out numbered tickets. Each time you pull, you get the next number: 41, then 42, then 43. The dispenser has a position built into it; it remembers what it handed out last, and it only ever moves forward. When the roll runs out, pulling gives you nothing more.

That dispenser is an iterator. Its one job is to answer the question "what is the next value?" and each answer advances its internal position. Once a number is pulled, it is gone; you cannot pull 41 again from the same dispenser.

Now picture the roll of tickets sitting in the stock room. The roll is not a dispenser. You cannot pull a number directly from the roll. But the roll is what you load into a dispenser, and you could load a fresh dispenser from the same kind of roll tomorrow and start again at the beginning.

That roll is an iterable. Its job is not to dispense values itself, but to produce a dispenser on demand. Ask it for one, and you get a brand-new iterator positioned at the start.

In Python these two roles map onto two methods. Asking an iterable for a dispenser is __iter__. Asking a dispenser for its next value is __next__. Hold the deli picture in mind: a dispenser gets used up, a roll can fill many dispensers.

The Contract Between Objects and for

A for statement does not require a list. It requires an object that follows the iterable interface: an object that can hand back an iterator when asked. That is the only thing for needs, which is why we can teach our own objects to cooperate with it.

The interaction has two layers:

Method Who defines it What it returns
__iter__ the iterable an iterator (a fresh dispenser)
__next__ the iterator the next value, or raises StopIteration

Two built-in functions are the polite front doors to these methods. The built-in iter(obj) calls obj.__iter__(). The built-in next(it) calls it.__next__(). You can call the dunder methods directly, but iter and next are the usual way, and the official examples below use both forms on purpose so you see they are interchangeable.

When you write for x in obj, Python does this once and then loops: it calls iter(obj) a single time to get an iterator, then it calls next on that iterator over and over, binding each result to x, until next raises StopIteration, which ends the loop quietly.

The Iterator Interface: __next__

We start with the dispenser, because it is the lower-level piece. The iterator interface is defined by a single method, __next__, which returns the next element of the underlying series. Each call to __next__ may do real work to retrieve or compute that element, and each call makes a mutating change: it advances the iterator's position so the next call returns the following value.

Here is the official iterator from the source, an iterator over letters of the alphabet in ASCII order. It walks from start up to but not including end.

# (1)
class LetterIter:
    """An iterator over letters of the alphabet in ASCII order."""
    def __init__(self, start='a', end='e'):
        self.next_letter = start
        self.end = end
    def __next__(self):
        if self.next_letter == self.end:
            raise StopIteration
        letter = self.next_letter
        self.next_letter = chr(ord(letter)+1)
        return letter

Two attributes carry all the state. The attribute self.next_letter holds the letter that the next call to __next__ will return; it is the dispenser's current position. The attribute self.end holds the stopping value, the first letter we do not return. The line self.next_letter = chr(ord(letter)+1) is how the position advances: ord turns a one-character string into its integer code point, adding 1 moves to the next code point, and chr turns that integer back into a character. So 'a' becomes 'b', 'b' becomes 'c', and so on.

Let us read the advancing line one token at a time, because it is the heart of the class.

self.next_letter = chr(ord(letter)+1)
├─ self.next_letter ... the attribute holding the position to return next
├─ = .................. assignment: rebind that attribute
├─ chr ............... built-in: integer code point to its character
├─ ( .................. begins chr's input
├─ ord ............... built-in: a character to its integer code point
├─ ( .................. begins ord's input
├─ letter ............ the character just returned this call
├─ ) .................. ends ord's input
├─ + .................. addition operator
├─ 1 .................. the integer one, the step to the next code point
├─ ) .................. ends chr's input
└─ performs ........... advances the position to the following letter

A Step-by-Step Trace of Iterator State

The point of an iterator is that it carries a position that mutates. Let us trace self.next_letter across the calls in (2), starting from LetterIter(), where start defaults to 'a' and end defaults to 'e'.

Call self.next_letter before Returns self.next_letter after
1st __next__ 'a' 'a' 'b'
2nd __next__ 'b' 'b' 'c'
3rd via next 'c' 'c' 'd'
4th __next__ 'd' 'd' 'e'
5th __next__ 'e' raises StopIteration 'e'

On the fifth call, self.next_letter equals self.end ('e'), so the guard at the top runs raise StopIteration before returning anything. The position does not advance because the method exits early. This is the dispenser hitting the end of its roll.

Making an Iterator Usable in a for Loop

There is a subtlety the source is careful about. To use an iterator in a for loop, the iterator must also have an __iter__ method, because for always begins by calling iter on whatever you give it. The Python documentation recommends that an iterator's __iter__ return the iterator itself, so that every iterator is also iterable.

Why return self? Because a dispenser is already a value-producer. If you ask a dispenser for a dispenser, the honest answer is "I am one — here I am." Returning self means the object hands itself back, position and all.

LetterIter object
  already has __next__ and a current position
  so __iter__ can simply return self

Here is a token-by-token reading of that recommended method.

def __iter__(self): return self
├─ def ............... begins a method definition
├─ __iter__ ......... the method for "give me an iterator"
├─ (self) ............ the one parameter: the object itself
├─ : .................. begins the method body
├─ return ............ hands a value back to the caller
├─ self .............. this very object, which already dispenses values
└─ performs ........... lets a `for` loop iterate this iterator directly

With that method added, the same LetterIter works directly in a for loop. Here for calls iter(letter_iter), gets the same object back, and then calls next on it repeatedly.

b

The Iterable Interface: __iter__

Now the other role. An object is iterable if it returns an iterator when its __iter__ method is invoked. An iterable represents a data collection and provides a fixed representation that may produce more than one iterator. The iterable itself does not get consumed; it is the stock room roll, and each __iter__ call loads a fresh dispenser.

The official Letters class is iterable but is not its own iterator. It stores only the rules (start and end) and builds a new LetterIter every time it is asked.

# (4)
class Letters:
    def __init__(self, start='a', end='e'):
        self.start = start
        self.end = end
    def __iter__(self):
        return LetterIter(self.start, self.end)

Watch the contrast in this official transcript. Two iterators are created from one iterable, b_to_k, and they advance independently. Pulling from first_iterator does not move second_iterator. The iterable b_to_k never changes; only the two dispensers do.

>>> b_to_k = Letters('b', 'k')
>>> first_iterator = b_to_k.__iter__()
>>> next(first_iterator)
'b'
>>> next(first_iterator)
'c'
>>> second_iterator = iter(b_to_k)
>>> second_iterator.__next__()
'b'
>>> first_iterator.__next__()
'd'
>>> first_iterator.__next__()
'e'
>>> second_iterator.__next__()
'c'
>>> second_iterator.__next__()
'd'

The iterable Letters instance b_to_k and the LetterIter iterator instances differ in exactly the way the deli picture predicts: the iterable does not change, while the iterator instances change with each call to next. The iterator tracks progress through the data; the iterable represents the data itself.

Because Letters knows how to produce an iterator, any built-in that consumes an iterable accepts it. The source shows map over b_to_k. Recall that map itself returns an iterator, so once it accepts b_to_k you can pull values from it with next, just like any other dispenser:

>>> caps = map(lambda x: x.upper(), b_to_k)
>>> next(caps)
'B'
>>> next(caps)
'C'

Iterator Versus Iterable, Side by Side

This is the distinction to lock in. An iterator carries a position and is consumed: after one full pass it is empty, and __next__ keeps raising StopIteration. An iterable carries no position; it manufactures a fresh iterator on each request, so it can be traversed again and again.

Iterator Iterable
Deli analogy the ticket dispenser the roll of tickets
Required method __next__ (and usually __iter__ returning self) __iter__ returning a new iterator
Carries a position? yes no
What happens on reuse already consumed; a second pass is empty fresh iterator each time; reusable

The next program makes the difference visible. A bare iterator is consumed after one pass; the second list call on it is empty. An iterable produces the same values every time.

Expected output:

['a', 'b', 'c', 'd']
[]
['a', 'b', 'c', 'd']
['a', 'b', 'c', 'd']

The first list(iterator) drains the dispenser; the second finds it empty. Both list(iterable) calls return the full sequence because each one loaded a fresh dispenser.

Creating Iterables with Yield

Writing a separate iterator class by hand, with its own __next__ and its own position attribute, is a lot of bookkeeping. Generators remove that work. Recall that any function containing a yield statement is a generator function: calling it returns a generator, which is an iterator that runs the function body, pausing at each yield and resuming on the next next.

The key move is that __iter__ can itself be a generator method. Because __iter__ is only required to return an iterator, and a generator is an iterator, putting yield inside __iter__ does exactly what we need: each call to iter(obj) starts the method over and produces a brand-new generator. The object becomes a reusable iterable with no hand-written __next__ at all.

Here is the official LettersWithYield. Its __iter__ is a generator method that walks from start up to end.

# (7)
class LettersWithYield:
    def __init__(self, start='a', end='e'):
        self.start = start
        self.end = end
    def __iter__(self):
        next_letter = self.start
        while next_letter < self.end:
            yield next_letter
            next_letter = chr(ord(next_letter)+1)

Notice what is gone compared with the Letters/LetterIter pair: there is no separate iterator class, no next_letter attribute on the object, and no explicit raise StopIteration. The local variable next_letter lives inside the generator, and the loop ending naturally stops iteration. Because __iter__ is a generator method, each call to iter(letters) produces a fresh generator with its own local next_letter, so the object is reusable.

Let us read the yield line in context, token by token.

yield next_letter
├─ yield ............. pauses the generator and emits a value
├─ next_letter ....... the local variable holding the current letter
└─ performs ........... hands this letter to next, then suspends here until resumed

Here is a runnable program. Because the object is iterable, not a one-shot iterator, two full passes both succeed.

Expected output:

['a', 'b', 'c', 'd']
['a', 'b', 'c', 'd']

A Trace of Generator Suspension

A generator method does not run top to bottom in one shot. It suspends at yield and resumes there. Tracing one pass of LettersWithYield().__iter__() makes the pausing concrete.

Event next_letter Action
iter(letters) called unset generator created, body not yet started
1st next 'a' run to yield, emit 'a', suspend
2nd next 'a' then 'b' advance to 'b', loop, yield 'b', suspend
3rd next 'b' then 'c' advance to 'c', yield 'c', suspend
4th next 'c' then 'd' advance to 'd', yield 'd', suspend
5th next 'd' then 'e' advance to 'e', 'e' < 'e' is false, loop ends, raise StopIteration

The local variable persists across the pauses because the generator keeps its own frame suspended between calls. When the while condition finally fails, the function returns, which signals StopIteration to whatever is driving the loop.

A Generator for Nested Pairs

Generators shine when the sequence is awkward to materialize as a stored list. The official all_pairs generator yields every ordered pair drawn from a single iterable, pairing each element with every element, including itself.

# (9)
def all_pairs(s):
    for item1 in s:
        for item2 in s:
            yield (item1, item2)

The structure is a nested loop: the outer loop fixes item1, and for each one the inner loop runs through every item2, yielding the pair (item1, item2). Because it yields one pair at a time, nothing forces the full collection of pairs to exist all at once; the pairs are produced on demand as something pulls them.

The official transcript runs it over the list [1, 2, 3]:

>>> list(all_pairs([1, 2, 3]))
[(1, 1), (1, 2), (1, 3), (2, 1), (2, 2), (2, 3), (3, 1), (3, 2), (3, 3)]

It also works over any iterable, including our LettersWithYield. Here the source takes only the first five pairs with a slice:

>>> letters = LettersWithYield()
>>> list(all_pairs(letters))[:5]
[('a', 'a'), ('a', 'b'), ('a', 'c'), ('a', 'd'), ('b', 'a')]

Here is a runnable version covering both transcripts:

Expected output:

[(1, 1), (1, 2), (1, 3), (2, 1), (2, 2), (2, 3), (3, 1), (3, 2), (3, 3)]

Because all_pairs(letters) walks through letters in the outer loop, and then walks through letters all over again for each item the outer loop reaches, it asks letters for a fresh iterator many times. That only works because letters is a reusable iterable. The inner loop calls iter(letters) afresh for each item1, getting a new generator each time. A one-shot iterator passed in s would be drained after the first inner pass.

An Unbounded Iterator: Positives

An iterator does not need a stopping value at all. The official Positives class is an iterator over the positive integers, which never raises StopIteration because the positive integers never run out.

# (11)
class Positives:
    def __init__(self):
        self.next_positive = 1;
    def __next__(self):
        result = self.next_positive
        self.next_positive += 1
        return result

The pattern is the familiar iterator pattern: save the value to return in result, advance the position with self.next_positive += 1, then return the saved value. There is no guard and no raise, so every call succeeds and the sequence goes on forever.

The official transcript pulls the first three values:

>>> p = Positives()
>>> next(p)
1
>>> next(p)
2
>>> next(p)
3

Because the sequence is infinite, you must never hand it to something that tries to consume all of it, such as list(Positives()) or a plain for loop with no break. You stay in control by pulling a finite number of values yourself.

Expected output:

1
2
3

The same infinite sequence can also be written as a generator, which many people find easier to read because the position is just a local variable. This is a bridge, not the official version above; both describe the same unbounded series.

Expected output:

1
2
3

The while True says there is no built-in end. The consumer decides how far to go, which is exactly why we only call next three times instead of asking for the whole sequence.

⚠Common Beginner Traps

One trap is confusing the iterator with the iterable. If your object has __next__ and its __iter__ returns self, it is a single dispenser with one shared position, and a second for loop over it will find it already empty. If you want repeated traversal, write __iter__ so it returns a new iterator (or make it a generator method).

A second trap is calling list or running an unbounded for over an infinite iterator such as Positives. There is no final value where collection can stop, so Python would try to pull values forever. Take a fixed number of values with explicit next calls instead.

A third trap is forgetting that consuming functions drain their input iterator. map, filter, and for all pull from the iterator you give them. If you pass the same iterator somewhere else afterward, it may already be empty, which is exactly what the iterator versus iterable example showed.

A fourth trap is treating __iter__ and __next__ as magic words with no model. __iter__ produces or returns a dispenser; __next__ asks a dispenser for one value and advances it. When you are unsure which method an object needs, ask whether it is the roll (needs __iter__ returning a new iterator) or the dispenser (needs __next__).

A fifth trap is assuming an iterator works in a for loop just because it has __next__. A for loop calls iter first, so an object with only __next__ raises TypeError: '...' object is not iterable the moment the loop tries to start. An iterator also needs an __iter__ method; the recommended one simply returns self.

§Summary

The iterable interface lets your own objects work with for, map, list, and every other sequence-processing tool. Two roles do the work, and keeping them apart is the key idea. An iterator defines __next__, carries a position, and is consumed as values are pulled; to use it in a for loop it also needs an __iter__ that returns self. An iterable defines __iter__ to return a fresh iterator, carries no position, and can therefore be traversed again and again. Generator methods collapse most of this bookkeeping: because a generator is an iterator, writing __iter__ with yield produces a reusable iterable with no hand-written __next__. And because an iterator need not ever stop, you can model unbounded series like the positive integers, as long as you pull from them a finite number of values at a time.

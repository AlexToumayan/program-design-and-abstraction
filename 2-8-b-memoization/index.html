Chapter 2/2.8 Efficiency

2.8 Memoization

◷ approx. 13 min read · 6 worked examples

In the previous lesson you watched the tree-recursive fib function do an absurd amount of repeated work: computing fib(19) called fib over thirteen thousand times, even though there are only twenty distinct answers it could ever need. By the end of this lesson you will be able to fix that. You will build a cache by hand, then package the same idea as a higher-order function named memo that wraps any function and remembers its answers, and finally apply memo to fib and prove with a call counter that each distinct input is now computed exactly once.

The single habit to carry through this lesson is this: a function that always returns the same answer for the same input never needs to compute that answer twice. Memoization is just the disciplined way to act on that fact.

An Answer Key Analogy

Picture a student grinding through a worksheet who keeps hitting the same arithmetic problem:

17 + 29 = ?
17 + 29 = ?
17 + 29 = ?

The first time, the student actually adds the numbers and gets 46. A careful student then writes that result in the margin as an answer key:

"17 + 29"  ->  46

Every time that exact problem shows up again, the student does not redo the addition. They glance at the margin and copy 46. The work happens once; every later appearance is a lookup.

Memoization gives a function exactly this kind of margin. The function keeps a private answer key, checks it before doing any real work, and only computes when the key has no entry yet.

A Manual Cache, Before Anything Fancy

Before wrapping recursion, it helps to see the answer-key idea on a plain function. Here is a function that is deliberately noisy so you can see when it actually runs:

Output:

computing square of 5
25
computing square of 5
25

The answer is 25 both times, but the message prints twice. Python re-runs the entire function body on the second call. It has no memory of the first one.

Now add an answer key by hand. A cache is just a storage place, and a dictionary is the natural choice: the input becomes the key, the computed result becomes the value.

Output:

computing square of 5
25
25
{5: 25}

The message prints only once. On the first call, 5 is not in cache, so the function computes 25 and stores it. On the second call, 5 is already a key, so the function skips the computation and hands back the stored value. The final line shows the answer key itself: {5: 25}.

Following the Key-Check Line

The line that does all the work is the membership test. Read it one token at a time:

if n not in cache:
├─ if ......... begins a conditional
├─ n .......... the input we were asked about
├─ not in ..... membership test, negated: true when the key is absent
├─ cache ...... the dictionary acting as our answer key
├─ : .......... ends the header, body runs only when the test is true
└─ runs the compute-and-store body only when this input has no cached answer

That one guard is the whole trick. When the answer is already known, the body is skipped entirely and the function falls straight through to return cache[n].

Turning the Cache into a Higher-Order Function

Writing a fresh cache and a fresh guard for every function would be tedious, and it leaks a global cache name into your program. The original text packages the pattern once, as a higher-order function. memo takes a function f and returns a new function that wraps f with an answer key. This is the official definition, reproduced exactly:

# (3)
def memo(f):
    cache = {}

    def memoized(n):
        if n not in cache:
            cache[n] = f(n)
        return cache[n]

    return memoized

The phrasing from the source is worth holding onto: a memoized function will store the return value for any arguments it has previously received.

Following the Line: the Memoized Wrapper

The heart of memo is the inner function it builds. The key line computes a fresh answer only when the cache has no entry for this input:

cache[n] = f(n)
├─ cache ...... the dictionary that lives in this call to memo
├─ [n] ........ the slot for this particular input
├─ = .......... assignment: bind that slot to the value on the right
├─ f .......... the original, un-memoized function
├─ (n) ........ call it on this input to compute the real answer
└─ stores the freshly computed answer under key n, so later calls can skip the work

And the line that every call reaches, whether it computed or not:

return cache[n]
├─ return ..... hand a value back to the caller
├─ cache ...... the answer key
├─ [n] ........ look up the slot for this input
└─ evaluates to the stored answer for n

The structural points to notice in (3):

  • memo(f) receives a function f and treats it as the slow, real computation.
  • cache = {} creates one dictionary, born in the frame that the call to memo opens.
  • memoized(n) is the replacement function that gets returned in place of f.
  • return memoized sends that replacement back to the caller.

The environment idea that makes this work: cache is created inside the call to memo, but it does not disappear when memo returns. Because the returned memoized function still refers to cache, that dictionary stays alive as part of the remembered environment of memoized. There is exactly one cache per call to memo, and it persists across every later call to the wrapper.

Applying memo as a Decorator

Because memo takes a function and returns a replacement function, it fits the exact shape of a decorator. The source flags this directly: memoization can be expressed naturally as a higher-order function, which can also be used as a decorator. A decorator is just the @ syntax that applies a higher-order function to a definition the moment it is written.

Here is fib wrapped with memo using decorator notation:

# (4)
@memo
def fib(n):
    if n == 0:
        return 0
    if n == 1:
        return 1
    return fib(n - 2) + fib(n - 1)

The @memo line above the def is the part to read carefully:

@memo
├─ @ ........... decorator marker: apply the function named below to this definition
├─ memo ........ the higher-order function to apply, our cache wrapper
└─ rebinds the name fib to memo(fib) as soon as the def finishes

The whole point is that this is not new behavior. Writing @memo directly above def fib is exactly equivalent to defining fib normally and then rebinding it through memo:

@memo
def fib(n):
    ...

is the same as

def fib(n):
    ...
fib = memo(fib)

So fib ends up bound to the memoized wrapper, the recursive calls inside the body (fib(n - 2) and fib(n - 1)) reach that wrapper, and every distinct input is computed once. This works because the name fib inside the body is looked up fresh on every call, and by the time any call runs, @memo has already rebound the global fib to the wrapper, so each inner fib(n - 2) lands on the cached version, not the raw one. The decorator form simply moves the rebinding up to the definition, so you cannot forget to apply it and you never expose the un-memoized version under the name fib. Everything you learned about the cache, the persistent environment, and immutable keys applies unchanged; only the wiring is written more compactly.

Why Cache Keys Must Be Immutable

The source attaches one constraint to this technique: the use of a dictionary requires that the argument to the memoized function be immutable.

The reason is mechanical. A dictionary finds a value by computing something from the key and using it to locate the slot. If a key could change after it was inserted, Python could no longer reliably find the entry again, because it would look in the slot the key used to map to. So dictionary keys must be values that never change. These are safe to use as keys:

0
19
"fib(19)"
(2, 3)

A list is mutable, so it cannot serve as a key:

[2, 3]

For the functions in this lesson the argument is always a single integer, which is immutable, so this constraint never bites. It matters as soon as you try to memoize a function whose argument is a list or another mutable object.

Memoizing Fibonacci

Now point memo at the function that needed it. Here we wire count and memo by hand rather than using @memo, so we can keep a separate handle on the counted version (counted_fib) and read its call_count to prove how few real computations actually happen. Here is the tree-recursive fib from the previous lesson, the count wrapper that tracks how many times a function runs, and the memo wrapper, all reproduced exactly, then wired together the way the source wires them:

This matches the original interactive session:

>>> counted_fib = count(fib)
>>> fib = memo(counted_fib)
>>> fib(19)
4181
>>> counted_fib.call_count
20
>>> fib(34)
5702887
>>> counted_fib.call_count
35

The wiring is worth reading slowly, because the order matters. First counted_fib = count(fib) wraps the real fib in a call counter, so every time the real computation runs, call_count ticks up. Then fib = memo(counted_fib) rebinds the name fib to a memoized wrapper around the counted version. From now on, calling fib first checks the cache; only on a miss does it fall through to counted_fib, which increments the counter and does the actual recursive work.

As the source puts it, the fib function is actually only called once for each unique input to fib. That is exactly what the counter proves.

Trace: Why the Count Is 20, Then 35

Computing fib(19) ultimately needs the answers for every input from 0 up through 19. That is 20 distinct values. Each one passes through the counter exactly once, on the single occasion it is actually computed; every other time that value is requested, the wrapper returns it from the cache without touching counted_fib. So after fib(19), the counter reads:

inputs computed once: 0, 1, 2, ..., 19   ->  20 distinct values
counted_fib.call_count == 20

Contrast that with the un-memoized version from the previous lesson, where the same fib(19) drove call_count to 13529.

Now the second call, fib(34), runs against a cache that is not empty. The entries for 0 through 19 are already there from the first call. The only inputs that still need real computation are the new ones:

already cached: 0 through 19
newly computed: 20 through 34   ->  15 new values

Fifteen fresh computations are added to the twenty already counted:

20+15=35

So the counter reads 35, exactly as the transcript shows. The cache is shared across both top-level calls, which is why the second call can stand on the results of the first.

The Computation Changes Shape

The deeper payoff is that memoization changes the shape of the process, not just its speed. Recall the duplicated work in the un-memoized tree for fib(5), where the right branch recomputes a subtree the left branch already built:

fib(5)
|-- fib(3)
`-- fib(4)
    |-- fib(2)
    `-- fib(3)   <- recomputed from scratch

In the source's words, in this computation of fib(5), the results for fib(2) and fib(3) are reused when computing fib(4) on the right branch of the tree. With memo in place, the second appearance of each input is no longer a recursive descent at all. It is a single dictionary lookup:

fib(5)
|-- compute fib(3)
`-- compute fib(4)
    |-- lookup fib(2)   <- already in cache
    `-- lookup fib(3)   <- already in cache

The wrapper still runs on every call, but the expensive recursive expansion underneath it happens only the first time each input is seen. A whole repeated subtree collapses into one constant-time read from the cache. That is why a process that was exploding in size becomes manageable: a new pattern of computation evolves, in which the duplicated branches simply vanish.

⚠Common Beginner Traps

Trap 1: Thinking memoization changes the answer. It must not. fib(19) still returns 4181 either way. Memoization changes how much repeated work is needed to get there, never the result itself.

Trap 2: Recreating the cache inside the inner function. If you write cache = {} inside memoized instead of inside memo, a brand-new empty cache is built on every single call, so nothing is ever remembered. The cache must live in the enclosing frame, created once when memo runs.

Trap 3: Memoizing a function whose result depends on outside state. Memoization assumes the same input always produces the same output. If a function depends on the current time, a random number, a file, or mutable global state, a cached answer can become stale and wrong.

Trap 4: Using a mutable value as a cache key. Lists and dictionaries cannot be dictionary keys; attempting cache[[2, 3]] = ... raises TypeError: unhashable type: 'list'. Use immutable values such as numbers, strings, or tuples.

Trap 5: Getting the wrapping order wrong. In counted_fib = count(fib) then fib = memo(counted_fib), the counter sits inside the cache, so it only ticks on a real computation. If you wrapped them the other way, the count would no longer measure distinct computations.

  2. Why does the cache dictionary keep its contents after the call to memo has already returned?
  3. After fib(19), why is counted_fib.call_count equal to 20 rather than 13529?
  4. Suppose you instead start fresh with counted_fib = count(fib) and fib = memo(counted_fib), then make the single call fib(6). What value does fib(6) return, and what is counted_fib.call_count afterward?

Show the answers
  1. On the first call, 5 is not in cache, so the body runs, prints the message, and stores cache[5] = 25. On the second call, 5 is already a key, so the if body is skipped entirely and the function returns the stored value without printing.
  2. The returned memoized function still refers to cache, so the dictionary remains part of memoized's enclosing environment and is not discarded. There is one such cache per call to memo.
  3. Computing fib(19) needs the distinct inputs 0 through 19, which is 20 values. Each is actually computed (and counted) only once; every repeated request is served from the cache, so counted_fib runs exactly 20 times.
  4. fib(6) returns 8. Computing it needs the distinct inputs 0 through 6, which is 7 values, each computed and counted exactly once, so counted_fib.call_count is 7.
  5. A dictionary locates an entry from its key, so the key must never change after insertion; a mutable key like a list could change and make its stored value impossible to find again.

§Summary

Memoization turns repeated questions into dictionary lookups by storing each result the first time it is computed. You saw it three ways: a manual cache guarded by an if n not in cache: test, the higher-order memo(f) that packages that pattern and returns a wrapper sharing one persistent cache through its enclosing environment, and the memoized fib, where a call counter proves that each distinct input is computed exactly once: 20 computations for fib(19), then 35 total after fib(34). The technique is especially powerful for tree-recursive functions, where it does not merely speed the process up but changes its shape, collapsing whole duplicated subtrees into single constant-time reads. It is safe precisely when the same input always yields the same answer and the argument is immutable.

▶Step-Through Checkpoint

This block is the lesson in miniature: memo wraps a plain squaring function, and three calls then put the cache to work. Stepping through it draws the environment diagram that makes the cache visible: the cache dictionary sits on the heap, tied to memoized's enclosing frame so it outlives the call to memo, and you watch it gain 4: 16 then 5: 25 while the second square(4) opens only a memoized frame that returns straight from that dict without ever opening a frame for the original square. Before you step through the block below, write down three predictions: how many frames will open for the original square over the whole run and how many for memoized, whether the line cache[n] = f(n) runs during the call that binds second, and what cache holds when the last line finishes.

Step through it one line at a time and check each prediction as the frames appear. The original square runs only twice, once for 4 and once for 5, while memoized opens a frame for all three calls. During the call that binds second, the n not in cache test is false, so the body is skipped and the call falls straight to return cache[n] without growing the cache. first, second, and third settle to 16, 16, and 25, and the cache kept alive by memoized ends at {4: 16, 5: 25}, alongside f, the original square function. The point to confirm is that cache is not rebuilt on each call. It belongs to the remembered environment of the returned function, which is why second = square(4) finds its answer already waiting.

4.8 · Parallel Computing — Coding Practice

◷ approx. 16 min read · 30 worked examples · best read in two passes

These problems build intuition for the deterministic models this section uses to reason about parallel programs without ever launching a real thread. You will apply an explicit schedule to shared state, replay the lost-update race by hand, decide when shared data is safe to leave unsynchronized, move work through a list-model queue and a list-model pipe, model a lock as a boolean flag and a barrier as two clean phases, and detect the circular wait that defines deadlock. Every model is plain Python you can run and trace — no real threads, no nondeterminism.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Run an Explicit Schedule

Once a second task is started, the order of visible effects is decided by a schedule, not by the order the lines were written. Model a schedule as a list of (who, step) pairs, walked in order. Write run_schedule(schedule) that returns a list of strings, one per entry, each formatted as "who -> step", in schedule order.

Examples

Input:  run_schedule([("main", "create other"), ("main", "start other"), ("other", "say hello"), ("main", "say hello")])
Output: ['main -> create other', 'main -> start other', 'other -> say hello', 'main -> say hello']
Explanation: each pair becomes one "who -> step" line, in the schedule's order.

Input:  run_schedule([("other", "say hello"), ("main", "say hello")])
Output: ['other -> say hello', 'main -> say hello']
Explanation: here the started task acts first, so its greeting line comes first.

Input:  run_schedule([])
Output: []
Explanation: an empty schedule produces no effects.

Constraints

  • schedule is a list of (who, step) tuples of strings.
  • Build the output in schedule order; do not sort or reorder.
  • Use string concatenation with " -> " between the two parts.

Two workers share a two-slot counter list. Output: [9, 4]

8. Detect a Circular Wait

Deadlock is a circular wait: each task waits for another, around a ring with no first mover. Model the wait relationships as a dict mapping each task to the single task it waits for. Write has_deadlock(waits_for, start) that follows the "waits for" links from start and returns True if the chain ever revisits a task (a cycle), or False if it reaches a task that waits for no one (a key not in the dict).

Examples

Input:  has_deadlock({"A": "B", "B": "C", "C": "A"}, "A")
Output: True
Explanation: A waits for B waits for C waits for A — the chain loops back, so it is a deadlock.

Input:  has_deadlock({"A": "B", "B": "C"}, "A")
Output: False
Explanation: A -> B -> C, and C waits for no one, so the chain ends; no cycle.

Input:  has_deadlock({"A": "A"}, "A")
Output: True
Explanation: A waits for itself — the smallest possible cycle.

Constraints

  • waits_for maps a task to the one task it waits for; a task missing from the dict waits for no one.
  • Follow links from start, remembering tasks already visited.
  • Return True on revisiting a task (cycle), False when the chain ends.

Hint

9. Replay an Arbitrary Interleaving

Generalize the lost-update replay to any schedule of read and write steps across several threads on one shared counter. Each thread has its own private snapshot; a read copies the live counter into that thread's snapshot, and a write stores snapshot + 1 back into the counter. Write replay_increments(num_threads, schedule) where schedule is a list of (thread_id, op) steps (op is "read" or "write"); run them in order and return the final counter value. The outcome depends entirely on how the reads and writes interleave.

Examples

Explanation: thread 0 fully finishes before thread 1 reads, so no update is lost — the counter reaches 2.

Solutions

1. Run an Explicit Schedule

Approach — walk the schedule in order. This is the deterministic model from 4.8-A: once a second task exists, the final order of effects follows the schedule, not the order the lines were written. Iterating the schedule list in order and formatting each (who, step) pair reproduces exactly that order of visible effects.

def run_schedule(schedule):
    log = []
    for who, step in schedule:
        log.append(who + " -> " + step)
    return log

Why it works. The list is processed front to back, so the produced lines appear in schedule order — the whole point of the model is that swapping entries in the schedule changes the output, just as a different real interleaving would.

Complexity. Time O(n), Space O(n) in the number of scheduled steps — one pass building one line each.

assert run_schedule([("main", "create other"), ("main", "start other"),
                     ("other", "say hello"), ("main", "say hello")]) == [
    "main -> create other", "main -> start other",
    "other -> say hello", "main -> say hello"]
assert run_schedule([("other", "say hello"), ("main", "say hello")]) == [
    "other -> say hello", "main -> say hello"]
assert run_schedule([("worker", "scan page")]) == ["worker -> scan page"]
assert run_schedule([]) == []

2. Replay the Lost Update

Approach — two stale reads, then two writes. This is the deterministic lost-update replay from 4.8-B. Both "threads" snapshot the shared counter before either writes, so both compute the same value + 1; the second write overwrites the first rather than building on it, and one increment disappears.

def lost_update(start):
    counter = [start]
    a_read = counter[0]
    b_read = counter[0]
    counter[0] = a_read + 1
    counter[0] = b_read + 1
    return counter[0]

Why it works. a_read and b_read are private copies frozen at start. Because neither read sees the other's write, both writes store start + 1, and the final write lands on top of the first — exactly the race condition the section warns about, played out by hand.

Complexity. Time O(1), Space O(1) — a fixed number of reads and writes.

assert lost_update(0) == 1
assert lost_update(5) == 6
assert lost_update(41) == 42
assert lost_update(99) == 100

3. Is the Shared Data Read-Only?

Approach — look for a write to the shared data. From 4.8-C: read-only data never needs synchronization, because with no writes the schedule cannot change any reader's result. So the data is safe exactly when no operation writes to the "shared" target.

def is_read_only(ops):
    for action, target in ops:
        if action == "write" and target == "shared":
            return False
    return True

Why it works. The only operation that can introduce a race is a write to data that other tasks read. A write to "local" data is private to its task, and any number of "read"s of shared data is safe, so the function returns False only on a ("write", "shared") and True otherwise — including for an empty list.

Complexity. Time O(n), Space O(1) — a single scan with early exit.

assert is_read_only([("read", "shared"), ("read", "shared")]) is True
assert is_read_only([("read", "shared"), ("write", "shared")]) is False
assert is_read_only([("read", "shared"), ("write", "local")]) is True
assert is_read_only([("write", "local"), ("read", "shared")]) is True
assert is_read_only([]) is True

4. Drain a Shared Work Queue

Approach — fill the back, drain the front. This is the single-threaded warm-up shape of the producer/consumer pattern from 4.8-D. A queue is first-in, first-out: appending puts work on the back, and repeatedly taking queue[0] removes it from the front, so items come out in the order they went in.

def drain_queue(items):
    queue = []
    for item in items:
        queue.append(item)
    consumed = []
    while len(queue) > 0:
        front = queue[0]
        del queue[0]
        consumed.append(front)
    return consumed

Why it works. Every append adds to the back and every del queue[0] removes from the front, which is precisely FIFO order, so consumed mirrors items. The real synchronized queue serializes concurrent producers and consumers, but the ordering it preserves is exactly this.

Complexity. Time O(n2), Space O(n) — each del queue[0] shifts the remaining items, so draining n items costs O(n2) overall; the result list holds n items.

assert drain_queue([0, 1, 2]) == [0, 1, 2]
assert drain_queue(["a", "b"]) == ["a", "b"]
assert drain_queue([]) == []
assert drain_queue([7]) == [7]
assert drain_queue([3, 1, 2]) == [3, 1, 2]

5. Lock-Guarded Check-and-Add

Approach — a boolean flag as the lock around the critical section. This models the web-crawler already_seen from 4.8-E: the check url not in seen and the seen.append(url) must run as one indivisible unit. The boolean lock_taken is the section's deterministic stand-in for the key — set True on entry, False on exit — with the check-and-add safely between them.

def register_seen(urls):
    seen = []
    lock_taken = False
    results = []
    for url in urls:
        lock_taken = True          # acquire the lock for the check-and-add
        if url not in seen:
            seen.append(url)
            results.append(False)  # url was new
        else:
            results.append(True)   # url was already seen
        lock_taken = False         # release the lock
    return results

Why it works. Because the membership test and the insertion live inside the same locked window, no other task can slip in between them, so each distinct URL is recorded exactly once. The first appearance of a URL returns False (new, now added); every repeat returns True (already present).

Complexity. Time O(n2), Space O(n) — each url not in seen scans the growing seen list, so n URLs cost O(n2); seen and results hold up to n entries.

assert register_seen(["a.html", "b.html", "a.html"]) == [False, False, True]
assert register_seen(["x", "x", "x"]) == [False, True, True]
assert register_seen([]) == []
assert register_seen(["p"]) == [False]
assert register_seen(["a", "b", "c", "a", "b"]) == [False, False, False, True, True]

6. One Barrier Round on Disjoint Slots

Approach — read phase, then write phase. This is the two-counter barrier model from 4.8-F. The barrier guarantees that all reads finish before any write begins, so both workers read the other slot first, and only then does each write its own slot. Because slot 0 and slot 1 are disjoint, the two writes never conflict and need no lock.

def barrier_round(counters, rounds):
    counters = list(counters)
    for _ in range(rounds):
        read0 = counters[1]
        read1 = counters[0]
        counters[0] = read0 + 1
        counters[1] = read1 + 1
    return counters

Why it works. Binding read0 and read1 before either assignment stands in for the first barrier: no write lands until both reads are taken, so neither worker can read a slot its partner has already bumped in the same round. Each round adds exactly one to each slot, and copying counters first keeps the caller's list untouched.

Complexity. Time O(rounds), Space O(1) — a constant amount of work per round on a fixed two-element list.

assert barrier_round([0, 0], 10) == [10, 10]
assert barrier_round([3, 8], 1) == [9, 4]
assert barrier_round([5, 5], 1) == [6, 6]
assert barrier_round([0, 0], 0) == [0, 0]
assert barrier_round([3, 8], 2) == [5, 10]

7. Consume a Pipe Until the Sentinel

Approach — receive until the None sentinel. This is the list-model pipe from 4.8-G: send appends to the back, recv reads and deletes the front, and the producer ends the stream with a None. The consumer loops, collecting each real value, and returns the moment it receives the sentinel.

def consume_pipe(messages):
    channel = list(messages)
    received = []
    while True:
        item = channel[0]
        del channel[0]
        if item is None:
            return received
        received.append(item)

Why it works. Reading channel[0] and deleting it models a blocking recv pulling the next message in order. The if item is None test is the agreed done signal: it ends the loop before any message behind the sentinel is ever received, exactly as the real consumer's return does.

Complexity. Time O(n2), Space O(n) — each del channel[0] shifts the rest, so receiving up to n messages costs O(n2); received holds the values before the sentinel.

assert consume_pipe([0, 1, 2, None]) == [0, 1, 2]
assert consume_pipe([5, 6, None, 7]) == [5, 6]
assert consume_pipe([None]) == []
assert consume_pipe([42, None, None]) == [42]
assert consume_pipe(["a", "b", "c", None]) == ["a", "b", "c"]

8. Detect a Circular Wait

Approach — follow the waits-for links and watch for a repeat. This is the deadlock model from 4.8-H: a circular wait is a cycle in the "waits for" graph. Starting at start, follow each task to the one it waits for; if the chain ever returns to a task already seen, the ring has closed and the program is deadlocked. If it reaches a task that waits for no one (a key absent from the dict), the chain ends and there is no deadlock.

def has_deadlock(waits_for, start):
    seen = []
    current = start
    while current in waits_for:
        if current in seen:
            return True
        seen.append(current)
        current = waits_for[current]
    return False

Why it works. Each task waits for at most one other, so following the links traces a single path. A task missing from waits_for waits for no one, ending the loop with False. Recording visited tasks in seen catches the moment the path revisits a task — the visible signature of a cycle with no first mover.

Complexity. Time O(n2), Space O(n) — up to n steps, each testing membership against the growing seen list of length up to n.

assert has_deadlock({"A": "B", "B": "C", "C": "A"}, "A") is True
assert has_deadlock({"A": "B", "B": "C"}, "A") is False
assert has_deadlock({"A": "A"}, "A") is True
assert has_deadlock({}, "A") is False
assert has_deadlock({"A": "B", "B": "A", "C": "A"}, "C") is True

9. Replay an Arbitrary Interleaving

Approach — private snapshots over a shared counter. This generalizes the lost-update replay (4.8-B) into the worst-case-pause tool from 4.8-I. Each thread owns a private snapshot; read freezes the live counter into that snapshot, and write stores snapshot + 1 back. Running an arbitrary schedule of these steps shows exactly how the final value depends on the interleaving.

def replay_increments(num_threads, schedule):
    counter = [0]
    snapshots = [0] * num_threads
    for tid, op in schedule:
        if op == "read":
            snapshots[tid] = counter[0]
        elif op == "write":
            counter[0] = snapshots[tid] + 1
    return counter[0]

Why it works. A thread's snapshot only changes on its own read, so a write always uses whatever value that thread last saw — even if another thread has since changed the counter. When two reads happen before either write (the worst-case pause), both snapshots hold the same old value, both writes store the same result, and an increment is lost. When each thread reads and writes before the next reads, every increment counts.

Complexity. Time O(s), Space O(t) — one pass over the s scheduled steps, with one snapshot per thread t.

assert replay_increments(2, [(0, "read"), (0, "write"),
                             (1, "read"), (1, "write")]) == 2
assert replay_increments(2, [(0, "read"), (1, "read"),
                             (0, "write"), (1, "write")]) == 1
assert replay_increments(3, [(0, "read"), (1, "read"), (2, "read"),
                             (0, "write"), (1, "write"), (2, "write")]) == 1
assert replay_increments(2, [(0, "read"), (0, "write"), (0, "read"),
                             (0, "write"), (1, "read"), (1, "write")]) == 3
assert replay_increments(1, []) == 0

10. Producer/Consumer Bookkeeping

Approach — track the unfinished count behind join. This models the synchronized-queue bookkeeping from 4.8-D: put raises an internal unfinished count, task_done lowers it, and join waits for it to reach zero. A get on an empty queue blocks the consumer. Simulating the operations in order reveals whether the program could exit cleanly, hang on join, or stall on an empty get.

def queue_audit(ops):
    queue = []
    unfinished = 0
    for op in ops:
        kind = op[0]
        if kind == "put":
            queue.append(op[1])
            unfinished = unfinished + 1
        elif kind == "get":
            if len(queue) == 0:
                return "blocked on get"
            del queue[0]
        elif kind == "task_done":
            unfinished = unfinished - 1
    if unfinished == 0:
        return "join returns"
    return "join blocks"

Why it works. The unfinished count is the exact bookkeeping that drives join: it rises on every put and falls on every task_done, independent of get (taking an item does not mark it done). So join can return only when every put item has been reported done. A get from an empty queue has no item to return, which models the consumer blocking — caught and reported immediately.

Complexity. Time O(n2), Space O(n) — one pass over n operations, but each get's del queue[0] shifts the remaining items; the queue holds up to n entries.

assert queue_audit([("put", "a"), ("get",), ("task_done",)]) == "join returns"
assert queue_audit([("put", "a"), ("put", "b"), ("get",),
                    ("task_done",)]) == "join blocks"
assert queue_audit([("get",)]) == "blocked on get"
assert queue_audit([("put", 0), ("put", 1), ("get",), ("task_done",),
                    ("get",), ("task_done",)]) == "join returns"
assert queue_audit([]) == "join returns"

Chapter 3/3.4 Interpreters for Languages with Combination

3.4.1 A Scheme-Syntax Calculator

◷ approx. 13 min read · 2 worked examples

Chapter 3 changes what you build. Until now you have written programs that compute answers. Now you will build a program that reads another program and computes its answer. A program that does that has a name you already know from day one: an interpreter.

To make that goal small enough to hold in your head, we start with a tiny language called the Calculator language. By the end of this lesson you will be able to do four things: state the two kinds of expression Calculator allows, read a Calculator expression by breaking it into tokens, predict the value of any arithmetic combination including the surprising zero-operand and one-operand cases, and explain the single rule the interpreter follows to evaluate a call expression. This lesson owns the language itself, its grammar, and what it computes. Later packets in this section show how the text becomes a tree, how that tree is parsed, and how an evaluator walks it.

A Calculator With One Strict Style

Picture a desk calculator that refuses every input except one exact style of instruction.

If you press the keys for ordinary school arithmetic, you write the operator between the numbers, which is called infix notation:

2 + 3

The Calculator language does not accept that. It uses prefix notation, where the operator comes first and the numbers follow:

(+ 2 3)

Read that out loud as a request: apply + to 2 and 3. The shape looks unfamiliar, but it is relentlessly regular. The operator always comes first. The numbers always come after it. The whole request is always wrapped in parentheses. That regularity is exactly why this language is a good first interpreter to build: there are no special cases in the grammar for the machine to worry about.

The Calculator language borrows this shape from Scheme. In fact Calculator shares Scheme's call-expression syntax and the behavior of its arithmetic operators, so everything you learn here transfers directly when you meet full Scheme.

Two Kinds of Expression, and Nothing Else

The entire Calculator language is built from just two kinds of expression. Holding both in mind is most of the work of this lesson.

  • A primitive expression is a number, such as 2, -3, or 15. A number stands for itself. There is nothing to compute.
  • A call expression is a parenthesized combination, such as (+ 2 3). It names an operator and some operands, and it asks the interpreter to compute a value by applying that operator.

That is the whole grammar. A Calculator program is a number, or a combination, where each operand inside a combination is itself a number or another combination. Because an operand can be another combination, combinations can nest as deeply as you like, and that nesting is where all the interesting structure comes from.

Calculator supports exactly four operators: addition +, subtraction -, multiplication *, and division /.

The Shape of a Call Expression

Every Calculator call expression has the same consistent form:

(operator operand-1 operand-2 ... operand-n)

In that template, operator, operand-1, and the rest are placeholders. They are not text you type literally; they stand for the real pieces you substitute in, such as +, 2, and 3. The rules are fixed:

  • the first item after ( is the operator
  • every item after the operator is an operand
  • the closing ) marks the end of the whole combination

One warning about the parentheses. In Python arithmetic, parentheses often mean "do this part first." Here they do not mean that. In Calculator, parentheses mark where one complete combination begins and ends. They are structure, not emphasis.

Here is the smallest real combination, with the interpreter's answer shown beneath the input:

> (+ 2 3)
5

The line starting with > is what you typed at the interpreter. The line below it is the value the interpreter computed: apply + to 2 and 3, which is 5.

Following the Line: A Call Expression Token by Token

When the interpreter meets (+ 2 3), it does not see a math problem the way you do. It sees a sequence of tokens read left to right. A token is one meaningful chunk of text, such as (, +, 2, or ). Walk the line one token at a time, then read the final row for the result:

(+ 2 3)
├─ ( ........... begins a call expression
├─ + ........... the operator: add the operands
├─ 2 ........... first operand, the number two
├─ 3 ........... second operand, the number three
├─ ) ........... ends the call expression
└─ evaluates to  5

The spaces are not decoration. They tell the interpreter where one token ends and the next begins. Without them, +2 and + 2 would be hard to tell apart. Here is a second combination read the same way:

(* 4 5)
├─ ( ........... begins a call expression
├─ * ........... the operator: multiply the operands
├─ 4 ........... first operand, the number four
├─ 5 ........... second operand, the number five
├─ ) ........... ends the call expression
└─ evaluates to  20

When you see any Calculator combination, the safe first question is always the same: what is the operator, and what are its operands?

The One Rule for Evaluating a Combination

There is a single rule that decides the value of every call expression, and it is worth memorizing:

To evaluate a call expression: evaluate each operand, then apply the operator to the resulting values.

For a flat expression like (+ 2 3), evaluating the operands is trivial because they are already numbers. The rule earns its keep when operands are themselves combinations, which is the case we build up to below. The same rule, applied over and over, handles every Calculator program no matter how deeply it nests.

Addition and Multiplication Take Any Number of Operands

Calculator's + and * each accept an arbitrary number of operands. Start with a warm-up you can read at a glance:

> (+ 2 3 4)
9
> (* 2 3 4)
24

Addition starts from the additive identity, 0 (the number you can add without changing anything), and adds each operand in turn, so (+ 2 3 4) is 0 + 2 + 3 + 4, which is 9. Multiplication starts from the multiplicative identity, 1 (the number you can multiply by without changing anything), and multiplies each operand in turn, so (* 2 3 4) is 1 * 2 * 3 * 4, which is 24.

That starting-from-the-identity idea explains a case that surprises most beginners: what happens with no operands at all?

> (+)
0
> (*)
1

This follows directly from what an identity value means:

0+x=x

and

1⋅x=x

If there are no operands to combine, the result is just the identity you started from. So (+) is 0 and (*) is 1.

Here is the official transcript from the source, which shows the same operators with four operands and with none:

> (+ 1 2 3 4)
10
> (+)
0
> (* 1 2 3 4)
24
> (*)
1

Subtraction and Division Depend on the First Operand

Subtraction and division behave differently from addition and multiplication, because the first operand sets the starting value rather than an identity. Each of them has two behaviors, one for a single operand and one for several.

With at least two operands, subtraction subtracts all the rest from the first. A warm-up:

> (- 10 1 2 3)
4

Read that as 10 - 1 - 2 - 3, evaluated left to right:

10 - 1 = 9
9 - 2 = 7
7 - 3 = 4

With a single operand, subtraction instead negates it: (- 3) means the opposite of 3. Division mirrors this exactly. With at least two operands it divides all the rest into the first; with a single operand it computes the multiplicative inverse, so (/ 10) means one divided by ten. A division warm-up read left to right:

> (/ 120 2 3 4 5)
1.0
120 / 2 = 60.0
60.0 / 3 = 20.0
20.0 / 4 = 5.0
5.0 / 5 = 1.0

Here is the official transcript from the source, which exercises both behaviors of - and /:

> (- 10 1 2 3)
4
> (- 3)
-3
> (/ 15 12)
1.25
> (/ 30 5 2)
3
> (/ 10)
0.1

Notice two details in that output. (- 3) produces -3, the negation. And (/ 10) produces 0.1, the reciprocal, written as a decimal because division generally yields a float.

Nested Combinations

Because an operand may itself be a combination, you can place a combination inside another combination. This is where the one evaluation rule does real work.

Start with a small bridge example, shown at the interpreter:

> (* (+ 1 2) (+ 2 3))
15

The interpreter does not guess. It applies the rule: evaluate each operand first. Both operands here are combinations, so evaluate them:

(* (+ 1 2) (+ 2 3))
   -------  -------
      3        5

Now both operands are plain numbers, and the outer combination becomes a flat one:

(* 3 5)

Applying * to 3 and 5 gives the final value 15. Here is a second bridge with division on the outside:

> (/ 40 (* 4 5))
2.0

The inner multiplication is an operand of the outer division, so it is evaluated first:

(* 4 5) -> 20

which leaves the flat combination:

(/ 40 20) -> 2.0

This is the same inside-out evaluation you already met with Python call expressions. The grammar is the only new part.

The source gives a densest nested example that uses all four operators at once. A call expression is evaluated by evaluating its operand sub-expressions, then applying the operator to the resulting arguments:

> (- 100 (* 7 (+ 8 (/ -12 -3))))
16.0

Take it apart from the inside out, one operand at a time:

(/ -12 -3) -> 4.0
(+ 8 4.0) -> 12.0
(* 7 12.0) -> 84.0
(- 100 84.0) -> 16.0

This single expression is the reason an interpreter for this language needs recursion: the same evaluate-the-operands rule has to call itself on the smaller combinations nested inside, exactly the kind of self-calling you saw earlier. The outer subtraction cannot run until the multiplication has a value; the multiplication cannot run until the addition has a value; the addition cannot run until the division has a value. Each combination waits on the combinations nested inside it.

A Step-by-Step Trace of the Machine Model

It helps to see the waiting explicitly. When the interpreter is handed this text:

(* (+ 1 2) (+ 2 3))

it cannot multiply right away. Following the one rule, it must produce values for both operands first, then apply the operator. Laid out as a tree, the values flow upward from the leaves to the root:

(* (+ 1 2) (+ 2 3))
       |         |
       3         5
        \       /
         (* 3 5)
            |
            15

The stages, in order, are: read the characters, split them into tokens, build the expression tree, evaluate the left operand to 3, evaluate the right operand to 5, apply the outer operator *, and return 15. The next packets in this section implement each of those stages in Python.

⚠Common Beginner Traps

  • Reading prefix notation as if it were Python infix. (+ 2 3) does not mean "open parenthesis, then plus, then two." It means "apply + to the operands 2 and 3."
  • Treating parentheses as optional or as emphasis. In Calculator, parentheses mark where a complete combination begins and ends. Removing them changes the structure or makes the input invalid.
  • Evaluating the outer operator before its operands. In (* (+ 1 2) (+ 2 3)), both inner + combinations must be evaluated to 3 and 5 before * can be applied.
  • Forgetting the identity values. (+) is 0 and (*) is 1 because those are the identities for addition and multiplication, and with no operands the result is just the identity.
  • Assuming every operator follows the same argument rules. Addition and multiplication accept zero operands; subtraction and division have a special one-operand behavior, negation and reciprocal, and otherwise work from the first operand.

§Summary

The Calculator language is a tiny Scheme-syntax language for arithmetic with four operators: +, -, *, and /. It has exactly two kinds of expression: primitive numbers, which stand for themselves, and call expressions, which are parenthesized combinations written in prefix form with the operator first. Addition and multiplication accept any number of operands and fall back to the identities 0 and 1 when given none; subtraction and division work from the first operand and treat a lone operand as negation and reciprocal. Every combination is evaluated by one rule, evaluate the operands and then apply the operator, and because operands may themselves be combinations, that rule recurses through any depth of nesting. The next step is to represent these expressions as data so an ordinary Python program can read and evaluate them.

2.2 · Data Abstraction — Coding Practice

◷ approx. 13 min read · 31 worked examples

These problems build fluency with data abstraction: designing constructors and selectors for two-field values, building operations on rational numbers and points through that interface, respecting the abstraction barrier instead of reaching into the representation, and even implementing a pair out of nothing but a function.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Throughout, a rational number is built and inspected only through rational(n, d), numer(x), and denom(x) (the constructor reduces to lowest terms). A point is built and inspected only through point(x, y), x_coord(p), and y_coord(p). Stay on the user side of the barrier: call the selectors, never x[0].

Easy

1. Reciprocal of a Rational

Examples

Output: 5
Explanation: 2/5 flipped is 5/2, whose numerator is 5.

Explanation: the denominator of 5/2 is 2.

Input:  reciprocal(rational(6, 9))

Explanation: 6/9 reduces to 2/3, and the reciprocal 3/2 is already reduced.

  • Build the result with rational; read parts with numer and denom. No x[0] / x[1].

10. Continued Fractions

A two-term continued fraction [a; b] means a + 1/b, which as a single fraction is (a*b + 1)/b. Explanation: 2 + 1/3 = 7/3.

  • a, b, c are positive integers.

Solutions

Setup

These problems are written against the section's rational-number interface. The constructor reduces every rational to lowest terms, and mul_rationals is the section's multiply operation. Define these once before running any solution below:

1. Reciprocal of a Rational

Approach — flip through the interface. Swapping a fraction's parts is a one-line operation if you stay on the user side of the barrier: read both parts with the selectors and hand them to the constructor in the opposite order. The constructor reduces the result, so 6/9 flipped comes back as [3, 2].

def reciprocal(x):
    return rational(denom(x), numer(x))

Why it works. The reciprocal of n/d is d/n. Passing denom(x) as the new numerator and numer(x) as the new denominator builds exactly that, and rational reduces it. Because it never touches x[0], this code would keep working if rationals were represented some other way.

Complexity. Time O(1), Space O(1) (plus the cost of one gcd inside the constructor) — a fixed amount of work.

assert numer(reciprocal(rational(2, 5))) == 5
assert denom(reciprocal(rational(2, 5))) == 2
assert numer(reciprocal(rational(3, 4))) == 4
assert denom(reciprocal(rational(3, 4))) == 3
# Reduction still happens through the constructor.
assert reciprocal(rational(6, 9)) == [3, 2]

2. Scale a Rational by an Integer

Approach — multiply the numerator only. Multiplying a fraction by a whole number k is the same as multiplying its numerator by k and leaving the denominator alone. Read the parts with the selectors, scale the numerator, and rebuild with rational, which reduces the result.

def scale_rational(x, k):
    return rational(numer(x) * k, denom(x))

Why it works. (n/d) * k = (n*k)/d. The constructor then reduces, so (2/5)*5 = 10/5 comes back as [2, 1]. The function speaks only the interface, never the list shape.

Complexity. Time O(1), Space O(1) — one multiplication and one constructor call.

assert scale_rational(rational(1, 3), 2) == [2, 3]
assert scale_rational(rational(2, 5), 5) == [2, 1]
assert numer(scale_rational(rational(3, 7), 0)) == 0
assert scale_rational(rational(1, 4), 8) == [2, 1]

3. A Constructor and Selectors for a 2D Point

Approach — a two-field abstraction, mirroring rationals. A point has the same shape as a rational number: two fields glued into one value. Represent it as a two-element list, with the constructor packing both coordinates and each selector pulling one back out by index. This is the representation layer, the one place allowed to know the list shape.

def point(x, y):
    return [x, y]

def x_coord(p):
    return p[0]

def y_coord(p):
    return p[1]

Why it works. The constructor and selectors satisfy the pair contract: x_coord(point(a, b)) is a and y_coord(point(a, b)) is b. Everything built on top of points can now ignore that they are lists.

Complexity. Time O(1), Space O(1) for each of the three operations — build or read one fixed-size container.

assert x_coord(point(3, 7)) == 3
assert y_coord(point(3, 7)) == 7
assert x_coord(point(0, 0)) == 0
assert y_coord(point(0, 0)) == 0

4. Is a Rational a Whole Number?

Approach — inspect the reduced denominator. Because rational reduces to lowest terms, any rational that equals an integer has denominator 1. So the test is a single selector call and a comparison — no division, which keeps everything exact.

def is_whole(x):
    return denom(x) == 1

Why it works. In lowest terms, n/d is an integer exactly when d == 1; for example 6/3 is stored as 2/1. Checking denom(x) == 1 answers the question using only the interface, with no float conversion that could mislead.

Complexity. Time O(1), Space O(1) — one selector and one comparison.

assert is_whole(rational(8, 1)) is True
assert is_whole(rational(6, 3)) is True
assert is_whole(rational(2, 5)) is False
assert is_whole(rational(7, 7)) is True
assert is_whole(rational(9, 4)) is False

5. Subtract Two Rationals

Approach — common-denominator rule with a minus. Subtraction is add_rationals with the cross terms subtracted instead of added. Name the four parts with multiple assignment, form nx*dy - ny*dx over the shared denominator dx*dy, and let the constructor reduce.

def sub_rationals(x, y):
    nx, dx = numer(x), denom(x)
    ny, dy = numer(y), denom(y)
    return rational(nx * dy - ny * dx, dx * dy)

Why it works. Putting both fractions over dx*dy gives nx*dy/(dx*dy) and ny*dx/(dx*dy); subtracting the numerators yields (nx*dy - ny*dx)/(dx*dy). Negative results are fine — 1/5 - 1/2 comes back as [-3, 10].

Complexity. Time O(1), Space O(1) — a fixed number of multiplications plus one constructor call.

assert sub_rationals(rational(1, 2), rational(1, 3)) == [1, 6]
assert sub_rationals(rational(3, 4), rational(1, 4)) == [1, 2]
assert sub_rationals(rational(2, 3), rational(2, 3)) == [0, 1]
assert sub_rationals(rational(1, 5), rational(1, 2)) == [-3, 10]

6. Midpoint of Two Points

Approach — average each coordinate, stored exactly. The midpoint's x-coordinate is the average of the two x-coordinates, which may not be a whole number, so represent it as a rational with denominator 2 instead of dividing. Do the same for y, then bundle both into a point. Two layers of abstraction stack here: points on the outside, rationals on the inside, each touched only through its own interface.

Why it works. The midpoint of x1 and x2 is (x1 + x2)/2. Handing the sum and 2 to rational keeps the coordinate exact and reduced, so the midpoint of 2 and 6 is stored as [4, 1], not a float. point then packages the two rational coordinates into one value.

Complexity. Time O(1), Space O(1) — two averages and one point.

m = midpoint(point(0, 0), point(3, 5))
assert x_coord(m) == [3, 2]
assert y_coord(m) == [5, 2]
m2 = midpoint(point(2, 4), point(6, 8))
assert x_coord(m2) == [4, 1]
assert y_coord(m2) == [6, 1]
m3 = midpoint(point(1, 1), point(1, 3))
assert x_coord(m3) == [1, 1]
assert y_coord(m3) == [2, 1]

7. Compare Two Rationals

Approach — cross-multiply with <. The equality test in this section cross-multiplies to avoid floats; the same trick orders two fractions. With positive denominators, a/b < c/d exactly when a*d < c*b, and that comparison stays in exact integer arithmetic.

def rational_less(x, y):
    return numer(x) * denom(y) < numer(y) * denom(x)

Why it works. Multiplying both sides of a/b < c/d by the positive quantity b*d clears the denominators without flipping the inequality, leaving a*d < c*b. Equal fractions like 1/2 and 2/4 give equal products, so rational_less correctly returns False for them.

Complexity. Time O(1), Space O(1) — two products and one comparison.

assert rational_less(rational(1, 3), rational(1, 2)) is True
assert rational_less(rational(2, 3), rational(1, 2)) is False
assert rational_less(rational(1, 2), rational(2, 4)) is False
assert rational_less(rational(3, 8), rational(5, 8)) is True
assert rational_less(rational(7, 10), rational(7, 9)) is True

8. Power of a Rational

Approach — recursion that multiplies through the interface. This is integer power with mul_rationals doing the multiplying. The base case k == 0 returns the rational 1/1. Otherwise multiply one copy of x onto x ** (k - 1), recurring with a smaller exponent until it bottoms out.

def pow_rational(x, k):
    if k == 0:
        return rational(1, 1)
    else:
        return mul_rationals(x, pow_rational(x, k - 1))

Why it works. By the recursive leap of faith, pow_rational(x, k - 1) is x multiplied k - 1 times; multiplying by one more x with mul_rationals gives x multiplied k times. The base case returns the multiplicative identity 1/1, and the constructor keeps each intermediate product reduced.

Complexity. Time O(k), Space O(k) — one frame per remaining exponent.

assert pow_rational(rational(2, 3), 0) == [1, 1]
assert pow_rational(rational(2, 3), 1) == [2, 3]
assert pow_rational(rational(2, 3), 2) == [4, 9]
assert pow_rational(rational(1, 2), 3) == [1, 8]
assert pow_rational(rational(3, 2), 3) == [27, 8]

9. A Rational Built on a Functional Pair

Approach — the representation is a closure, the operations do not care. pair(a, b) returns an inner get that remembers a and b; select(p, i) runs that function with the chosen index. Build the fraction interface on top of pair/select, then write frac_add exactly like add_rationals but with the new constructor and selectors. Nothing here indexes a list, and the addition still produces the right numerator and denominator — proof that data is defined by behavior, not shape.

Why it works. The functional pair satisfies select(pair(a, b), 0) == a and select(pair(a, b), 1) == b, which is the entire pair contract. Because frac_add reaches a fraction only through frac_numer, frac_denom, and make_frac, it behaves identically to add_rationals even though the storage is a function. make_frac does not reduce, so 1/3 + 1/3 comes back as 6/9.

Complexity. Time O(1), Space O(1) per operation — constructing a closure and calling it are constant-time; frac_add does a fixed amount of arithmetic.

half = make_frac(1, 2)
third = make_frac(1, 3)
assert frac_numer(half) == 1
assert frac_denom(half) == 2
s = frac_add(half, third)
assert frac_numer(s) == 5
assert frac_denom(s) == 6
s2 = frac_add(third, third)
assert frac_numer(s2) == 6
assert frac_denom(s2) == 9

10. Continued Fractions

Approach — collapse from the inside out, on the rational interface. A two-term continued fraction a + 1/b is the single fraction (a*b + 1)/b. For three terms, evaluate the inner [b; c] first with continued, then add a to its reciprocal: if the tail is p/q, then a + q/p = (a*p + q)/p. Reading the tail's parts with numer and denom keeps continued3 on the interface and lets it reuse continued.

def continued(a, b):
    return rational(a * b + 1, b)

def continued3(a, b, c):
    tail = continued(b, c)
    return rational(a * numer(tail) + denom(tail), numer(tail))

Why it works. continued is just the algebra a + 1/b = (a*b + 1)/b. In continued3, the tail continued(b, c) is the rational p/q, and a + 1/(p/q) = a + q/p = (a*p + q)/p, which is rational(a * numer(tail) + denom(tail), numer(tail)). Building from the innermost term outward is the natural recursion of continued fractions, and every step stays in exact integer arithmetic.

Complexity. Time O(1), Space O(1) — a fixed number of selector and constructor calls (the tail is computed once).

assert continued(2, 3) == [7, 3]
assert continued(1, 1) == [2, 1]
assert continued3(1, 2, 3) == [10, 7]
assert continued3(3, 1, 1) == [7, 2]
assert continued3(2, 1, 2) == [8, 3]

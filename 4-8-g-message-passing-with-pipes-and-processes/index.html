Chapter 4/4.8 Parallel Computing

4.8.7 Message Passing

◷ approx. 22 min read · 7 worked examples · best read in two passes

Every tool you have met so far in this chapter, locks, synchronized queues, and barriers, exists to make it safe for several tasks to touch the same data. This lesson teaches the opposite strategy: do not let them touch the same data at all. Instead, give each task its own private world and let the tasks communicate by sending messages.

By the end of this lesson you will be able to do four things: explain why separate processes do not share memory and why that is sometimes an advantage, create a one-way channel between two processes with Pipe, use send and recv to move values across that channel and explain why recv makes a process wait, and read the official producer-consumer program and the official duplex-pipe program one line at a time. You will also see, concretely, why two processes that both try to receive first can hang forever.

An Intuitive Picture: Sealed Envelopes Between Offices

Imagine two offices in different buildings. Each office keeps its own private notebook on its own desk. Neither office can reach across town and scribble in the other's notebook, that is the whole point of being separate.

So how do they coordinate? They mail sealed envelopes. When the first office wants the second to know a number, it writes the number on a slip, seals it in an envelope, and drops it in the mailbox. The second office opens the envelope when it arrives and copies the number into its own notebook.

Three features of this picture map exactly onto message passing between processes, so hold onto them:

  • Nothing is shared. A copy of the slip travels in the envelope. The original stays in the first office. After the exchange there are two separate slips in two separate notebooks, not one slip both offices are editing.
  • Receiving means waiting. If the second office checks an empty mailbox, it cannot proceed. It waits by the mailbox until an envelope shows up. There is no way to "open mail that has not arrived."
  • You need a way to say "that was the last letter." Otherwise the second office waits by the mailbox forever, expecting more.

That last point is the source of most beginner bugs, and we will give it a name shortly.

The Formal Idea: Separate Interpreters, No Shared Data

A final mechanism to avoid improper mutation of shared data is to entirely avoid concurrent access to the same data. In Python, using multiprocessing rather than threading naturally produces this situation, because processes run in separate interpreters, each with its own data. Two threads in one process can both see the same list object; two processes cannot, because each one is a separate running copy of Python with its own memory.

That separation is exactly the safety property we want. There is no shared list for two tasks to corrupt, because there is no shared list at all. The cost is that the convenience of shared memory is gone too. Any state that more than one process needs must be communicated explicitly, by passing messages between processes. Nothing is shared by accident, which also means nothing is shared for free.

The channel Python gives us for those messages is a pipe.

Pipes: A Channel With Two Ends

The Pipe class in the multiprocessing module provides a communication channel between processes. A pipe has two ends, and you get both of them at once when you create it. By default a pipe is duplex, meaning it is a two-way channel, both ends can send and both ends can receive. Passing the argument False instead gives you a one-way channel, where one end sends and the other end receives.

You move data with two methods on a pipe end:

  • The send method sends an object over the channel.
  • The recv method receives an object from the channel.

The detail that surprises beginners is that recv is blocking. A process that calls recv will wait until an object is received. It does not return an empty value or an error when the channel is empty; it simply pauses at that line until something arrives, just like the office worker waiting by the mailbox.

Before the official code, here is the small vocabulary this section uses.

Term What it means
Pipe() A factory that returns the two ends of a channel, as a pair. Duplex by default.
Pipe(False) The same factory, but the channel is one-way: one end sends, the other receives.
send(obj) A method on a pipe end that places obj into the channel. The object is copied to the other process; memory is not shared.
recv() A method on a pipe end that waits for and returns the next object from the channel. Blocking: if nothing has arrived, the process pauses here.
Sentinel An agreed-upon special value, here None, that means "no more real messages."

Progressive Examples

Warm-Up: One Send, One Receive

Before the full program, picture the smallest possible exchange across a one-way pipe. One end sends a single number; the other end receives it. In words:

sender end:    put 7 into the channel
receiver end:  wait for the channel, get 7

There is no loop and no done signal here, just one value crossing once. The receiver waits, the sender sends 7, the receiver wakes up holding 7. That single hop, wait then receive, is the atom that every larger pattern is built from.

Bridge: A Plain-Python Model of a Pipe (My Own Simpler Example)

This bridge is my own simplified stand-in, not source code. It is not how real pipes work, and it does not use multiple processes. Its only job is to make send and recv feel concrete by modeling a pipe as an ordinary list of messages: send appends to the back, and recv removes from the front. Because it runs in a single process with no real waiting, you can trace it by hand with total certainty.

This prints got an item: 10 and then done signal is: None. The first message is real work; the second message, None, is the agreed-upon "no more real messages" value. Keep this picture, a queue of messages with a final None, in mind as you read the official program, which does the same thing across two real processes.

The real pipe differs in one crucial way: when the list-model recv runs on an empty list it would raise an error, but a real recv on an empty channel waits instead. That waiting is what makes message passing a coordination tool and not just a data structure.

The Official Example, Verbatim: Producer and Consumer

This is the official program from the source, reproduced exactly. One process produces the numbers 0 through 9 and sends them down a one-way pipe; a second process consumes them and prints each one. The shutdown is handled by sending None as the last message.

# (5)
def process_consume(in_pipe):
    while True:
        item = in_pipe.recv()
        if item is None:
            return
        print('got an item:', item)

def process_produce():
    pipe = multiprocessing.Pipe(False)
    consumer = multiprocessing.Process(target=process_consume, args=(pipe[0],))
    consumer.start()
    for i in range(10):
        pipe[1].send(i)
    pipe[1].send(None) # done signal

process_produce()

Read it in two halves, matching the office picture.

process_consume is the receiving office. It loops forever. Each pass, it calls in_pipe.recv(), which waits for the next object. If the object it receives is None, that is the agreed signal that no more real work is coming, so it returns and the loop ends. Otherwise the object is a real item, and it prints got an item: followed by that item.

process_produce is the sending office and the manager. It creates a one-way pipe with multiprocessing.Pipe(False). It then creates a separate consumer process whose target is process_consume, and, crucially, it hands the receiving end pipe[0] to that process as an argument. It calls consumer.start() to launch the consumer, then sends the numbers 0 through 9 on the sending end pipe[1]. Finally it sends None as the done signal.

The line consumer = multiprocessing.Process(target=process_consume, args=(pipe[0],)) describes a second process: target names the function that process will run, and args lists the values handed to that function. Here args takes a tuple of arguments, so even a single argument is written with a trailing comma, as in (pipe[0],). The call consumer.start() then actually launches it, like sending the second office worker off to their building to begin work.

Two points the source emphasizes are worth stating plainly. First, the program uses a None message to signal the end of communication; without that sentinel, the consumer's recv would wait forever for an eleventh message that never comes. Second, one end of the pipe is passed in as an argument to the target function when the consumer process is created. This is necessary, because state must be explicitly shared between processes. There is no shared variable the consumer could reach on its own; it can only see the pipe end it was handed.

A Numbered Trace of the Official Program

Because two processes run at once here, it helps to walk the exchange as a sequence of events. (This is a time-ordered protocol, so it is a numbered trace, not a token anatomy.) A representative ordering is:

  1. The producer creates the one-way pipe, getting pipe[0] (receive end) and pipe[1] (send end).
  2. The producer creates the consumer process, handing it pipe[0], and calls start().
  3. The consumer begins its loop and calls recv() on pipe[0]. The channel is empty, so it waits.
  4. The producer sends 0 on pipe[1]. A copy of 0 enters the channel.
  5. The consumer wakes, receives 0, sees it is not None, and prints got an item: 0. It loops and calls recv() again.
  6. Steps 4 and 5 repeat for 1, 2, ..., 9, printing got an item: for each.
  7. The producer sends None on pipe[1].
  8. The consumer receives None, the if item is None test is true, and it returns. Its loop ends and the process finishes.

Notice that step 3 can happen before step 4: the consumer often reaches recv and waits before the producer has sent anything. That is fine, and it is exactly the behavior recv is designed for. The blocking is what keeps the consumer correctly idle until there is work.

Shared Memory Versus Messages

It is worth stepping back to compare the two coordination strategies this chapter has taught.

With shared memory (the threads-with-locks world), communication is cheap: two threads simply read and write the same object, and a value written by one is instantly visible to the other. The difficulty is safety. Because both can touch the same object at the same time, you must add locks, queues, or barriers to prevent races, and getting that protection right is hard.

With message passing (the processes-with-pipes world), safety is clearer: each process owns its own data, so there is no shared object to corrupt and far less to accidentally share. The cost moves to communication. Every value another process needs must be sent and received explicitly, and the design work shifts from "which lock protects this object?" to "which messages must move, and when must each process wait?"

Shared memory (threads) Message passing (processes)
How tasks communicate Read and write the same object Send and receive copies over a channel
What is shared The data itself Nothing automatically; only what you send
Main risk Races on shared mutable state A process waiting forever for a message
Main safeguard Locks, queues, barriers A clear protocol with a done signal
Where the hard work goes Protecting shared state correctly Designing the messages and the waits

Neither side is free. Message passing does not remove difficulty; it relocates it from synchronization to protocol design.

The Official Duplex-Pipe Example

The producer-consumer program above used a one-way pipe, Pipe(False). Recall that a pipe is duplex by default, so Pipe() with no argument gives a two-way channel where both ends can send and receive. A two-way channel is convenient, but it makes a new mistake possible, and the source uses it to set up a deadlock between two processes. We reproduce that program here because the duplex pipe is part of this lesson's scope. The full treatment of deadlock, what it is and how to avoid it, belongs to the synchronization-pitfalls lesson; here we focus on the pipe mechanics that cause it.

As an example, two processes are set up to deadlock. Suppose they share a duplex pipe and attempt to communicate with each other as follows. This is the official code, reproduced exactly:

# (6)
def deadlock(in_pipe, out_pipe):
    item = in_pipe.recv()
    print('got an item:', item)
    out_pipe.send(item + 1)

def create_deadlock():
    pipe = multiprocessing.Pipe()
    other = multiprocessing.Process(target=deadlock, args=(pipe[0], pipe[1]))
    other.start()
    deadlock(pipe[1], pipe[0])

create_deadlock()

Look closely at what each side tries to do first. The deadlock function receives first (in_pipe.recv()), then prints, then sends (out_pipe.send(item + 1)). create_deadlock builds a default, duplex pipe, starts a second process running deadlock with the ends in one arrangement, and then calls deadlock itself with the ends swapped. In the child, in_pipe is pipe[0]; in the parent, in_pipe is pipe[1]. The ends are swapped so each process receives on a different end, but because in_pipe.recv() is the first line either way, both processes call recv before either calls send. So both the child process and the parent are running the same deadlock function, and the very first thing each one does is call recv.

Both processes attempt to receive data first. Recall that the recv method blocks until an item is available. Since neither process has sent anything, both will wait indefinitely for the other to send it data, resulting in deadlock. Each is frozen on its own recv, waiting for a message the other will never get the chance to send.

A Numbered Trace of the Deadlock

  1. create_deadlock creates a default duplex pipe with both ends, pipe[0] and pipe[1].
  2. It starts the other process running deadlock(pipe[0], pipe[1]).
  3. It then calls deadlock(pipe[1], pipe[0]) in the parent.
  4. The other process reaches its first line, item = in_pipe.recv(), on its receiving end. Nothing has been sent, so it waits.
  5. The parent reaches its first line, item = in_pipe.recv(), on its receiving end. Nothing has been sent, so it waits too.
  6. Now both processes are paused at recv. The only line that could unblock either one is the send later in deadlock, but neither process can reach its send, because neither can get past its recv.
  7. Nothing changes, ever. Both processes wait forever. This frozen, mutually-waiting state is the deadlock.

The contrast with the working producer-consumer program is the lesson: there, one side committed to sending before the other needed to receive, so the channel was never empty when it mattered. Here, both sides insist on receiving first, and a channel that starts empty stays empty.

Where Pipes Lead: The Circular Particle Pipeline

The barriers lesson ended by promising that the same particle simulator you met there would return here, rebuilt out of processes and pipes instead of threads and a shared barrier. This is where that promise is kept. The source closes its treatment of message passing with exactly this real-world application, and we quote it verbatim:

> The multiprocess version of the particle simulator uses pipes to communicate particle positions between processes in each timestep. In fact, it uses pipes to set up an entire circular pipeline between processes, in order to minimize communication.

Every word of that sentence is now something you can unpack with the mechanics from this lesson. "Pipes to communicate particle positions between processes" is just send and recv moving copies of data across the channels you built above; recall that a sent value is copied, not shared, so each process ends a timestep holding its own picture of where the particles are. "In each timestep" means this exchange repeats once per simulated tick of the clock, the same per-step rhythm the threaded version used between its barriers. The new idea is the shape of the wiring: a circular pipeline, also called a ring.

The following picture is my own bridge, not source code, offered only to make the word "circular" concrete. Arrange the processes in a circle. Give each process a one-way pipe to its clockwise neighbor: each process holds a sending end aimed at the next process and a receiving end fed by the previous one. On each timestep, every process sends its own particles' positions to its neighbor and receives the previous neighbor's positions, then passes those along on the next hop. After one full trip around the ring, every process has seen every other process's particles, but no process ever had to broadcast to all the others at once. That is the meaning of "in order to minimize communication": instead of each of the processes talking to every other process directly, each one talks only to its single neighbor, and the data walks the rest of the way around the circle hand to hand.

Notice that the ring is built entirely from the parts this lesson already gave you, with nothing new required: it is a handful of one-way pipes, the same send and recv calls, the same fact that sent values are copied rather than shared, and a protocol that says which neighbor each process talks to and when. The full simulator code is beyond this lesson's scope, but its communication backbone is not: it is pipes, used exactly the way the producer-consumer program used them, wired into a circle so that separate processes can update shared information without ever sharing memory.

⚠Common Beginner Traps

  • Thinking message passing means no waiting. recv is blocking. A process that calls it stops until a message arrives. Waiting is a feature, not a bug, but you must expect it.
  • Forgetting the done signal. Without a sentinel such as None, a consumer's loop calls recv one more time than there are messages and waits forever. Always send an agreed end-of-stream value.
  • Mentally sharing objects between processes. A sent value is copied, not shared. After send, the sender and receiver hold separate objects. Do not imagine one object that both processes are editing.
  • Both sides receiving first. If two processes each call recv before either calls send, neither can make progress. That is precisely the duplex-pipe deadlock.
  • Expecting a process to reach state it was not handed. A new process can only see the pipe ends (and other arguments) explicitly passed to it. State must be shared explicitly; there is no hidden shared variable to fall back on. If a child relies on an outside variable instead, it works on its own separate copy, so any change it makes silently never appears in the parent.
  • Believing message passing removes complexity. It moves the complexity from locks and races to protocol design: which messages move, when each side waits, and what signal means "done."

  7. The ring lets each process talk only to its single neighbor instead of broadcasting to every other process, which minimizes communication while still letting every process eventually see every other's particle positions as the data is passed around the circle each timestep. It is built from the same pieces as the producer-consumer program: one-way pipes plus send and recv, with copied (not shared) values.

§Summary

Message passing coordinates parallel work by communication instead of shared mutation. Because processes run in separate interpreters with their own data, they share nothing automatically; any state more than one process needs must be sent explicitly. A Pipe gives two ends of a channel, duplex (two-way) by default or one-way with Pipe(False). You move values with send, which delivers a copy, and recv, which blocks until a value arrives. The official producer-consumer program shows the healthy pattern: produce items, send them, and finish with a None sentinel so the consumer's blocking recv loop knows to stop. The official duplex-pipe program shows the failure mode: when both processes try to recv first, the channel stays empty and both wait forever, a deadlock. These same pipe mechanics scale up to a real application: the multiprocess particle simulator uses pipes to pass particle positions between processes each timestep, wired into a circular pipeline so each process communicates only with its neighbor and no one broadcasts to everyone at once. The trade-off against shared memory is clean: message passing trades the safety hazards of shared state for the design work of choosing the right messages, the right waits, and the right done signal.

Stepping through it draws the environment diagram standing in for the channel: the global frame holds channel pointing at one shared list object on the heap, and you can watch each send frame append to the back of that list while each recv frame reads and deletes from its front, so the produce and consume calls coordinate entirely through that single shared list rather than any value passing directly between them.

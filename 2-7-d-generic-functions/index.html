Chapter 2/2.7 Object Abstraction

2.7 Generic Functions

◷ approx. 17 min read · 12 worked examples · best read in two passes

A generic function is one operation that works on values of more than one type. You have already used several without thinking about it: len works on strings and lists, repr works on almost everything, and the Complex.add method from the previous packet works on both rectangular and polar complex numbers. In each case, one name hides several possible behaviors.

By the end of this lesson you will be able to do four things. You will recognize when two values are "the same kind" versus "different kinds." You will use type dispatching to inspect the kinds of the arguments and pick the matching code path. You will attach a type tag to a class so a generic operation can compare kinds with a simple string. And you will use coercion to convert one kind into another so a single shared operation can handle both.

The running goal is the one that has driven this whole chapter: let different kinds of objects work together without breaking the abstraction barriers that keep each kind self-contained.

A Cafeteria With Different Payment Cards

Picture the checkout machine at a cafeteria. It accepts three things you might tap on the reader:

student ID cards
staff ID cards
guest credit cards

You want one button on the machine, and you want it to mean the same thing every time:

charge(card, amount)

But behind that one button, each card goes through a different process. A student ID bills a prepaid meal plan. A staff ID routes through payroll deduction. A guest credit card runs through the bank. The cashier presses one button; the machine quietly does the right thing for whatever was tapped.

There are two honest ways to build that machine. The first is type dispatching: the machine looks at what kind of card it is, then runs the matching procedure.

If the card is a student ID, use meal-plan billing.
If the card is a staff ID, use payroll billing.
If the card is a guest card, use credit billing.

The second is coercion: the machine converts every supported card into one common payment request, then runs a single billing procedure on that request.

Convert the card into a standard payment request,
then process that one kind of request.

Both designs are real and both appear in serious software. The right choice depends on whether the kinds are genuinely separate or whether one can be safely turned into another. That is exactly the choice we will make for numbers.

The Number Interface From Last Time

We start from the same small superclass we built in the previous packet. It says that the + and * operators should defer to ordinary methods named add and mul.

# (1)
class Number:
    def __add__(self, other):
        return self.add(other)

    def __mul__(self, other):
        return self.mul(other)

In words, this class promises two translations:

x + y   ->   x.add(y)
x * y   ->   x.mul(y)

As long as both operands are the same general kind of number, this is enough: x.add(y) knows what to do because x and y speak the same internal language. The trouble begins when they do not, which is the whole subject of this lesson.

Rational Numbers as Exact Fractions

To create that trouble, we introduce a second kind of number. A rational number is a ratio of two integers,

dn​

with d=0. Unlike a float, a rational number is stored as an exact pair of integers, so it never rounds.

One detail of history matters here. The original Composing Programs text imported the greatest-common-divisor helper from the fractions module:

from fractions import gcd

That location was removed from modern Python. The current home for the same function is the math module, so this course uses:

from math import gcd

The behavior is identical for our needs: gcd(a, b) returns the greatest common divisor of two integers, which the constructor uses to reduce the fraction to lowest terms.

Here is the Rational class, with the add and mul methods that implement the Number interface.

Expected output:

Rational(1, 2)
Rational(1, 6)

Following the Line: The Addition Result

The one line that does the arithmetic in add is worth slowing down on. Here it is broken into its tokens.

return Rational(nx * dy + ny * dx, dx * dy)
├─ return ........... hand a value back to the caller
├─ Rational( ........ build a new exact fraction
├─ nx * dy + ny * dx  the new numerator: cross-multiply, then add
├─ , ................ separates numerator from denominator
├─ dx * dy .......... the new denominator: the common denominator
├─ ) ................ finishes the constructor call
└─ evaluates to ..... a reduced Rational, e.g. Rational(1, 2)

This is the schoolbook rule for adding fractions, written once:

dx​nx​​+dy​ny​​=dx​dy​nx​dy​+ny​dx​​

Multiplication is simpler, just numerator times numerator over denominator times denominator:

dx​nx​​×dy​ny​​=dx​dy​nx​ny​​

The constructor then divides both parts by their greatest common divisor, which is why the printed answers are already in lowest terms. Tracing the first call by hand:

52​+101​=5⋅102⋅10+1⋅5​=5025​=21​

Notice that none of this needed type dispatching. Both operands are Rational, so the inherited Number.__add__ calls Rational.add directly. The shared interface alone carries us, exactly as it did for complex numbers.

The Cross-Type Problem

Now keep both kinds of number in the same program at once:

Complex numbers
Rational numbers

Mathematically, mixing them is perfectly ordinary. A rational number sits on the real axis of the complex plane, so this expression has a clear value:

(1.5+0i)+23​=3+0i

But the Number class from (1) cannot evaluate it. When Python runs complex_value + rational_value, it calls complex_value.add(rational_value), and Complex.add immediately reaches for other.real and other.imag. A Rational has no such attributes; it has numer and denom. The method call

complex_value.add(rational_value)

is simply not the same as the case it was written for:

complex_value.add(other_complex_value)

What we need is a generic operation that looks at the kinds of both arguments and chooses behavior to match. That is type dispatching.

Type Dispatching With isinstance

The most direct way to ask "what kind is this object?" is the built-in function isinstance. It takes an object and a class, and returns True when the object's class either is that class or inherits from it.

The complex-number classes below are exactly the design from the previous packet, reproduced so this example runs on its own.

Following the Line: An isinstance Check

├─ isinstance .... the built-in kind-checking function
├─ ( ............. begins the inputs
├─ c ............. the object to inspect (a ComplexRI)
├─ , ............. separates the object from the class
├─ Complex ....... the class to test membership against
├─ ) ............. ends the inputs

Read the three results in order. The object c was built by ComplexRI, so isinstance(c, ComplexRI) is True. It is also an instance of Complex, because ComplexRI inherits from Complex, so the second check is True as well. It is not an instance of ComplexMA, because rectangular and polar are sibling classes, both children of Complex but neither a parent of the other. That last fact is the one beginners trip on: sharing a parent does not make two classes instances of each other.

A Simple Type-Dispatch Function

Here is the smallest useful dispatcher: a function that answers "is this complex number actually real?" using a different rule for each representation.

# (4)
def is_real(c):
    """Return whether c is a real number with no imaginary part."""
    if isinstance(c, ComplexRI):
        return c.imag == 0
    elif isinstance(c, ComplexMA):
        return c.angle % pi == 0


print(is_real(ComplexRI(1, 1)))
print(is_real(ComplexMA(2, pi)))

Expected output:

False
True

The body of is_real is two paths guarded by isinstance. A rectangular number is real exactly when its imaginary part is zero. A polar number is real exactly when its angle is a whole-number multiple of π, meaning it points straight along the real axis in one direction or the other.

Type Real-number test
ComplexRI imaginary part is 0
ComplexMA angle is a multiple of π

Here angle % pi is the float remainder left after dividing the angle by π; it lands on 0 exactly when the angle is a whole-number multiple of π (0, π, 2π, ...), which points straight along the real axis.

That pattern, in one phrase, is type dispatching:

inspect the type, then select the matching code path

Type Tags for Arithmetic

Dispatching with isinstance works, but for arithmetic the original text uses a lighter tool: a type tag. A type tag is just a short string attached to a class that names the general kind of number an instance represents. Two values can then be compared by tag instead of by walking the class hierarchy.

The text attaches the tags after the classes already exist, by assigning a class attribute. We reproduce the Rational class here so the example is complete, then attach both tags.

# (5)
from math import gcd


class Rational(Number):
    def __init__(self, numer, denom):
        g = gcd(numer, denom)
        self.numer = numer // g
        self.denom = denom // g

    def __repr__(self):
        return "Rational({0}, {1})".format(self.numer, self.denom)

    def add(self, other):
        nx, dx = self.numer, self.denom
        ny, dy = other.numer, other.denom
        return Rational(nx * dy + ny * dx, dx * dy)

    def mul(self, other):
        numer = self.numer * other.numer
        denom = self.denom * other.denom
        return Rational(numer, denom)


Rational.type_tag = "rat"
Complex.type_tag = "com"

print(Rational(2, 5).type_tag == Rational(1, 2).type_tag)
print(ComplexRI(1, 1).type_tag == ComplexMA(2, pi / 2).type_tag)
print(Rational(2, 5).type_tag == ComplexRI(1, 1).type_tag)

Expected output:

True
True
False

Following the Line: Attaching a Tag

Rational.type_tag = "rat"
├─ Rational ........ the class object itself
├─ .type_tag ....... a new attribute named on the class
├─ = ............... assignment: bind the attribute to a value
├─ "rat" ........... the string label for "rational kind"
└─ action: every Rational instance now reports type_tag == "rat"

This is a statement, not an expression, so it produces no value; its job is the binding. Because type_tag lives on the class, every existing and future Rational instance sees it through normal attribute lookup, with no change to __init__.

The three comparisons confirm the design:

ComplexRI and ComplexMA both report "com"
Rational values report "rat"

So two rationals match, the two complex representations match each other, and a rational does not match a complex. The system can now treat both complex forms as one abstract kind while keeping rationals distinct, using nothing fancier than string equality.

Cross-Type Operations

Tags let us detect a mismatch, but detecting it is not the same as handling it. To actually add a complex number and a rational number, we write functions that reach into both representations at once. The trick is to view the rational number as a real number by computing r.numer / r.denom.

For addition, only the real part changes:

(a+bi)+dn​=(a+dn​)+bi

Multiplication uses polar form. In the complex plane, a real number always has a nonnegative magnitude; its sign is carried by the angle instead. A positive real has angle 0, and a negative real points the opposite way, angle π. That is the reason for the guard in (6):

if r_magnitude < 0:
    r_magnitude, r_angle = -r_magnitude, pi

If the rational is negative, we flip its magnitude positive and absorb the minus sign into the angle. Both operations are commutative — order doesn't change the answer, so a+b equals b+a — which means the rational_and_complex versions can just call their partners with the arguments swapped. That keeps one real implementation per operation instead of two.

A Dispatching Number Class

Now we rebuild Number so its operators check type tags before acting. The structure is the same for both operators, so read __add__ carefully and __mul__ falls out by analogy.

# (7)
class Number:
    def __add__(self, other):
        if self.type_tag == other.type_tag:
            return self.add(other)
        elif (self.type_tag, other.type_tag) in self.adders:
            return self.cross_apply(other, self.adders)

    def __mul__(self, other):
        if self.type_tag == other.type_tag:
            return self.mul(other)
        elif (self.type_tag, other.type_tag) in self.multipliers:
            return self.cross_apply(other, self.multipliers)

    def cross_apply(self, other, cross_fns):
        cross_fn = cross_fns[(self.type_tag, other.type_tag)]
        return cross_fn(self, other)

    adders = {
        ("com", "rat"): add_complex_and_rational,
        ("rat", "com"): add_rational_and_complex,
    }

    multipliers = {
        ("com", "rat"): mul_complex_and_rational,
        ("rat", "com"): mul_rational_and_complex,
    }

__add__ considers two cases. If the two tags match, it assumes the first operand's add method can take the second directly, the same-kind path. If they differ, it looks up the ordered pair of tags in the adders dictionary; when a function is registered there, cross_apply fetches it and calls it. __mul__ mirrors this with multipliers.

Following the Line: The Tag-Match Test

if self.type_tag == other.type_tag:
├─ if ............... begin a conditional
├─ self.type_tag ... the tag of the left operand, e.g. "com"
├─ == ............... compare the two tags for equality
├─ other.type_tag .. the tag of the right operand, e.g. "rat"
├─ : ................ ends the condition, begins its block
└─ action: when the tags are equal, take the same-kind add path

Following the Line: The Cross-Type Lookup

cross_fn = cross_fns[(self.type_tag, other.type_tag)]
├─ cross_fn ............ a local name for the function we are about to find
├─ = ................... bind that name to the lookup result
├─ cross_fns ........... the dictionary passed in (adders or multipliers)
├─ [ ................... begin a dictionary lookup
├─ (self.type_tag, other.type_tag)  the ordered pair of tags, the key
├─ ] ................... end the lookup
└─ action: bind cross_fn to the function registered for that tag pair

The adders dictionary maps a pair of tags to the function that handles that pair. The entry

("com", "rat") -> add_complex_and_rational

says: when the left operand is complex and the right is rational, use add_complex_and_rational. So when Python evaluates

ComplexRI(1.5, 0) + Rational(3, 2)

the tag pair is ("com", "rat"), the lookup succeeds, and that exact function runs. The approach is extensible: a new subclass of Number can join the system just by declaring a type tag and adding entries to Number.adders and Number.multipliers, or by defining its own adders and multipliers in the subclass.

Complete Dispatching Example

This block assembles every piece into one runnable program. The classes are defined first, then the four cross-type functions, then the dictionaries are installed onto Number, and finally two mixed-type expressions are printed.

Expected output:

ComplexRI(3, 0)
ComplexMA(2, 1.5 * pi)

This version works, but it carries a real cost: every new pair of types may demand brand-new cross-type functions, and the number of pairs grows quickly as you add kinds. That cost is what coercion is designed to cut.

Tracing the Dispatch: ComplexRI(1.5, 0) + Rational(3, 2)

Before moving on, step the first mixed expression in the block above one line at a time. Watch the "com" and "rat" tags differ, then the frame for Number.__add__ open onto a Number.adders lookup and a cross_apply call, and finally a single add_complex_and_rational frame open at the bottom of that chain to compute 1.5 + 3 / 2 and build the result ComplexRI. This is the mental model you want for every dispatched operation.

Stated as a one-line pipeline, dispatching is always:

operator -> special method -> tag check -> dictionary lookup -> selected function

Coercion

Coercion takes the second road from the cafeteria. Instead of writing one function per pair of types, it converts one type into another and then reuses the operation that already exists.

convert one type into another type, then use the ordinary operation

For our numbers, the natural conversion is one-directional: a rational number is exactly a complex number whose imaginary part is zero.

dn​↦dn​+0i
# (9)
def rational_to_complex(r):
    return ComplexRI(r.numer / r.denom, 0)

Instead of the four separate functions the dispatching version needed,

complex + rational
rational + complex
complex * rational
rational * complex

we now need just one conversion,

rational -> complex

After converting, both addition and multiplication can run on the existing Complex.add and Complex.mul. The conversion depends only on the types involved, not on which operation you are about to perform, which is exactly why one function can replace four.

A Coercing Number Class

This Number class implements the second design. Its operators coerce both operands to a common type first, then call the same-kind method.

# (10)
class Number:
    def __add__(self, other):
        x, y = self.coerce(other)
        return x.add(y)

    def __mul__(self, other):
        x, y = self.coerce(other)
        return x.mul(y)

    def coerce(self, other):
        if self.type_tag == other.type_tag:
            return self, other
        elif (self.type_tag, other.type_tag) in self.coercions:
            return (self.coerce_to(other.type_tag), other)
        elif (other.type_tag, self.type_tag) in self.coercions:
            return (self, other.coerce_to(self.type_tag))

    def coerce_to(self, other_tag):
        coercion_fn = self.coercions[(self.type_tag, other_tag)]
        return coercion_fn(self)

    coercions = {('rat', 'com'): rational_to_complex}

The coerce method returns two values that share a type tag. It checks three cases in order: if the tags already match, return the operands unchanged; if self can be coerced toward other, convert self; otherwise, if other can be coerced toward self, convert other. The actual conversion happens in coerce_to, which looks up the conversion function and applies it.

Following the Line: Coercing the Left Operand

return (self.coerce_to(other.type_tag), other)
├─ return ........... hand a pair back to the operator
├─ ( ................ begin a two-element tuple
├─ self.coerce_to(other.type_tag)  convert self into other's kind
├─ , ................ separates the two tuple elements
├─ other ............ the right operand, left unchanged
├─ ) ................ close the tuple
└─ evaluates to ..... two values that now share a type tag

The coercions dictionary holds exactly one entry:

('rat', 'com') -> rational_to_complex

In words: if one value is rational and the other is complex, turn the rational one into a complex one. There is deliberately no ('com', 'rat') entry, because an arbitrary complex number cannot be converted exactly to a rational; a nonzero imaginary part has nowhere to go. Just one entry replaces all four cross-type functions from the dispatching version, because the conversion is reused across every operation.

Dispatching vs Coercion

Technique Core idea Strength Cost
Type dispatching choose the operation by the pair of types very explicit, no information lost one function per pair per operation
Coercion convert to a common type, then operate once one function per pair of types, reused across operations conversion may lose information

Coercion shines when one type naturally becomes another and the conversion is exact. It also extends in two directions worth knowing about. A scheme can coerce two different types each into a third common type, the way a rhombus and a rectangle are neither a special case of the other yet both are quadrilaterals. And coercion can be iterative, routing through intermediate types: an integer can become a real number by first becoming rational and then becoming real, which cuts the total number of conversion functions a program must define.

The danger is information loss. In our case rational numbers are exact, but converting one to a complex number goes through floating-point division,

r.numer / r.denom

and that can round. For instance, 1/3 has no exact finite binary floating-point representation. As a historical note, early versions of Python even had a built-in __coerce__ special method on objects, but the complexity of an automatic, language-wide coercion system did not earn its keep, and it was removed. Modern Python instead lets each operator coerce its own arguments as needed, which is the spirit of what we built here.

⚠Common Beginner Traps

Trap 1: Treating type tags as built-in magic. A type tag is nothing but an attribute you chose to put on a class. Python does not know type_tag means anything special; your dispatch code is what gives it meaning. You could have named it kind or flavor and the system would work the same. Because nothing is automatic, a new Number subclass that never defines type_tag (or misspells it) raises AttributeError: object has no attribute 'type_tag' the moment the dispatcher reads it.

Trap 2: Forgetting both argument orders. Addition is commutative in math, but the dictionary looks up an ordered pair of tags. You must register both ("com", "rat") and ("rat", "com") (or coerce, which sidesteps the issue), or one of the two operand orders will silently fall through.

Trap 3: Assuming coercion is always safe. Converting can lose information. Turning an exact Rational into a ComplexRI uses floating-point division, so the exactness is gone the moment you coerce. Choose coercion only when the conversion preserves what you care about.

Trap 4: Returning None silently. In the simple Number classes here, if no tag matches and no entry is found, __add__ falls off the end and returns None. That produces a confusing error later, far from the real cause. Production code would raise a clear TypeError at the point of the mismatch.

Trap 5: Confusing sibling classes with parent-child classes. isinstance(ComplexRI(1, 1), ComplexMA) is False. Sharing the parent Complex does not make one a kind of the other; only inheritance up the chain counts.

  1. Both operands are Rational, so the inherited Number.__add__ sees matching tags (or, in (2), simply calls self.add(other)) and runs Rational.add directly, with no cross-type logic needed.
  2. It returns True, telling you c is a kind of Complex. That is because ComplexRI inherits from Complex, and isinstance follows the inheritance chain.
  3. Because they are two representations, rectangular and polar, of the same abstract kind: complex numbers. The shared tag "com" lets the system treat them as one type.
  4. The operands can appear in either order, and the dictionary key is an ordered pair of tags. Without both entries, one ordering would fail to find a function.

§Summary

A generic function lets one operation serve many kinds of data. This lesson added two new ways to build one, on top of the shared interfaces from the previous packet. Type dispatching inspects the kinds of the arguments, by isinstance or by a lightweight type tag, and selects the matching code path, often through a dictionary keyed on a pair of tags. Coercion instead converts one kind into another and then reuses a single shared operation, which needs only one conversion function per pair of types but risks losing information when the conversion is inexact. Dispatching is explicit and lossless but multiplies functions; coercion is compact and reusable but only as safe as its conversions. Both serve the same end that has run through the whole chapter: letting different objects cooperate while each kind stays behind its own abstraction barrier.

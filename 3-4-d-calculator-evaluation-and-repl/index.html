Chapter 3/3.4 Interpreters for Languages with Combination

3.4.4 Calculator Evaluation and the REPL

◷ approx. 19 min read · 10 worked examples · best read in two passes

The previous packets gave the interpreter a way to turn raw Calculator text into an expression tree made of Pair and nil objects. By the end of this lesson you will be able to do five things: explain what it means to evaluate an expression tree, describe how the evaluator splits its work between two cooperating functions named calc_eval and calc_apply, trace a nested expression by hand from the inside out, read the official Calculator evaluator line by line, and wrap the whole interpreter in a read-eval-print loop that survives a bad input instead of crashing.

Everything here builds on one habit you have been practicing since the beginning of this course: when something is nested, finish the smaller pieces first, then combine their results.

The Cook and the Ingredient Prep Team

Imagine a kitchen. A recipe card says:

make a sandwich using toasted bread and sliced tomato

The cook cannot assemble the sandwich until two smaller jobs are done first: the bread has to be toasted and the tomato has to be sliced. Only when both ingredients are ready can the cook perform the final step of putting the sandwich together. The smaller tasks must be completed before the larger task can finish.

Calculator evaluation works exactly the same way. For the expression:

(* (+ 1 2) (+ 2 3))

the outer multiplication is the cook's final assembly step. It cannot happen until both inner additions have produced their values:

(+ 1 2) -> 3
(+ 2 3) -> 5
(* 3 5) -> 15

Keep this picture. The whole evaluator is just this idea made precise: prepare the ingredients (evaluate the operands), then assemble (apply the operator).

Evaluation Gives Meaning to a Tree

By this point the interpreter has a structured expression tree. Parsing is finished. Nothing has been computed yet. The tree for (* (+ 1 2) (+ 2 3)) is only a description of structure: a combination whose operator is * and whose two operands are themselves combinations.

Evaluation is the process that answers the central question:

What value does this expression produce?

For Calculator there are only two legal kinds of expression, and the evaluator must handle each one:

  • A number is a primitive expression. There is nothing smaller inside it to work out, so it stands for itself.
  • A call expression is a Pair representing a well-formed Scheme list: an operator followed by zero or more operands. It must be worked out by evaluating its operands and then applying its operator.

Two Cooperating Jobs

calc_eval calls calc_apply and also calls itself on each operand; calc_apply does not call back, so the recursion all lives in calc_eval.

The official interpreter separates evaluation into two cooperating functions, each with a narrow responsibility:

  • calc_eval evaluates an expression. It understands expression structure.
  • calc_apply applies an operator to already-evaluated arguments. It understands arithmetic.

Keeping these jobs apart is what keeps the interpreter understandable. calc_eval asks one question:

Is this expression a number, or is it a combination?

calc_apply asks a different question:

Given an operator symbol and a list of argument values, what result should arithmetic produce?

Together they form a call-and-response pattern. When calc_eval meets a combination, it evaluates each operand, then hands the operator and the finished argument values to calc_apply, which returns the arithmetic result:

calc_eval sees a combination
calc_eval evaluates each operand
calc_eval calls calc_apply
calc_apply returns the arithmetic result
calc_eval returns that result

The two functions are mutually recursive in spirit: calc_eval calls calc_apply, and calc_eval also calls itself on each operand. That self-call is the entire reason the evaluator can handle nesting of any depth.

Evaluating Primitive Expressions

Numbers evaluate to themselves. There is no smaller expression to evaluate inside 3; it is already a value.

calc> 3
3
calc> 2.5
2.5

In interpreter language, expressions that require no further evaluation step are called self-evaluating. The only self-evaluating expressions in Calculator are numbers. A larger language might also treat strings or boolean values as self-evaluating, but Calculator does not.

Here is the structure of evaluating a number, walked one token at a time. The evaluator reads the single token, recognizes its type, and returns it unchanged:

calc_eval(3)
├─ calc_eval ... the evaluator: decides what to do based on the expression's type
├─ 3 .......... a number token, which is self-evaluating
└─ evaluates to  3

Evaluating Call Expressions

A call expression is more work. The evaluator must first evaluate every operand, then apply the operator to those values.

Start with the simplest call, one where the operands are already numbers:

calc> (+ 2 3)
5

Here is that call expression walked one token at a time. Read each branch row as "what this token contributes," and the final row as the value the whole line produces:

(+ 2 3)
├─ + .......... the operator symbol, passed to calc_apply
├─ 2 .......... first operand, evaluated by calc_eval to 2
├─ 3 .......... second operand, evaluated by calc_eval to 3
└─ evaluates to  5

The interpreter does this in two stages. First it evaluates the operands, which here are already numbers and so evaluate to themselves. Then it applies + to the resulting values:

expression: (+ 2 3)
operator: +
operands before evaluation: 2, 3
operands after evaluation: 2, 3
apply + to 2 and 3
result: 5

Now nest one expression inside another:

calc> (* (+ 1 2) (+ 2 3))
15

The operands are no longer numbers; they are themselves combinations. Walk the outer line one token at a time. Notice that the two operand tokens are whole subexpressions, and each branch row records the value that subexpression evaluates to:

(* (+ 1 2) (+ 2 3))
├─ * .......... the operator symbol, passed to calc_apply
├─ (+ 1 2) .... first operand, evaluated by calc_eval to 3
├─ (+ 2 3) .... second operand, evaluated by calc_eval to 5
└─ evaluates to  15

So the work reduces to three steps, two preparations and one assembly:

evaluate (+ 1 2) -> 3
evaluate (+ 2 3) -> 5
apply * to 3 and 5 -> 15

The Official Evaluator

The scalc module implements an evaluator for the Calculator language. The calc_eval function takes an expression as an argument and returns its value. For Calculator, the only two legal syntactic forms are numbers and call expressions, which are Pair instances representing well-formed Scheme lists. Numbers are self-evaluating, so they can be returned directly from calc_eval. Call expressions require function application.

Here is the official calc_eval, reproduced exactly:

# (1)
def calc_eval(exp):
    """Evaluate a Calculator expression."""
    if type(exp) in (int, float):
        return simplify(exp)
    elif isinstance(exp, Pair):
        arguments = exp.second.map(calc_eval)
        return simplify(calc_apply(exp.first, arguments))
    else:
        raise TypeError(exp + ' is not a number or call expression')

Read it slowly, one branch at a time.

  • If exp is an int or a float, it is a number. Return it (after simplify tidies the display).
  • If exp is a Pair, it is a combination. Then exp.first is the operator and exp.second is the operand list.
  • The line exp.second.map(calc_eval) is the heart of the function. It maps calc_eval across every operand, producing a Scheme list of finished argument values.
  • calc_apply(exp.first, arguments) then performs the arithmetic, and simplify cleans up the numeric result for display.
  • Anything that is neither a number nor a Pair is not a legal expression, so the evaluator raises a TypeError.

Call expressions are evaluated by first recursively mapping calc_eval across the list of operands, which computes a list of arguments, and then applying the operator to those arguments in a second function, calc_apply.

The single most important line is:

# (2)
arguments = exp.second.map(calc_eval)

That line says: before applying the operator, evaluate every operand expression. Because calc_eval is what gets mapped, each operand is evaluated by the same rules, which is precisely what lets the evaluator descend into nesting of any depth.

Here is that key line walked one token at a time, so the recursion is explicit. The final row gives the value bound to arguments:

arguments = exp.second.map(calc_eval)
├─ exp.second ........ the operand list (everything after the operator)
├─ .map(calc_eval) ... evaluate each operand with calc_eval
├─ arguments = ....... bind the resulting list of values to the name arguments
└─ result ............ a Scheme list of finished argument values

Applying Operators

The Calculator language is simple enough that the logic of applying each operator fits in the body of a single function. In calc_apply, each conditional clause corresponds to applying one operator. Here is the official calc_apply, reproduced exactly, including the argument-count checks:

# (3)
def calc_apply(operator, args):
    """Apply the named operator to a list of args."""
    if not isinstance(operator, str):
        raise TypeError(str(operator) + ' is not a symbol')
    if operator == '+':
        return reduce(add, args, 0)
    elif operator == '-':
        if len(args) == 0:
            raise TypeError(operator + ' requires at least 1 argument')
        elif len(args) == 1:
            return -args.first
        else:
            return reduce(sub, args.second, args.first)
    elif operator == '*':
        return reduce(mul, args, 1)
    elif operator == '/':
        if len(args) == 0:
            raise TypeError(operator + ' requires at least 1 argument')
        elif len(args) == 1:
            return 1/args.first
        else:
            return reduce(truediv, args.second, args.first)
    else:
        raise TypeError(operator + ' is an unknown operator')

Each suite computes the result of a different operator, or raises an appropriate TypeError when the wrong number of arguments is given. The operator rules encoded here are:

  • + adds all its arguments, starting from the identity value 0.
  • * multiplies all its arguments, starting from the identity value 1.
  • - negates a single argument, or subtracts every later argument from the first.
  • / reciprocates a single argument, or divides every later argument into the first.

calc_apply can be called directly, but it must be passed a list of values as arguments rather than a list of operand expressions. The helper as_scheme_list builds the Pair list the interpreter expects:

>>> calc_apply('+', as_scheme_list(1, 2, 3))
6
>>> calc_apply('-', as_scheme_list(10, 1, 2, 3))
4
>>> calc_apply('*', nil)
1
>>> calc_apply('*', as_scheme_list(1, 2, 3, 4, 5))
120
>>> calc_apply('/', as_scheme_list(40, 5))
8.0

Notice the third line: calc_apply('*', nil) applies * to the empty list and returns the identity value 1. The last line divides 40 by 5 and produces the float 8.0.

The Helper Functions

The page says the definitions of simplify, reduce, and as_scheme_list appear in the model. Here they are, reproduced exactly so the evaluator above is complete.

simplify is what makes (/ 40 5) print as 8 rather than 8.0 in the REPL. It returns an int when a float has no fractional part, and otherwise returns its argument unchanged:

# (4)
def simplify(value):
    """Return an int if value is an integer, or value otherwise.

    >>> simplify(8.0)
    8
    >>> simplify(2.3)
    2.3
    >>> simplify('+')
    '+'
    """
    if isinstance(value, float) and int(value) == value:
        return int(value)
    return value

reduce walks a Scheme list of Pair objects, combining the elements with a two-argument function and a starting value. It is the engine behind +, -, *, and /:

# (5)
def reduce(fn, scheme_list, start):
    """Reduce a recursive list of Pairs using fn and a start value.

    >>> reduce(add, as_scheme_list(1, 2, 3), 0)
    6
    """
    if scheme_list is nil:
        return start
    return reduce(fn, scheme_list.second, fn(start, scheme_list.first))

as_scheme_list builds the nested Pair structure the interpreter uses, given ordinary Python values:

# (6)
def as_scheme_list(*args):
    """Return a recursive list of Pairs that contains the elements of args.

    >>> as_scheme_list(1, 2, 3)
    Pair(1, Pair(2, Pair(3, nil)))
    """
    if len(args) == 0:
        return nil
    return Pair(args[0], as_scheme_list(*args[1:]))

Evaluating a Nested Tree Directly

The role of calc_eval is to make proper calls to calc_apply by first computing the value of each operand sub-expression before passing it as an argument. Because of that, calc_eval can accept a nested expression and work the whole thing out. The official text builds the expression tree for (+ (* 3 4) 5) and evaluates it:

>>> print(exp)
(+ (* 3 4) 5)
>>> calc_eval(exp)
17

This transcript is the full interpreter pipeline running after parsing. The expression tree already exists as nested Pair objects, so calc_eval can walk the tree: it first evaluates the operand (* 3 4) to 12, then applies + to 12 and 5 to produce 17.

The structure of calc_eval is an example of dispatching on type: the function chooses what to do based on the form of the expression. A number requires no further evaluation step, while a Pair triggers the recursive evaluate-then-apply process.

The official evaluator depends on the Pair, nil, and reader machinery from the previous packets. To see the evaluate-then-apply cycle run on its own, here is a small self-contained Python version that mirrors the official design while staying short enough to trace. It defines a minimal Pair, a minimal nil, and its own calc_eval and calc_apply:

Expected output:

This version uses the same load-bearing line as the official evaluator, arguments = exp.second.map(calc_eval), so the recursion you trace here is the recursion that runs in the real interpreter.

Why calc_apply Receives Values, Not Trees

A common beginner shortcut is to imagine that calc_apply can look inside any expression and figure it out for itself. In this design, it cannot, and that limitation is on purpose.

calc_apply is deliberately narrow. It expects:

operator symbol + already evaluated argument values

For example, in:

(* (+ 1 2) (+ 2 3))

calc_apply should never receive the two inner expression trees (+ 1 2) and (+ 2 3). Instead, calc_eval must first turn those trees into values:

(+ 1 2) -> 3
(+ 2 3) -> 5

Only then is calc_apply asked the question it knows how to answer:

apply * to the values 3 and 5

This division of labor is what keeps the interpreter understandable. One function handles structure; the other handles arithmetic, and neither has to do the other's job.

The Evaluation Trace for a Nested Expression

Now trace the full machine model for:

(* (+ 1 2) (+ 2 3))

Read this indented trace the way the machine runs it: each deeper indentation is a recursive call to calc_eval, and a call cannot return until its inner calls have returned.

calc_eval((* (+ 1 2) (+ 2 3)))
  operator is *
  operands are (+ 1 2) and (+ 2 3)

  calc_eval((+ 1 2))
    operator is +
    operands are 1 and 2
    calc_eval(1) -> 1
    calc_eval(2) -> 2
    calc_apply(+, (1 2)) -> 3

  calc_eval((+ 2 3))
    operator is +
    operands are 2 and 3
    calc_eval(2) -> 2
    calc_eval(3) -> 3
    calc_apply(+, (2 3)) -> 5

  calc_apply(*, (3 5)) -> 15

This is the same mental model as any recursive function: solve the smaller subexpressions first, then combine their results. The outer calc_apply(*, ...) is the cook's final assembly step, and it runs last.

The Read-Eval-Print Loop

An interpreter is rarely run on a single expression. It normally runs in a loop, reading one expression after another. A typical way to interact with an interpreter is through a read-eval-print loop, or REPL: a mode of interaction that reads an expression, evaluates it, and prints the result for the user. The Python interactive session you have used all course is itself an example of such a loop.

The name spells out four stages:

Read
Eval
Print
Loop

Each stage names one step, and you have already built three of the four in earlier packets:

  • Read: parse the user's text into an expression tree, using scheme_read.
  • Eval: evaluate the expression tree, using calc_eval.
  • Print: display the result, using print.
  • Loop: ask for the next input, using while True.

An implementation of a REPL can be largely independent of the interpreter it uses. The official read_eval_print_loop buffers input from the user, constructs an expression using the language-specific scheme_read function, then prints the result of applying calc_eval to that expression:

# (8)
def read_eval_print_loop():
    """Run a read-eval-print loop for calculator."""
    while True:
        src = buffer_input()
        while src.more_on_line:
            expression = scheme_read(src)
            print(calc_eval(expression))

That function is short precisely because the previous packets already built the pieces it calls. The outer while True is the Loop. The inner while src.more_on_line lets the reader pull several complete expressions out of one input line. scheme_read does the Read, calc_eval does the Eval, and print does the Print.

This version contains all the essential components of an interactive interface. An example session looks like this:

> (* 1 2 3)
6
> (+)
0
> (+ 2 (/ 4 8))
2.5
> (+ 2 2) (* 3 3)
4
9
> (+ 1
     (- 23)
     (* 4 2.5))
-12

These cases show several interpreter behaviors at once:

  • (* 1 2 3) evaluates a multi-operand expression and returns 6.
  • (+) uses the identity value for addition, 0.
  • (+ 2 (/ 4 8)) evaluates the inner division first, giving 0.5, then adds to get 2.5.
  • (+ 2 2) (* 3 3) shows that the inner while src.more_on_line reads two complete expressions from one line and prints 4 then 9.
  • The final multi-line example shows why the reader buffers input: the interpreter waits until it has read enough tokens to complete the expression, even across line breaks, before evaluating it to -12.

Errors in the REPL

The loop above has no mechanism for termination or error handling. A good interactive interpreter should not disappear every time the user makes one mistake. It should report the problem and then continue listening for the next input.

We can improve the interface by reporting errors and by letting the user exit deliberately. To do this, the original suite of the while statement is placed inside a try statement. The first except clause handles the SyntaxError and ValueError exceptions raised by scheme_read, as well as the TypeError and ZeroDivisionError exceptions raised by calc_eval. The second except clause exits the loop when the user signals a keyboard interrupt or an end-of-file:

# (9)
def read_eval_print_loop():
    """Run a read-eval-print loop for calculator."""
    while True:
        try:
            src = buffer_input()
            while src.more_on_line:
                expression = scheme_read(src)
                print(calc_eval(expression))
        except (SyntaxError, TypeError, ValueError, ZeroDivisionError) as err:
            print(type(err).__name__ + ':', err)
        except (KeyboardInterrupt, EOFError):  # <Control>-D, etc.
            print('Calculation completed.')
            return

This loop reports errors without exiting. Rather than ending the program on an error, restarting the loop after an error message lets users revise their expressions. Upon importing the readline module, users can even recall their previous inputs using the up arrow or Control-P.

That last detail is a small but real quality-of-life feature, and it costs almost nothing. readline is a standard library module that hooks into the line-editing the REPL does while it reads input. Once it is imported, the loop's buffer_input step gains an input history: each line you submit is remembered, so pressing the up arrow (or Control-P) walks backward through what you already typed instead of forcing you to retype a long expression. Nothing in read_eval_print_loop has to change for this to work; importing the module is enough, because the recall happens during the Read step before scheme_read ever sees the text. So after a mistake you can press up, edit the recalled line, and resubmit:

> (+ 1 2
SyntaxError: unexpected end of file
> (+ 1 2)        # recalled with the up arrow, then closed
3

The result is an informative error-reporting interface:

> )
SyntaxError: unexpected token: )
> 2.3.4
ValueError: invalid numeral: 2.3.4
> +
TypeError: + is not a number or call expression
> (/ 5)
TypeError: / requires exactly 2 arguments
> (/ 1 0)
ZeroDivisionError: division by zero

Look at where each error comes from. The machine path is:

read user input
if parsing fails, report a syntax or numeral error
otherwise evaluate the expression
if evaluation fails, report the type or zero-division error
return to the prompt

The first two errors, SyntaxError and ValueError, are raised during the Read step by scheme_read: a stray ) has no matching open parenthesis, and 2.3.4 is not a valid numeral. The last three are raised during the Eval step by calc_eval and calc_apply: a bare + is neither a number nor a call expression, (/ 5) is reported as needing exactly 2 arguments, and (/ 1 0) divides by zero. One detail worth flagging so you don't think you misread the code: this session comes from a stricter variant of calc_apply that requires / to receive exactly two arguments, which is why (/ 5) is rejected here. The calc_apply shown earlier in this lesson is more permissive and treats a single argument to / as a reciprocal, so the same input would not raise there. This connects directly to the work on exceptions in Section 3.3: exceptions let the interpreter recover from one bad input and continue the interactive session.

As we generalize this interpreter to richer languages later, the read_eval_print_loop will be parameterized by a parsing function, an evaluation function, and the exception types the try statement handles. Beyond those substitutions, every REPL can be built from this same structure.

▶Step-Through Checkpoint

The block below is a stripped-down evaluator that uses ordinary Python lists in place of Pair, so the evaluate-then-apply cycle runs with no extra machinery. The outer frame stays open on the stack while the inner evaluate frames for ["+", 1, 2] and ["+", 2, 3] run to completion. ⚠Common Beginner Traps

Trap 1: Applying the operator before evaluating the operands. For nested expressions the operands are themselves full expressions, so the interpreter must evaluate them into values first.

Trap 2: Thinking calc_eval and calc_apply do the same job. calc_eval handles expression structure and decides what kind of expression it is looking at. calc_apply handles the arithmetic rules for one already-identified operator.

Trap 3: Forgetting that numbers evaluate to themselves. The number 3 does not need to be parsed into anything smaller. It is self-evaluating and is returned directly.

Trap 4: Passing operand trees to calc_apply. calc_apply expects finished values such as 3 and 5, never the trees (+ 1 2) and (+ 2 3). If a Pair reaches the arithmetic, the operator tries to combine it with a number and Python raises TypeError: unsupported operand type(s). Turning trees into values is calc_eval's job.

Trap 5: Treating REPL errors as fatal. An interactive interpreter should report an error and keep listening for the next input, which is exactly what the try statement in the loop provides.

Trap 6: Losing track of where exceptions belong. Parsing errors such as a stray ) or the numeral 2.3.4 happen during the Read step. Arithmetic errors such as dividing by zero happen during the Eval step. A single REPL can catch both and report each clearly.

  2. Operands can be nested expressions, so each operand must be evaluated into a value before the current operator can be applied. The self-call is what makes nesting of any depth work.

§Summary

Evaluation gives meaning to an expression tree. In the Calculator interpreter, calc_eval recursively evaluates expressions by dispatching on their type, and calc_apply applies an operator to already-evaluated arguments. Numbers are self-evaluating, while call expressions are handled by evaluating their operands first and then applying the operator, so nested combinations work because the evaluator finishes the smaller subexpressions before the outer operator runs. The helper functions simplify, reduce, and as_scheme_list round out the evaluator. A read-eval-print loop connects reading, evaluation, printing, and looping into an interactive session, and wrapping the loop in a try statement lets the interpreter report a bad input and keep going instead of crashing.

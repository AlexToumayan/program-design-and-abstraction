2.5 · Object-Oriented Programming — Coding Practice

◷ approx. 13 min read · 30 worked examples

These problems build fluency with classes: writing an __init__ constructor, storing and mutating instance attributes, defining methods that share behavior through self, distinguishing instance attributes from shared class attributes, and building inheritance hierarchies that override methods, delegate to a base class, and resolve names across multiple parents.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Tally Counter

Examples

Input:  c = Counter(); c.click(); c.click(); c.click()
Output: 3
Explanation: three clicks raise count from 0 to 3; the last click returns 3.

Input:  c = Counter(); c.click(); c.reset(); c.count
Output: 0
Explanation: reset puts the count back to 0.

Input:  c = Counter(); d = Counter(); c.click(); d.count
Output: 0
Explanation: each counter has its own count, so clicking c does not touch d.

Constraints

  • Define a class with __init__, click, and reset.
  • Store the running total in the instance attribute self.count; two counters must be independent.

5. Stack

Output: True
Explanation: a brand-new stack holds nothing.

Constraints

  • Wrap a list in the instance attribute self.items.
  • Use list methods (append, pop, indexing, len) inside the class.

Hint

push is self.items.append(value); pop is self.items.pop(); peek is self.items[-1].

10. Combining Two Parents

Build a class that inherits from two parents at once. Flyer has a class attribute medium = 'air' and a method move() returning 'soaring through the ' + self.medium. Swimmer has a class attribute medium = 'water' and a method dive() returning 'plunging into the ' + self.medium. Define Duck(Flyer, Swimmer) with an empty body. The duck inherits move from Flyer and dive from Swimmer. Both parents define medium, so Python must break the tie: because Flyer is listed first, Duck.medium is 'air', and that is what dive reads too.

Examples

Input:  Duck().move()
Output: 'soaring through the air'
Explanation: move is inherited from Flyer, and Flyer's medium wins the tie.

Input:  Duck().dive()
Output: 'plunging into the air'
Explanation: dive comes from Swimmer, but self.medium resolves to 'air' because Flyer is listed first.

Input:  [k.__name__ for k in Duck.mro()]
Output: ['Duck', 'Flyer', 'Swimmer', 'object']
Explanation: the method resolution order lists Duck, then its parents left to right, then object.

Constraints

  • Duck inherits from both Flyer and Swimmer, with Flyer listed first.
  • Do not define any methods or attributes in Duck's body; it is pass.
  • Both parents define medium; the first-listed parent must win the tie.

Starter

class Flyer:
    pass

class Swimmer:
    pass

class Duck(Flyer, Swimmer):
    pass
Hint

Listing Flyer before Swimmer puts it first in the method resolution order, so its medium is found before Swimmer's. Inspect the order with Duck.mro().

Solutions

1. Tally Counter

Approach — instance state that a method mutates. A counter is a tiny object that remembers a number across calls. __init__ plants self.count = 0; click reads, increments, and writes back self.count, then returns it; reset zeroes it. Because count is an instance attribute, every Counter carries its own, so clicking one never disturbs another.

class Counter:
    def __init__(self):
        self.count = 0

    def click(self):
        self.count = self.count + 1
        return self.count

    def reset(self):
        self.count = 0

Why it works. The object persists between calls: click changes the same self.count each time, so repeated identical calls produce a rising sequence. Two counters are distinct objects with distinct count attributes.

Complexity. Time O(1), Space O(1) per call — a single attribute read and write.

c = Counter()
assert c.count == 0
assert c.click() == 1
assert c.click() == 2
assert c.click() == 3
c.reset()
assert c.count == 0
assert c.click() == 1
# A second counter is independent.
d = Counter()
assert d.count == 0
c.click()
assert d.count == 0

2. Thermostat

Approach — methods that reassign one attribute. The thermostat stores a single instance attribute, target. warmer and cooler each reassign self.target and return the updated value, exactly the shape of a deposit/withdraw pair. Distinct thermostats hold independent targets.

class Thermostat:
    def __init__(self, target):
        self.target = target

    def warmer(self):
        self.target = self.target + 1
        return self.target

    def cooler(self):
        self.target = self.target - 1
        return self.target

Why it works. Each call mutates the object's own target, and the object remembers that change for the next call. Because the attribute belongs to the instance, warming t cannot affect u.

Complexity. Time O(1), Space O(1) per call.

t = Thermostat(68)
assert t.target == 68
assert t.warmer() == 69
assert t.warmer() == 70
assert t.cooler() == 69
assert t.target == 69
# Two thermostats keep separate targets.
u = Thermostat(60)
t.warmer()
assert u.target == 60

3. Rectangle

Approach — store data, compute on demand. The constructor records width and height as instance attributes; the three methods derive their answers from those two values whenever called. Nothing is precomputed, so the methods always reflect the current dimensions through self.

class Rectangle:
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height

    def perimeter(self):
        return 2 * (self.width + self.height)

    def is_square(self):
        return self.width == self.height

Why it works. Each method reads self.width and self.height, so it computes the right answer for whichever rectangle it was called on. is_square returns the boolean result of the equality comparison directly.

Complexity. Time O(1), Space O(1) per method — a fixed amount of arithmetic.

r = Rectangle(3, 4)
assert r.area() == 12
assert r.perimeter() == 14
assert r.is_square() is False
s = Rectangle(5, 5)
assert s.area() == 25
assert s.is_square() is True

4. Game Character

Approach — a class attribute for the shared default. The starting health is the same for everyone, so it belongs on the class, not on each instance: start_health = 100 sits in the class body. __init__ copies that shared value into a per-instance self.health, which then changes independently as the character takes damage. take_damage mutates self.health; is_alive compares it to 0.

class Character:
    start_health = 100

    def __init__(self, name):
        self.name = name
        self.health = Character.start_health

    def take_damage(self, amount):
        self.health = self.health - amount
        return self.health

    def is_alive(self):
        return self.health > 0

Why it works. start_health lives once on the class and is read through Character.start_health when each character is born, so every new character starts at 100 without storing a separate copy of the default. After construction, self.health is an ordinary instance attribute that damage mutates privately.

Complexity. Time O(1), Space O(1) per call.

hero = Character('Mara')
assert hero.name == 'Mara'
assert hero.health == 100
assert hero.start_health == 100
assert hero.take_damage(30) == 70
assert hero.take_damage(80) == -10
assert hero.is_alive() is False
foe = Character('Drax')
assert foe.health == 100
assert foe.is_alive() is True

5. Stack

Approach — an object that wraps a mutable list. The stack hides a list behind a small interface. push appends to the end (the top); pop removes from the end; peek reads the last item with index -1; size and is_empty report on len. Encapsulating the list means callers use the stack interface — push, pop, peek — without touching items directly.

class Stack:
    def __init__(self):
        self.items = []

    def push(self, value):
        self.items.append(value)

    def pop(self):
        return self.items.pop()

    def peek(self):
        return self.items[-1]

    def size(self):
        return len(self.items)

    def is_empty(self):
        return len(self.items) == 0

Why it works. Appending and popping at the same end of the list gives last-in, first-out order: the most recently pushed value is at index -1, so it is what peek shows and what pop removes first.

Complexity. Time O(1) per push/pop/peek/size/is_empty (amortized for list append), Space O(n) for the stored items.

st = Stack()
assert st.is_empty() is True
st.push(1)
st.push(2)
st.push(3)
assert st.size() == 3
assert st.peek() == 3
assert st.pop() == 3
assert st.pop() == 2
assert st.size() == 1
assert st.is_empty() is False
assert st.pop() == 1
assert st.is_empty() is True

6. Auto-Assigned ID Badges

Approach — a mutable class attribute as a shared counter. The "next free ID" is not a property of any one employee; it belongs to the whole company, so it is a class attribute. Each constructor reads Employee.next_id, stamps it onto self.id, then advances Employee.next_id. Assigning to the class (not to self) updates the single shared counter that the next instance will read.

class Employee:
    next_id = 1

    def __init__(self, name):
        self.name = name
        self.id = Employee.next_id
        Employee.next_id = Employee.next_id + 1

Why it works. Because next_id lives on the class, every construction reads and then bumps the same value, so IDs come out 1, 2, 3, ... in creation order. Writing Employee.next_id = ... targets the class attribute; writing it on self would instead create a per-instance attribute and break the sharing.

Complexity. Time O(1), Space O(1) per construction.

a = Employee('Ada')
b = Employee('Linus')
c2 = Employee('Grace')
assert a.id == 1
assert b.id == 2
assert c2.id == 3
assert Employee.next_id == 4
assert a.name == 'Ada'
assert c2.name == 'Grace'

7. Shapes That Report Their Sides

Approach — inherit the method, override the class attribute. describe is written once on Shape and reads self.sides. Each subclass changes nothing but the sides class attribute, so the inherited describe automatically reports the right number through attribute lookup: it finds sides on the subclass before reaching Shape.

class Shape:
    sides = 0

    def __init__(self, name):
        self.name = name

    def describe(self):
        return self.name + ' has ' + str(self.sides) + ' sides'

class Triangle(Shape):
    sides = 3

class Square(Shape):
    sides = 4

Why it works. When describe evaluates self.sides on a Triangle, lookup checks the instance (no sides there), then the Triangle class, where it finds 3; on a Square it finds 4; on a plain Shape it falls through to 0. One inherited method serves all three because it reads the value through self.

Complexity. Time O(1), Space O(1) per describe (ignoring string size).

sh = Shape('blob')
tr = Triangle('warning sign')
sq = Square('window')
assert sh.describe() == 'blob has 0 sides'
assert tr.describe() == 'warning sign has 3 sides'
assert sq.describe() == 'window has 4 sides'
assert tr.sides == 3
assert sq.sides == 4
# describe was inherited unchanged; only the class attribute differs.
assert isinstance(tr, Shape) is True

8. Gift Card With a Fee

Approach — override and delegate to the base. FeeWallet keeps every part of Wallet except spend. Its spend adds the fee to the requested amount and hands the larger charge to Wallet.spend(self, amount + self.fee), reusing the base class's overdraft check and balance update. This is the CheckingAccount.withdraw pattern: write only the difference, then call the parent through the class, passing self explicitly.

class Wallet:
    def __init__(self):
        self.balance = 0

    def add(self, amount):
        self.balance = self.balance + amount
        return self.balance

    def spend(self, amount):
        if amount > self.balance:
            return 'Declined'
        self.balance = self.balance - amount
        return self.balance

class FeeWallet(Wallet):
    fee = 2

    def spend(self, amount):
        return Wallet.spend(self, amount + self.fee)

Why it works. Wallet.spend reached through the class is a plain function, so FeeWallet.spend passes self by hand. The base method still does the comparison and the subtraction, but now against amount + self.fee, so the fee is removed too — and the declined case fires when the charged total exceeds the balance. Reading the fee through self.fee lets a future subclass change it.

Complexity. Time O(1), Space O(1) per call — one delegated method call.

w = Wallet()
w.add(50)
assert w.spend(20) == 30
assert w.spend(100) == 'Declined'
assert w.balance == 30

fw = FeeWallet()
fw.add(50)
assert fw.spend(20) == 28
assert fw.spend(100) == 'Declined'
assert fw.balance == 28
# add is inherited unchanged.
assert fw.add(10) == 38

9. Loyalty Card Hierarchy

Approach — three-level inheritance with delegation up one rung. price is defined once on Member and reads self.discount, so Silver only has to override the class attribute to get a 10% discount. Gold overrides both the discount and price: it calls Silver.price(self, amount) to apply the percentage discount, then subtracts a flat 5. The subtle point is that Silver.price still reads self.discount, and self is a Gold, so the 0.2 discount is the one applied — not Silver's 0.1.

class Member:
    discount = 0

    def __init__(self, name):
        self.name = name

    def price(self, amount):
        return amount - amount * self.discount

class Silver(Member):
    discount = 0.1

class Gold(Silver):
    discount = 0.2

    def price(self, amount):
        base = Silver.price(self, amount)
        return base - 5

Why it works. Method lookup and self binding are separate: Gold.price reaches into Silver.price for its body, but self stays the Gold object the whole time, so self.discount resolves to 0.2. Silver.price therefore returns 100 - 100 * 0.2 = 80.0, and Gold.price subtracts 5 to give 75.0. A plain Member has discount 0, leaving the price unchanged.

Complexity. Time O(1), Space O(1) per call — a fixed number of delegated calls (depth at most three).

m = Member('Pat')
s2 = Silver('Quinn')
g = Gold('Rory')
assert m.price(100) == 100
assert s2.price(100) == 90.0
assert g.price(100) == 75.0
assert g.discount == 0.2
# Gold reuses Silver.price (which applies the 0.2 discount) then subtracts 5.
assert g.price(50) == 35.0
assert s2.discount == 0.1

10. Combining Two Parents

Approach — multiple inheritance with a left-to-right tie-break. Duck lists two parents and writes nothing of its own, so it inherits move from Flyer and dive from Swimmer. Both parents define medium, which forces Python to consult the method resolution order: Duck, then Flyer, then Swimmer, then object. Because Flyer comes first, its medium = 'air' wins, and every method that reads self.medium sees 'air', including dive, which lives on Swimmer.

class Flyer:
    medium = 'air'

    def move(self):
        return 'soaring through the ' + self.medium

class Swimmer:
    medium = 'water'

    def dive(self):
        return 'plunging into the ' + self.medium

class Duck(Flyer, Swimmer):
    pass

Why it works. The MRO is a single linear order Python searches for any attribute. For medium the search stops at Flyer (first listed) before reaching Swimmer, so the tie resolves to 'air'. The object never changes class: dive runs with self a Duck, so its self.medium resolves through the same MRO to 'air' even though the method body lives on Swimmer. Names defined by only one parent — move, dive — have no tie to resolve.

Complexity. Time O(1), Space O(1) per call; building the MRO is linear in the number of classes.

d2 = Duck()
# move comes from Flyer, dive comes from Swimmer.
assert d2.move() == 'soaring through the air'
assert d2.dive() == 'plunging into the air'
# Both parents define medium; Flyer is listed first, so it wins the tie.
assert d2.medium == 'air'
assert Duck.medium == 'air'
assert [k.__name__ for k in Duck.mro()] == ['Duck', 'Flyer', 'Swimmer', 'object']

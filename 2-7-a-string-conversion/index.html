Chapter 2/2.7 Object Abstraction

2.7 String Conversion

◷ approx. 15 min read · 4 worked examples · best read in two passes

By now you have built objects that hold state and respond to methods. This lesson is about a quieter ability that every well-behaved object should have: it should be able to turn itself into a string.

By the end you will be able to do four things. Explain why Python wants two different string forms for the same value, not one. Use the str and repr functions and predict which one you get in each situation. Read the __repr__ and __str__ methods that sit behind those functions, and write them on your own small class. And trace, step by step, how a bare call expression (a function call) like repr(x) ends up running a method defined on x.

That last idea is the real payoff. The function repr works on integers, floats, strings, lists, dates, and classes you have not written yet. It manages that not by knowing about every type in advance, but by asking each object to represent itself. Watching how that works is your first close look at a polymorphic function.

Two Name Tags for the Same Object

Imagine a single artifact in a museum, and notice that it travels with two different labels.

On the public display card, the artifact reads:

Bronze coin, 4th century BCE

That label is written for a visitor. It is short, it is in plain language, and it tells a human what they are looking at. Nobody could rebuild the museum's records from it, and nobody needs to.

Inside the museum's catalog database, the very same coin is recorded as:

Artifact(id=38291, material='bronze', century=-4)

That label is written for the staff and the software. It is less conversational, but it is exact: a curator could read it and recreate the database entry character for character. It looks less like a sentence and more like an instruction for rebuilding the object.

Same coin, two audiences, two labels. Python has exactly this split, and it gives each label its own function.

Function Written for Goal
str(value) humans reading output readable text
repr(value) programmers and Python tools precise, expression-like text

Sometimes the two labels are identical, and Python shows you the same string for both. The interesting cases are when they differ, because that is when the human view and the programmer view have genuinely different jobs to do.

The Formal Rule Behind repr

Python ships with a one-line description of what repr is supposed to do. It is the docstring attached to the function, and it states the goal precisely:

repr(object) -> string
Return the canonical string representation of the object.
For most object types, eval(repr(object)) == object.

The last line is the heart of it. Read it slowly as a round trip:

eval(repr(x))=x

First repr(x) produces a string that looks like a Python expression. Then eval takes that string and runs it as code. (eval is a built-in function that takes a string and runs it as if you had typed it as Python code — the same evaluation step from the read-eval-print loop, now available as a function. You do not need to reach for eval yourself; it appears here only to state precisely what repr promises.) For most types, the value you get back is equal to the object you started with. The repr string is, in other words, a recipe for rebuilding the object.

Two words in that rule are worth holding onto. Canonical means the one standard, official representation, not a casual one. And for most object types is a deliberate hedge: it is a goal, not a guarantee. Some objects cannot be described by a short expression that rebuilds them. For those, Python falls back to a plain description wrapped in angle brackets, like <built-in function min>, which you are meant to read as "here is a description, not code that eval will rebuild."

The str function carries no such obligation. It is free to favor readability over reconstruction, which is exactly why the two functions can disagree.

The Numeric and Function Transcripts

The official text opens with two short interactive sessions. Here is the first, exactly as Python responds at the prompt:

>>> 12e12
12000000000000.0
>>> print(repr(12e12))
12000000000000.0

The literal 12e12 is scientific notation. It means:

12×1012=12000000000000.0

Python stores that as a floating-point number, so both lines show the same thing. Typing 12e12 on its own makes the interpreter display the value's representation automatically, and print(repr(12e12)) asks for that representation explicitly and prints it. For a number like this the two coincide, because a numeric literal is already both readable and a valid Python expression.

Follow the second line one token at a time:

print(repr(12e12))
├─ print ......... built-in that displays its argument as text
├─ ( ............. begins print's argument
├─ repr .......... built-in that returns the canonical representation string
├─ ( ............. begins repr's argument
├─ 12e12 ......... the float twelve-trillion-point-zero
├─ ) ............. closes repr's call, producing "12000000000000.0"
├─ ) ............. closes print's call
└─ prints ........ 12000000000000.0

The second transcript shows the angle-bracket fallback:

>>> repr(min)
'<built-in function min>'

The name min refers to a built-in function object. Notice it is written with no parentheses: we are handing the function itself to repr, not calling it. A function is also an object, so it too has a representation. There is no short Python expression that rebuilds the exact built-in min, so repr cannot honor the round-trip rule here. Instead it returns a description. Notice the outer single quotes in the output: at the prompt, repr(min) is itself an expression whose value is a string, and Python shows you that string complete with quotes. The angle brackets inside are the signal:

<...>  →  a description of the object, not code eval can rebuild

Follow the line:

repr(min)
├─ repr .......... built-in that returns the canonical representation string
├─ ( ............. begins the call
├─ min ........... looks up the built-in function object named min
├─ ) ............. closes the call, asking min to represent itself
└─ evaluates to .. '<built-in function min>'

Dates Make the Two Forms Disagree

The clearest case is a date, because here the human label and the programmer label are genuinely different. This is the official transcript, line for line:

>>> from datetime import date
>>> tues = date(2011, 9, 12)
>>> repr(tues)
'datetime.date(2011, 9, 12)'
>>> str(tues)
'2011-09-12'

The two outputs describe the same object two ways:

Expression Output Why it looks that way
repr(tues) datetime.date(2011, 9, 12) reads like the constructor call that built it
str(tues) 2011-09-12 reads like a date a person would write

The repr form is programmer-facing. It says, in effect, "this object came from datetime.date with the arguments 2011, 9, and 12" — which is exactly the recipe the round-trip rule promises. The str form is human-facing. It drops the machinery and just shows the calendar date in year-month-day order.

Follow the Line: the import

from datetime import date
├─ from .......... start an import that pulls specific names out of a module
├─ datetime ...... the module to look inside
├─ import ........ bring the following name into the current environment
├─ date .......... the constructor to bind here, so date(...) now works
└─ action ........ binds the name date in the current frame

This is a statement, so its final branch states the action it performs rather than a value.

Follow the Line: the constructor call

date(2011, 9, 12)
├─ date .......... looks up the constructor function named date
├─ ( ............. begins the call
├─ 2011 .......... first argument, the year
├─ , ............. separates arguments
├─ 9 ............. second argument, the month
├─ , ............. separates arguments
├─ 12 ............ third argument, the day
├─ ) ............. closes the call and runs the constructor
└─ evaluates to .. a date object for September 12, 2011

Follow the Line: the assignment

tues = date(2011, 9, 12)
├─ tues .......... a name in the current frame
├─ = ............. bind the name on the left to the value on the right
├─ date(...) ..... the date object produced by the constructor call
└─ action ........ binds tues to that date object

The Methods Behind str and repr

Now the object-abstraction idea. Defining repr so it works on every type is a real problem, including types that did not exist when repr was written. The text names the property it wants: repr should be a generic or polymorphic function, from the roots poly (many) and morph (forms) — one function applied to many forms of data.

The solution is elegant. The repr function always invokes a method called __repr__ on its argument. So repr(x) does not contain a giant table of types. It simply asks x to represent itself, by calling x.__repr__(). We can confirm this directly at the prompt:

>>> tues.__repr__()
'datetime.date(2011, 9, 12)'

The str constructor works the same way: it invokes a method called __str__ on its argument.

>>> tues.__str__()
'2011-09-12'

These two transcripts return exactly what repr(tues) and str(tues) returned, because that is precisely what those functions called. The double underscores are pronounced "dunder," short for "double underscore," so __repr__ is read aloud as "dunder repr." The names that match are:

repr(x)   →   x.__repr__()    Python-facing representation
str(x)    →   x.__str__()     human-facing string

This is also why dot expressions are powerful beyond simple attribute access: by defining __repr__ and __str__ in your own classes, you extend the reach of the existing repr and str functions to objects that did not exist when those functions were written. The general principle the text draws out is this: certain functions should apply to many data types, and one clean way to build such a function is to use a shared attribute name with a different definition in each class.

Follow the Line: calling the method directly

tues.__repr__()
├─ tues .......... looks up the date object bound to tues
├─ . ............. attribute access on that object
├─ __repr__ ...... finds the representation method on the object's class
├─ ( ............. begins the call, with self bound to tues
├─ ) ............. closes the call, running the method
└─ evaluates to .. 'datetime.date(2011, 9, 12)'

A Warm-Up: repr Versus str on Values You Know

Before writing a class, predict these by hand. They use only types you have already met.

This prints:

7
7
hi
'hi'

The integer 7 looks the same either way: 7 is both readable and a valid expression. The string 'hi' is where they split. As a human-facing string, str('hi') is just the characters hi. As a programmer-facing representation, repr('hi') keeps the quotes, because 'hi' with quotes is the Python expression that rebuilds the string, while hi without quotes would look like a variable name. That single difference, quotes or no quotes, is the whole reason repr matters for strings.

A Bridge You Can Verify by Hand: a Coordinate Class

Now a small class that controls both of its string forms. Read each method and predict the output before you check it.

Work it out by hand. After p = Point(2, 5), the object has self.x equal to 2 and self.y equal to 5. The __repr__ method substitutes those into 'Point({0}, {1})', giving the recipe-like string Point(2, 5). The __str__ method substitutes them into '({0}, {1})', giving the plain pair (2, 5). The third line, print(p), has no repr or str written around it, so which form does it use? print displays the human-facing string, so it calls str(p), which calls p.__str__(). So:

Point(2, 5)
(2, 5)
(2, 5)

The thing to lock in: print(p) and str(p) agree, and both differ from repr(p). When you want the precise form, you must ask for repr explicitly. One twist to keep in mind for later: when a container like a list is displayed, Python uses repr on each item inside it, even though the list itself is shown the human-facing way.

The Official Pattern: a Class That Defines Both Methods

The bridge above used plain substitution. A polished __repr__ goes one step further and uses repr on its own attributes, so that string attributes come out with their quotes. This class shows the full pattern.

This prints:

Ticket('Spring Concert', 'B12')
Spring Concert, seat B12
Spring Concert, seat B12

The object stores two attributes:

event  →  'Spring Concert'
seat   →  'B12'

The __repr__ method returns a recipe-like string, Ticket('Spring Concert', 'B12'), that names the class and shows both attributes as quoted strings. The __str__ method returns the friendlier Spring Concert, seat B12. And print(ticket), asking for the human-facing form, calls str(ticket), which calls ticket.__str__(). As with dates, the precise form and the readable form differ, and print chooses the readable one.

Why the !r Appears

Inside the __repr__ method, the format fields are written {0!r} and {1!r} rather than plain {0} and {1}. Inside a format string, !r means "apply repr to this value before inserting it." Compare the two versions on the same data:

'Ticket({0}, {1})'      →   Ticket(Spring Concert, B12)
'Ticket({0!r}, {1!r})'  →   Ticket('Spring Concert', 'B12')

The second version is the one you want, because it puts the quotes back. With the quotes, the output reads like a Python expression that would rebuild the ticket, honoring the spirit of the round-trip rule. Without them, you could not tell event and seat were strings. This is the same quotes-or-no-quotes difference you saw with repr('hi'), now used deliberately inside a class.

Follow the Line: the format call inside __repr__

'Ticket({0!r}, {1!r})'.format(self.event, self.seat)
├─ 'Ticket({0!r}, {1!r})' .. a format string with two replacement fields
├─ .format ................. fills the fields with the given arguments
├─ ( ....................... begins the call
├─ self.event .............. value for field {0}, inserted via repr
├─ , ....................... separates arguments
├─ self.seat ............... value for field {1}, inserted via repr
├─ ) ....................... closes the call
└─ evaluates to ............ "Ticket('Spring Concert', 'B12')"

Tracing How print(ticket) Reaches a Method

The deepest point of this lesson is that a bare call expression ends up running a method on the object. Trace print(ticket) from (3) as a dispatch sequence, watching where each name is looked up.

print(ticket)
   │
   1. evaluate the argument: look up ticket in the global frame
   │     → the Ticket instance with event='Spring Concert', seat='B12'
   │
   2. print needs human-facing text, so it calls str(ticket)
   │
   3. str(ticket) invokes ticket.__str__()
   │     → look up __str__ on the object's class, Ticket
   │
   4. call __str__ with self bound to the ticket instance
   │     → read self.event and self.seat
   │     → return 'Spring Concert, seat B12'
   │
   5. print displays that string, without surrounding quotes
         → Spring Concert, seat B12

Notice what the machine never does: it never "understands" that a ticket is a thing you take to a concert. It only knows that ticket is a name bound to an instance, and that the instance's class supplies a __str__ method. The readable text does not appear by magic. The object receives a message, runs the matching method, and returns text. Swap in repr(ticket) and the only change is step 3: repr invokes ticket.__repr__() instead, and you get the quoted recipe form.

This is exactly why repr and str are called polymorphic. The same two function names drive completely different methods depending on the object in front of them, and your class joins that system simply by defining the methods with the matching names.

⚠Common Beginner Traps

Trap 1: Thinking repr and str print on their own. Both functions return a string; they do not display anything by themselves. You only see output because you are at an interactive prompt, which prints the value of an expression, or because you passed the result to print. Writing repr(x) inside a script with nothing around it computes a string and throws it away.

Trap 2: Forgetting that print uses the human-facing form. Plain print(x) calls str(x), not repr(x). If you wanted the precise, quoted form, you have to ask for it: print(repr(x)).

Trap 3: Confusing the quotes in the output with part of the value. At the prompt, repr(min) shows '<built-in function min>' with outer quotes because the value of that expression is a string, and the interpreter shows strings with quotes. The actual characters of the string do not include those outer quotes.

Trap 4: Returning a non-string from __repr__ or __str__. Both methods must return a string. Returning a number, a list, or another object raises a TypeError when Python tries to use the result as text.

Trap 5: Writing a __repr__ that is too vague. A __repr__ that returns just 'ticket' is readable but useless to another programmer, because it hides the data inside the object. A good __repr__ names the class and shows the important attributes, ideally as a string eval could rebuild — which is what the !r fields are for.

✓Check Your Understanding

  1. State the round-trip goal of repr in symbols, and explain why repr(min) does not satisfy it.
  2. Why does repr(date(2011, 9, 12)) mention datetime, while str(date(2011, 9, 12)) does not?
  3. Suppose a Color class defines __str__ to return self.name and __repr__ to return 'Color({0!r})'.format(self.name), and you make c = Color('teal'). What does print(c) show, and what does print([c]) show?
  4. In the Ticket example, which method does print(ticket) ultimately run, and why that one?
  5. What does !r do inside a format string, and why does the Ticket.__repr__ method use it?
  6. What goes wrong if a __str__ method returns the integer 42?
Show the answers
  1. The goal is eval(repr(x))=x: the representation string, when evaluated, rebuilds an equal object. repr(min) cannot satisfy it because there is no short Python expression that rebuilds the exact built-in min, so Python returns the angle-bracket description <built-in function min> instead, which is not runnable code.
  2. repr is programmer-facing and aims to look like the constructor call that built the object, so it includes the module and class, datetime.date(...). str is human-facing and chooses the plain calendar form 2011-09-12, dropping the machinery.
  3. print(c) shows teal, because print uses the human-facing form and calls str(c), which runs __str__. print([c]) shows [Color('teal')], because displaying a list uses repr on each element, so the element comes out through __repr__ even though print was called on the list as a whole.
  4. It runs Ticket.__str__. print displays human-facing text, so it calls str(ticket), which invokes ticket.__str__(). The printed result is a display of the object, not the object itself.
  5. !r applies repr to the value before inserting it into the string, which puts quotes around strings. Ticket.__repr__ uses it so the output reads like a Python expression, Ticket('Spring Concert', 'B12'), that shows the attributes are strings and could rebuild the object.
  6. __str__ must return a string. Returning 42 raises a TypeError when Python tries to use the method's result as text.

§Summary

Every object should be able to describe itself as a string, and Python asks for two descriptions, not one. The repr function returns the canonical, programmer-facing representation, with the goal that eval(repr(x))=x rebuilds the object; when no such expression exists, it returns an angle-bracket description instead. The str function returns readable, human-facing text and is free to favor clarity over reconstruction. With dates the two genuinely differ: datetime.date(2011, 9, 12) versus 2011-09-12.

The mechanism behind both is the same. repr(x) always invokes x.__repr__(), and str(x) always invokes x.__str__(). This makes them polymorphic functions: one name, many forms, because each class supplies its own method under a shared name. By writing __repr__ and __str__ on your own classes, you extend these built-in functions to objects that did not exist when they were written. Keep in mind that print(x) uses the human-facing str(x), that both functions only return strings rather than printing on their own, and that an !r field forces the programmer-facing form inside a format string.

This is the first close look at a generic function in this section. Next we widen the idea: Python defines many special method names that let your objects join truth tests, indexing, call expressions, and arithmetic operators.

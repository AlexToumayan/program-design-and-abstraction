Chapter 4/4.5 Unification

4.5.1 Pattern Matching and Variable Bindings

◷ approx. 14 min read · 1 worked example

In the previous section you learned to state facts and ask queries in the logic language. This lesson explains the single operation that makes those queries work: matching a pattern against an expression and recording what each variable must mean.

By the end you will be able to do four things: say precisely what it means for a pattern to match an expression, fill in a variable to make a pattern agree with a fact, handle a variable that appears more than once, and read a match as the construction of a small table of variable bindings. This is the primitive that the rest of this section builds on, so it is worth getting exactly right.

The Operation Behind Every Query

The query interpreter answers a question by comparing it against facts you have already stated. The comparison is not a simple "are these two things identical?" test, because a query usually contains a variable and a fact usually does not. The interpreter has to line them up and figure out what the variable would have to mean.

That operation has a name. As the original text puts it:

Unification is a general method of matching a query to a fact, each of which may contain variables.

In this lesson we study the simpler half of unification, called pattern matching, where only one side (the query pattern) contains variables and the other side (the fact) is a plain expression. The next lessons generalize to the case where both sides have variables.

An Intuitive Picture: A Worksheet With Blanks

Picture a worksheet line printed with a blank you have to fill in:

parent(abraham, ____)

Now picture a fact card that is fully filled in:

parent(abraham, barack)

To make the worksheet line agree with the fact card, you write barack in the blank. You do not get to edit the fact card; the card is fixed truth. Your only freedom is what you write in the blank.

That is exactly pattern matching. The worksheet line is the pattern, the blank is a variable, and the fact card is the expression. Matching means choosing what goes in each blank so that the filled-in pattern reads identically to the expression. The list of choices you wrote in the blanks is the result you actually care about.

Two more details from this picture will carry through the whole lesson. First, a blank can hold a whole chunk, not just one word: a blank could be filled with (a b), an entire small list. Second, if the same blank label appears twice on the worksheet, you must write the same thing in both copies. We will see both of these below.

Recalling the Fact and the Query

In the previous section you wrote a fact, an unconditional truth, like this:

(fact (parent abraham barack))

And you asked a query, a question, like this:

(query (parent abraham ?child))

The query carries a variable, ?child. The fact does not. Following the original text: the query (query (parent abraham ?child)) and the fact (fact (parent abraham barack)) match, if the variable ?child takes the value barack.

Notice the shape of the reasoning. The interpreter walks the two expressions together, token by token. The symbol parent lines up with parent, and abraham lines up with abraham. The only spot that differs is the last one, where the pattern has the variable ?child and the fact has the plain symbol barack. The variable is free to become whatever stands across from it, so it becomes barack, and the match succeeds.

The choice the interpreter just made has a name. A binding is a stored meaning for a variable. You can read it as a sentence: "for the rest of this attempt, this variable means this value."

?child -> barack

The Formal Meaning of a Pattern

A pattern is an expression that may contain variables. The original text gives the exact rule for when a pattern matches:

A pattern matches some expression (a possibly nested Scheme list) if there is a binding of variable names to values such that substituting those values into the pattern yields the expression.

Read that rule slowly, because every word is doing work.

It asks whether there is a binding, not whether you happened to guess one. Matching is a search for a set of meanings that works.

It says substituting those values into the pattern yields the expression. The test is concrete: take your bindings, plug each value in for its variable, and check whether the pattern now reads as the exact expression. If it does, the match holds. If no set of bindings can make them identical, the match fails.

This is not the assignment you know from Python. A line like x = 3 overwrites a name in memory. A binding here is a logical agreement discovered by lining up two expressions; nothing in your program is being mutated, and the symbols inside the expression are never evaluated as Python names.

Anatomy: How a Pattern Matches the Fact

Here is the query pattern, walked one token at a time. Each branch row annotates a single token; the final row gives the result of the whole line.

(parent abraham ?child)
├─ (       .......... opens the Scheme list
├─ parent  .......... a plain symbol; must match the symbol parent
├─ abraham .......... a plain symbol; must match the symbol abraham
├─ ?child  .......... a variable; free to take a value
├─ )       .......... closes the list
└─ matches (parent abraham barack)  with  ?child -> barack

The first three tokens are fixed: parent and abraham are plain symbols, so they only agree if the expression has the identical symbols in those spots. The token ?child begins with ?, which marks it as a variable, so it is the one slot allowed to bend. It takes the value sitting across from it in the fact, barack, and that single binding makes the filled-in pattern read exactly as the fact.

A Warm-Up Match

Start with the smallest possible match before the official nested examples. Suppose the pattern and the expression are:

pattern:    (?name is happy)
expression: (maya is happy)

Walk the three elements together. The element is is a plain symbol and lines up with is; happy lines up with happy; the element ?name is a variable, so it takes whatever is across from it, which is maya. The one binding is:

?name -> maya

Apply the formal test to be sure. Substitute maya for ?name in the pattern and you get (maya is happy), which is exactly the expression. The match holds.

A Bridge: One Blank Used Twice

Before the official example, here is a slightly harder case that isolates the one new idea in it: a variable that appears more than once. Take this pattern and expression:

pattern:    (?p likes ?p)
expression: (sam likes sam)

The first ?p lines up with sam, so you record ?p -> sam. Now the second ?p appears. You do not get a fresh choice. The same variable must keep the same value, so the second ?p is already committed to sam, and the expression does have sam in that spot, so the match succeeds.

Contrast it with (sam likes joan). The first ?p would bind to sam, but the second ?p is now fixed at sam and the expression has joan there. One variable cannot mean two different things in a single match, so this one fails. That single rule, repeated variables must agree, is the whole point of the official example that follows.

The Official Example: A Repeated Variable

The original text gives this nested Scheme-list example. The expression is a list of three elements, and so is the pattern:

expression: ((a b) c (a b))
pattern:    (?x c ?x)

The pattern matches the expression with the binding:

?x -> (a b)

Here is why, walked element by element. The pattern's first element is ?x, a variable, so it takes the first element of the expression, the whole sublist (a b). That is the "a blank can hold a chunk" detail from the worksheet picture: ?x is bound not to a single symbol but to an entire list. The pattern's second element is the plain symbol c, which lines up with the expression's c. The pattern's third element is ?x again, the same variable, which is already committed to (a b); the expression's third element is also (a b), so they agree and the match holds.

Apply the formal test. Substitute (a b) for both copies of ?x in (?x c ?x) and you get ((a b) c (a b)), exactly the expression. The match holds, and there is one binding, ?x -> (a b).

(?x c ?x)
├─ (   .......... opens the list
├─ ?x  .......... a variable; takes the first element of the expression
├─ c   .......... a plain symbol; must match the symbol c
├─ ?x  .......... the same variable; must keep its committed value
├─ )   .......... closes the list
└─ matches ((a b) c (a b))  with  ?x -> (a b)

One thing did not happen here that a Python reader might expect: the interpreter never tried to evaluate a, b, or c as names. They are plain symbols inside Scheme-style data. Matching compares structure and symbols; it never runs them.

The Same Expression, A Different Pattern

The original text then matches that same expression against a second pattern:

expression: ((a b) c (a b))
pattern:    ((a ?y) ?z (a b))

This match produces two bindings:

?y -> b
?z -> c

Walk the three elements again. The pattern's first element is itself a small list, (a ?y), which must line up against the expression's first element (a b). Going inside that sublist: the symbol a matches a, and the variable ?y takes b, giving ?y -> b. The pattern's second element is the variable ?z, which takes the expression's second element c, giving ?z -> c. The pattern's third element is the plain list (a b), which has no variables and matches the expression's third element (a b) literally.

((a ?y) ?z (a b))
├─ (        .......... opens the outer list
├─ (a ?y)   .......... a sublist; a matches a, and ?y takes b
├─ ?z       .......... a variable; takes the second element c
├─ (a b)    .......... a literal sublist; must match (a b) exactly
├─ )        .......... closes the outer list
└─ matches ((a b) c (a b))  with  ?y -> b, ?z -> c

Substitute b for ?y and c for ?z in ((a ?y) ?z (a b)) and you get ((a b) c (a b)), the expression. The match holds with the two bindings above. So a single expression can be matched by more than one pattern, each producing its own table of bindings.

A Step-by-Step Trace: Building the Table of Bindings

It helps to watch a match as the interpreter filling a small table, one element at a time. The table is the interpreter's notebook of what each variable currently means. Trace the official repeated-variable match.

Pattern:    (?x c ?x)
Expression: ((a b) c (a b))

Step  Pattern element  Expression element  Table of bindings after this step
------------------------------------------------------------------------------
1     ?x               (a b)               ?x -> (a b)
2     c                c                   ?x -> (a b)
3     ?x               (a b)               ?x -> (a b)

At step 1 the table is empty, the variable ?x is unbound, so it takes (a b) and the table gains one entry. At step 2 the plain symbol c matches c and changes nothing in the table. At step 3 the variable ?x is already in the table, so instead of choosing again the interpreter looks up its value and checks it against the expression element; (a b) equals (a b), the check passes, and the match succeeds. The final table, the set of bindings, is the useful result of the whole match.

⚠Common Beginner Traps

  • Thinking ?x is a Python variable. It is a symbol in the logic language that the query interpreter treats specially during matching; it has nothing to do with Python name binding.
  • Treating symbols like a, b, and c as Python names to be evaluated. In this setting they are plain data inside Scheme-style expressions, compared as symbols and never run.
  • Forgetting that a repeated variable must keep the same value. A second appearance of ?x does not get a fresh choice; it must agree with the value already stored, and if the expression has a different value in that spot the whole match fails.
  • Believing a successful match just returns True. The result that matters is the table of bindings produced along the way.
  • Forgetting that a single blank can hold a whole list. In the official example ?x binds to (a b), an entire sublist, not a single symbol.
  • Assuming pattern matching is the whole interpreter. It is only the primitive operation, applied many times while the interpreter searches.

✓Check Your Understanding

  1. Using the formal rule, what does it mean for a pattern to match an expression?
  2. Why does (?x c ?x) match ((a b) c (a b)), and what is the binding?
  3. Why would (?x c ?x) fail to match ((a b) c (a c))?
  4. In ((a ?y) ?z (a b)) matched against ((a b) c (a b)), which variables receive bindings, and to what?
  5. Does the pattern (?a b ?a) match the expression (m b m), and if so with what binding? What about (m b n)?
Show the answers
  1. A pattern matches an expression if there is a binding of variable names to values such that substituting those values into the pattern yields exactly that expression.
  2. The first ?x takes the first element (a b), so ?x -> (a b). The second ?x is the same variable and must keep that value; the expression's third element is also (a b), so it agrees and the match succeeds. The binding is ?x -> (a b).
  3. The first ?x would bind to (a b), but the third element of the expression is (a c), which the now-committed ?x cannot match. One variable cannot have two different values in a single match, so the match fails.
  4. ?y receives b (from inside the sublist (a ?y) against (a b)), and ?z receives c (the second element). The surrounding symbols and the literal sublist (a b) must match exactly and produce no bindings.
  5. It matches (m b m) with the single binding ?a -> m: the first ?a takes m, the plain symbol b lines up, and the second ?a is already committed to m, which agrees with the expression's third element. It does not match (m b n): the second ?a is fixed at m but the expression has n there, so one variable would need two values and the match fails.

§Summary

Pattern matching compares a pattern against an expression and records a binding for each variable so that substituting those bindings into the pattern yields the expression exactly. A binding is a stored meaning for a variable, written variable -> value, and a single blank can hold a whole list such as (a b). A variable that appears more than once is a constraint: every appearance must take the same value within one match, which is why (?x c ?x) matches ((a b) c (a b)) but not ((a b) c (a c)). The same expression can be matched by different patterns, each yielding its own table of bindings. The real output of a match is that table, not a bare yes-or-no. Unification, in the next lessons, generalizes this to the case where both sides may contain variables.

▶Step-Through Checkpoint

The next block is a tiny, self-contained model of the matching machinery: it walks a pattern and an expression element by element, binds each variable the first time it appears, and checks the stored value every later time. Stepping through it draws the environment diagram the match builds: each match_element call opens a frame for pattern_part and expression_part, but the single bindings dict is created in the global frame, so every match_element frame reaches up to that one shared table and you watch '?x' appear in it as the call at i = 0 returns. Before you step through it, write down three predictions: how many local frames will open and close over the whole run, which frame the bindings table lives in as it fills, and exactly what the two print lines at the bottom will show. Don't worry if you're unsure of an answer; the point is to compare your guess against what the stepper shows, and the walkthrough right after this block confirms all three.

Step through the block one line at a time and check each prediction as the frames appear. Three local frames open and close, one call to match_element per element, and the bindings table they all consult sits in the global frame; no call gets a private copy. At i = 0 the table gains '?x': ['a', 'b']. At i = 1 the plain symbol "c" is compared directly and the table is never touched. At i = 2 the function does not bind again: "?x" is already stored, so it compares the saved ['a', 'b'] against the current ['a', 'b'], and they are equal. The program prints True and the final table {'?x': ['a', 'b']}. If you change the last element of expression to ["a", "c"] and rerun, the second ?x check fails and matched becomes False, exactly the repeated-variable constraint.

Chapter 4/4.3 Declarative Programming

4.3.6 Aggregation and Grouping

◷ approx. 16 min read · 1 worked example · best read in two passes

Until now, almost every row in your results came from a single input row, or from a single row built by joining two tables. One row in, one row out. This lesson is about the opposite move: collapsing many rows into a single summary value, and doing it group by group.

By the end you will be able to do five things. Write an aggregate function like max, min, count, sum, or avg that summarizes a whole column into one number. Count rows with count(*) and count unique values with count(distinct ...). Split a table into groups with group by so an aggregate applies to each group separately. Group by more than one expression at once. And choose correctly between where and having, the two filters that act at different moments in the process.

The mental shift is the whole lesson: stop thinking "one row at a time" and start thinking "a pile of rows, summarized."

An Intuitive Analogy

Picture a teacher holding a stack of graded exams, one paper per student. Most questions you could ask are about a single paper: "show me Maria's score," "which papers scored above 90." Those are the row-at-a-time questions you already know how to answer with where.

Now the principal walks in and asks, "What was the highest score on the exam?" The teacher does not hand over one student's paper as an answer. Instead she fans through the entire stack, keeping a running "biggest so far" in her head, and reports a single number. That single number was computed from many papers but belongs to none of them in particular. That is aggregation.

Then the principal asks something sharper: "What was the highest score in each class period?" Now the teacher first sorts the stack into piles, one pile per period, and computes a highest score within each pile. One number comes out per pile. That sorting-into-piles step is group by, and the per-pile summary is an aggregate applied to each group.

Hold onto those two pictures: fanning through one stack for a single summary, and splitting into piles before summarizing each pile.

Aggregate Functions

A select statement can do more than join, project, and filter individual rows. It can also perform aggregation across many rows at once. An aggregate function reads down a column and folds all of its values into one result.

The four aggregate functions used most often are:

Function What it returns
max the largest value in the column
min the smallest value in the column
count the number of values (or rows)
sum the total of the values
avg the average (mean) of the values

avg belongs to the same family: it adds the values and divides by how many there are. You can apply several aggregate functions to the same set of rows by listing more than one column in the select clause. One important rule controls which rows get folded in: only rows that survive the where clause are considered in the aggregation.

We will work with the animals table the official text uses. Create it once and every query below runs against it:

create table animals as
  select "dog" as name, 4 as legs, 20 as weight union
  select "cat"        , 4        , 10           union
  select "ferret"     , 4        , 10           union
  select "t-rex"      , 2        , 12000        union
  select "penguin"    , 2        , 10           union
  select "bird"       , 2        , 6;

Here is that same table as a grid so you can see every value at a glance:

name legs weight
dog 4 20
cat 4 10
ferret 4 10
t-rex 2 12000
penguin 2 10
bird 2 6

Following an Aggregate Query Token by Token

Before running anything, read a select that contains an aggregate the same careful way you read any line of code: one token at a time, ending with what it produces. Here is select max(legs) from animals; broken into its pieces:

select max(legs) from animals
├─ select ...... begin a query; describe the columns to return
├─ max ......... aggregate function: keep the largest value seen
├─ ( ........... begin the aggregate's input
├─ legs ........ the column to summarize
├─ ) ........... end the aggregate's input
├─ from ........ name the source table
├─ animals ..... read rows from the animals table
└─ returns ..... one row, one column: the largest legs value, 4

Notice the final row. An aggregate query does not return one row per animal; it returns a single row holding the summary. Every token before the last describes how that one value is built; the last row is the result.

Warm-Up: One Aggregate Over a Tiny Table

Start smaller than the official example so the idea is unmistakable. Imagine a two-row table of just legs values, 4 and 2. Asking for max(legs) fans through both values and keeps the larger:

max over [4, 2]
├─ 4 ........... largest so far is 4
├─ 2 ........... 2 is not larger; largest stays 4
└─ evaluates to  4

That is the entire mechanism. min keeps the smaller instead, sum keeps a running total, and count keeps a running tally. The official table just has more rows.

Bridge: Summing By Hand Before Trusting sum

sum(weight) is worth slowing down on, because the answer looks surprising until you add it up yourself. The weights are 20, 10, 10, 12000, 10, 6. Add them in order:

20+10+10+12000+10+6=12056

The single huge value (the t-rex at 12000) dominates the total, which is exactly why the sum is so large. Now you can predict the official output instead of being surprised by it.

The Official Aggregate Queries

Here are the official aggregate queries verbatim, run against the animals table:

sqlite> select max(legs) from animals;
4
sqlite> select sum(weight) from animals;
12056
sqlite> select min(legs), max(weight) from animals where name <> "t-rex";
2|20

Read the third query carefully, because it combines two ideas. It applies two aggregate functions at once, min(legs) and max(weight), producing a row with two columns. It also filters first: name <> "t-rex" keeps every animal except the t-rex (<> is SQL's "not equal"). Because the t-rex is excluded before aggregation, max(weight) is computed over the remaining animals, whose heaviest is the dog at 20. The smallest leg count among those same rows is 2, so the result is 2|20. This shows the rule in action: only rows that pass where reach the aggregate.

Counting Rows and Counting Distinct Values

count deserves its own section because it has two special behaviors that trip people up.

The first is count(*). The * is not a column; it is a signal that means "count the rows themselves," regardless of any particular column's contents. The second is the distinct keyword, which removes repeated values from a column before the aggregate sees them, so each value is considered only once.

Read select count(distinct legs) from animals; token by token:

select count(distinct legs) from animals
├─ select ...... begin a query
├─ count ....... aggregate function: tally how many values
├─ ( ........... begin the aggregate's input
├─ distinct .... drop repeated values first; keep each one once
├─ legs ........ the column whose distinct values are tallied
├─ ) ........... end the aggregate's input
├─ from ........ name the source table
├─ animals ..... read rows from the animals table
└─ returns ..... one row: how many distinct legs values exist, 2

The animals table has only two distinct leg counts, 2 and 4, so count(distinct legs) is 2, even though six rows have a legs value. Here are the three official counting queries:

sqlite> select count(legs) from animals;
6
sqlite> select count(*) from animals;
6
sqlite> select count(distinct legs) from animals;
2

Compare them directly. There are six rows, so count(*) is 6. Every row has a non-empty legs value, so count(legs) is also 6 here. But there are only two different leg counts, so count(distinct legs) is 2. The gap between 6 and 2 is the entire point of distinct: it counts variety, not volume.

Grouping Rows With group by

Each query so far returned a table with a single row, because each one summarized the whole table at once. The group by and having clauses change that. They partition rows into groups and let you keep only a subset of those groups. Crucially, any aggregate function in the having clause or the column list then applies to each group independently, not to the entire table.

This is the "sort into piles, then summarize each pile" picture from the analogy. Read the clause token by token:

select legs, max(weight) from animals group by legs
├─ select ........ begin a query
├─ legs .......... return the grouping value for each group
├─ , ............. separate the columns
├─ max(weight) ... per group, the largest weight in that group
├─ from .......... name the source table
├─ animals ....... read rows from the animals table
├─ group ......... begin the grouping clause
├─ by ............ specify what to group on
├─ legs .......... form one group per distinct legs value
└─ returns ....... one row per group: its legs value and its max weight

To compute the heaviest four-legged animal and the heaviest two-legged animal at the same time, group the rows by legs. That forms a two-legged group (t-rex, penguin, bird) and a four-legged group (dog, cat, ferret), and computes the maximum weight inside each. The second query below lists the legs values that have at least two rows sharing the same weight.

sqlite> select legs, max(weight) from animals group by legs;
2|12000
4|20
sqlite> select weight from animals group by weight having count(*)>1;
10

A Step-by-Step Trace of the Grouped Query

It is worth watching the first grouped query build its rows, because the model becomes obvious once you see it staged.

Step one, partition the six rows by their legs value:

legs group rows in the group their weights
2 t-rex, penguin, bird 12000, 10, 6
4 dog, cat, ferret 20, 10, 10

Step two, compute max(weight) inside each group on its own:

legs max(weight)
2 12000
4 20

Step three, emit one result row per group, which is exactly the official output 2|12000 and 4|20. The two-legged maximum is 12000 (the t-rex), and the four-legged maximum is 20 (the dog).

How having Filters Whole Groups

The second query, select weight from animals group by weight having count(*)>1;, groups by weight and then keeps only groups containing more than one row. Trace it the same way. Grouping by weight makes these groups:

weight group rows in the group count(*)
6 bird 1
10 cat, ferret, penguin 3
20 dog 1
12000 t-rex 1

The condition count(*) > 1 keeps only the group whose row count exceeds one. Only the weight 10 qualifies, because three animals weigh 10. So the single output row is 10. The key insight: count(*) > 1 cannot be evaluated for a row in isolation. It can only be checked after a group has been formed, which is exactly why it lives in having and not where.

Grouping by Multiple Expressions

The group by clause is not limited to one column. You can list several columns, or even full expressions, and a separate group is formed for every unique combination of the resulting values. By convention the grouping expression also appears in the column list, so you can tell which result row came from which group.

Three official queries show the range of what group by accepts:

sqlite> select max(name) from animals group by legs, weight order by name;
bird
dog
ferret
penguin
t-rex
sqlite> select max(name), legs, weight from animals group by legs, weight
  ....>   having max(weight) < 100;
bird|2|6
penguin|2|10
ferret|4|10
dog|4|20
sqlite> select count(*), weight/legs from animals group by weight/legs;
2|2
1|3
2|5
1|6000

The first two queries group by the pair legs, weight. Two rows land in the same group only if they agree on both values. In this table every animal has a distinct (legs, weight) pair except cat and ferret (both 4, 10), so there are five groups, and max(name) picks the alphabetically largest name in each. The second query adds having max(weight) < 100, which drops the t-rex's group (its weight 12000 is not under 100), leaving four rows.

The third query groups by an expression, weight/legs. SQLite does integer division here, so each animal's quotient is computed first, and animals that share a quotient share a group:

animal weight legs weight/legs
bird 6 2 3
penguin 10 2 5
t-rex 12000 2 6000
dog 20 4 5
cat 10 4 2
ferret 10 4 2

Group by that quotient and the counts fall out: quotient 2 has two animals (cat, ferret), quotient 3 has one (bird), quotient 5 has two (penguin, dog), and quotient 6000 has one (t-rex). That matches the official output rows 2|2, 1|3, 2|5, and 1|6000, where each row is the count followed by the shared quotient.

where Versus having

Both where and having filter, and the syntax looks similar, so it is easy to reach for the wrong one. The difference is when each filter runs.

where:  filters individual rows, before any grouping or aggregation
having: filters whole groups, after aggregates have been computed

A having clause can contain the same kind of row condition a where clause can, but it can also call aggregate functions like count and max. For the fastest execution and the clearest code, follow this rule: if a condition depends only on a single row's contents, put it in where; reserve having for conditions that genuinely require aggregation, such as demanding a minimum count for a group.

Use where when the test depends on one row:

select name from animals where legs = 4;

Use having when the test depends on a group summary:

select weight from animals group by weight having count(*) > 1;

The expression count(*) > 1 is impossible to evaluate before grouping, because a count only exists once a group has been assembled. That single fact tells you which clause to use every time.

Non-Aggregate Columns in Grouped Queries

There is a subtle case the official text warns about. When you use group by, your column list can include expressions that do not aggregate. In some situations SQLite will fill such a column with a value taken from the same row that produced an aggregate. For example, this query reports the name of the animal that has the maximal weight:

sqlite> select name, max(weight) from animals;
t-rex|12000
sqlite> select name, legs, max(weight) from animals group by legs;
t-rex|2|12000
dog|4|20

In the first query, max(weight) is 12000, and SQLite pairs it with t-rex, the name from the very row that held that maximum. In the second, grouping by legs produces a two-legged row and a four-legged row, and each non-aggregate name comes from that group's maximum-weight row: t-rex for the heaviest two-legged animal and dog for the heaviest four-legged one.

This is convenient, but it is only reliable when the chosen row is unambiguous. Whenever the row that corresponds to the aggregation is unclear, for instance when you aggregate with count instead of max, the value placed in a non-aggregate column may be arbitrary. For the clearest and most predictable code, a select statement that includes a group by clause should include at least one aggregate column, and should include non-aggregate columns only when their contents are determined by the aggregation.

⚠Common Beginner Traps

  • Thinking an aggregate runs on one row at a time. max, min, count, sum, and avg summarize a collection of rows into a single value.
  • Confusing count(legs) with count(distinct legs). The first counts how many rows have a legs value; the second counts how many different leg values exist.
  • Writing where count(*) > 1. An aggregate condition cannot be checked per row, so SQLite rejects it with the error misuse of aggregate function count(); the condition belongs in having, not where.
  • Forgetting that only rows passing the where clause are included in an aggregate, as in min(legs), max(weight) ... where name <> "t-rex".
  • Expecting a grouped query to return one row per input row. It returns one row per group.
  • Treating group by weight/legs as if it grouped by weight or legs alone. It groups by the value of the whole expression, so animals with the same quotient share a group.
  • Selecting an arbitrary non-aggregate column in a grouped query and trusting the value. It is predictable only when the aggregation determines which row it comes from.

✓Check Your Understanding

  1. Why does select sum(weight) from animals; return a single row instead of one row per animal?
  2. The animals table has six rows with legs values, yet count(distinct legs) is 2. Explain the difference between that result and count(legs).
  3. Predict the full output of select count(*), legs from animals group by legs;, including how many rows it has and the values in each. Which animals contribute to each row?
  4. In select legs, max(weight) from animals group by legs;, why does the result have exactly two rows, and where do the values 12000 and 20 come from?
  5. The query select count(*), weight/legs from animals group by weight/legs; returns the row 2|2. Which animals produced that row, and what do the two numbers mean?
Show the answers
  1. With no group by, sum(weight) aggregates over the entire selected table at once, folding all six weight values into one total (12056). Aggregation produces a single summary value, so the result is one row.
  2. count(legs) counts how many rows have a legs value, which is 6. count(distinct legs) first removes repeated values, leaving only the two distinct leg counts 2 and 4, then counts those, giving 2. One measures volume; the other measures variety.
  3. Grouping by legs forms two groups, so the output has two rows: 3|2 and 3|4. The two-legged group (t-rex, penguin, bird) has three rows, giving count(*) of 3 paired with legs value 2; the four-legged group (dog, cat, ferret) also has three rows, giving 3 paired with legs value 4.
  4. Grouping by legs creates one group per distinct leg count, and there are two distinct counts (2 and 4), so two rows result. Within the two-legged group the maximum weight is 12000 (the t-rex); within the four-legged group it is 20 (the dog).
  5. The quotient weight/legs equals 2 for both cat (10/4) and ferret (10/4) under integer division, so those two animals form one group. The row 2|2 means that group has a count of 2 rows sharing the quotient 2.

§Summary

Aggregation collapses many rows into a single summary value. The aggregate functions max, min, count, sum, and avg fold a column down to one result, and only rows that survive the where clause are included. count(*) counts rows, while count(distinct ...) counts how many different values a column holds. The group by clause partitions rows into groups so that aggregates apply to each group independently, and you can group by several columns or by a full expression, with one group per unique combination of values. having filters whole groups after aggregation and may itself call aggregate functions, whereas where filters individual rows beforehand; the deciding question is always whether the condition needs a group summary. Finally, in a grouped query, include at least one aggregate column and trust a non-aggregate column only when the aggregation determines which row supplies it.

Chapter 2/2.9 Recursive Objects

2.9 Recursive Objects and the Link Class

◷ approx. 14 min read · 6 worked examples

By the end of this lesson you will be able to do four things: say precisely what a recursive object is, build a linked list out of small Link objects and explain what each object stores, trace how bracket indexing like s[2] walks the chain one link at a time, and reason about a linked list that holds another linked list as its data.

The single mental move that makes all of this click is the same one from earlier lessons, turned on data instead of code: when a structure looks like it contains itself, stop and ask "what is the first piece, and what is the rest?" If the rest has the same shape as the whole, you are looking at a recursive object.

A Chain of Index Cards

Picture a stack of index cards on a desk. Each card holds two things written on it:

the value for this position
an arrow pointing to the next card

The last card's arrow does not point to another card. Instead it points to a small label that reads "no more cards." That label is what tells you the chain has ended.

This is exactly the picture of a linked list. A linked list is not one long Python list sitting in memory. It is a chain of small objects, each holding one value and a pointer to the next object:

[3 | -]---> [4 | -]---> [5 | empty]

Each object in the chain knows only two facts:

  • its own first value
  • the rest of the chain, which is another linked list

That second fact is what makes the object recursive: the rest of a linked list is itself a linked list, all the way down until you hit the "no more cards" label. A recursive object is an object that can contain another object of the same kind, and a linked list is the cleanest first example of one.

The Link Class

We capture the index-card idea with a class named Link. Each instance is one card. The first attribute is the value written on the card, and the rest attribute is the arrow to the next card. To represent the "no more cards" label we use an empty tuple stored on the class itself as Link.empty.

3
3
4
5

The double-underscore names are special methods. You never call them by name; Python calls them for you whenever you use the matching built-in syntax. Two of them appear right above:

len(s)     ->  Python calls  s.__len__()
s[1]       ->  Python calls  s.__getitem__(1)

The class attribute empty = () lives on the class, not on any one instance. Every link can refer to the same shared end marker as Link.empty, and there is only ever one of it.

Anatomy: the class header

The first line of a class definition names the new kind of object you are about to describe.

class Link:
├─ class ....... keyword that begins a class definition
├─ Link ........ the name bound to the new class object
├─ : ........... begins the indented suite of class statements
└─ defines a new class named Link whose suite describes its attributes and methods

Anatomy: the constructor

The __init__ method runs once for each new link, right after the object is created, to fill in its two attributes.

def __init__(self, first, rest=empty):
├─ def ......... keyword that begins a function definition
├─ __init__ .... the special method Python runs to initialize a new instance
├─ self ........ the new Link object being initialized
├─ first ....... the value stored at this position in the chain
├─ rest ........ the remaining linked list, defaulting to empty when omitted
└─ defines the initializer that stores first and rest on each new Link

The body's first line is an assertion that guards the structure:

assert rest is Link.empty or isinstance(rest, Link)
├─ assert ...................... raise an error unless the condition is true
├─ rest is Link.empty ......... true when rest is exactly the end marker
├─ or ......................... succeed if either side is true
├─ isinstance(rest, Link) ..... true when rest is another Link object
└─ enforces that rest is either the empty sentinel or another Link, never a bare value

This assertion is the whole reason the chain stays well-formed. The rest of a linked list must be either the empty sentinel or another Link. If you tried to pass a number or a string as rest, the assertion would stop you immediately, because the chain would otherwise have no proper endpoint and no way to continue.

Anatomy: the indexing method

__getitem__ is the method Python runs when you write square brackets after a link. It is recursive: index zero is answered directly, and every larger index is handed to the rest of the chain.

def __getitem__(self, i):
├─ def ............. keyword that begins a function definition
├─ __getitem__ .... the special method Python runs for bracket indexing
├─ self ........... the Link object being indexed
├─ i .............. the integer index requested
└─ defines how a Link responds to s[i] by walking toward position i

The recursive line that does the walking:

return self.rest[i-1]
├─ return ............ hand a value back to the caller
├─ self.rest ......... the rest of the chain, itself a Link
├─ [i-1] ............. ask that shorter chain for one position earlier
└─ evaluates to the element at index i by delegating to the rest at index i minus one

Each recursive call moves one link to the right and asks for an index that is one smaller. When the index finally reaches zero, the if i == 0 branch returns that link's first and the walk stops.

Building the Chain: From the Inside Out

Before trusting the official three-element list, build the smallest possible chain by hand and check it.

Expected output:

7
True
1

Link(7) builds a one-element chain. Because we left rest off, it defaulted to Link.empty, so one.rest is Link.empty is True and the length is exactly 1. This is the base of every linked list: a single card whose arrow points at the end marker.

Now grow it to two elements, which is small enough to still verify in your head.

Expected output:

8
9
2

Link(8, Link(9)) is two cards. The outer card holds 8 and points to an inner card that holds 9 and points to the end marker. Indexing pair[1] cannot answer directly, so it delegates: pair.rest[0], which is the inner card's first, namely 9. That delegation is the same machinery the official example relies on, just with one fewer link.

Anatomy: constructing the official chain

The construction expression nests three calls. Read it as one expression whose value is the outermost link.

s = Link(3, Link(4, Link(5)))
├─ s ........................... the name being bound
├─ = ........................... binds the name on the left to the value on the right
├─ Link(3, ... ) ............... builds the outer link with first 3
├─ Link(4, ... ) ............... builds the middle link with first 4
├─ Link(5) ..................... builds the innermost link with first 5 and rest empty
└─ binds s to the outermost Link, the head of a three-element chain

Python must evaluate the arguments before it can call a function, so the innermost Link(5) is built first, then Link(4, Link(5)), then the outer Link(3, ...). The chain is assembled from the inside out even though you read it from the outside in.

Here is that same official example run interactively, which also shows what a link looks like before we give it a custom representation:

>>> s = Link(3, Link(4, Link(5)))
>>> len(s)
3
>>> s[1]
4

len(s) is 3 because __len__ adds one for the current link and recurses into the rest until it reaches the empty tuple, whose length is 0. s[1] is 4 because the index walks one link to the right and then returns that link's first value.

How Indexing Walks the Chain

Indexing a linked list is not a jump to a slot the way indexing a Python list is. It is a walk. To compute s[2], Python starts at the head and follows rest pointers, shrinking the index by one at each step, until the index hits zero.

Anatomy: an indexing expression

s[2]
├─ s ........... the head link of the chain
├─ [2] ......... request the element at index 2
└─ evaluates to 5

Here is the full unfolding, one delegation per line:

s[2]
= s.rest[1]        index 2 was not 0, so ask the rest for index 1
= s.rest.rest[0]   index 1 was not 0, so ask the next rest for index 0
= 5                index 0, so return that link's first value

Each line moves exactly one link to the right and subtracts one from the index. The walk is guaranteed to end because the index decreases by one every step and the chain is finite. This is the linked-list cost you will see again and again: reaching index i takes Θ(i) steps, because there is no way to skip ahead.

What Python Stores

It is worth being concrete about what lives in memory after the construction line runs, because the recursive picture is easy to misread. The global frame holds only two bindings:

Global frame
Link -> the class object
s    -> the first (outer) Link object

The name s does not hold all three numbers. It holds a reference to the first object only. Each object then points to the next:

first (outer) Link    first -> 3    rest -> middle Link
middle Link           first -> 4    rest -> inner Link
inner (last) Link     first -> 5    rest -> Link.empty

This is the heart of the machine model. The numbers 3, 4, and 5 are not packed together in one place. They are scattered across three separate objects, and the only thing tying them into a sequence is the chain of rest pointers. Every operation on a linked list, whether it is len, indexing, or printing, has to follow those pointers to do its job.

Giving Links a Printed Representation

If you print a link before doing anything special, Python falls back to a generic description that looks roughly like an object sitting at a memory address. That is not useful for reading a chain. The original text fixes this by defining a function that turns a linked list into a string shaped like the constructor call that would rebuild it.

Before installing it anywhere, the original text first defines link_expression and calls it directly to see what it produces. Notice that what comes back is an ordinary str, which is why the interpreter shows it wrapped in quotation marks:

>>> def link_expression(s):
...     if s.rest is Link.empty:
...         rest = ''
...     else:
...         rest = ', ' + link_expression(s.rest)
...     return 'Link({0}{1})'.format(s.first, rest)
...
>>> link_expression(s)
'Link(3, Link(4, Link(5)))'

The surrounding single quotes are the tell: link_expression(s) is a function that returns a string value, exactly like any expression that evaluates to a str. That is different from print(...), which has no value of its own and instead writes characters to the screen with no quotes. Holding onto that distinction matters in the next step, because we are about to hand this string-returning function to Python as a link's official printed form.

# (4)
def link_expression(s):
    if s.rest is Link.empty:
        rest = ''
    else:
        rest = ', ' + link_expression(s.rest)
    return 'Link({0}{1})'.format(s.first, rest)


Link.__repr__ = link_expression

print(s)
print(s.rest)

Expected output:

Link(3, Link(4, Link(5)))
Link(4, Link(5))

You learned earlier that print uses __str__. When a class defines __repr__ but no __str__, as Link does, Python falls back to __repr__ for printing too, so print(s) now shows our constructor-shaped form.

The function is itself recursive. For the last link, whose rest is the empty sentinel, it uses an empty string and the result is just Link(5). For any other link it prefixes ', ' and recursively calls link_expression on the rest, so the inner constructor calls nest exactly the way the objects do.

Anatomy: rebinding the representation method

Link.__repr__ = link_expression
├─ Link .............. the class object
├─ .__repr__ ......... the method Python uses for the official (repr) form, which print also falls back to when a class has no __str__
├─ = ................. binds that attribute to a new value
├─ link_expression ... the function to use, not called but referenced
└─ installs link_expression as the method Python runs to represent any Link

After this assignment, the function lives on the class as __repr__, so it applies to every Link object, including ones already built. From now on, whenever Python needs the official string form of a link, it calls link_expression. Notice that the right side is the function name with no parentheses: you are storing the function itself, not the result of calling it.

Run interactively, the same example reads:

>>> s
Link(3, Link(4, Link(5)))
>>> s.rest
Link(4, Link(5))

Links Can Contain Links

The first value of a link can be any Python value at all, and that includes another linked list. This is what lets a linked list hold structured data, not just flat numbers.

# (5)
s_first = Link(s, Link(6))

print(s_first.first.first)
print(s_first.first.rest.first)
print(s_first.rest.first)
print(s_first)

Expected output:

3
4
6
Link(Link(3, Link(4, Link(5))), Link(6))

Here s_first is a two-element chain. Its first element is the entire linked list s, and its second element is the number 6. Reading the access paths one dot at a time:

s_first.first              -> the linked list s
s_first.first.first        -> 3 (the head value of s)
s_first.first.rest.first   -> 4 (the second value of s)
s_first.rest.first         -> 6 (the second value of s_first)

The crucial point is identity, not copying. When you write Link(s, Link(6)), Python stores a reference to the very same object that s already names. No copy of the three-element chain is made:

s_first -> outer Link
  first -> the SAME object that s points to
  rest  -> Link(6)

Because s and s_first.first are the same object, length and indexing flow straight through to it:

>>> len(s_first)
2
>>> len(s_first[0])
3
>>> s_first[0][2]
5

len(s_first) is 2 because the outer chain has two elements, s and 6. len(s_first[0]) is 3 because s_first[0] is s, whose chain has three elements. And s_first[0][2] first retrieves s, then walks s to index 2, landing on 5. None of these operations copies anything; they all follow pointers to the one shared linked-list object.

Tracing a Nested Access by Hand

The expression s_first.first.rest.first is the kind of dotted path that is easy to misread, so trace it as a sequence of single hops. Each dot moves you to exactly one attribute of the object you are currently holding.

start:                    s_first        (the outer Link)
.first ->                 s              (the inner linked list)
.rest  ->                 Link(4, Link(5))   (rest of s)
.first ->                 4              (head value of that rest)

Read aloud: "from s_first, go to its first, then to that object's rest, then read that object's first." The path ends at 4. The discipline that prevents mistakes is to never read two dots at once. Hold one object, take one hop, then look again.

⚠Common Beginner Traps

Trap 1: thinking rest is a single value. The rest attribute is the entire remaining chain, not the next number. It is always either Link.empty or another Link object. The assertion in __init__ enforces exactly that: writing Link(1, 2) raises AssertionError, because 2 is neither the empty sentinel nor a Link.

Trap 2: forgetting the empty sentinel. Every chain needs an endpoint. In this implementation it is Link.empty, an empty tuple shared on the class. Without it, __len__ and __getitem__ would have nothing to stop at and would recurse forever.

Trap 3: thinking nested constructor calls run left to right. For Link(3, Link(4, Link(5))), Python must finish building Link(5) before it can build the link that points to it, so the chain is assembled inside out even though you read it outside in.

Trap 4: thinking s_first.first copies s. It does not. It stores a reference to the same linked-list object. Changes seen through one name would be visible through the other, because there is only one object.

Trap 5: expecting indexing to be a jump. s[2] does not leap to a slot. It walks the chain link by link via __getitem__, which is why reaching a later index costs more than reaching an earlier one.

✓Check Your Understanding

  1. For t = Link(2, Link(4, Link(6, Link(8)))), what does t[2] evaluate to, and how many times does __getitem__ run to produce it?
  2. What does Link.empty represent, and why is it stored on the class rather than on each instance?
  3. When you evaluate s[2], why does it call __getitem__ more than once?
  4. What does the line Link.__repr__ = link_expression change, and why is there no pair of parentheses after link_expression?
  5. After s_first = Link(s, Link(6)), why does s_first.first.rest.first evaluate to 4?
Show the answers
  1. t[2] is 6. Index 2 is not 0, so the head delegates to t.rest[1], which delegates again to t.rest.rest[0], which returns that link's first. __getitem__ runs three times in all.
  2. Link.empty is the end-of-chain marker, an empty tuple. It lives on the class so every link shares one single end marker rather than each carrying its own copy.
  3. __getitem__ answers index 0 directly but delegates any larger index to the rest of the chain with index i - 1. So s[2] asks the rest for index 1, which asks the next rest for index 0, which finally returns a first value. Each delegation is one more call.
  4. It installs link_expression as the class's __repr__, so Python uses it to produce the printed form of every Link. There are no parentheses because you are binding the function itself to __repr__, not calling it and binding its result.
  5. Follow one dot at a time: s_first.first is the linked list s, s.rest is the chain Link(4, Link(5)), and that chain's first is 4.

§Summary

A recursive object is an object that can contain another object of the same kind, and the linked list is the cleanest example. The Link class makes each list element a small object holding two things: a first value and a rest that is either the shared Link.empty sentinel or another Link. Because the rest has the same shape as the whole, both the structure and the methods over it are recursive. __len__ adds one and recurses into the rest, and __getitem__ answers index zero directly while delegating larger indices to the rest, which means indexing is a walk down the chain rather than a jump. Defining __repr__ as link_expression gives links a readable constructor-shaped printout. And because first can be any value, a linked list can hold another linked list; when it does, it stores a shared reference, not a copy, so length and indexing flow straight through to the same underlying object.

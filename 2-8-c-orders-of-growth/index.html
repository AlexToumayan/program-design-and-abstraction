Chapter 2/2.8 Efficiency

2.8 Orders of Growth

◷ approx. 18 min read · 3 worked examples · best read in two passes

Two programs can solve the exact same problem and still behave completely differently as the input grows: one stays fast forever, the other slows to a crawl. By the end of this lesson you will be able to do four things: describe what an order of growth is and why it matters more than a stopwatch, read the count_factors function and count how much work it does, write down a growth claim using Theta notation, and prove a Theta claim by sandwiching the real work between two simple bounds.

You do not need any new Python syntax for this. What you need is a way of thinking: stop asking "how many seconds did this take," and start asking "what shape does the work follow as the input gets large." That second question gives an answer that stays true on any machine, this year or in ten years.

A Scaling Analogy

Imagine you run the cleaning crew for a school, and the number of students keeps changing. You can pick a cleaning plan, and each plan reacts differently as the school grows.

Plan A: no matter how many students enroll, clean one front office.
Plan B: for each student, wipe down that student's one desk.
Plan C: for each student, walk them past every other student to compare lockers.

Plan A never gets harder. Whether the school has 50 students or 50,000, the work is the same: one office. That is constant work.

Plan B gets harder in step with the school. Twice the students means twice the desks. That is linear work: the work tracks the input one-for-one.

Plan C explodes. Adding a single new student forces a new comparison against everyone already there, so the work grows far faster than the head count. That is quadratic work.

The point of this lesson is to give precise names to these patterns, so that when you look at a piece of code you can say which pattern it follows before you ever run it. An order of growth is the shape of that curve, not a measurement in seconds.

A Program That Counts Factors

The original text introduces orders of growth with count_factors, a function that counts how many positive integers evenly divide an input n. A factor of n is a number that divides it with no remainder.

For example, the factors of 12 are:

1, 2, 3, 4, 6, 12

There are six of them, so count_factors(12) should return 6.

The clever part is how the function avoids checking every number from 1 up to n. Factors come in pairs that multiply to n:

1⋅12=12
2⋅6=12
3⋅4=12

Notice that in each pair, one partner is small and one is large, and they meet in the middle near the square root of n. This is the exact insight the original text uses: if k divides n and k<n​, then there is another factor j=n/k such that j>n​. So once you have checked every possible small factor up to n​, every large factor has already been accounted for as a partner. You never have to look past the square root.

A Warm-Up Version You Can Read Slowly

Before the official function, here is a slower but more obvious version that checks every number from 1 to n directly. It is easy to believe it is correct, which makes it a good thing to compare against.

This prints 6. It walks k across every value from 1 to n and adds one each time k divides n evenly. It is correct, but it does roughly n units of work. For n = 1000000 that is a million checks. The official version does the same job in far fewer steps by stopping at the square root and counting both partners of each pair at once.

The Official Version

Following the Lines of the Loop

This function packs a lot into a few lines. Walk through each new line one token at a time. The final row of each block states what that line does.

The first line inside the function computes the boundary that lets us stop early:

sqrt_n = sqrt(n)
├─ sqrt_n ...... the name we are binding
├─ = ........... assignment: bind the name on the left
├─ sqrt ........ the imported square-root function
├─ ( ........... begins the inputs
├─ n ........... the input whose square root we want
├─ ) ........... ends the inputs
└─ binds sqrt_n to the square root of n, the point where factor pairs meet

The next line creates two names at once with a single assignment:

k, factors = 1, 0
├─ k ........... the candidate factor we will test, starting low
├─ , ........... separates the names on the left
├─ factors ..... the running count of factors found
├─ = ........... assignment: bind both names at once
├─ 1 ........... the first value, bound to k
├─ , ........... separates the values on the right
├─ 0 ........... the second value, bound to factors
└─ binds k to 1 and factors to 0 in a single step

The loop header decides how many times the body repeats:

while k < sqrt_n:
├─ while ....... repeat the body as long as the test is true
├─ k ........... the current candidate factor
├─ < ........... the less-than comparison
├─ sqrt_n ...... the square-root boundary
├─ : ........... begins the indented loop body
└─ repeats the body while k is still below the square root of n

Inside the loop, the if checks whether the current k divides n:

if n % k == 0:
├─ if .......... run the body only when the test is true
├─ n ........... the number being divided
├─ % ........... the remainder operator
├─ k ........... the candidate factor
├─ == .......... the equality comparison
├─ 0 ........... the value zero
├─ : ........... begins the indented if body
└─ runs the next line only when k divides n with no remainder

When k is a factor, the count jumps by two, not one, because the partner is counted at the same time:

factors += 2
├─ factors ..... the running count of factors
├─ += .......... add to the name and rebind it
├─ 2 ........... the amount to add
└─ counts both k and its partner n / k in one move

After each candidate, k steps up to the next integer:

k += 1
├─ k ........... the current candidate factor
├─ += .......... add to the name and rebind it
├─ 1 ........... the amount to add
└─ advances k to the next candidate factor

After the loop, one line handles a special case that pairs would otherwise miss:

if k * k == n:
├─ if .......... run the body only when the test is true
├─ k ........... the value k stopped at, near the square root
├─ * ........... the multiplication operator
├─ k ........... the same value again
├─ == .......... the equality comparison
├─ n ........... the original input
├─ : ........... begins the indented if body
└─ checks whether n is a perfect square, whose middle factor has no distinct partner

When n is a perfect square, its middle factor is its own partner, so it is counted exactly once:

factors += 1
├─ factors ..... the running count of factors
├─ += .......... add to the name and rebind it
├─ 1 ........... the amount to add
└─ counts the single middle factor of a perfect square

Finally the function hands its answer back:

return factors
├─ return ...... send a value back to the caller
├─ factors ..... the final count to return
└─ returns the total number of factors of n

Two operators here are worth a closer look. The % operator gives the remainder after division, so n % k == 0 is true exactly when k divides n cleanly. And += is shorthand: factors += 2 means "take the current value of factors, add 2, and rebind factors to the result." The colon at the end of while and if tells Python that an indented block follows, and the indentation marks which lines belong to that block.

Tracing the Work by Hand

To see why the loop count matters, trace count_factors(12) step by step. First, sqrt(12) is about 3.464, so sqrt_n is roughly 3.464. The loop runs while k < 3.464, meaning k takes the values 1, 2, and 3, then stops.

step k before n % k == 0? action factors after
1 1 12 % 1 == 0, yes factors += 2 2
2 2 12 % 2 == 0, yes factors += 2 4
3 3 12 % 3 == 0, yes factors += 2 6
stop 4 4 < 3.464 is false exit loop 6

After the loop, k is 4. The final check asks whether k * k == n, that is 16 == 12, which is false, so nothing is added. The function returns 6, which matches the six factors we listed by hand. Each factors += 2 quietly counted a pair: 1 paired with 12, 2 paired with 6, and 3 paired with 4.

Now picture the same trace for count_factors(576). Here sqrt(576) is exactly 24.0, so the loop runs for k from 1 up to 23, then exits when k reaches 24 because 24 < 24.0 is false. At that point k * k == n is 576 == 576, which is true, so the middle factor 24 is counted once. The frame has the same shape as the 12 trace, but k must travel much farther: about two dozen iterations instead of three. That distance, the number of values k must visit, is the thing an order-of-growth analysis measures.

Counting the Work

How much time does it take to evaluate count_factors? The exact answer depends on the machine, but we can say something durable about the amount of computation involved. The body of the while statement runs once for each value of k below the square root, so it runs the greatest integer less than n​ times. The statements before and after the loop each run exactly once.

If we let w be the number of statements inside the while body and v be the number of statements outside it, then the total number of statements executed is roughly:

w⋅n​+v

This formula is not exact, but it captures the idea: the dominant term scales with n​, while v is a fixed startup cost that does not grow.

A more exact description is hard to pin down, and that is the whole motivation for what comes next. The constants w and v are not really constant, because the assignment to factors sometimes runs and sometimes does not, depending on whether the current k is a factor. Rather than chase those details, an order-of-growth analysis lets us gloss over them and focus on the general shape of growth. For count_factors, that shape is square-root growth: the time to compute count_factors(n) scales at the rate n​, within a margin of some constant factors.

Theta Notation

To make "scales at the rate of" precise, we use Theta notation. Let n be a parameter that measures the size of the input to some process, and let R(n) be the amount of some resource the process requires for an input of size n.

So far we have taken n to be the number a function is computing with, but it does not have to be. If the goal were to compute an approximation to a square root, for instance, n might instead be the number of digits of accuracy required. The resource R(n) is just as flexible. It might measure any of these:

R(n) could measure the amount of memory used.
R(n) could measure the number of elementary machine steps performed.
R(n) could measure the number of function calls made.
R(n) could measure the number of active frames at once.

On a machine that does a fixed number of steps at a time, the time required to evaluate an expression is proportional to the number of elementary steps performed during evaluation, so counting steps is a reasonable stand-in for counting time.

We say that R(n) has order of growth Θ(f(n)), written R(n)=Θ(f(n)) and pronounced "theta of f(n)," if there are positive constants k1​ and k2​, independent of n, such that

k1​⋅f(n)≤R(n)≤k2​⋅f(n)

for any value of n larger than some minimum m. In other words, once n is large enough, the value R(n) is always sandwiched between two values that both scale with f(n): a lower bound k1​⋅f(n) and an upper bound k2​⋅f(n).

Read in plain language, R(n)=Θ(f(n)) says that R(n) grows at the same general rate as f(n), allowing constant-factor differences. This is not a prediction of seconds on a particular laptop. It is a more durable claim about shape: when the input gets much larger, what curve does the work follow? If one implementation takes about 3n+20 simple steps and another takes about 100n+5 simple steps, both are still linear, both are Θ(n). The constants matter in real engineering, but the category tells us the long-run shape, and that is what survives a change of hardware.

The Sandwich Picture

The two bounds in the definition act like guard rails. Think of R(n) as a curve that, once n is large enough, is forced to stay between them.

upper rail:  k2 * f(n)   <- R(n) never rises above this
actual work: R(n)        <- the real resource use, trapped in between
lower rail:  k1 * f(n)   <- R(n) never falls below this

Two features of this picture matter. First, the rails are both constant multiples of the same f(n), so they have the same shape, just stretched by different amounts. That is what makes Theta a statement about shape rather than exact size. Second, the rails only have to trap R(n) after n passes the minimum m. For small inputs, anything goes. This is exactly why startup costs and tiny-input quirks do not affect an order-of-growth claim: the claim is about what happens once the input is large.

Proving the Bound for count_factors

We can use the definition directly to show that the number of steps required to evaluate count_factors(n) grows as Θ(n​), just by inspecting the function body. We need a lower bound and an upper bound.

For the lower bound, choose k1​=1 and m=0, so the claim is that count_factors(n) requires at least 1⋅n​ steps for any n>0. There are at least 4 lines executed outside the while statement, each taking at least 1 step. Inside the loop there are at least two lines plus the while header itself, and each of those takes at least one step. We deliberately undercount here to keep the bound a guaranteed minimum, skipping lines like factors += 2 that do not always run; the upper bound below does the opposite and overcounts by assuming every conditional line fires, which is why its line counts come out slightly higher. The while body is evaluated at least n​−1 times. Composing these, the process requires at least

4+3⋅(n​−1)

steps. Multiplying out, 4+3(n​−1)=1+3n​, which is comfortably larger than 1⋅n​, so the lower bound holds with k1​=1.

For the upper bound, assume that any single line in the body of count_factors requires at most p steps. This assumption is not true for every line of Python in general, but it does hold here. Some Python lines can hide a lot of work, for instance an operation that touches every element of a huge list, but every line here does just one fixed-size arithmetic or comparison step, so a single constant p safely caps all of them. Then evaluating count_factors(n) can require at most

p⋅(5+4n​)

steps, because there are 5 lines outside the while statement and 4 within it, counting the header. This bound holds even if every if header turns out to be true. Now choose k2​=5p, and the steps required are always smaller than k2​⋅n​.

We have produced constants k1​, k2​, and m that satisfy the definition, so the argument is complete:

R(n)=Θ(n​)

This does not mean every run takes exactly n​ steps. It means the dominant growth shape is square-root growth, and no machine's constant factors can change that category.

Why Square Root Is a Big Win

Square-root growth is dramatically better than checking every number up to n. The reason is how the two react when the input gets much larger. If you multiply n by 100, then n​ only grows by a factor of 10.

n n​ approximate loop iterations
100 10 about 10
10,000 100 about 100
1,000,000 1,000 about 1,000

Compare that last row to the warm-up version from earlier: at one million, the slow function does about a million checks, while count_factors does about a thousand. Same answer, a thousand times less work, and the gap only widens as n grows. That is the practical payoff of caring about orders of growth.

⚠Common Beginner Traps

Trap 1: Treating Theta as exact timing. The claim Θ(n​) does not tell you how many seconds the program takes or how many steps a specific run uses. It tells you how the required work scales as the input grows.

Trap 2: Forgetting that constant factors are allowed. If one machine performs twice as many primitive steps per line as another, the growth category is unchanged. Both Θ(n) and a program that does 500n steps are still Θ(n). The whole point of the constants k1​ and k2​ is to absorb those differences.

Trap 3: Trusting small inputs. Startup work and one-time costs can dominate tiny examples, which is exactly why the definition only requires the bounds to hold for n larger than some minimum m. Growth analysis describes large inputs, not the first few.

Trap 4: Confusing n with n​. The input size is still n. What grows like n​ is the loop count, the number of candidate factors the function examines. The function is fast precisely because those two are different.

Trap 5: Forgetting the perfect-square case. The loop counts factors in pairs with factors += 2. When n is a perfect square, its middle factor pairs with itself and would be double-counted, so the loop stops just before it and the final if k * k == n adds it back exactly once.

✓Check Your Understanding

  1. Why does count_factors stop the loop near n​ instead of going all the way to n?
  2. What does the test n % k == 0 check, and why does a true result trigger factors += 2?
  3. Why does the perfect-square case at the end add only 1 to factors?
  4. Trace count_factors(49). How many times does the while body run, what value does factors hold at the moment the loop exits but before the final if, and what does the function return?
  5. State in plain language what R(n)=Θ(n​) means for count_factors.
  6. In the upper-bound proof, why is it valid to assume each line costs at most p steps, and how is k2​ then chosen?
Show the answers
  1. Factors come in pairs whose partners meet at n​: if k<n​ divides n, its partner n/k is greater than n​. After checking every candidate up to the square root, every larger factor has already been counted as a partner, so there is nothing left to find.
  2. It checks whether k divides n with no remainder, which means k is a factor. When it is, the function adds 2 because it has found both k and its distinct partner n / k at the same time.
  3. For a perfect square, the square-root factor is its own partner. Counting it with the pair logic would count the same factor twice, so the loop excludes it and the final if adds it exactly once.
  4. Since 49​=7.0, the loop runs while k < 7.0, so the body runs six times for k equal to 1, 2, 3, 4, 5, 6. Only k = 1 divides 49, so factors is 2 when the loop exits with k at 7. Then k * k == n is 49 == 49, which is true, so the middle factor 7 is added once and the function returns 3 (the factors 1, 7, 49).
  5. It means the number of steps count_factors(n) performs grows at the same general rate as n​ once n is large, ignoring constant factors and small startup costs. Multiplying n by 100 multiplies the work by about 10.
  6. Each line of count_factors performs a bounded amount of work, so some constant p caps the cost of any single line. With 5 lines outside the loop and 4 within it running about n​ times, the total is at most p⋅(5+4n​) steps; choosing k2​=5p makes this fit under k2​⋅n​.

§Summary

An order of growth describes the shape of a program's resource use as the input grows, which is a more durable question than how many seconds it takes on one machine. The count_factors function makes this concrete: by stopping its loop at n​ and counting factor pairs, it examines only about n​ candidates instead of n. We made the claim precise with Theta notation, where R(n)=Θ(f(n)) means positive constants k1​ and k2​ sandwich R(n) between k1​⋅f(n) and k2​⋅f(n) for all large enough n. Inspecting the function body, a lower bound of 4+3(n​−1) steps and an upper bound of p⋅(5+4n​) steps prove that count_factors(n) is Θ(n​). The constants and startup costs wash out; the square-root shape is what remains.

▶Step-Through Checkpoint

This is the same function whose growth we just proved is Θ(n​), run here on 36 so the entire process fits in a handful of steps. Stepping through it draws the environment diagram of an iterative process: a single count_factors frame stays open the whole time, and you watch k, factors, and sqrt_n rebind in place as the loop turns, so the only growing quantity is the number of times the body fires, the n​-shaped count, not any pile of stacked frames. Before you step through the block below, write down three predictions: how many times the while body will run, what factors holds at the moment the loop exits but before the final if runs, and what value result is bound to when the last line finishes.

Step through it one line at a time and check each prediction as the bindings change. Inside the frame for the call, sqrt_n is bound to 6.0, so the body runs five times, taking k through 1, 2, 3, 4, 5 and stopping when 6 < 6.0 is false. Along the way, factors rises by two each time k divides 36: 1 pairs with 36, 2 with 18, 3 with 12, and 4 with 9, while 5 is skipped because 36 % 5 is not 0, leaving factors at 8 when the loop exits. Those five trips through the body, instead of thirty-six, are the n​-shaped quantity this whole lesson measures.

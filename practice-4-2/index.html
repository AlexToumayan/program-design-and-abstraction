4.2 · Iterators and Generators — Coding Practice

◷ approx. 12 min read · 31 worked examples

These problems build fluency with implicit sequences: driving an iterator by hand with iter and next, catching StopIteration, writing generator functions that yield values one at a time (including infinite ones you sample with a bounded helper), building your own iterator and iterable classes, and forcing values out of a lazily computed Stream.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Drain an Iterator

Examples

Input:  drain([1, 2, 3, 4])
Output: 10
Explanation: 1 + 2 + 3 + 4, accumulated by pulling next until StopIteration.

Input:  drain([10])
Output: 10
Explanation: one value, then the iterator is exhausted.

Input:  drain([])
Output: 0
Explanation: the very first next raises StopIteration, so the total stays 0.

  • Drive the walk yourself with iter and next; catch StopIteration with a try statement (do not use a for loop or sum).
  • Values are numbers; the input may be any iterable, including a range.

Hint

Solutions

Setup

These problems use two bounded helpers (so tests can sample infinite generators and streams) and the lazy Stream class:

def take(iterable, k):
    """Return a list of the first k values produced by an iterable."""
    it = iter(iterable)
    result = []
    while k > 0:
        result.append(next(it))
        k = k - 1
    return result


class Stream:
    """A lazily computed linked list."""
    class empty:
        def __repr__(self):
            return 'Stream.empty'
    empty = empty()
    def __init__(self, first, compute_rest=lambda: empty):
        assert callable(compute_rest), 'compute_rest must be callable.'
        self.first = first
        self._compute_rest = compute_rest
    @property
    def rest(self):
        if self._compute_rest is not None:
            self._rest = self._compute_rest()
            self._compute_rest = None
        return self._rest


def first_k(s, k):
    result = []
    while s is not Stream.empty and k > 0:
        result.append(s.first)
        s, k = s.rest, k - 1
    return result

1. Drain an Iterator

Approach — drive the iterator protocol by hand. This is the for-loop machinery written out: get an iterator with iter, pull values with next, and treat StopIteration as the stop signal — exactly the contract from this section.

def drain(values):
    it = iter(values)
    total = 0
    while True:
        try:
            total = total + next(it)
        except StopIteration:
            return total

Why it works. Each next(it) returns one value and advances the iterator's position; when the values run out, next raises StopIteration, and the except block returns the total accumulated so far. An empty input raises on the very first next, so the total is still 0.

Complexity. Time O(n), Space O(1). One pass over the values; only a running total is kept.

assert drain([1, 2, 3, 4]) == 10
assert drain([10]) == 10
assert drain([]) == 0
assert drain(range(1, 5)) == 10

2. Count Up to a Limit

Approach — generator with a counter. A yield statement turns this into a generator function: calling it builds a paused generator, and each next resumes the while loop to produce the next integer.

def count_up(start, stop):
    current = start
    while current < stop:
        yield current
        current = current + 1

Why it works. The body runs straight-line code, pausing at yield current to hand back one value and resuming right after it on the next request. When current reaches stop the while condition fails, the function returns, and the generator raises StopIteration, which list reads as the end.

Complexity. Time O(n) for n = stop - start, Space O(1). Values are produced one at a time; no list is held inside the generator.

assert list(count_up(0, 4)) == [0, 1, 2, 3]
assert list(count_up(5, 8)) == [5, 6, 7]
assert list(count_up(3, 3)) == []
assert next(count_up(10, 20)) == 10

3. Repeat Each Value

Approach — nested loop in a generator. The outer for walks the values; an inner while yields each one times times. Because the body uses yield, the whole thing is a lazy producer.

def repeat_each(values, times):
    for value in values:
        count = 0
        while count < times:
            yield value
            count = count + 1

Why it works. For each value, the inner loop yields it once per pass until count reaches times, then the outer loop moves on. When times is 0 the inner condition is false immediately, so that value is skipped entirely.

Complexity. Time O(n⋅t) for n values and t = times, Space O(1). One value held at a time; output size drives the time.

assert list(repeat_each([1, 2], 3)) == [1, 1, 1, 2, 2, 2]
assert list(repeat_each(['a'], 2)) == ['a', 'a']
assert list(repeat_each([7, 8, 9], 1)) == [7, 8, 9]
assert list(repeat_each([1, 2], 0)) == []

4. Second Value of an Iterable

Approach — one iterator, two next calls. A single iterator carries a position that advances on every next, so two calls return two different values: the first discarded, the second returned.

def second(values):
    it = iter(values)
    next(it)
    return next(it)

Why it works. iter(values) creates one iterator positioned before the first value. The first next(it) consumes the first value and moves the position forward; the second next(it) starts from that later position and returns the second value. Indexing is avoided, so non-indexable iterables like ranges and strings work too.

Complexity. Time O(1), Space O(1). Only two values are ever pulled.

assert second([10, 20, 30]) == 20
assert second('python') == 'y'
assert second(range(100, 200)) == 101
assert second([1, 2]) == 2

5. Running Maximum

Approach — generator carrying a best-so-far. One variable remembers the largest value seen; the generator yields it after each new input, so the result never decreases.

def running_max(values):
    best = None
    for value in values:
        if best is None or value > best:
            best = value
        yield best

Why it works. best starts at None; the first element always replaces it (the best is None test), and later elements replace it only when they are strictly larger. The single yield best per pass produces the running maximum lazily. An empty input never enters the loop, so nothing is yielded.

Complexity. Time O(n), Space O(1). One pass; a single best-so-far variable.

assert list(running_max([3, 1, 4, 1, 5, 9, 2, 6])) == [3, 3, 4, 4, 5, 9, 9, 9]
assert list(running_max([5, 4, 3])) == [5, 5, 5]
assert list(running_max([1])) == [1]
assert list(running_max([])) == []

6. Flatten a Nested List

Approach — recursive generator. For an atom, yield it directly; for a sublist, loop over a recursive call and re-yield each value it produces, descending to any depth.

def flatten(nested):
    for element in nested:
        if type(element) == list:
            for value in flatten(element):
                yield value
        else:
            yield element

Why it works. Looping for value in flatten(element): yield value drives the recursive generator and forwards each of its values to the caller, so deeply nested lists are expanded one atom at a time, left to right. Because it is a generator, values stream out lazily — you can take a prefix without flattening the whole structure.

Complexity. Time O(n) in the total number of elements and sublist nodes, Space O(d) for nesting depth d. Each node is visited once; depth bounds the generator call stack.

assert list(flatten([1, [2, 3], 4])) == [1, 2, 3, 4]
assert list(flatten([[1, [2, [3]]], 4, [5, 6]])) == [1, 2, 3, 4, 5, 6]
assert list(flatten([])) == []
assert list(flatten([[], [1], [], [2, 3]])) == [1, 2, 3]
assert take(flatten([[1, 2], [3, 4], [5, 6]]), 3) == [1, 2, 3]

7. Multiples Generator

Approach — unbounded generator. A while True loop with yield describes an endless sequence; the consumer decides how far to go, here with the take helper.

def multiples(n):
    current = n
    while True:
        yield current
        current = current + n

Why it works. The generator never reaches an end condition, so it keeps yielding n, 2n, 3n, ... — but no work happens until a value is requested. take(multiples(n), k) pulls exactly k values and stops, which is why an infinite generator is safe to use as long as you never force all of it.

Complexity. Time O(k) to take k values, Space O(1). Each value is one addition; only current is stored.

assert take(multiples(3), 5) == [3, 6, 9, 12, 15]
assert take(multiples(7), 3) == [7, 14, 21]
assert take(multiples(1), 4) == [1, 2, 3, 4]
gen = multiples(10)
assert next(gen) == 10
assert next(gen) == 20

8. Countdown Iterator Class

Approach — a custom iterator class. The class implements the iterator interface directly: __next__ returns one value and advances an attribute, raising StopIteration at the end, and __iter__ returns self so a for loop (and list) can drive it.

class Countdown:
    def __init__(self, start):
        self.current = start

    def __next__(self):
        if self.current <= 0:
            raise StopIteration
        value = self.current
        self.current = self.current - 1
        return value

    def __iter__(self):
        return self

Why it works. self.current is the mutable position. Each __next__ saves the value, decrements the attribute, and returns the saved value; once self.current hits 0 the guard raises StopIteration. Returning self from __iter__ makes the iterator iterable, which is why it works in for/list — and why a single instance is consumed after one pass.

Complexity. Time O(start) to exhaust, Space O(1). One attribute tracks the position.

assert list(Countdown(4)) == [4, 3, 2, 1]
assert list(Countdown(1)) == [1]
assert list(Countdown(0)) == []
c = Countdown(3)
assert next(c) == 3
assert next(c) == 2
# An iterator is consumed: a second pass over the same object is empty.
drained = Countdown(2)
assert list(drained) == [2, 1]
assert list(drained) == []

9. Reusable Stepped Range

Approach — separate iterable and iterator classes. StepRange is the iterable (it stores only the rules and builds a fresh iterator on each __iter__); StepRangeIter is the iterator (it carries the moving position in __next__). This split is exactly the iterable-vs-iterator distinction from this section, and it makes the range reusable.

class StepRangeIter:
    def __init__(self, start, stop, step):
        self.current = start
        self.stop = stop
        self.step = step

    def __next__(self):
        if self.current >= self.stop:
            raise StopIteration
        value = self.current
        self.current = self.current + self.step
        return value

    def __iter__(self):
        return self

class StepRange:
    def __init__(self, start, stop, step):
        self.start = start
        self.stop = stop
        self.step = step

    def __iter__(self):
        return StepRangeIter(self.start, self.stop, self.step)

Why it works. Each iter(step_range) constructs a new StepRangeIter starting at start, so every traversal begins fresh and independently — two iterators advance separately, and a second list(...) sees the full sequence again. The iterator's __next__ raises StopIteration as soon as self.current >= self.stop. Because StepRange is iterable, it slots into sum, map, and for unchanged.

Complexity. Time O((stop−start)/step) per traversal, Space O(1) per iterator. Each iterator holds three numbers.

r = StepRange(0, 10, 2)
assert list(r) == [0, 2, 4, 6, 8]
# Reusable: each pass starts fresh because __iter__ builds a new iterator.
assert list(r) == [0, 2, 4, 6, 8]
assert list(StepRange(1, 8, 3)) == [1, 4, 7]
assert list(StepRange(5, 5, 1)) == []
# Two iterators from one iterable advance independently.
a = iter(r)
b = iter(r)
assert next(a) == 0
assert next(a) == 2
assert next(b) == 0
# Works anywhere an iterable works.
assert sum(StepRange(0, 10, 2)) == 20
assert list(map(lambda x: x * x, StepRange(1, 5, 1))) == [1, 4, 9, 16]

10. Lazy Stream of Triangular Numbers

Approach — an infinite stream with a deferred rest. Like integer_stream from this section, each node stores its value eagerly and hides the recursive call inside a lambda, so only the part of the stream you force is ever built. A small build(n, total) helper carries the running sum.

def triangulars():
    def build(n, total):
        return Stream(total, lambda: build(n + 1, total + n + 1))
    return build(1, 1)

Why it works. build(n, total) returns one node whose first is the current triangular number total; its compute_rest is a lambda that, when forced, calls build(n + 1, total + n + 1) — adding the next integer n + 1 to the running total. Because the recursive call lives inside the lambda, triangulars() returns immediately and the next node is built only when .rest is accessed. The Stream.rest property caches each forced node, so traversing the stream twice yields the same values.

Complexity. Time O(k) to read k terms, Space O(k) for the forced-and-cached prefix. Each forced node is one addition; cached nodes persist.

t = triangulars()
assert first_k(t, 6) == [1, 3, 6, 10, 15, 21]
# The stream is not consumed: forcing caches, so a second pass is identical.
assert first_k(t, 6) == [1, 3, 6, 10, 15, 21]
assert t.first == 1
assert t.rest.first == 3
assert t.rest.rest.first == 6
assert first_k(triangulars(), 1) == [1]
assert first_k(triangulars(), 10) == [1, 3, 6, 10, 15, 21, 28, 36, 45, 55]

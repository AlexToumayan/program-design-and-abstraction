2.1 · Native Data Types — Coding Practice

◷ approx. 11 min read · 30 worked examples

These problems drill the core ideas of Section 2.1: naming the native types (int, float, complex, bool, str, NoneType), asking type what category a value belongs to, writing type predicates and boolean comparisons, knowing why float equality is a trap, and treating strings as values you can convert, concatenate, measure with len, format, and transform with string methods. Every solution is a short def — no sequences, indexing, or loops are needed yet.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Type Name

Define type_name(value) that returns the name of value's type as a plain string — "int", "float", "str", and so on. This is the lesson's habit ("before you trust a value, ask what type it is") packaged into a function. The type object knows its own name through its .__name__ attribute.

Examples

Input:  type_name(7)
Output: 'int'
Explanation: 7 is a whole number, so its type is int.
Input:  type_name("7")
Output: 'str'
Explanation: the quotation marks make it text, so the type is str, not int.
Input:  type_name(1 + 2j)
Output: 'complex'
Explanation: the j suffix marks an imaginary part, so the value is complex.

Constraints

  • value may be any single Python value.
  • Return the type's name as a string, not the <class '...'> object itself.

Starter

def type_name(value):
    pass
Hint

type(value) gives the class; every class carries its short name in .__name__.

  • value may be any single Python value.

10. Numbers vs. Text

The lesson's sharpest trap is that "4" + "5" is "45" while 4 + 5 is 9 — the same + symbol, completely different behavior, decided by the types. Define two functions over two numeric strings: add_as_numbers(a, b) converts both to int and adds them, while join_as_text(a, b) concatenates them as text. The pair makes the difference impossible to miss.

Examples

Input:  add_as_numbers("4", "5")
Output: 9
Explanation: int("4") + int("5") = 4 + 5 = 9.
Input:  join_as_text("4", "5")
Output: '45'
Explanation: "+" on two strings glues them end to end.
Input:  add_as_numbers("12", "30")
Output: 42
Explanation: converting first gives 12 + 30 = 42, not "1230".

Constraints

  • a and b are strings that contain only digits (no sign, no decimal point).
  • add_as_numbers must convert each with int(...) before adding; join_as_text must concatenate the strings unchanged.

Starter

def add_as_numbers(a, b):
    pass

def join_as_text(a, b):
    pass
Hint

Convert with int(...) to add as numbers; leave the strings alone to join as text.

Solutions

1. Type Name

Approach — ask type, then read its name. Section 2.1 introduces type as the function that reports a value's class. The class object also stores its own short name in .__name__, so type(value).__name__ turns the <class 'int'> answer into the plain string "int".

def type_name(value):
    return type(value).__name__

Why it works. type(7) is the class int; int.__name__ is the string "int". The same chain names every native type — float, str, bool, complex, and NoneType for None.

Complexity. Time O(1), Space O(1). One type lookup and one attribute read.

assert type_name(7) == "int"
assert type_name(7.0) == "float"
assert type_name("7") == "str"
assert type_name(True) == "bool"
assert type_name(1 + 2j) == "complex"
assert type_name(None) == "NoneType"

2. Is It a Whole Number?

Approach — compare the type, not the value. The lesson stresses that 2, 2.0, and "2" are three different objects even though they look alike. Checking type(value) == int asks the category question directly, so a float or a str that merely resembles a whole number is correctly rejected.

def is_whole_number(value):
    return type(value) == int

Why it works. type(5) is int, so the comparison is True; type(5.0) is float and type("5") is str, so both are False. Even True, whose type is bool rather than int, gives False — the predicate looks at the stored type, nothing else.

Complexity. Time O(1), Space O(1). One type call and one comparison.

assert is_whole_number(5) == True
assert is_whole_number(-12) == True
assert is_whole_number(5.0) == False
assert is_whole_number("5") == False
assert is_whole_number(True) == False

3. Label the Sign

Approach — comparisons feeding a branch. Each comparison (n > 0, n < 0) evaluates to a bool, exactly the truth values the lesson lists among the native types. if/elif/else reads those booleans in order and returns the matching word, so every number lands in exactly one of the three cases.

def label_sign(n):
    if n > 0:
        return "positive"
    elif n < 0:
        return "negative"
    else:
        return "zero"

Why it works. A number is either greater than, less than, or equal to zero. The first two tests catch the first two cases; whatever reaches the else must equal zero, which covers both 0 and 0.0.

Complexity. Time O(1), Space O(1). At most two comparisons.

assert label_sign(7) == "positive"
assert label_sign(-3.5) == "negative"
assert label_sign(0) == "zero"
assert label_sign(0.0) == "zero"
assert label_sign(0.0001) == "positive"

4. Shout It

Approach — a string method plus concatenation. Strings are values you can transform: .upper() returns a new all-caps copy, and + joins two strings end to end (the same concatenation the lesson contrasts with numeric addition). Combining them builds the shouted word.

def shout(word):
    return word.upper() + "!"

Why it works. "go".upper() produces "GO", and concatenating "!" gives "GO!". Digits and other non-letters are unaffected by .upper(), so "cs61a" becomes "CS61A" before the "!" is added.

Complexity. Time O(n), Space O(n) in the length of word (uppercasing builds a new string).

assert shout("go") == "GO!"
assert shout("Bears") == "BEARS!"
assert shout("cs61a") == "CS61A!"
assert shout("Win") == "WIN!"

5. Close Enough?

Approach — tolerance instead of exact equality. The lesson's float pitfall is that 1 / 3 * 7 * 3 stores as 6.999999999999999, so == 7 is False. Comparing the absolute difference to a tiny tolerance treats values that are mathematically equal but stored slightly apart as the same.

def close_enough(x, y):
    return abs(x - y) < 1e-9

Why it works. abs(x - y) is the gap between the two stored floats; if that gap is below 1e-9, the difference is just representation noise, so the answer is True. Genuinely different values like 1.0 and 1.5 have a gap of 0.5, far above the tolerance.

Complexity. Time O(1), Space O(1). One subtraction, one absolute value, one comparison.

assert close_enough(1 / 3 * 7 * 3, 7) == True
assert close_enough(0.1 + 0.2, 0.3) == True
assert close_enough(7 / 3 * 3, 7.0) == True
assert close_enough(1.0, 1.5) == False
assert (1 / 3 * 7 * 3 == 7) == False

6. Price Tag

Approach — format a float into fixed-point text. Formatting is how you turn a numeric value into a readable string. The :.2f specifier inside an f-string rounds to two decimals and pads with trailing zeros, so any int or float of dollars prints like money.

def price_tag(dollars):
    return f"${dollars:.2f}"

Why it works. f"${dollars:.2f}" evaluates dollars, formats it with exactly two fractional digits, and drops the result after the $. So 3 becomes "$3.00" and 0.1 becomes "$0.10", with the trailing zero shown.

Complexity. Time O(1), Space O(1). One format operation building a short string.

assert price_tag(3) == "$3.00"
assert price_tag(19.5) == "$19.50"
assert price_tag(0.1) == "$0.10"
assert price_tag(1234.5) == "$1234.50"

7. Count the Digits

Approach — convert to a string, then measure it. Without indexing or loops, the way to count digits is to render the number as text with str(...) and take its len(...). This leans on two native-type ideas at once: a number can be converted to a string, and a string has a length.

def digit_count(n):
    return len(str(n))

Why it works. str(100) is the three-character string "100", and len("100") is 3. Because n is a non-negative integer, every character of its string form is a digit, so the length is exactly the digit count.

Complexity. Time O(d), Space O(d) in the number of digits d (the cost of building the string).

assert digit_count(0) == 1
assert digit_count(7) == 1
assert digit_count(100) == 3
assert digit_count(99999) == 5
assert digit_count(2026) == 4

8. Normalize a Name

Approach — chain two string methods. Each string method returns a brand-new string, so you can call the next method on that result. .strip() removes surrounding whitespace; calling .title() on what it returns capitalizes the first letter of each word and lowercases the rest.

def normalize_name(raw):
    return raw.strip().title()

Why it works. For " ada lovelace ", .strip() yields "ada lovelace", and .title() turns that into "Ada Lovelace". Because .title() lowercases every non-initial letter, even all-caps input like "GRACE HOPPER" comes out as "Grace Hopper".

Complexity. Time O(n), Space O(n) in the length of raw (each method scans and rebuilds the string).

assert normalize_name("  ada lovelace ") == "Ada Lovelace"
assert normalize_name("GRACE HOPPER") == "Grace Hopper"
assert normalize_name("alan TURING") == "Alan Turing"
assert normalize_name("katherine johnson") == "Katherine Johnson"

9. Receipt Line

Approach — convert each type, then concatenate. A string can only be +'d onto another string, so the int quantity and the float total must become text first. str(qty) renders the count, and format(total, ".2f") renders the price with two decimals; concatenation stitches the labeled pieces together.

def receipt_line(item, qty, unit_price):
    total = qty * unit_price
    return item + " x" + str(qty) + " = $" + format(total, ".2f")

Why it works. A local name total holds qty * unit_price. The return expression joins the string item, the literal " x", the converted str(qty), the literal " = $", and the formatted total — so ("Coffee", 3, 2.5) becomes "Coffee x3 = $7.50".

Complexity. Time O(n), Space O(n) in the total length of the assembled string.

assert receipt_line("Coffee", 3, 2.5) == "Coffee x3 = $7.50"
assert receipt_line("Bagel", 2, 1.25) == "Bagel x2 = $2.50"
assert receipt_line("Tea", 1, 3.0) == "Tea x1 = $3.00"
assert receipt_line("Muffin", 4, 2.0) == "Muffin x4 = $8.00"

10. Numbers vs. Text

Approach — let the types decide what + means. This is the lesson's "4" + "5" versus 4 + 5 trap turned into two functions. add_as_numbers converts each string to an int so + performs arithmetic; join_as_text leaves them as strings so + performs concatenation. Same operator, opposite results — controlled entirely by type.

def add_as_numbers(a, b):
    return int(a) + int(b)

def join_as_text(a, b):
    return a + b

Why it works. int("4") and int("5") are the numbers 4 and 5, and 4 + 5 is 9. Leaving "4" and "5" as strings makes + glue them into "45". The two functions over the same inputs lay the difference side by side.

Complexity. Time O(n), Space O(n) in the length of the inputs (conversion and concatenation scan the characters).

assert add_as_numbers("4", "5") == 9
assert join_as_text("4", "5") == "45"
assert add_as_numbers("12", "30") == 42
assert join_as_text("12", "30") == "1230"
assert add_as_numbers("100", "0") == 100
assert join_as_text("100", "0") == "1000"

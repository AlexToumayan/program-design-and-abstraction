Chapter 1/1.2 Elements of Programming

1.2 Elements of Programming: Expressions, Calls, Imports, and Evaluation

◷ approx. 16 min read · 16 worked examples · best read in two passes

In the last lesson you learned that every line of Python does one of two jobs: an expression computes a value, and a statement carries out an action. This lesson zooms all the way in on expressions, because expressions are where computation actually happens.

By the end you will be able to do four things: recognize the simplest expressions Python understands, read a call expression (the shape function(arguments)) one piece at a time, bring in extra functions from Python's library with an import statement, and trace a deeply nested expression by hand using the exact procedure the interpreter follows. That last skill is the real prize. Once you can predict the value of a nested expression before you press Enter, you are no longer guessing at Python — you are running it in your head.

The habit from lesson 1.1 still applies: when code looks confusing, slow down and break it into the smallest meaningful pieces.

Three Ways a Language Lets You Build

Think about how you describe directions to a friend. You start with words everyone already knows ("left," "two blocks," "the bakery"). You combine those words into phrases ("turn left two blocks past the bakery"). And you give a long route a single name ("my usual way home") so you never have to spell it out again.

Every powerful programming language gives you those same three abilities, and Python is no exception:

  • Primitive expressions are the built-in pieces you start with, like the number 42. You do not build them; they are simply there.
  • Means of combination let you join pieces into larger expressions, the way 1/2 + 1/4 joins two values with an operator.
  • Means of abstraction let you wrap something complicated in a single name and reuse it as one unit.

In this lesson we work entirely with the first two: the primitive pieces, and the main way Python combines them. Naming things — the third ability — is the heart of the next lessons. Across all three, one thing stays constant: the machine reads exactly the symbols you type, with no common sense filling in the gaps. Your job is to read those symbols the same literal way Python does.

Primitive Expressions: Values That Already Stand Alone

A primitive expression is one Python can evaluate without looking anything up. The most basic kind is a number. When you type a number, you are typing the numerals that represent it in base 10, and Python evaluates them straight to the value:

# (1)
42

There is nothing to combine and no function to call here. The expression in (1) is already a value, so the interpreter just echoes it back:

42

A number with a decimal point is also a primitive expression, and it evaluates to a float, Python's representation for numbers with a fractional part:

# (2)
3.14

The point to hold onto is that primitive expressions are the floor. Everything else in this lesson is built by combining pieces like these.

Combining Values With Operators

Numbers become interesting when you combine them. Joining expressions with a mathematical operator (+, -, *, or /) produces a compound expression, which the interpreter then evaluates to a single value. These use infix notation, meaning the operator sits in between its operands:

1 + 2
├─ 1 .......... the left operand (a primitive value)
├─ + .......... the operator, sitting between its operands
├─ 2 .......... the right operand (a primitive value)
└─ evaluates to  3

Start with a case that looks stranger than it is. Minus signs near each other do not all mean the same thing:

# (3)
-1 - -1

There are three tokens here, and the middle one is doing different work than the outer two. Follow the line:

-1 - -1
├─ -1 ......... the literal value negative one (left operand)
├─ -  ......... the subtraction operator
├─ -1 ......... the literal value negative one (right operand)
└─ evaluates to  0

So Python computes "negative one, minus negative one." Subtracting a negative is the same as adding a positive:

−1−(−1)=−1+1=0

and the interpreter displays:

0

Now a longer compound expression. This one chains several divisions and additions:

# (4)
1/2 + 1/4 + 1/8 + 1/16 + 1/32 + 1/64 + 1/128

Division binds more tightly than addition, so Python computes each fraction first, then adds the seven results:

0.5+0.25+0.125+0.0625+0.03125+0.015625+0.0078125=0.9921875

The displayed value is:

0.9921875

The important machine idea is that Python is not reading this left to right like an English sentence. It parses the whole expression according to its grammar and precedence rules, then evaluates the pieces in the order those rules require. Python offers many ways to form compound expressions, and you will meet more as the course goes on; operators are just the first one.

Call Expressions: Applying a Function to Arguments

The most important kind of compound expression is the call expression, which applies a function to some arguments. The idea comes straight from algebra: a function is a mapping from input arguments to an output value. The built-in max function, for example, maps its inputs to a single output — the largest of them. Python writes function application the same way conventional mathematics does:

# (5)
max(7.5, 9.5)

A call expression has named parts. The operator is the expression before the parentheses, and it specifies the function. Inside the parentheses is a comma-separated list of operand expressions, whose values become the arguments. Follow the line:

max(7.5, 9.5)
├─ max ........ the operator: names the built-in function that returns the largest input
├─ (   ........ opens the call and begins the operand list
├─ 7.5 ....... first operand, supplies the argument 7.5
├─ ,   ........ separates one operand from the next
├─ 9.5 ....... second operand, supplies the argument 9.5
├─ )   ........ closes the call
└─ evaluates to  9.5

When this call is evaluated, we say the function max is called with arguments 7.5 and 9.5, and returns the value 9.5. The value of the whole call expression is whatever the function returns.

The most common beginner mistake is to leap from the code straight to an answer without asking what max actually does. The name max is not a magic word that means "do something with these numbers." It is bound to a built-in function whose one job is to inspect its arguments and hand back the greatest one. So Python is not computing 7.5 + 9.5, and it is not averaging them. The comma keeps the two values separate, and max compares them:

Is 7.5 larger than 9.5? No.
Is 9.5 larger than 7.5? Yes.
Return 9.5.

A beginner-safe way to read the line out loud is:

Call the function named max.
Give it 7.5 and 9.5 as two separate inputs.
Let max compare those inputs.
Use the larger input as the value of the whole expression.

This is the start of a habit that pays off all semester:

Do not ask only, "What is the answer?"
Ask, "What function is Python applying, and to which argument values?"

Order Matters, and Functions Can Take Many Arguments

Function notation has real advantages over the mix of symbols used in everyday math, and two of them show up immediately.

First, the order of arguments matters, because the function decides what each position means. The pow function raises its first argument to the power of its second:

# (6)
pow(100, 2)

This asks for 1002, so Python returns:

10000

Swap the roles of the two numbers and you get a completely different computation. Here the base is 2 and the exponent is 100:

# (7)
pow(2, 100)

This asks for 2100, and Python's integers grow as large as they need to, so the result is an exact, very large integer:

1267650600228229401496703205376

The rule is not "small numbers behave and big numbers are special." The rule is simply that pow(base, exponent) receives two arguments and returns the base raised to that exponent.

Second, a function may take an arbitrary number of arguments, and no ambiguity arises because the function name always comes first. Here max receives four arguments:

# (8)
max(1, -2, 3, -4)

The commas mark the four boundaries:

1
-2
3
-4

The single pair of parentheses still encloses one call, and max returns the largest value:

3

Nested Call Expressions: Inside Out

Function notation extends cleanly to nested expressions, where the operands are themselves call expressions. Unlike a long chain of infix operators, the structure of the nesting is fully spelled out by the parentheses, so there is never any doubt about what belongs to what.

This is the official nested example. Read it as a structure, not as a sentence:

# (9)
max(min(1, -2), min(pow(3, 5), -4))

Drawing the structure as a tree makes the nesting visible. Each branch is a subexpression the outer call is waiting on:

max
├── min
│   ├── 1
│   └── -2
└── min
    ├── pow
    │   ├── 3
    │   └── 5
    └── -4

Here is the key idea, and it is a machine rule rather than a study tip: a call expression cannot apply its function until every one of its operands has produced a value. The outer max is like a judge who cannot rule yet, because neither thing handed to it is a finished value — both are still unfinished calls. So Python completes the smaller calls first and lets their values rise up to the outer call:

First argument for max:
  min(1, -2) gives -2

Second argument for max:
  pow(3, 5) gives 243
  min(243, -4) gives -4

Now max has two real argument values:
  max(-2, -4) gives -2

So the whole expression evaluates to:

-2

There is, in principle, no limit to how deep this nesting can go. The limit is human: people get lost in multi-level nesting fast. Part of your job as a programmer is to structure expressions so that you, your partners, and whoever reads your code next can still follow them.

Importing Library Functions

Python knows a very large number of functions, but it does not make every name available the moment you start the interpreter. Instead it organizes functions and other values into modules, which together form the Python Library. To use the names in a module, you import them with an import statement.

The math module, for instance, provides familiar mathematical functions. Follow the line of an import statement:

# (10)
from math import sqrt

├─ from   ........ begins an import statement
├─ math   ........ names the module to take from (a collection of definitions)
├─ import ........ marks which names to pull into the current environment — the running collection of name-to-value bindings you pictured as the global frame in 1.1
├─ sqrt   ........ the name being imported
└─ performs  binds the name sqrt in the current environment to the square-root function

Notice the final line: an import statement is a statement, so it carries out an action — it binds a name — rather than producing a value to print. After (10) runs, the name sqrt is available, and you can call it like any other function:

# (11)
sqrt(256)

16.0

The result is 16.0 rather than 16 because many math-library functions return floats. Once a function is imported, you can call it as many times as you like.

The operator module provides functions that correspond to the infix operators. You import several names at once by separating them with commas:

# (12)
from operator import add, sub, mul

After (12), add, sub, and mul are callable functions. Calling add is just addition written in function form:

# (13)
add(14, 28)

42

Because these are ordinary functions, they nest exactly like max and pow did. Here is the official nested-operator example:

# (14)
sub(100, mul(7, add(8, 4)))

Trace it inside out:

add(8, 4)       -> 12
mul(7, 12)      -> 84
sub(100, 84)    -> 16

so the value is:

16

An import statement names a module (such as operator or math) and then lists the named attributes of that module to import (such as sqrt). There is no difference between using an operator function like add and using the operator symbol +; by convention, most programmers reach for the symbols and infix notation for simple arithmetic. The Python Library documentation lists what every module defines, but early on you will usually learn more about a function by experimenting with it than by reading reference docs written for experts.

The Evaluation Procedure Python Actually Follows

One goal of this chapter is to think clearly about procedures — step-by-step processes. A great first example is hiding in plain sight: when the interpreter evaluates a nested call expression, it is itself following a procedure. Here it is, exactly as Python applies it.

To evaluate a call expression, Python will do the following:

  1. Evaluate the operator and operand subexpressions, then
  2. Apply the function that is the value of the operator subexpression to the arguments that are the values of the operand subexpressions.

This short procedure already teaches something deep. Step 1 says that to evaluate a call expression you must first evaluate other expressions — and some of those may themselves be call expressions. So the procedure is recursive: one of its own steps is to invoke the very same rule again. Recursive is just the name for what you already saw — a rule that uses itself — and you do not need to master recursion here; a whole later section is devoted to it. For now it simply explains why nested expressions get evaluated inside out. The rule keeps applying itself to the operands until it reaches pieces that need no further evaluation.

Those simplest pieces are the primitive cases, and they end the recursion:

A numeral evaluates to the number it names.
A name evaluates to the value associated with that name in the current environment.

So a numeral like 2 evaluates straight to the number two, and a name like add evaluates to whatever it is currently bound to. This is why an environment matters: it is meaningless to ask for the value of add(x, 1) without knowing the environment that gives meaning to x (or even to add). The environment is the context that makes the symbols mean something.

A final boundary on this procedure: it evaluates call expressions, numerals, and names — and nothing else. It does not handle a statement such as x = 3. An assignment does not return a value or apply a function to arguments; its purpose is to bind a name. In general, statements are not evaluated but executed — they do not produce a value, they make a change. Each kind of expression or statement has its own evaluation or execution procedure.

Follow the Line: One Step of the Procedure

It helps to see the two-step rule applied to a single call. Take the innermost call from the next example, add(1, 10), and walk it through the procedure:

add(1, 10)
├─ add ........ Step 1: evaluate the operator -> the built-in add function
├─ 1   ........ Step 1: evaluate this operand (a numeral) -> the number 1
├─ 10  ........ Step 1: evaluate this operand (a numeral) -> the number 10
└─ apply ...... Step 2: apply add to 1 and 10 -> 11

Every interior call in a nested expression is handled by repeating exactly this, with the values percolating upward as each call finishes.

The Expression Tree, Step by Step

Here is the official example that ties the whole procedure together. It is deeper than the others, so the tree does real work:

# (15)
sub(pow(2, add(1, 10)), pow(2, 5))

Evaluating this requires the procedure to be applied four times — once for each call expression. Drawing every expression we evaluate reveals the hierarchical structure of the process. This drawing is called an expression tree. In computer science, trees grow from the top down. Each point in the tree is a node, and here each node is an expression paired with its value:

sub(pow(2, add(1, 10)), pow(2, 5))  -> 2016
├── pow(2, add(1, 10))  -> 2048
│   ├── 2
│   └── add(1, 10)  -> 11
│       ├── 1
│       └── 10
└── pow(2, 5)  -> 32
    ├── 2
    └── 5

The leaf expressions — nodes with no branches stemming from them — are either functions or numbers, the primitive cases that evaluate directly. The interior nodes each have two parts: the call expression our rule is applied to, and the result of that expression. You can picture the values percolating upward, starting from the terminal nodes and combining at higher and higher levels.

Working from the leaves up:

add(1, 10)          -> 11
pow(2, 11)          -> 2048
pow(2, 5)           -> 32
sub(2048, 32)       -> 2016

So the root, the full expression at the top, evaluates to:

2016

Reading the tree from the bottom up is the same thing as evaluating the expression inside out. They are two views of one process.

⚠Common Beginner Traps

Mistake What Goes Wrong Safer Question
Treating commas like arithmetic max(7.5, 9.5) compares two arguments; it does not add them What function is being called, and what are its separate inputs?
Evaluating nested calls left to right An outer call needs every operand's value before it can run, so the answer comes last, not first Which complete innermost call can Python evaluate first?
Treating parentheses as decoration They mark where a call begins and ends; an unbalanced pair raises a SyntaxError Which parenthesis opens this call, and where does it close?
Assuming argument order is flexible pow(2, 100) and pow(100, 2) give very different values Which argument is the function treating as first, and as second?
Forgetting to import a name Python raises a NameError because the environment has no binding for it Has this name been defined or imported before this line runs?
Expecting an import to print or call something An import only binds a name; it performs no call on its own What does this statement bind, and what still needs to be called afterward?
Reading a name with no environment in mind add(x, 1) raises a NameError until x and add are bound to values What environment gives each name in this expression a value?

✓Check Your Understanding

  1. What two steps make up Python's procedure for evaluating a call expression?
  2. Why is that evaluation procedure described as recursive?
  3. In max(min(1, -2), min(pow(3, 5), -4)), why must Python evaluate pow(3, 5) before the outer max call can finish?
  4. What does from math import sqrt add to the current environment, and does it call sqrt?
  5. In the procedure's primitive cases, what does a numeral evaluate to, and what does a name evaluate to?
  6. Assuming add, sub, and mul have been imported from operator, trace sub(add(2, 3), mul(2, 4)) inside out and give the value of each call and of the whole expression.
Show the answers
  1. First, evaluate the operator and the operand subexpressions; then apply the function that is the value of the operator to the arguments that are the values of the operands.
  2. Because step one requires evaluating other expressions, and when those are themselves call expressions the same rule must be applied again — the procedure invokes itself.
  3. The outer max cannot run until both of its operands are finished values. The second operand is min(pow(3, 5), -4), which cannot finish until pow(3, 5) produces 243.
  4. It binds the name sqrt in the current environment to the square-root function from the math module. It does not call sqrt; you must call it yourself afterward.
  5. A numeral evaluates to the number it names; a name evaluates to the value associated with that name in the current environment.
  6. The two inner calls must finish first: add(2, 3) gives 5 and mul(2, 4) gives 8. The outer call then has both operand values, so sub(5, 8) gives -3, the value of the whole expression.

§Summary

Computation lives in expressions. Primitive expressions like 42 already stand for a value; compound expressions combine values, either with infix operators like 1/2 + 1/4 or, most importantly, with call expressions that apply a function to arguments. A call expression has an operator that names the function and operand expressions whose values become the arguments, and its value is whatever the function returns. Import statements bind extra function names from the Python Library, such as sqrt from math or add, sub, and mul from operator. The interpreter evaluates every call by the same two-step rule — evaluate the operator and operands, then apply the function — and because that rule invokes itself on nested operands, deep expressions are evaluated inside out, with values percolating up an expression tree from its numeral-and-name leaves to its root. Master that procedure and you can predict any nested expression by hand, which is the whole point: you stop guessing at Python and start running it in your head.

▶Step-Through Checkpoint

Step through the block below slowly. It imports the operator functions, then rebuilds the official expression from (15) as separate assignment steps. Watching it as an environment diagram shows a single global frame filling up: the import drops add, sub, mul, and pow in at once, then each assignment adds one more binding — inner, left, right — while the final sub(left, right) call leaves the frame untouched, so the diagram makes plain which lines grow the frame and which only compute a value. Before you step, write down three predictions: how many names the first line adds to the global frame, how many bindings that frame holds when the run ends, and whether the last line adds a binding of its own or only produces a value.

Check each prediction as you step. The import binds add, sub, mul, and pow in a single step — four names at once, including mul, which nothing here calls — then inner becomes 11, left becomes 2048, and right becomes 32, for seven bindings in all. The last line is a bare call expression, so it binds nothing: sub(left, right) evaluates to 2016, the same value the nested expression sub(pow(2, add(1, 10)), pow(2, 5)) produces in one line. Seeing the intermediate values named makes the inside-out order concrete: each call had to finish before the one that depended on it could run.

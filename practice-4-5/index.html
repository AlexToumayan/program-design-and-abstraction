4.5 · Unification — Coding Practice

◷ approx. 12 min read · 31 worked examples

These problems build fluency with the unification machinery: telling variables from atoms, matching a pattern against an expression and recording the bindings, looking a variable up through a frame, resolving a binding chain into a variable-free answer, running the six-case unify algorithm where both sides may hold variables, and renaming a fact's variables so a search can reuse it safely. Logic expressions are modeled as nested Python lists, a binding frame is a dictionary, and a ?-variable is a string beginning with ?.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Classify a Token

A logic expression is built from three kinds of token: a variable (a string beginning with ?), an atom (any other plain value such as a symbol or number-symbol), and a list (a Python list of more tokens). Write classify(token) that returns the string "variable", "atom", or "list" for a single token.

Examples

Output: 'variable'
Explanation: a string starting with ? is a variable.

Input:  classify("abraham")
Output: 'atom'
Explanation: a plain symbol with no leading ? is an atom.

Input:  classify(["a", "b"])
Output: 'list'
Explanation: a Python list is the third kind of token.

  • A variable is a str whose first character is ?; check with .startswith("?").
  • The bare string "?" counts as a variable; "color?" is an atom (the ? is not first).

2. Match One Element

Pattern matching walks a pattern and an expression together, one element at a time, recording a binding the first time a variable appears and checking it every later time. Write match_element(pattern_part, expression_part, bindings) that returns True/False and mutates the bindings dictionary: if pattern_part is a variable, bind it (or, if already bound, succeed only when the stored value equals expression_part); otherwise succeed only when the two parts are equal.

Examples

Input:  b = {}; match_element("?name", "maya", b)
Output: True   (and b becomes {'?name': 'maya'})
Explanation: a fresh variable takes the value across from it.

Input:  b = {"?name": "maya"}; match_element("?name", "sam", b)
Output: False
Explanation: ?name is already maya, so it cannot also be sam.

Input:  match_element("is", "is", {})
Output: True
Explanation: two equal atoms match and record nothing.

Constraints

  • Reuse the same is_var test from Problem 1 (a str starting with ?).
  • A repeated variable must keep its first value; do not overwrite an existing binding.

Output: None
Explanation: the repeated ?x is fixed at (a b) but the third position is (a c).

  • Lengths must be equal; a length mismatch returns None.
  • Start from an empty dictionary and feed it through every element in order.

6. The Unification Algorithm

Write unify_lists(e, f, env), the six-case algorithm from the section. Look each side up first; succeed if they are now equal; if either side is an unbound variable bind it to the other and succeed; if either side is a non-list atom (and they are unequal) fail; otherwise both are lists, so fail on a length mismatch and otherwise recurse element by element. Record bindings by mutating env, and return True/False.

Explanation: the atoms b and c are unequal with no inner structure to recurse into.

7. Resolve an Expression

After unification, a variable may be bound to an expression that still mentions other variables (?x -> (a ?y c)). Write resolve(expr, env) that returns a fully resolved copy of expr: replace every variable by its value, repeatedly, until no bound variable remains. An unbound variable is left as itself.

Solutions

Setup

These problems model logic expressions as nested Python lists, a binding frame as a dictionary, and a ?-variable as a string beginning with ?. Problems 6–10 reuse the shared machinery the section builds in Python:

1. Classify a Token

Approach — the three kinds of token. The matcher distinguishes a ?-variable from a plain atom from a sublist. Mirror that split directly: test for a ?-prefixed string first, then for a list, then default to atom.

def classify(token):
    if isinstance(token, str) and token.startswith("?"):
        return "variable"
    if isinstance(token, list):
        return "list"
    return "atom"

Why it works. The variable test is the exact is_var rule (a str beginning with ?), so "?" is a variable while "color?" is an atom. Anything that is a list is the list case; everything else is an atom.

Complexity. Time O(1), Space O(1) — a couple of type checks on one token.

assert classify("?child") == "variable"
assert classify("abraham") == "atom"
assert classify("?") == "variable"
assert classify(["a", "b"]) == "list"
assert classify(["?x"]) == "list"
assert classify("color?") == "atom"

2. Match One Element

Approach — bind once, then check. This is the primitive from 4.5-A: a fresh variable takes whatever sits across from it; a variable seen before must keep its first value; an atom must equal the expression part exactly.

def match_element(pattern_part, expression_part, bindings):
    if is_var(pattern_part):
        if pattern_part in bindings:
            return bindings[pattern_part] == expression_part
        bindings[pattern_part] = expression_part
        return True
    return pattern_part == expression_part

Why it works. The first time a variable appears it is recorded and the match succeeds. A later appearance is checked against the stored value, which is exactly the rule that a repeated variable must mean the same thing each time. A non-variable simply compares for equality.

Complexity. Time O(1), Space O(1) — one dictionary read or write per call.

b = {}
assert match_element("?name", "maya", b) is True
assert b == {"?name": "maya"}
assert match_element("?name", "maya", b) is True       # same value: agrees
assert match_element("?name", "sam", b) is False       # different value: fails
assert match_element("is", "is", {}) is True
assert match_element("is", "was", {}) is False
b2 = {}
assert match_element("?x", ["a", "b"], b2) is True
assert b2 == {"?x": ["a", "b"]}

3. Match a Whole Pattern

Approach — feed one shared frame through every position. Pattern matching over a whole list is just match_element applied at each index, all writing into one bindings dictionary so a repeated variable is constrained.

def match_pattern(pattern, expression):
    if len(pattern) != len(expression):
        return None
    bindings = {}
    for i in range(len(pattern)):
        if not match_element(pattern[i], expression[i], bindings):
            return None
    return bindings

Why it works. A length check rules out mismatched shapes first. The single bindings dictionary carries each variable's value forward, so the second ?x is checked against the first; one disagreement returns None, otherwise the completed table is the answer.

Complexity. Time O(n), Space O(n) for n positions — one pass building a table with at most one entry per variable.

assert match_pattern(["parent", "abraham", "?child"],
                     ["parent", "abraham", "barack"]) == {"?child": "barack"}
assert match_pattern(["?x", "c", "?x"],
                     [["a", "b"], "c", ["a", "b"]]) == {"?x": ["a", "b"]}
assert match_pattern(["?x", "c", "?x"],
                     [["a", "b"], "c", ["a", "c"]]) is None
assert match_pattern(["?p", "likes", "?p"],
                     ["sam", "likes", "joan"]) is None
assert match_pattern(["a", "b"], ["a", "b", "c"]) is None

4. Look Up a Variable

Approach — walk the binding chain. A frame may point one variable at another, so resolving a name means following the chain until you reach a value that is not a bound variable. This is "case 1" of unify, isolated.

def lookup_value(name, env):
    while is_var(name) and name in env:
        name = env[name]
    return name

Why it works. The loop continues only while name is a variable that has a binding, so it stops the moment it reaches an atom, a list, or an unbound variable. With {"?x": "?y", "?y": "delano"} it hops ?x -> ?y -> delano and returns delano.

Complexity. Time O(d) for a chain of depth d, Space O(1) — it rebinds one local name in place.

assert lookup_value("?x", {"?x": "barack"}) == "barack"
assert lookup_value("?x", {"?x": "?y", "?y": "delano"}) == "delano"
assert lookup_value("?x", {}) == "?x"
assert lookup_value("abraham", {"?x": "barack"}) == "abraham"
assert lookup_value("?x", {"?x": ["a", "b"]}) == ["a", "b"]

5. Does a Variable Occur?

Approach — structural recursion. A variable occurs in an expression if the expression is that variable, or if it is a list with the variable somewhere inside. That is a base case plus a recursive descent into sublists.

def occurs(var, expr):
    if expr == var:
        return True
    if isinstance(expr, list):
        for part in expr:
            if occurs(var, part):
                return True
    return False

Why it works. The first check catches a direct hit at any depth. The loop recurses into each element, so a variable buried in a nested sublist is still found. If neither fires, the variable is absent and the function returns False.

Complexity. Time O(n), Space O(h) for n tokens and nesting depth h — it touches each token once and recurses as deep as the nesting.

assert occurs("?x", ["a", "?x", "c"]) is True
assert occurs("?x", "?x") is True
assert occurs("?x", ["a", ["b", "?x"], "c"]) is True
assert occurs("?x", ["a", "?y", "c"]) is False
assert occurs("?x", "abraham") is False
assert occurs("?x", []) is False

6. The Unification Algorithm

Approach — the six cases in order. This is the heart of 4.5-C. Look both sides up first so a bound variable is replaced by its value; succeed on equality; bind an unbound e or f; fail on unequal atoms; otherwise recurse into two lists, requiring equal length and a successful unification at every position.

def unify_lists(e, f, env):
    e = lookup(e, env)
    f = lookup(f, env)
    if e == f:
        return True
    if is_var(e):
        env[e] = f
        return True
    if is_var(f):
        env[f] = e
        return True
    if not isinstance(e, list) or not isinstance(f, list):
        return False
    if len(e) != len(f):
        return False
    for i in range(len(e)):
        if not unify_lists(e[i], f[i], env):
            return False
    return True

Why it works. The opening lookup is what makes a repeated variable reach across positions: on its second appearance ?x is replaced by (a ?y c), turning the comparison into a list-against-list recursion that forces ?y -> b and ?z -> c. Unequal atoms have no inner structure, so case 5 fails them; equal-length lists unify exactly when every element does.

Complexity. Time O(n), Space O(h) in the size n of the expressions and nesting depth h — each pair of corresponding nodes is unified once.

env = {}
assert unify_lists(["?x", "?x"], [["a", "?y", "c"], ["a", "b", "?z"]], env) is True
assert env == {"?x": ["a", "?y", "c"], "?y": "b", "?z": "c"}

env = {}
assert unify_lists(["?x", "c", "?x"], [["a", "b"], "c", ["a", "b"]], env) is True
assert env == {"?x": ["a", "b"]}

env = {}
assert unify_lists("?x", "7", env) is True
assert env == {"?x": "7"}

env = {}
assert unify_lists(["a", "?y", "b"], ["a", "d", "b"], env) is True
assert env == {"?y": "d"}

env = {}
assert unify_lists(["a", "b"], ["a", "c"], env) is False

7. Resolve an Expression

Approach — bind by another name. Resolving an expression means repeatedly replacing every variable by its value until none remain. Looking a token up handles a chain; recursing into lists handles structure; doing both handles a variable whose value still mentions variables.

def resolve(expr, env):
    expr = lookup(expr, env)
    if isinstance(expr, list):
        return [resolve(part, env) for part in expr]
    return expr

Why it works. lookup collapses any variable chain to its end. If that end is a list, each part is resolved in turn, so ?x -> (a ?y c) keeps going and turns ?y into b. An unbound variable or atom is its own resolution. Running it on (?x ?x) reproduces the variable-free ((a b c) (a b c)) from the section.

Complexity. Time O(n), Space O(n) in the size n of the resolved result — it rebuilds the structure with every variable replaced.

env = {"?x": ["a", "?y", "c"], "?y": "b", "?z": "c"}
assert resolve(["?x", "?x"], env) == [["a", "b", "c"], ["a", "b", "c"]]
assert resolve("?z", env) == "c"
assert resolve(["?y", "?z"], env) == ["b", "c"]
assert resolve(["plain", "?missing"], env) == ["plain", "?missing"]
assert resolve("loner", {}) == "loner"

8. Unify Into a Fresh Frame

Approach — split the two outputs. As 4.5-B stresses, unify returns a Boolean and writes the bindings into a frame. This wrapper hands the caller the frame on success and None on failure, so a single return value carries both pieces of news.

def unifies(e, f):
    env = {}
    if unify(e, f, env):
        return env
    return None

Why it works. A fresh dictionary per call keeps attempts isolated, so leftover bindings from a failed match never leak. When unify returns True the frame already holds every binding it discovered; when it returns False the wrapper reports None instead of a half-filled frame.

Complexity. Time O(n), Space O(n) — it does one unification and returns its frame.

assert unifies(["?x", "?x"], ["7", "7"]) == {"?x": "7"}
assert unifies(["?a", "b"], ["c", "?d"]) == {"?a": "c", "?d": "b"}
assert unifies(["?x", "?x"], ["a", "b"]) is None
assert unifies(["f", "?x", "?y"], ["f", "1", "?y"]) == {"?x": "1"}
assert unifies("?x", ["a", "?y"]) == {"?x": ["a", "?y"]}

9. Unify Against a Renamed Fact

Approach — rename, then unify. 4.5-E explains that unification assumes the two sides share no variable names, so a search stamps a unique id onto a fact's variables before using it. Rename the fact with rename_variables, then unify the unchanged query against the renamed copy into a fresh frame.

def fresh_unify(query, fact, fact_id):
    fact = rename_variables(fact, fact_id)
    env = {}
    if unify(query, fact, env):
        return env
    return None

Why it works. After renaming, the fact's ?x becomes ?x_7, so it can no longer be confused with the query's own ?x; the query variable is then free to bind to the renamed fact variable. The same fresh-frame discipline as Problem 8 keeps the result clean.

Complexity. Time O(n), Space O(n) — one renaming pass plus one unification over expressions of size n.

assert fresh_unify(["ancestor", "?x", "clinton"],
                   ["ancestor", "?x", "?y"], 7) == \
    {"?x": "?x_7", "?y_7": "clinton"}
assert fresh_unify(["parent", "?x", "?y"],
                   ["parent", "fillmore", "abraham"], 3) == \
    {"?x": "fillmore", "?y": "abraham"}
assert fresh_unify(["p", "?x", "?x"],
                   ["p", "?a", "?b"], 1) == {"?x": "?a_1", "?a_1": "?b_1"}
assert fresh_unify(["q", "a"], ["q", "?z"], 9) == {"?z_9": "a"}

10. Solve a Single-Clause Query

Approach — rename, unify, resolve, collect. This is the one-clause core of the search procedure from 4.5-E. For each fact, rename its variables with its index, unify the query into a fresh frame, and on success resolve the query through that frame into a fully ground answer, skipping duplicates.

def solve(query, facts):
    results = []
    for index in range(len(facts)):
        fact = rename_variables(facts[index], index)
        env = {}
        if unify(query, fact, env):
            answer = bind(query, env)
            if answer not in results:
                results.append(answer)
    return results

Why it works. Renaming each fact with its index keeps facts from sharing variables across iterations. A fresh frame per fact sandboxes each attempt, exactly the non-destructive discipline of the real search. On a match, bind(query, env) walks the query and substitutes every variable, producing the variable-free fact that satisfied it; the duplicate check keeps the answer list distinct.

Complexity. Time O(m⋅n), Space O(m⋅n) for m facts of size n — one rename-unify-resolve per fact.

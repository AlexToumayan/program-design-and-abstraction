Chapter 1/1.7 Recursive Functions

1.7 Mutual Recursion and Recursive Printing

◷ approx. 16 min read · 5 worked examples · best read in two passes

You already know that a recursive function can call itself. This lesson covers two ideas that grow out of that.

The first is mutual recursion: a recursive process that is split across two functions that call each other instead of one function that calls itself. By the end you will be able to read a pair like is_even and is_odd, explain why each call still gets its own frame, and turn a mutually recursive pair back into a single function when you want to.

The second is recursive printing: using print inside a recursive function to make the shape of the computation visible. You will learn the single most important habit for this, which is that code written before a recursive call runs on the way down, and code written after it runs on the way back up. That one distinction explains the entire output of the cascade function below.

Two Functions Passing the Same Task Back and Forth

Picture two clerks at adjacent desks who refuse to do each other's job. Clerk E only knows how to answer "is this number even?" and Clerk O only knows how to answer "is this number odd?" Neither one can solve a problem alone. When Clerk E is handed the number 4, the only move E knows is to subtract one and slide the number 3 over to Clerk O, asking "is 3 odd?" O subtracts one and slides 2 back to E. The number shrinks by one with every handoff until it reaches 0, where each clerk has a memorized answer: E knows 0 is even, so E can answer without passing anything along.

That back-and-forth between two functions that depend on each other is mutual recursion. The work is genuinely shared: neither function is complete on its own, but together they decide the question.

Even and Odd, Defined in Terms of Each Other

The cleanest example of mutual recursion is the definition of even and odd for non-negative integers. We can describe these two properties entirely in terms of each other, plus one fact we just know outright:

  • A number is even if it is one more than an odd number.
  • A number is odd if it is one more than an even number.
  • 0 is even.

Notice that the first two bullets are circular on purpose. "Even" is defined using "odd," and "odd" is defined using "even." The third bullet is what stops the circle from spinning forever. It is the base case: a fact that needs no further question.

Reading the definition the way the code will use it, each property points at the smaller number just below it. To decide whether 4 is even, look at whether 3 is odd. To decide whether 3 is odd, look at whether 2 is even. The number marches down toward 0, and 0 has a fixed answer.

Each function has the same shape you saw in ordinary recursion: a base case (n == 0) and a recursive case. The only new thing is that the recursive case calls the other function, not itself.

Follow the Line: The Recursive Case of is_even

No new Python rule appears in mutual recursion. The line return is_odd(n-1) is just a return statement wrapped around a call expression, the same machinery you already know. What is worth slowing down on is exactly what each piece does, because the call hands control to a different function:

return is_odd(n-1)
├─ n-1 ............ build the next-smaller number to ask about
├─ is_odd(...) .... ask the partner function whether that number is odd
├─ return ........ hand that partner's answer straight back to whoever called is_even
└─ performs ...... return the result of the is_odd call as this call's own value

The whole statement does not compute anything new on its own. It delegates: it builds a smaller argument, calls the partner, and returns whatever the partner returns. The is_odd function works the same way in reverse. This is why mutual recursion feels no harder than ordinary recursion once you see it: the next frame simply belongs to a different function.

Tracing is_even(4) by Hand

Let us walk the handoffs down to the base case. Each function checks n == 0; if not, it calls its partner with n-1.

is_even(4)
  n is not 0, so return is_odd(3)

is_odd(3)
  n is not 0, so return is_even(2)

is_even(2)
  n is not 0, so return is_odd(1)

is_odd(1)
  n is not 0, so return is_even(0)

is_even(0)
  base case: return True

The argument shrinks by one at every step and the function name flips even, odd, even, odd. When is_even(0) finally hits its base case, it produces True. Now that value travels back up the chain. Because every recursive case is just return <partner call>, nobody modifies the answer on the way up; each call returns exactly what it received:

is_even(0) -> True
is_odd(1)  -> True
is_even(2) -> True
is_odd(3)  -> True
is_even(4) -> True

So result is bound to True. That is correct: 4 really is even.

A Bridge You Can Check: is_even(3)

Run the same machine on an odd number to convince yourself it reports odd numbers correctly. The handoffs are identical, but the base case lands on the other function:

is_even(3)
  n is not 0, so return is_odd(2)

is_odd(2)
  n is not 0, so return is_even(1)

is_even(1)
  n is not 0, so return is_odd(0)

is_odd(0)
  base case: return False

This time the chain bottoms out in is_odd(0), whose base case is return False. That False rides back up unchanged, so is_even(3) is False. The two base cases, True in is_even and False in is_odd, are what give the pair its opposite answers.

Turning Two Functions Back Into One

Mutual recursion is a convenience, not a new kind of power. Anything you can do with two functions calling each other, you can do with one function calling itself. To see this, you can inline one function into the other: take the body of is_odd, paste it where is_odd(n-1) was called, and replace each n in that pasted body with n-1 to account for the argument that was passed in.

Here is the result from the text, written as an interactive session exactly as it appears in the original:

>>> def is_even(n):
        if n == 0:
            return True
        else:
            if (n-1) == 0:
                return False
            else:
                return is_even((n-1)-1)

Read what happened to the structure. The outer if n == 0: return True is the original is_even base case, untouched. The else branch used to be return is_odd(n-1); in its place we now have the entire body of is_odd, with every n rewritten as n-1. So is_odd's base case if n == 0: return False became if (n-1) == 0: return False, and is_odd's recursive case return is_even(n-1) became return is_even((n-1)-1). We leave it written as (n-1)-1 rather than simplifying to n-2 so you can see exactly where it came from — they are the same number.

This single function behaves identically to the pair. The lesson is that mutual recursion is no more mysterious than simple recursion. What it buys you is abstraction: keeping is_even and is_odd as separate, individually meaningful functions is easier to read and modify than one function with a nested base case buried inside it.

Recursive Printing: Making the Call Stack Visible

Return values are not the only thing a recursive function can produce. It can also print, and where you place the print relative to the recursive call lets you watch the computation unfold. The text demonstrates this with cascade, a function that prints all prefixes of a number from largest, down to the single leading digit, and back up to largest. A "prefix" of 2013 means 2013, then 201, then 20, then 2, formed by repeatedly dropping the last digit with n // 10.

Here is the official definition, reproduced exactly, including its docstring:

# (2)
def cascade(n):
    """Print a cascade of prefixes of n."""
    if n < 10:
        print(n)
    else:
        print(n)
        cascade(n//10)
        print(n)

The base case is a single-digit number (n < 10), which is simply printed. Otherwise a recursive call to cascade(n//10) is placed between two calls to print(n). That sandwich is the whole trick.

Calling cascade(2013) produces this session, exactly as in the text:

>>> cascade(2013)
2013
201
20
2
20
201
2013

Follow the Line: n // 10

The shrinking step in cascade is n // 10. The // operator is floor division: it divides and throws away any remainder, so it chops off the last digit of an integer.

cascade(n//10)
├─ n .......... the current number, for example 2013
├─ // 10 ...... floor-divide by ten, dropping the final digit
├─ n//10 ...... the prefix with one fewer digit, here 201
├─ cascade(...) call cascade on that shorter prefix
└─ evaluates to  None (cascade returns nothing; it prints as a side effect)

Notice the final line: cascade has no return statement, so its value is None. It is run entirely for its side effect of printing. Keep that in mind for the traps section, because mixing up "prints" and "returns" is a classic mistake.

The Order Is the Lesson

Why does 2013 appear at both the top and the bottom of the output, while 2 appears only once in the middle? Look at where each print(n) sits relative to the recursive call.

In the else branch, the first print(n) runs before cascade(n//10). A line before a recursive call runs while the call stack is still growing — on the way down. The second print(n) runs after cascade(n//10) returns. A line after a recursive call runs while the stack is unwinding — on the way back up.

So the descent prints the prefixes large to small, and the ascent prints them small to large:

printed on the way down (before the call):  2013, 201, 20, 2
printed on the way back up (after the call):     20, 201, 2013

The single digit 2 is the base case. It has no second print waiting after it, because the base case branch only prints once and never recurses. That is why 2 sits alone at the fold in the middle. This before-versus-after rule is the most important habit in recursive printing: code before the recursive call runs as frames are created, and code after the recursive call runs as frames return.

A More Compact Cascade

You do not have to write base cases before recursive calls. Look again at cascade: print(n) appears in both branches of the conditional, the n < 10 branch and the else branch. Anything that happens no matter which branch you take can be lifted out and run before the conditional. Pulling that shared print(n) to the top gives a shorter version with identical behavior. This is the official compact form from the text, reproduced exactly:

# (3)
def cascade(n):
    """Print a cascade of prefixes of n."""
    print(n)
    if n >= 10:
        cascade(n//10)
        print(n)

The base case did not disappear; it just changed shape. Before, the base case was "if n < 10, only print." Now the base case is expressed as the absence of recursion: when n < 10, the condition n >= 10 is false, so the if block is skipped and the function simply returns after its single print(n). The first print(n) still runs on the way down for every call, and the print(n) inside the if still runs on the way back up only for the multi-digit calls. Trace cascade(2013) against this version and you will get the same seven lines.

Both versions are worth being able to read. The first makes the base case explicit and obvious for a beginner; the second is more compact and is the kind of factoring you will see in real code.

The Pebble Game: Mutual Recursion as Turn-Taking

Mutual recursion shines when each function represents a different state or role, and the recursive call is what switches between them. The text illustrates this with a two-player game.

There are n pebbles on a table. The two players take turns removing either one or two pebbles, and whoever removes the final pebble wins. Alice and Bob each follow a fixed, simple strategy:

  • Alice always removes a single pebble.
  • Bob removes two pebbles when an even number of pebbles is on the table, and one pebble otherwise.

Given n starting pebbles with Alice going first, who wins? We can answer this by giving each player their own function. Keeping the strategies in separate functions means you could change Alice's strategy without touching Bob's, preserving the abstraction barrier between them. Because the game alternates turns, each function ends by calling the other.

Here are the two functions, reproduced exactly as a session in the original text:

>>> def play_alice(n):
        if n == 0:
            print("Bob wins!")
        else:
            play_bob(n-1)
>>> def play_bob(n):
        if n == 0:
            print("Alice wins!")
        elif is_even(n):
            play_alice(n-2)
        else:
            play_alice(n-1)

The argument n is the number of pebbles left, and the function you are in is whose turn it is. The base case is n == 0: there are no pebbles to take, which means the previous player took the last one and won. That is why play_alice prints "Bob wins!" when it is handed an empty table — if it is Alice's turn and nothing is left, Bob must have cleared the table on the move before.

Running it reproduces the result from the text:

>>> play_alice(20)
Bob wins!

One detail worth flagging: play_bob contains two different recursive calls, play_alice(n-2) and play_alice(n-1), one in each branch of its conditional. That is allowed. But notice that only one of them runs on any given turn, because they sit in separate branches. So each call to play_bob calls play_alice at most once. The next lesson explores what happens when a single call makes several recursive calls at once.

Walking the Pebble Game from 20

You can verify Bob wins! by hand. Track whose turn it is and how many pebbles remain. Remember Alice always takes 1, and Bob takes 2 when the count is even and 1 when it is odd.

play_alice(20)  Alice takes 1  ->  play_bob(19)
play_bob(19)    19 is odd, Bob takes 1  ->  play_alice(18)
play_alice(18)  Alice takes 1  ->  play_bob(17)
play_bob(17)    17 is odd, Bob takes 1  ->  play_alice(16)
play_alice(16)  Alice takes 1  ->  play_bob(15)
play_bob(15)    15 is odd, Bob takes 1  ->  play_alice(14)
...
play_alice(2)   Alice takes 1  ->  play_bob(1)
play_bob(1)     1 is odd, Bob takes 1  ->  play_alice(0)
play_alice(0)   n == 0, print "Bob wins!"

From 20, the count stays even on Alice's turns and odd on Bob's turns the whole way down, so each player keeps taking a single pebble. The table empties on Bob's move, and when Alice is finally handed 0 pebbles she announces that Bob won.

Why Split a Process Across Functions

The single-function rewrite of is_even showed that mutual recursion adds no raw power. So why use it? Because naming each state keeps the program honest. Instead of one tangled function that has to track every possible situation with extra parameters and nested conditionals, you give each situation its own clearly named function:

  • play_alice versus play_bob — whose turn it is.
  • is_even versus is_odd — which property you are currently checking.

Each function stays small and individually meaningful, and the recursive call is the moment the program moves to the next state. That is the practical payoff of mutual recursion: abstraction inside a recursive process.

⚠Common Beginner Traps

Mistake What Goes Wrong Safer Question
Forgetting a base case in one of the partner functions One side keeps calling forever, never reaching n == 0, until Python raises RecursionError: maximum recursion depth exceeded Does every possible state eventually hit a base case?
Calling the wrong partner (or calling yourself instead of the partner) The state machine is wrong, so the answer is wrong even though it terminates After this action, whose turn or which state comes next?
Confusing print order with return order Output appears in the wrong sequence Is this line before or after the recursive call?
Using print when the problem wants a return The function's value is None, so a grader checking the returned value sees nothing Is the task asking me to display a value or to compute one?
Assuming the base case must come before the recursive call You miss simpler factorings like the compact cascade Can I express the base case as "do not recurse" instead?

✓Check Your Understanding

  1. Trace is_even(5). Which function reaches the base case, and what does is_even(5) return?
  2. Why does cascade(2013) print 2013 twice but 2 only once?
  3. Which lines of cascade run while the call stack is growing, and which run while it is unwinding?
  4. The compact cascade has no explicit if n < 10 base case. How does it still stop?
  5. In the pebble game, why does play_alice print "Bob wins!" exactly when it is called with n == 0?
  6. What does it mean to "turn mutual recursion into a single recursive function," and what do you gain by keeping the two functions separate instead?
Show the answers
  1. The handoffs go is_even(5) to is_odd(4) to is_even(3) to is_odd(2) to is_even(1) to is_odd(0), so is_odd reaches the base case and returns False. Every recursive case passes that value straight back up unchanged, so is_even(5) returns False.
  2. The first print(n) runs before the recursive call (on the way down) and the second print(n) runs after the call returns (on the way back up), so every multi-digit prefix is printed twice. 2 is the base case, which prints once and never recurses, so it has no second print waiting.
  3. The line before the recursive call (the first print(n)) runs while the stack is growing; the line after the recursive call (the second print(n)) runs while the stack is unwinding.
  4. The base case is written as the absence of recursion. When n < 10, the condition n >= 10 is false, so the if block is skipped, the function prints once and returns without recursing.
  5. n == 0 means no pebbles remain on Alice's turn, which means the previous player, Bob, removed the last pebble. Removing the final pebble wins, so Bob is the winner.
  6. You inline one function's body into the other, replacing n with n-1 to match the argument that was passed, producing one function that calls itself with nested base cases. Keeping the functions separate preserves abstraction: each stays small, individually meaningful, and easy to modify on its own.

§Summary

Mutual recursion splits a single recursive process across two or more functions that call each other, like is_even and is_odd, or the turn-taking play_alice and play_bob. Each call still receives its own frame; the only difference from ordinary recursion is that the next frame may belong to a different function. Any mutually recursive pair can be collapsed into one self-calling function by inlining, which shows that mutual recursion adds convenience and abstraction rather than new power. Recursive printing uses print to expose the shape of a computation: in cascade, a print before the recursive call runs on the way down and a print after it runs on the way back up, which is why the prefixes appear large-to-small-to-large. That before-versus-after rule — code before the recursive call runs as frames are created, code after runs as frames return — is the key habit for reasoning about any recursive function that produces output.

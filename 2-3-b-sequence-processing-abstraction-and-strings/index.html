Chapter 2/2.3 Sequences

2.3 Sequence Processing, Abstraction, and Strings

◷ approx. 18 min read · 30 worked examples · best read in two passes

You already know how to store many values in a sequence and how to walk over them with a loop. This lesson is about the next question: what work do you actually do to a sequence once you have one? Almost every interesting sequence program is built from three moves. You transform each element into a new element, you filter by keeping only the elements that pass a test, or you aggregate many elements down into one result. By the end of this lesson you will be able to read and write a list comprehension, recognize when a comprehension is doing a transform versus a filter, express the same three moves as higher-order functions that take other functions as arguments, and treat a string as just another sequence that responds to length, indexing, membership, and slicing.

The habit underneath all of it is the same one you have been building: name the pattern before you write the code. Once you can say out loud "this is a filter that keeps the divisors of n," the Python almost writes itself.

Processing as an Assembly Line

Picture a small workshop with a conveyor belt. A bin of raw items rolls in from the left, passes a few stations, and leaves as something finished on the right. Each station does one predictable job. The painting station coats every item that goes by. The inspection station shoves the cracked ones off the belt so only the good ones continue. The tally station at the very end does not pass items along at all; it just counts what made it through and writes a single number on a clipboard.

Those three stations are exactly the three moves of sequence processing. Painting every item is a transform: same number of things come out, but each one is changed. Shoving off the cracked ones is a filter: the survivors are unchanged, but there are fewer of them. The tally is an aggregate: many things go in, one summary comes out.

input sequence --> transform / filter / aggregate --> output

The whole skill is learning to look at a problem and see which stations it needs, and in what order.

Transforming a Sequence

Start with a list of odd numbers.

# (1)
odds = [1, 3, 5, 7, 9]

We want a brand-new list where every element is one larger. Rather than writing a loop that appends to an empty list, Python lets us describe the whole new list in one expression.

# (2)
[x + 1 for x in odds]

The value of this expression is:

[2, 4, 6, 8, 10]

That expression is a list comprehension. It reads almost like English: "the value x + 1 for each x in odds." It builds and returns a new list and never touches odds.

The Anatomy of a List Comprehension

A comprehension packs a loop and a result rule into one line, so it pays to slow down and see each part for what it does.

[x + 1 for x in odds]
├─ [ ......... start building a brand-new list
├─ x + 1 ..... the rule that produces each new element
├─ for ....... begin looping
├─ x ......... the name bound to the current element each pass
├─ in ........ connect that name to the source it pulls from
├─ odds ...... the source sequence being walked
├─ ] ......... finish the list
└─ evaluates to  [2, 4, 6, 8, 10]

The trick to reading any comprehension is that the parts run in a different order than they are written. Python reads the for ... in ... part first to get the values, and only then applies the front rule to each one. So even though x + 1 appears first on the page, it happens second in time.

Here is the machine running, step by step.

Start with an empty result list.

x = 1  ->  append x + 1, which is 2.
x = 3  ->  append x + 1, which is 4.
x = 5  ->  append x + 1, which is 6.
x = 7  ->  append x + 1, which is 8.
x = 9  ->  append x + 1, which is 10.

Return [2, 4, 6, 8, 10].

One more thing worth saying plainly: the for keyword here is not the start of a for statement. It is part of the comprehension because it lives inside the square brackets. That is also why nothing gets printed during the loop and why odds comes out unchanged.

Filtering a Sequence

A comprehension can carry an optional if clause at the end. The if acts as the inspection station: an element is added to the result only when the condition is true for it.

# (3)
[x for x in odds if 25 % x == 0]

The piece 25 % x is the remainder when 25 is divided by x. When that remainder is 0, it means x divides 25 evenly. So this comprehension keeps exactly the elements of odds that are divisors of 25. The value is:

[1, 5]

Here is the anatomy, now with the filter on the end.

[x for x in odds if 25 % x == 0]
├─ [ ................. start building a new list
├─ x ................. the element to keep, unchanged
├─ for x in odds ..... walk each element of odds, binding it to x
├─ if 25 % x == 0 .... keep x only when 25 divided by x leaves no remainder
├─ ] ................. finish the list
└─ evaluates to  [1, 5]

Trace it and you can see which elements survive the inspection.

x = 1:  25 % 1 == 0   True, keep 1
x = 3:  25 % 3 == 1   False, skip 3
x = 5:  25 % 5 == 0   True, keep 5
x = 7:  25 % 7 == 4   False, skip 7
x = 9:  25 % 9 == 7   False, skip 9

Notice the front rule here is just x, not x + 1. A filter keeps elements as they are; only the count changes. You can transform and filter in the same comprehension, but here we only want to filter.

The General Form

It helps to have the full template in front of you so every comprehension you meet maps onto the same skeleton.

[<map expression> for <name> in <sequence expression> if <filter expression>]

To evaluate it, Python first evaluates the <sequence expression>, which must give back a sequence. Then, for each element in order, it binds <name> to that element, evaluates the <filter expression>, and if the result is a true value it evaluates the <map expression> and adds that value to the result list. The if part is optional; leave it off and every element is kept.

Divisors as a Sequence Program

Filtering shines when the test is a small computation. Here is a function that returns the positive divisors of n, not counting n itself.

# (4)
def divisors(n):
    return [1] + [x for x in range(2, n) if n % x == 0]

Two design choices are worth pinning down. The result starts with [1] because every positive integer is divisible by 1, and the comprehension begins its search at 2, so 1 is added separately rather than discovered. And range(2, n) stops just before n, because we want the divisors that are strictly less than n itself.

Walk the smaller case by hand first so the official outputs are not a surprise. For divisors(4), the comprehension searches range(2, 4), which is 2 and 3. Now 4 % 2 == 0 so 2 is kept, and 4 % 3 == 1 so 3 is skipped. Prepend [1] and you get [1, 2]. The source checks exactly this small case before the larger one.

# (5)
divisors(4)
divisors(12)

The results are:

[1, 2]
[1, 2, 3, 4, 6]

Perfect Numbers

A perfect number is a positive integer that equals the sum of its proper divisors. The classic first example is six, because:

6=1+2+3

Because divisors already hands back the proper divisors, and sum already adds a list, we can describe a search for perfect numbers as a single filtering comprehension over a range.

# (6)
[n for n in range(1, 1000) if sum(divisors(n)) == n]

This produces:

[6, 28, 496]

Read the structure as a search, because that pattern will come back constantly. Consider each candidate n in a range, compute a property of n, and keep n only when the property holds.

Consider each n in a range.
Compute a property of n, here sum(divisors(n)).
Keep n only when that property equals n.

The number theory convention for whether 1 counts as perfect varies between textbooks, and that is not the lesson. The lesson is that a single sequence expression can describe an entire search over a thousand candidates.

Aggregating a Sequence

The third move collapses a whole sequence into one value. The built-in sum is the workhorse aggregator, and you just saw it used inside the perfect-number search. On its own it adds the elements of a list.

# (7)
sum([2, 3, 4])

The result is:

9

It takes an optional second argument that supplies the starting value instead of the default 0.

# (8)
sum([2, 3, 4], 5)

The result is:

14

The machine model for the second call makes the starting value visible.

Start with 5.
Add 2 -> 7.
Add 3 -> 10.
Add 4 -> 14.
Return 14.

The same shape powers min, max, and all: many values flow in, one value comes out.

Reusing divisors: The Minimum Perimeter

A good sequence program often becomes a building block for the next one. Suppose you want the smallest possible perimeter of a rectangle whose area is a given whole number. If the area is 80, the whole-number side pairs are limited.

1 by 80
2 by 40
4 by 20
5 by 16
8 by 10

The perimeter of a rectangle with width w and height h is:

2w+2h

The design splits the work into three small functions and reuses divisors to enumerate the candidate heights.

# (9)
def width(area, height):
    assert area % height == 0
    return area // height

def perimeter(width, height):
    return 2 * width + 2 * height

def minimum_perimeter(area):
    heights = divisors(area)
    perimeters = [perimeter(width(area, h), h) for h in heights]
    return min(perimeters)

Each function owns one job. divisors(area) lists the heights worth trying, width(area, h) recovers the matching width, perimeter(...) scores each rectangle, and min(...) chooses the best score. The smallest perimeter for area 80 comes from the 8 by 10 rectangle:

2⋅8+2⋅10=36

The source walks the exact numeric checkpoints, and it is worth running them in your head before you trust the function as a whole.

# (10)
area = 80
width(area, 5)
perimeter(16, 5)
perimeter(10, 8)
minimum_perimeter(area)
[minimum_perimeter(n) for n in range(1, 10)]

The results are:

16
42
36
36
[4, 6, 8, 8, 12, 10, 16, 12, 12]

The first line confirms that the width paired with height 5 is 16, the next two score two different rectangles, the fourth confirms the overall minimum is 36, and the last applies the whole function across a range to produce one minimum per area.

Sequence Abstraction

Transform, filter, and aggregate show up so often that it is worth naming the tools that match each one. The table is a map from "what move am I making?" to "what reaches for it."

Pattern Question it answers Common tool
Transform What should each element become? map or a list comprehension
Filter Which elements should remain? filter or an if clause
Aggregate How do we combine all elements? sum, min, max, all, reduce

For most everyday code, list comprehensions are the easiest to read, because the loop is right there in the brackets. Here are a transform and a filter side by side as comprehensions.

Higher-Order Sequence Functions

The same three moves can be captured as functions that take other functions as arguments. This is the more abstract view, and it is where the patterns stop being syntax and become reusable tools. We can write our own transform and filter that accept the per-element work as a parameter.

# (12)
def apply_to_all(map_fn, s):
    return [map_fn(x) for x in s]

def keep_if(filter_fn, s):
    return [x for x in s if filter_fn(x)]

apply_to_all takes a one-argument function map_fn and applies it to every element of s. keep_if takes a one-argument predicate filter_fn and keeps the elements for which it returns a true value. Both are just comprehensions with the per-element rule pulled out into a parameter.

Aggregation gets the same treatment. Many forms of aggregation are the same loop: hold a running result, and combine it with each element using a two-argument function.

# (13)
def reduce(reduce_fn, s, initial):
    reduced = initial
    for x in s:
        reduced = reduce_fn(reduced, x)
    return reduced

To see reduce work, feed it multiplication. The operator module exposes the multiply operation as a function named mul, so reduce(mul, ...) multiplies a sequence together starting from 1.

# (14)
from operator import mul
reduce(mul, [2, 4, 8], 1)

The result is:

64

Trace it once: start at 1, multiply by 2 to get 2, multiply by 4 to get 8, multiply by 8 to get 64. That running-result pattern is exactly what sum, min, and max are doing under the hood.

Rebuilding the Perfect-Number Search Abstractly

The payoff of these tools is that the earlier divisor and perfect-number program can be rebuilt out of keep_if and reduce instead of raw comprehensions and sum. The logic is identical; only the spelling changes.

The result is:

[1, 2, 3, 4, 6]

Here divides_n is a small unnamed function built with lambda. It captures n and reports whether a given x divides n, which is precisely the predicate keep_if needs.

The word lambda is just a way to write a tiny function inline, without giving it a name with def.

lambda x: n % x == 0
├─ lambda ......... create a small unnamed function
├─ x .............. its one parameter
├─ : .............. separates the parameter from the body
├─ n % x == 0 ..... the expression it returns
└─ evaluates to  a function value you can call later

Now aggregate the divisors with reduce over addition rather than sum. The operator module also provides add as a function, so reduce(add, ..., 0) sums a sequence starting from 0.

# (16)
from operator import add
def sum_of_divisors(n):
    return reduce(add, divisors_of(n), 0)

def perfect(n):
    return sum_of_divisors(n) == n

keep_if(perfect, range(1, 1000))

The result is the familiar perfect-number list:

[1, 6, 28, 496]

This is the same search you wrote earlier as one comprehension, now assembled from named, reusable parts. Either spelling is fine; the value of the abstract version is that keep_if, reduce, and a predicate can be recombined for completely different problems.

The Built-in Versions

You do not have to define apply_to_all, keep_if, and reduce yourself, because Python ships equivalents. The transform tool is map, the filter tool is filter, and the aggregate tool is reduce from the functools module. Both map and filter return lazy objects, so wrapping them in list(...) forces the actual list.

# (17)
apply_to_all = lambda map_fn, s: list(map(map_fn, s))
keep_if = lambda filter_fn, s: list(filter(filter_fn, s))

The built-in reduce lives in functools, and in that version the starting value is optional. A product function follows the same shape as the earlier multiply-reduce.

The result is:

120

In real programs you will reach for list comprehensions more often than for map and filter, but it is worth knowing both, because the higher-order versions are how these patterns generalize once the per-element work itself becomes a value you pass around.

Strings Are Sequences

Before leaving sequences, it is worth recalling that the sequence abstraction is broader than lists. Membership and slicing, for instance, work on any sequence. Take a list of digits.

The results are:

True
True
[1, 8]
[8, 2, 8]

The in and not in operators test membership, and a slice digits[start:stop] returns the contiguous span from start up to but not including stop. Leaving the stop off, as in digits[1:], means "to the end."

Now the key point: a string is also a sequence, where the elements are characters. Bind a city name.

# (20)
city = "Berkeley"

You can picture it as a row of characters, each with an index counting from zero.

index:   0   1   2   3   4   5   6   7
       +---+---+---+---+---+---+---+---+
char:  | B | e | r | k | e | l | e | y |
       +---+---+---+---+---+---+---+---+

Because it is a sequence, the same operations you use on lists apply. The official transcript exercises length, indexing, concatenation, repetition, and membership all on this one string.

The results are:

8
'k'
'Berkeley, CA'
'Shabu Shabu '
True

A few of these deserve a note. Indexing a string returns a one-character string, not some separate character type, so city[3] is the string 'k'. The + operator concatenates two strings into a longer one, and * with an integer repeats a string. Membership on strings is a little different from membership on lists: 'here' in "Where's Waldo?" is true because "here" appears as a contiguous run of characters inside the larger string, starting at its second character. On a list, in only checks whole elements, but on a string it checks substrings.

String Literals

Python lets you write a string with single quotes, double quotes, or triple quotes, and all three produce ordinary string values.

# (22)
'I am string!'
"I've got an apostrophe"
'您好'

Python treats each of these as a string value. The choice of quote is mostly about avoiding conflicts: if the text itself contains an apostrophe, wrapping it in double quotes saves you from fighting the single quote, which is why "I've got an apostrophe" is convenient. The third example is a deliberate reminder that a string is a sequence of characters in general, not specifically English text.

Triple quotes are for strings that span more than one physical line.

# (23)
"""The Zen of Python
claims, "Readability counts."
Read more: import this."""

The value Python builds is:

'The Zen of Python\nclaims, "Readability counts."\nRead more: import this.'

The \n you see in the value is the newline character: Python stored the line breaks from the source right inside the one string. That is the whole syntax lesson here, that triple quotes let one string value stretch across several physical lines, with each break preserved as a \n in the single value Python builds.

Converting Values to Strings

You cannot add a number directly to a string, because + has no meaning across those two types.

# (24)
2 + " is a number"

An integer and a string have no shared +, so Python rejects this. The fix is to first turn the number into its string form with the str function, which builds a string representation of any value.

# (25)
str(2) + " is a number"

The result is:

'2 is a number'

This is especially handy when you want to splice a non-string value into a message. The official example converts both a number and a whole list to strings before concatenating them.

# (26)
str(2) + ' is an element of ' + str(digits)

The result is:

'2 is an element of [1, 8, 2, 8]'

Notice that str(digits) turned the entire list into the text [1, 8, 2, 8]. The str function knows how to render lists, numbers, and most other values as readable text, which is exactly what concatenation needs.

⚠Common Beginner Traps

The first trap is reading a comprehension front to back as if it ran in that order. In [x * x for x in range(5)], the for x in range(5) part is what supplies values, and it effectively happens first; the front x * x is the rule applied to each value. Read the for clause first, then the front rule.

# (27)
[x * x for x in range(5)]

The second trap is forgetting that range(1, 1000) stops before 1000, so it covers 1 through 999. Off-by-one confusion here is a frequent source of wrong answers in searches like the perfect-number one.

The third trap is reaching for sum to join strings.

# (28)
sum(["a", "b"])

This does not concatenate text; sum is built for numbers and starts from 0, which has no meaning added to a string, so the call fails with TypeError: unsupported operand type(s) for +: 'int' and 'str'. Joining strings uses different tools entirely.

The fourth trap is confusing an index with the element stored there. The index is a position; the result of indexing is the value at that position.

# (29)
city[3]   # 'k', not 3

A fifth, quieter trap: membership behaves differently on strings than on lists. On a list, 8 in digits checks for the whole element 8. On a string, 'her' in 'Where' checks for a contiguous substring. Do not assume the two are the same test.

  3. sum([2, 3, 4], 5) uses 5 as the starting value instead of the default 0, then adds the elements: 5+2+3+4=14.
  4. city[0] is 'B'. A string is a sequence of characters, indexing starts at 0, and indexing a string returns a one-character string, so position 0 gives the string 'B'.
  5. Python cannot apply + to an integer and a string because they have no shared meaning for that operator. The smallest fix is to convert the number with str(2), giving str(2) + " apples", which yields '2 apples'.
  6. keep_if(filter_fn, s) is the higher-order version of the comprehension's if clause: both keep only the elements for which a predicate is true. The comprehension writes the predicate inline as the if expression, while keep_if takes it as a function argument, so keep_if(perfect, ...) is the same filter with the test named perfect.

§Summary

Sequence processing is built from three repeated moves: transforming each element, filtering to keep some elements, and aggregating many elements into one result. List comprehensions express transform and filter compactly while keeping the loop visible, with the for ... in ... clause supplying values, an optional if clause filtering them, and a front expression building each result. The same moves generalize into higher-order functions, where apply_to_all, keep_if, and reduce (and their built-in cousins map, filter, and functools.reduce) take the per-element work as a function argument, which is how a single tool can be reused across very different problems. Strings turn out to be sequences of characters, so length, indexing, membership, slicing, and concatenation all carry over, with the one wrinkle that string membership tests substrings rather than whole elements. Once you can name which move a problem needs, both the comprehension and the higher-order spelling follow directly.

▶Step-Through Checkpoint

This program runs the same transformation two ways so you can compare a hand-built loop with a comprehension. The manual loop is longer on purpose: each pass builds a one-element list, concatenates it onto the running result, and rebinds the name to the new list, which is exactly the machine model a comprehension hides. Watching this as an environment diagram is the point of the exercise: inside the squares_two_ways frame you see manual get rebound to a brand-new list object on every pass, and then comprehension snap to an equal list in one step, so the diagram lays the comprehension's hidden rebinding right next to the loop's explicit one. Before you step through the block below, write down three predictions: what manual holds after the loop's third pass, what list comprehension is bound to once its line finishes, and what value the call on the last line returns.

Step through it one line at a time and check each prediction as the bindings change. Inside the call to squares_two_ways, you should have seen manual grow one element per pass, reaching [1, 9, 25] after the third pass and [1, 9, 25, 49, 81] after the last, while comprehension arrives at that same list without the rebinding spelled out step by step. The final call returns True, because both routes produce identical lists. The point to take away is that the comprehension and the loop are two spellings of one computation; the comprehension is shorter only because you already understand the loop it stands for.

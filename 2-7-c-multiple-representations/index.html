Chapter 2/2.7 Object Abstraction

2.7 Multiple Representations

◷ approx. 14 min read · 7 worked examples

By the end of this lesson you will be able to do four things: explain why one kind of abstract data might have more than one good concrete representation, describe what an interface is and why it lets different representations coexist, read and write the @property decorator so a computed value can look like a stored attribute, and trace a single arithmetic operation across two objects that store their data in completely different ways.

The running example is the complex number. Python already ships a built-in complex type, so we are not building this to use it. We are building it because it is the cleanest place to watch a single design idea work: when two representations agree on what messages they answer, code written against those messages does not care which representation it is talking to.

Two Maps of the Same Place

Imagine giving someone directions to a park.

One person says:

Go 3 blocks east and 4 blocks north.

Another person says:

Walk 5 blocks at about 53 degrees north of east.

Both descriptions land you at the exact same corner. They are two representations of one destination:

  • rectangular: how far east and how far north
  • polar: how far in total, and in which direction

Neither one is "the real location." Each is a faithful description, and each is convenient for a different task. Rectangular is easy when you want to combine two trips ("go east 3, then east 2 more"). Polar is easy when you want to talk about distance and heading.

Complex numbers carry exactly these two descriptions, and we will see that each makes a different operation easy.

Representation Stored information Good for
Rectangular real part and imaginary part addition
Polar magnitude and angle multiplication

Complex Numbers as Points

A complex number can be written as:

c=a+bi

Here a is the real part, b is the imaginary part, and i is the imaginary unit with the defining property i2=−1. We can picture a+bi as a point on a plane: a measured along the real axis, b measured along the imaginary axis.

imaginary axis
     ^
     |
  b  |          point (a, b)
     |            *
     |
-----+----------------> real axis
     |
     |      a

In rectangular form, that point is named by its two coordinates:

(real,imag)

In polar form, the same point is named by its distance from the origin and its direction:

(magnitude,angle)

The magnitude is the straight-line distance from the origin, which the Pythagorean theorem gives as:

magnitude=real2+imag2​

The angle is the direction measured from the positive real axis.

Why does each form suit a different operation? Adding complex numbers means adding their real and imaginary coordinates separately, which is trivial in rectangular form:

(a+bi)+(c+di)=(a+c)+(b+d)i

Picture each complex number as an arrow (a "vector") drawn from the origin out to its point. It turns out that multiplying two complex numbers stretches and rotates these arrows: one arrow grows by the length of the other and turns by the angle of the other. You can take this as a given; the payoff is that it makes polar form (length and angle) the natural fit for multiplication. That is exactly what polar form records, so multiplication just multiplies magnitudes and adds angles:

(r1​,θ1​)×(r2​,θ2​)=(r1​r2​, θ1​+θ2​)

So neither representation is best for everything. That is the whole motivation for keeping both.

A Shared Numeric Superclass

We start at the most general level. Whatever a number is, we want to be able to add it and multiply it with the ordinary + and * operators. Here is a class that promises exactly that and nothing more.

# (1)
class Number:
    def __add__(self, other):
        return self.add(other)

    def __mul__(self, other):
        return self.mul(other)

Number does not define add or mul. It only says how the operators should be handled: when Python sees x + y and x is a Number, it calls x.__add__(y), which turns around and calls x.add(y). The arithmetic itself is left to whatever subclass x actually belongs to.

This is message passing. The operator + is a message; different subclasses can answer it in different ways by supplying their own add. Number is never meant to be instantiated on its own. It is a superclass that gives every numeric subclass the same operator surface, so the choice of + versus add lives in one place.

Anatomy: the operator method body

Here is the single line inside __add__, broken into its tokens. Read it left to right and notice that the whole point of the line is to forward the work to another method.

return self.add(other)
├─ return ........ hand a value back to whoever evaluated x + y
├─ self .......... the left operand (the object on the left of +)
├─ .add .......... look up an add method on that left operand
├─ (other) ....... pass the right operand as the argument
└─ evaluates to .. whatever self.add(other) returns

So the chain for an expression like x + y is short and mechanical:

x + y          ->  x.__add__(y)   ->  x.add(y)

The special method __add__ is just the bridge from operator syntax to the class-specific add method. __mul__ does the identical thing for *.

The Complex Interface

We decided addition is easiest in rectangular form and multiplication is easiest in polar form. So we want every complex-number object, no matter how it stores itself internally, to be able to answer four attribute messages:

real
imag
magnitude
angle

That set of shared attribute names, together with the promise of what each one means, is the interface. Any object that answers all four can stand in for a complex number.

Once we trust that interface, we can write the arithmetic once, in a Complex class, without ever asking how a given operand is stored.

# (2)
class Number:
    def __add__(self, other):
        return self.add(other)

    def __mul__(self, other):
        return self.mul(other)


class Complex(Number):
    def add(self, other):
        return ComplexRI(self.real + other.real, self.imag + other.imag)

    def mul(self, other):
        magnitude = self.magnitude * other.magnitude
        return ComplexMA(magnitude, self.angle + other.angle)

Look at what each method assumes. add reads self.real, other.real, self.imag, and other.imag, and builds a rectangular result. mul reads self.magnitude, other.magnitude, self.angle, and other.angle, and builds a polar result. Neither method asks "are you rectangular or polar?" Each asks only "can you give me these interface attributes?" That question is the abstraction barrier. Below it, a representation is free to store whatever it likes, as long as it can produce the four answers.

Rectangular Representation

The first concrete representation stores real and imag directly, and computes magnitude and angle on demand.

Expected output:

5
13.0
9
15.0

The object stores only two things:

real -> 5
imag -> 12

It does not store magnitude anywhere. When you ask for ri.magnitude, the @property method runs and computes it from the stored coordinates. For the point (5,12):

52+122​=25+144​=169​=13

Now the line ri.real = 9 changes the stored real part. The object becomes:

real -> 9
imag -> 12

Because magnitude is computed fresh every time, asking again gives a different answer:

92+122​=81+144​=225​=15

That is why the second print(ri.magnitude) shows 15.0 instead of 13.0. A stored copy of the magnitude would have gone stale; a computed property never does.

The angle property leans on atan2, and the polar class ahead uses sin and cos. These are the standard trig functions from Python's math module, and the angle they work with is measured in radians, where pi/2 is a quarter turn (straight up) and pi is a half turn (pointing left). You do not need to compute these by hand — trust that they convert between the rectangular and polar forms.

The @property Decorator

The decorator is what lets a method masquerade as an attribute. Here is the relevant construct, broken into tokens line by line.

First the decorator line:

@property
├─ @ ............ the decorator marker for the line below
├─ property .... the built-in that turns a method into a computed attribute
└─ applies to ... the next def, so reads of that name run the method

Then the method header it decorates:

def magnitude(self):
├─ def ......... begin a method definition
├─ magnitude ... the attribute name users will read as ri.magnitude
├─ (self) ...... the instance whose stored data the body will read
├─ : ........... starts the indented body
└─ defines ..... the value returned when someone reads ri.magnitude

Without @property, magnitude would be an ordinary method, and you would have to call it with parentheses:

ri.magnitude()

With @property, the parentheses disappear and the access reads like plain data:

ri.magnitude

That matters here because the interface promises four attribute names — real, imag, magnitude, angle — and they should all look the same to a user. Some happen to be stored and some happen to be computed, but from the outside they are uniform. The user reads ri.angle exactly the way they read ri.real, with no hint that one is a lookup and the other is a calculation.

A tiny bridge you can verify by hand

Before the polar class, here is a stripped-down property you can check in your head. It stores a temperature in Celsius and exposes Fahrenheit as a computed attribute.

Expected output:

212.0
32.0

Only celsius is stored. Reading t.fahrenheit runs the method: 100×9/5+32=212.0. After t.celsius = 0, the same read recomputes to 0×9/5+32=32.0. This is the identical pattern ComplexRI uses for magnitude and angle — store one form, compute the other on every read.

Polar Representation

The second concrete representation flips which pair is stored. It keeps magnitude and angle directly, and computes real and imag on demand.

# (5)
from math import sin, cos, pi


class ComplexMA(Complex):
    def __init__(self, magnitude, angle):
        self.magnitude = magnitude
        self.angle = angle

    @property
    def real(self):
        return self.magnitude * cos(self.angle)

    @property
    def imag(self):
        return self.magnitude * sin(self.angle)

    def __repr__(self):
        return "ComplexMA({0:g}, {1:g} * pi)".format(
            self.magnitude,
            self.angle / pi,
        )


ma = ComplexMA(2, pi / 2)
print(ma.imag)
ma.angle = pi
print(ma.real)

Expected output:

2.0
-2.0

When ma = ComplexMA(2, pi / 2), the object stores:

magnitude -> 2
angle     -> pi / 2

An angle of π/2 points straight up the imaginary axis. A vector of length 2 pointing straight up has imaginary coordinate 2.0, which is what ma.imag computes through 2 * sin(pi/2).

Then ma.angle = pi rotates the vector to point left along the real axis. Length 2 pointing left has real coordinate -2.0, which ma.real computes through 2 * cos(pi). The magnitude was untouched; only the angle changed, and the computed real followed automatically.

Notice this class is the mirror image of ComplexRI. There, real and imag were stored and magnitude, angle were properties. Here, magnitude and angle are stored and real, imag are properties. Both classes expose the same four names.

Combining Both Representations

Both representations now honor the same interface, but each stores a different pair and computes the other.

Attribute ComplexRI ComplexMA
real stored computed
imag stored computed
magnitude computed stored
angle computed stored

Because the interface matches, the generic add and mul in Complex work no matter which representation each operand uses. Here is the whole system assembled, with two mixed operations at the bottom.

Expected output:

ComplexRI(1, 4)
ComplexMA(1, 1 * pi)

Work through the first line, ComplexRI(1, 2) + ComplexMA(2, pi / 2). The rectangular operand has real=1 and imag=2. The polar operand has magnitude 2 and angle π/2, which points straight up, so its computed coordinates are real=0 and imag=2. Addition adds coordinates:

(1+0)+(2+2)i=1+4i

Complex.add builds a rectangular result, so the answer prints as ComplexRI(1, 4). The polar operand was read purely through its real and imag properties; add never knew it was stored in polar form.

Now the second line, ComplexRI(0, 1) * ComplexRI(0, 1). The number ComplexRI(0, 1) is i. Multiplying it by itself:

i×i=i2=−1

Complex.mul builds a polar result by multiplying magnitudes and adding angles. Each operand i has magnitude 1 and angle π/2, so the product has magnitude 1×1=1 and angle π/2+π/2=π. A vector of length 1 at angle π is exactly -1, and it prints as ComplexMA(1, 1 * pi).

Method-Dispatch Trace: One Addition Across Two Representations

This is the moment that justifies the whole design, so trace it slowly. Evaluate:

ComplexRI(1, 2) + ComplexMA(2, pi / 2)

Right before the +, two objects exist with different stored attributes:

left operand: a ComplexRI instance
  real -> 1
  imag -> 2

right operand: a ComplexMA instance
  magnitude -> 2
  angle     -> pi / 2

Now follow how Python resolves the +, step by step, paying attention to where each method is found:

1. x + y triggers x.__add__(y); x is a ComplexRI, which has no
   __add__, so Python climbs to Complex, still none, then to
   Number and finds Number.__add__.
2. Number.__add__ runs self.add(other).
3. self.add looks up add on the ComplexRI instance: not on the
   instance, not on ComplexRI, found on Complex as Complex.add.
4. Complex.add reads self.real and self.imag.
   self is the ComplexRI, so these are the stored values 1 and 2.
5. Complex.add reads other.real and other.imag.
   other is the ComplexMA, which has NO stored real or imag, so
   each read runs a @property: real = 2*cos(pi/2) = 0,
   imag = 2*sin(pi/2) = 2.
6. Complex.add computes (1 + 0) and (2 + 2), giving 1 and 4.
7. Complex.add returns ComplexRI(1, 4).

The key is step 5. The exact same line of code, other.real, was a plain attribute lookup for the left operand and a property computation for the right operand, and Complex.add could not tell the difference. That is the abstraction win: one arithmetic algorithm, written once, drives two unrelated storage strategies.

⚠Common Beginner Traps

Trap 1: Thinking every attribute must be stored. An attribute can be produced by a @property method at read time. From the outside, ri.magnitude looks like stored data even though it is recomputed on every access.

Trap 2: Adding parentheses to a property. Because magnitude is a property, you read it as ri.magnitude, not ri.magnitude(). Adding parentheses tries to call the number you got back and raises TypeError: 'float' object is not callable.

Trap 3: Letting the interface drift. If one representation forgets to provide angle, then Complex.mul breaks the moment it touches that object, because the shared arithmetic depends on all four names being present.

Trap 4: Confusing representation with meaning. ComplexRI(0, 1) and ComplexMA(1, pi / 2) are two representations of the same mathematical number i. Different storage, identical value.

Trap 5: Checking the representation too early. The point of the shared interface is that Complex.add and Complex.mul never ask whether an operand is rectangular or polar. The first instinct to write if isinstance(other, ComplexRI) defeats the abstraction.

  2. Which attributes does ComplexRI store, and which does it compute?
  3. What does the @property decorator change about how magnitude is accessed?
  4. Why can ComplexRI(1, 2) + ComplexMA(2, pi / 2) succeed even though the two objects store different attributes?
  5. After ri = ComplexRI(5, 12), why does ri.magnitude change from 13.0 to 15.0 once you run ri.real = 9?
  6. Where is the add method actually found when you write ComplexRI(1, 2) + ComplexMA(2, pi / 2)?
Show the answers
  1. ComplexMA(6, 1 * pi). Both operands are polar, so Complex.mul multiplies the magnitudes (2×3=6) and adds the angles (π/2+π/2=π), and mul returns a ComplexMA.
  2. ComplexRI stores real and imag, and computes magnitude and angle with @property methods from those stored coordinates.
  3. With @property, you read it as ri.magnitude with no parentheses, so a computed value looks and is used like a stored attribute.
  4. Both classes honor the same interface: each can answer real, imag, magnitude, and angle. Complex.add only needs real and imag, and the polar object supplies them through computed properties.
  5. magnitude is not stored; it is recomputed on every read. After ri.real = 9 the stored real part is 9, so the next computation gives 92+122​=15.0.
  6. Python looks for add on the ComplexRI instance, then on ComplexRI, and finds it on the shared Complex superclass as Complex.add.

§Summary

Multiple representations let one kind of abstract data live in more than one concrete form. The mechanism that makes this safe is a shared interface: a set of attribute names, plus the promise of what each means. For complex numbers the interface is real, imag, magnitude, and angle. The rectangular class ComplexRI stores real and imag and computes the rest; the polar class ComplexMA stores magnitude and angle and computes the rest. The @property decorator is what lets a computed value be read like a stored attribute, so both representations look identical from the outside. Because the generic add and mul in Complex speak only to the interface, the arithmetic is written once and works across both representations through message passing. The interface is also additive: a new representation just needs to answer the same four names to join the system. Next we widen the question past complex numbers: what happens when the operands are genuinely different numeric types, such as a complex number and a rational number, and we need generic functions to combine them.

▶Step-Through Checkpoint

Both representations now meet in one mixed addition. The environment diagram lays bare the abstraction win: a ComplexRI object holds real and imag on the heap while a ComplexMA object holds magnitude and angle, and as the + climbs to Number.__add__ and then Complex.add, the reads of other.real and other.imag open @property frames on the polar object that compute coordinates the diagram never stored. Before you step through the block below, write down two predictions: how many ComplexRI.__init__ frames will open over the whole run, and what exact value the polar operand's real property will return when Complex.add reads it.

Step through it one line at a time and check each prediction as the frames appear. The + runs Number.__add__, then Complex.add, whose property reads on the polar operand return exactly 2.0 for imag but not quite 0 for real: 2 * cos(pi/2) lands near 1.22e-16, a floating-point sliver the {0:g} format hides when ComplexRI(1, 4) prints. Here 1.22e-16 means 0.000...122, a tiny rounding error from how computers store decimals, and the {0:g} in the repr (a number-formatting code) rounds it away so the result still prints as a clean 1 + 4i. You should also have seen ComplexRI.__init__ open exactly twice, once for the left operand and once for the result, and Complex.add never knew the right operand was polar.

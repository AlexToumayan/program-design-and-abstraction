Chapter 2/2.9 Recursive Objects

2.9 The Tree Class and Shared Structure

◷ approx. 14 min read · 8 worked examples

You already know how to build a recursive object: a linked list is an object that holds one value and a reference to another object of the same kind. A tree is the next step up. Instead of pointing to exactly one smaller object, a tree node points to a whole sequence of smaller trees.

By the end of this lesson you will be able to do four things. You will read and write the Tree class, which stores a label and a sequence of branches. You will write recursive functions that walk a tree by walking each branch, using sum_labels as the model. You will see why building trees out of objects lets two different parts of one tree point at the same shared object. And you will measure why that sharing matters, by counting how many Tree objects a computation actually creates.

The habit underneath all of this is the same one recursion always asks for: trust that the function works on a smaller piece, combine the smaller answers, and stop at the base case.

A Family Tree of Index Cards

Picture a stack of index cards describing a family. The top card names one person. Clipped underneath it are a few more cards, one for each of that person's children. Each of those child cards may itself have cards clipped underneath it, for grandchildren, and so on. There is no special "child card" material — a child card is just another card that happens to be clipped below a parent.

That picture is exactly a tree. The top card is the root. The writing on a card is its label. The cards clipped directly underneath a card are its branches. The crucial part is that each branch is itself a complete card-with-cards-underneath — a whole tree in its own right. A card with nothing clipped under it is a leaf.

This is what we mean when we say a tree is a recursive object: a tree is a label together with a sequence of branches, and every one of those branches is again a tree.

Trees as Objects With Labels and Branches

Earlier in the chapter, trees were built out of built-in sequences. Here we represent a tree as an instance of a class we define ourselves. A tree is any object that carries a label and a sequence of branches, where every branch is also a tree.

The word label is worth pinning down. In some tree definitions, the meaningful values live only at the leaves. Here, every node carries a value, including the nodes in the middle. A value stored at the root of a subtree, not just at a leaf, is called the label of that subtree. So the root has a label, and each branch has its own label, all the way down.

Two pieces of vocabulary will repeat throughout the lesson:

  • The label is the value stored at this node.
  • The branches are the sequence of subtrees hanging off this node. A tree with no branches is a leaf.

The Tree Class

Here is the class. It stores the label, checks that every branch really is a Tree, keeps the branches, and adds two conveniences: a readable printed form and a test for whether a tree is a leaf.

# (1)
class Tree:
    def __init__(self, label, branches=()):
        self.label = label
        for branch in branches:
            assert isinstance(branch, Tree)
        self.branches = branches
    def __repr__(self):
        if self.branches:
            return 'Tree({0}, {1})'.format(self.label, repr(self.branches))
        else:
            return 'Tree({0})'.format(repr(self.label))
    def is_leaf(self):
        return not self.branches

The default value branches=() is an empty tuple, which is what you get when you build a leaf by writing Tree(5) with no branches given. The for loop runs the assertion once per branch; if any branch is not a Tree, the assertion stops construction immediately. Notice that the constructor stores self.branches = branches directly — it does not convert the branches into a list or copy them. Whatever sequence you hand in is kept as is.

Reading the Constructor Header

The header line of __init__ names the data every tree carries. Read it one token at a time.

def __init__(self, label, branches=()):
├─ def .............. begins a function (method) definition
├─ __init__ ......... the constructor, run when a Tree is built
├─ ( ................ begins the parameter list
├─ self ............. the new Tree object being initialized
├─ , ................ separates parameters
├─ label ............ the value to store at this node
├─ , ................ separates parameters
├─ branches ......... the sequence of subtrees for this node
├─ =() .............. default: an empty tuple, so a leaf needs no branches
├─ ) ................ ends the parameter list
├─ : ................ begins the method body
└─ defines how a new Tree stores its label and branches

Reading the Branch Check

This is the line that protects the structure. It guarantees that everything inside branches is itself a tree, so recursive code can rely on it.

assert isinstance(branch, Tree)
├─ assert .......... raise an error unless the condition is true
├─ isinstance ...... the built-in type test
├─ ( ............... begins the inputs
├─ branch .......... one element pulled from branches by the loop
├─ , ............... separates the inputs
├─ Tree ............ the class we are requiring branch to be
├─ ) ............... ends the inputs
└─ stops construction unless this branch is itself a Tree

Reading the is_leaf Method

A leaf is a tree with no branches. The method says exactly that, using the fact that an empty sequence is falsy in Python.

return not self.branches
├─ return .......... hand a value back to the caller
├─ not ............. logical negation: turns truthy into False, falsy into True
├─ self ............ the tree this method was called on
├─ . ............... attribute access
├─ branches ........ this tree's sequence of subtrees
└─ evaluates to True when branches is empty, False otherwise

A Warm-Up: The Smallest Tree

The smallest possible tree is a single leaf. Build one and ask whether it is a leaf.

# (2)
smallest = Tree(5)
smallest.label
smallest.is_leaf()

Because no branches were passed, branches defaults to the empty tuple (). So smallest.label is 5, and smallest.is_leaf() returns True. There is nothing recursive happening yet; this is just one node with a value and an empty branch sequence.

A Bridge You Can Check by Hand

Now build a tree that has structure but is still small enough to read at a glance: a root labeled 3 with two branches, where the second branch has two leaves of its own.

# (3)
t = Tree(3, (Tree(1), Tree(2, (Tree(1), Tree(1)))))
t
t.label
t.branches[0].label
t.branches[1].is_leaf()
t.branches[1].branches[0].label

Walk the access expressions slowly. The whole object t prints using __repr__:

Tree(3, (Tree(1), Tree(2, (Tree(1), Tree(1)))))

Each later line drills one step deeper into the structure. The expression t.branches[1].branches[0].label reads as: take t, go to its branches, pick branch index 1 (the Tree(2, ...)), go to that subtree's branches, pick branch index 0 (a Tree(1)), and read its label.

t.branches[1].branches[0].label
├─ t ............... the whole tree, labeled 3
├─ .branches ....... its sequence of two subtrees
├─ [1] ............. the second subtree, Tree(2, (...))
├─ .branches ....... that subtree's sequence of two leaves
├─ [0] ............. its first leaf, Tree(1)
├─ .label .......... the value stored at that leaf
└─ evaluates to 1

Running all five lines gives:

Tree(3, (Tree(1), Tree(2, (Tree(1), Tree(1)))))
3
1
False
1

The fourth answer is False because t.branches[1] is the Tree(2, ...) node, and it has branches, so it is not a leaf.

The Official Example: Fibonacci Trees

You built this same fib_tree in 2.3-C over the list-based tree ADT; here is the same idea expressed with the Tree class, which now lets two branches literally share one object.

The textbook uses trees to record the shape of a recursive computation. The function fib_tree(n) returns a Tree whose label is the nth Fibonacci number and whose branches hold the two smaller Fibonacci trees that were combined to compute it. This is the expression tree for the recursive definition of fib.

# (4)
def fib_tree(n):
    if n == 1:
        return Tree(0)
    elif n == 2:
        return Tree(1)
    else:
        left = fib_tree(n-2)
        right = fib_tree(n-1)
        return Tree(left.label + right.label, (left, right))

The two base cases are leaves: fib_tree(1) is Tree(0) and fib_tree(2) is Tree(1). For larger n, the function builds the two subtrees first, then makes a new tree whose label is the sum of the two subtree labels and whose branches are those two subtrees, kept as a tuple (left, right).

# (5)
fib_tree(5)

The result records the entire recursive trace:

Tree(3, (Tree(1, (Tree(0), Tree(1))), Tree(2, (Tree(1), Tree(1, (Tree(0), Tree(1)))))))

The root label is 3 because the fifth Fibonacci number in this numbering is 3:

fib(5)=fib(3)+fib(4)=1+2=3

The left branch is fib_tree(3) and the right branch is fib_tree(4). Each of those is again a full tree showing how its own value was built, all the way down to the Tree(0) and Tree(1) leaves.

Summing the Labels of a Tree

Trees are processed with recursive functions, and the shape of the function mirrors the shape of the data. To total every label in a tree, add this node's label to the totals of all its branches.

# (6)
def sum_labels(t):
    """Sum the labels of a Tree instance, which may be None."""
    return t.label + sum([sum_labels(b) for b in t.branches])

The docstring mentions a None case for completeness; every tree in this lesson is a real Tree, so you can read the body as always receiving a Tree and not go hunting for missing None-handling.

The list comprehension calls sum_labels on each branch, producing a list of branch totals. The built-in sum adds that list up, and we add this node's own label on top. For a leaf, t.branches is empty, so the comprehension produces an empty list, sum([]) is 0, and the result is just the leaf's label. That empty case is the base case — it is what stops the recursion.

Read the body as one expression rather than a sequence of steps:

return t.label + sum([sum_labels(b) for b in t.branches])
├─ return .......... hand the total back to the caller
├─ t.label ......... the value stored at this node
├─ + ............... add this node's value to the branch total
├─ sum ............. built-in that adds a list of numbers
├─ ( ............... begins sum's input
├─ [sum_labels(b) for b in t.branches] .. a list of each branch's own total
├─ ) ............... ends sum's input
└─ evaluates to this node's label plus the totals of every branch

Apply it to the official Fibonacci tree:

# (7)
sum_labels(fib_tree(5))

10

Tracing sum_labels on a Small Tree

The fib_tree(5) total is hard to verify by eye, so trace a tree you can check: a root labeled 1 with leaf branches Tree(2) and Tree(3). By hand the total is 1+2+3=6. Here is how the recursion produces that.

sum_labels(Tree(1, (Tree(2), Tree(3))))
  t.label = 1
  branch totals:
    sum_labels(Tree(2)) -> 2 + sum([]) -> 2 + 0 -> 2
    sum_labels(Tree(3)) -> 3 + sum([]) -> 3 + 0 -> 3
  return 1 + sum([2, 3]) -> 1 + 5 -> 6

Each leaf hits the base case where sum([]) is 0, so it returns its own label. The root then adds its label to the sum of the branch totals. The same pattern runs on fib_tree(5); there are simply many more nodes, and repeated subtrees like Tree(1, (Tree(0), Tree(1))) get counted every time they appear.

Shared Structure: When Two Branches Are the Same Object

Because trees are objects, two different places in a tree can hold a reference to the very same object, not merely two equal-looking copies. The textbook makes this happen by applying memo to fib_tree. The memoized version stores each result the first time it is computed and returns that exact stored object on later calls with the same argument. So when two larger trees both need fib_tree(33), they end up sharing one Tree object as a branch.

>>> fib_tree = memo(fib_tree)
>>> big_fib_tree = fib_tree(35)
>>> big_fib_tree.label
5702887
>>> big_fib_tree.branches[0] is big_fib_tree.branches[1].branches[1]
True
>>> sum_labels = memo(sum_labels)
>>> sum_labels(big_fib_tree)
142587180

The identity test in the middle is the heart of the idea. The left branch of big_fib_tree is fib_tree(33), and the right branch of big_fib_tree is fib_tree(34), whose own right branch is again fib_tree(33). Two different paths through the tree arrive at the same subproblem. Without memoization those would be two separate objects; with memoization they are one shared object, so is reports True.

The last two lines apply memo to sum_labels as well. Here memo caches by the Tree object itself rather than by a number, but it works the same way you saw in 2.8 — compute the total once, then return the stored result on later calls with that same tree.

Reading the Identity Check

The is operator asks about object identity, not equality. Two trees can have identical labels and branches and still be different objects; is is True only when both sides name the exact same object in memory.

big_fib_tree.branches[0] is big_fib_tree.branches[1].branches[1]
├─ big_fib_tree.branches[0] ............... the left branch, fib_tree(33)
├─ is ..................................... identity test: same object?
├─ big_fib_tree.branches[1].branches[1] ... the right branch's right branch, also fib_tree(33)
└─ evaluates to True because memoization made both the one shared object

Why Sharing Matters: Counting Objects

The payoff of sharing is concrete. To build fib_tree(35) the naive way, every repeated subproblem is rebuilt from scratch, and the number of Tree objects created explodes: 18,454,929 of them. With the memoized fib_tree, each distinct subproblem is built once and then reused as a branch wherever it is needed, so only 35 Tree objects are ever created.

That is the difference between exponential work and linear work, made visible as a count of objects. Sharing also changes how you must think about mutation: if many branches point at one shared object and you mutate it through one reference, every other reference sees the change. Identity, not just equality, becomes part of the mental model once objects can be shared.

⚠Common Beginner Traps

Trap 1: Treating a branch as if it were a label. A branch is a whole Tree object, not a number. Writing t.branches[0] + 1 raises TypeError: unsupported operand type(s) for +: 'Tree' and 'int', and printing t.branches[0] shows a whole Tree(...) rather than a value. To reach the value inside it, go through .label: t.branches[0] is a tree, and t.branches[0].label is its value.

Trap 2: Assuming the constructor converts branches to a list. It does not. The line self.branches = branches stores exactly the sequence you passed in. If you pass a tuple, branches is a tuple; the default for a leaf is the empty tuple ().

Trap 3: Confusing equality with identity. Two trees built separately can have the same labels and the same branch structure and still be different objects. == would compare contents; is asks whether they are literally the same object. The shared-structure example is True only because memoization produced one object used in two places.

Trap 4: Forgetting the base case in tree recursion. In sum_labels, the base case is hidden in plain sight: a leaf has an empty branches, so the comprehension is empty and sum([]) is 0. If your tree function never bottoms out, check that an empty branch sequence actually stops the recursion.

Trap 5: Expecting fib_tree's numbering to start at zero. In this definition, fib_tree(1) is Tree(0) and fib_tree(2) is Tree(1). The base cases are on n == 1 and n == 2, not n == 0.

Show the answers
  1. The root stores the label 3. Its branches are a sequence of two Tree objects, Tree(1) and Tree(2), both of which are leaves.
  2. It returns 12. The root contributes 4, the leaf Tree(2) contributes 2, the subtree Tree(5, (Tree(1),)) contributes its label 5 plus its one leaf 1, and 4+2+5+1=12.

§Summary

A Tree is a recursive object that stores a label and a sequence of branches, where every branch is again a Tree. The class checks that branches are trees, keeps the branch sequence exactly as given, prints itself readably with __repr__, and reports leaves with is_leaf. Functions over trees are recursive in the same shape as the data: fib_tree builds a tree from two smaller trees, and sum_labels totals a node's label with the totals of its branches, bottoming out when the branch sequence is empty. Because branches are objects, two parts of one tree can share a single object; applying memo to fib_tree makes the repeated Fibonacci subtrees literally the same object, which is confirms. That sharing is not a curiosity: it cuts fib_tree(35) from 18,454,929 created objects down to 35, turning identity into a practical tool, not just a definition.

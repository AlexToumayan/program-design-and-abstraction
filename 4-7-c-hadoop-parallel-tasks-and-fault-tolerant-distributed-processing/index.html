Chapter 4/4.7 Distributed Data Processing

4.7.3 Distributed Implementation: Hadoop, Parallel Tasks, and Fault Tolerance

◷ approx. 19 min read · 4 worked examples · best read in two passes

In the previous lesson you built a working MapReduce on one machine. You wrote a mapper that reads standard input and writes standard output, you wrote a reducer that does the same, and you wired them together with a Unix pipe. It runs. It produces the right vowel counts. But it runs on one computer, reading one stream, doing one line at a time.

This lesson answers the next question: what changes when the dataset is too big for one machine, and the work has to be spread across a cluster of machines that can crash, lose network connections, and stop in the middle of a task?

By the end you will be able to explain what Hadoop is, describe the three things it buys you over your local pipeline (speed, fault tolerance, and monitoring), read the exact command that runs the vowel-count job on a cluster, and say precisely why the same mapper and reducer files you already wrote can be reused without a single edit. The deepest idea here is one you have met before, now paying off: because your map and reduce functions are pure, a failed task can simply be run again.

A note before we start: this packet does not require you to install Hadoop or run anything on a cluster. The shell commands here are shown so you can read the interface, not so you can execute them. Where a command appears, it is reproduced exactly as the official text writes it.

An Intuitive Analogy: The Grading Coordinator

Imagine you are running grading for an exam taken by ten thousand students. You cannot grade ten thousand exams yourself in any reasonable time, so you hire a room full of graders.

You do not grade. Your job is coordination. You split the stack of exams into bundles and hand one bundle to each grader. You watch the room. If a grader gets sick halfway through their bundle and walks out, you notice that their bundle never came back finished, so you hand that same bundle to a different grader who is free. At the end you collect every grader's tallies and add them up into one final result. You also keep a board at the front showing how many bundles are done, how many are still out, and whether the whole operation is still moving.

Three things make this work, and they map exactly onto what is coming:

  • Speed comes from many graders working at once, not from any one grader becoming faster. Twenty graders finish roughly twenty times sooner than one.
  • Recovery from a lost grader is safe only because grading a given bundle gives the same tally no matter who does it. A bundle handed to a replacement produces the same numbers the first grader would have. If grading depended on the grader's mood or on the order bundles were graded in, you could not safely re-hand a bundle.
  • The board at the front is monitoring. Without it, you would have no idea whether the job is halfway done or stuck.

In this analogy, you are the framework. The graders are tasks running on machines. The bundles are chunks of the input. The unchanging "same bundle gives the same tally" rule is purity. Hadoop is the software that plays your coordinator role automatically, so that the people writing the actual grading instructions, the map and reduce functions, never have to think about sick graders or the board at the front.

What Hadoop Is

The official text introduces the distributed implementation in one sentence, and it is worth reading slowly:

Hadoop is the name of an open-source implementation of the MapReduce framework that executes MapReduce applications on a cluster of machines, distributing input data and computation for efficient parallel processing.

Unpack that. A cluster is a group of computers working together as one system. "Open-source implementation of the MapReduce framework" means Hadoop is real, downloadable software that plays the coordinator role we have been describing in the abstract since the start of this section. You still supply the map and reduce functions, the application logic. Hadoop supplies everything else: splitting the input, placing work on machines, moving intermediate data between map tasks and reduce tasks, retrying failures, and reporting progress.

This is the same separation of concerns you saw in the very first lesson of this section, now made concrete in a named tool:

your map function + your reduce function + a large input
    -> Hadoop splits the input and distributes it across the cluster
    -> Hadoop runs map tasks and reduce tasks on many machines
    -> Hadoop stores the final output for you to read

The application code describes the computation. The framework coordinates the cluster. Nothing about that division is new; what is new is that a production-grade framework exists to hold up the framework side.

The Streaming Interface: Reuse Without Editing

Here is the part that should feel like a reward for the work you did in the previous lesson. The official text continues:

Its streaming interface allows arbitrary Unix programs to define the map and reduce functions. In fact, our count_vowels_mapper.py and sum_reducer.py can be used directly with a Hadoop installation to compute vowel counts on large text corpora.

Read that twice. The exact two files you wrote for the one-machine pipeline, with no changes, can serve as the map and reduce functions for a job spread across a cluster:

count_vowels_mapper.py
sum_reducer.py

Why does this work? Because of the contract those programs already follow. A Hadoop streaming mapper or reducer is just an ordinary program that reads lines from standard input and writes lines to standard output. That is precisely the contract your local mapper and reducer already honor. They do not know or care whether the bytes on standard input arrived through a Unix pipe on your laptop or were fed in by Hadoop from a chunk of a giant corpus on some machine in a data center. The program reads, transforms, writes. The source of the input and the destination of the output are someone else's problem.

This is abstraction doing exactly its job. The same artifact runs in two wildly different environments because both environments agree on one narrow interface: text in on standard input, text out on standard output. You did not write "local code" in the last lesson. You wrote map and reduce functions that happen to also run on a cluster.

The streaming interface is a documented feature of Hadoop with rules and options well beyond what we need here. For more information on the Hadoop streaming interface and use of the system, consult the Hadoop Streaming Documentation. We do not reproduce that documentation in this packet; the point for now is simply that it exists and describes the same standard-input/standard-output contract your two files already satisfy.

Follow the Line: The Streaming Reuse Claim

When the text says the local files "can be used directly," it is making a precise claim about one of them. Read the mapper's own name token by token, the way we break any line into its smallest meaningful pieces:

count_vowels_mapper.py
├─ count_vowels .............. what the program computes: a count of vowels
├─ _mapper ................... its role: the map step of the MapReduce job
├─ .py ....................... a Python script, run as an ordinary Unix program
└─ result ................... reads standard input, writes standard output, so Hadoop's streaming interface accepts it unchanged

The single fact that makes the result row true is the input/output contract. The mapper never named a file, never opened a socket, never assumed it was the only process running. It read standard input and wrote standard output, so it slots into Hadoop's streaming interface unchanged, and sum_reducer.py reuses unchanged for exactly the same reason.

The Three Advantages Over a Local Pipeline

Your local pipeline works, but it is, in the words of the official text, "simplistic." Hadoop offers several advantages over it. The text names exactly three. We will take them in order, and we will be careful to claim only what the source claims.

Speed

The first is speed: map and reduce functions are applied in parallel using different tasks on different machines running simultaneously.

This is the grading room. If a corpus has millions of lines, one mapper grinding through them one at a time is slow. Hadoop splits the input and runs many map tasks at the same time, each on its own machine, each chewing through its own slice. Reduce work can likewise be split across tasks, since different keys can be combined independently of one another.

The crucial mental correction is this: the job is not faster because any single line became easier to process. Counting the vowels in one line takes exactly as long as it always did. The job is faster because many lines are being processed at the same moment on different machines. Picture the input split into chunks: one chunk goes to a map task on machine A, another to a map task on machine B, a third to a map task on machine C, and all of those tasks run simultaneously rather than one after another. Total wall-clock time drops because the work overlaps. Parallelism multiplies throughput; it does not change the cost of one unit of work.

Fault Tolerance

The second is fault tolerance: when a task fails for any reason, its result can be recomputed by another task in order to complete the overall computation.

Distributed systems must assume failure as a normal event, not a rare disaster. With enough machines, at any given moment something is probably broken: a disk dies, a network link drops, a process is killed. The official guarantee is simple and strong: when a task fails for any reason, its result can be recomputed by another task, and the overall computation can still complete.

Notice the exact wording. The source does not describe a detection mechanism in detail; it states the property. A failed task's result can be recomputed by another task. That is the lost grader handed off to a free one.

And here is the payoff for a property you learned to insist on lessons ago. This safety to recompute exists only because the map and reduce functions are pure. A pure function returns the same output for the same input every time, with no hidden state and no side effects. So running a failed task's input again, possibly on a different machine, produces exactly the result the failed task would have produced. The retry does not change the meaning of the job. If the functions could behave differently on a second run, recomputation would be a gamble instead of a guarantee. Purity is what turns "try again" from a hope into a correctness property.

Monitoring

The third is monitoring: the framework provides a user interface for tracking the progress of a MapReduce application.

A real job may have thousands of tasks spread over many machines, and a human running it needs to know whether it is making progress or quietly stuck. Hadoop provides a user interface for tracking the progress of the application. That is the board at the front of the grading room, and you get it for free. You do not build a progress-tracking system for every job you write; the framework provides one.

Running the Vowel Count on Hadoop

The official text gives the setup at a high level and then the exact command. The setup, in its own words, is to install Hadoop, point the framework at your installation, get your data into the cluster's storage, and run. Reproducing the steps faithfully:

install Hadoop
change the assignment statement of HADOOP to the root of your local installation
copy a collection of text files into the Hadoop distributed file system
run the MapReduce job

A few things are worth pinning down so you read these steps correctly. The text refers to "the provided mapreduce.py module" as the support code, and HADOOP is a variable inside that setup whose assignment you edit to point at where Hadoop lives on your machine. The "Hadoop distributed file system" is the cluster's own storage, spread across the machines; your input has to live there, not on your laptop's local disk, before the job can run on it.

Then comes the command itself. This is reproduced exactly as the official text writes it, and because it is a shell command, not Python, it is fenced as a shell session with no numbered label:

$ python3 mr.py run count_vowels_mapper.py sum_reducer.py [input] [output]

The text then tells you what the last two arguments are: where [input] and [output] are directories in the Hadoop file system.

Follow the Line: The Hadoop Run Command

This single command line carries the whole job. Walk it argument by argument, exactly as written. The leading $ is the shell prompt; everything after it is what you would type.

$ python3 mr.py run count_vowels_mapper.py sum_reducer.py [input] [output]
├─ python3 ................... run the Python 3 interpreter
├─ mr.py ..................... the MapReduce driver script you invoke
├─ run ....................... the subcommand telling it to run a job
├─ count_vowels_mapper.py .... the map program (your unchanged local mapper)
├─ sum_reducer.py ............ the reduce program (your unchanged local reducer)
├─ [input] ................... a directory in the Hadoop file system holding the input
├─ [output] .................. a directory in the Hadoop file system for the results
└─ action ................... Hadoop runs the vowel-count job across the cluster

Two cautions about reading this line. First, [input] and [output] are placeholders, not Python lists and not literal text you type with the brackets. They stand for real directory paths in the Hadoop file system that you substitute in. The square brackets are the convention for "fill this in," the same way function_name(argument) uses placeholders rather than literal names. Second, you may have noticed the support code is called mapreduce.py in the setup prose but the command invokes mr.py. That is how the official text presents it, and we reproduce it faithfully rather than silently smoothing it over; treat mr.py as the script name you actually run, exactly as the command shows.

Why This Is Not Just a Faster Pipe

It is tempting to think Hadoop is "the same as my pipeline, but quicker." It is not, and the difference is the real lesson of this section.

Your local run is a single stream on a single machine. In the previous lesson you ran the concrete job like this:

$ cat haiku.txt | ./count_vowels_mapper.py | sort | ./sum_reducer.py

One process reads one file, pipes its output to a sort, pipes that to one reducer. One machine, one stream, start to finish.

The distributed version replaces that single stream with many map tasks and reduce tasks running on many machines at once, reading from and writing to storage that is itself spread across the cluster, with the framework retrying any task that fails and reporting progress the whole time. None of that difficulty went away. Machines really do crash; data really is scattered; tasks really do get retried. What changed is that you do not have to see any of it. The command you type is barely longer than the pipeline, and the two files you point it at are byte-for-byte the same files you wrote for one machine.

That is the central idea: abstraction can make a genuinely hard system usable without making the underlying difficulty disappear. The complexity is still there. The framework is holding it for you.

A Step-by-Step Trace: One Task Fails and Is Recomputed

Let us trace the fault-tolerance story concretely, because it is the heart of why purity matters. Suppose the input is split into three chunks, and a job runs map tasks on three machines.

Step 1. Hadoop splits the input into chunk A, chunk B, chunk C.
Step 2. Hadoop starts a map task for each chunk on a different machine, all at once.
Step 3. The tasks for A and C finish and hand back their key-value pairs.
Step 4. The machine running the task for B crashes before that task finishes.
Step 5. Because the result for chunk B never completed, Hadoop runs that task again
        on another machine, feeding it the same chunk B.
Step 6. The retried task produces the same pairs the failed one would have, because
        the map function is pure: same input, same output.
Step 7. With results for A, B, and C all present, grouping and reducing proceed,
        and the overall computation completes.

Step 6 is the whole point. The retry is correct, not lucky, because the map function depends only on its input. If counting vowels in chunk B could give different answers on a second run, step 6 would corrupt the result instead of completing it. Purity is the property that makes "run it again somewhere else" a safe move rather than a roll of the dice.

Progressive Examples: Modeling a Pure, Retryable Task

The official code for this section is the two files you already wrote, reused unchanged, plus the run command above. To build intuition for the fault-tolerance idea in plain Python you can run anywhere, here is a small model in three stages. It is not Hadoop; it is a model of why a pure task is safe to retry.

Warm-up: a pure task is just a function whose output depends only on its input. Run it twice, get the same answer twice.

Both runs print the same number, because count_a reads only its argument. That is purity in miniature, and it is the only property the retry logic needs.

Bridge: wrap the task in a record that reports success or failure, so we can simulate a machine going down mid-task.

The first record is done with a real result; the second is failed with no result. The input chunk is still known either way, which is exactly what makes recovery possible: to recompute, we only need the chunk and the pure function.

Official idea, made runnable: split the input into chunks, run a task per chunk, let one fail, then recompute the failed one from its original chunk and sum every result.

The first attempt for task-2 produces no result. The recovery loop finds the failed record, looks up that task's original input chunk, and recomputes the count. The final sum is correct because count_a is pure: rerunning it on the same chunk gives the same answer the failed task would have. Swap fail=True onto a different task, or remove it, and the total comes out the same, which is the property fault tolerance relies on.

⚠Common Beginner Traps

  • Thinking Hadoop changes the map or reduce logic. It does not. Your mapper and reducer are the application logic; Hadoop changes only how that logic is executed across a cluster. The same two files run locally and on Hadoop.
  • Thinking "parallel" means each task is smarter or each line is easier. Parallelism means many ordinary tasks run at the same moment. One line of counting costs exactly what it always did; the speedup comes from overlap, not from any task becoming faster.
  • Assuming failures are rare enough to ignore. Across many machines, partial failure is the normal case. A distributed framework is built around the expectation that tasks will fail and must be redone.
  • Missing why purity is what makes retries safe. Recomputing a failed task is correct only because a pure function gives the same output for the same input. Drop purity and "try again" stops being a guarantee.
  • Treating [input] and [output] as Python variables or literal text. In the run command they are placeholders for directories in the Hadoop file system; you substitute real paths, brackets and all removed. Type them literally and the job fails to start, because no directory named [input] exists in the file system.
  • Expecting this packet to actually run a cluster. The Hadoop commands are shown to teach the interface. Running them requires an installed Hadoop, which this lesson does not assume.

§Summary

Hadoop is an open-source implementation of MapReduce that runs jobs across a cluster of machines. Through its streaming interface, the very same Unix-style mapper and reducer you wrote for one machine, count_vowels_mapper.py and sum_reducer.py, can be used directly on large corpora, because they only ever read standard input and write standard output. Hadoop offers three advantages over a simplistic local pipeline: speed, from running map and reduce tasks in parallel on many machines at once; fault tolerance, because a failed task's result can be recomputed by another task, which is safe precisely because the functions are pure; and monitoring, through a user interface that tracks the job's progress. You launch the vowel-count job with $ python3 mr.py run count_vowels_mapper.py sum_reducer.py [input] [output], where [input] and [output] are directories in the Hadoop file system. The lesson underneath all of it: a good abstraction can make a genuinely hard distributed system usable without pretending the difficulty is gone. The framework is holding the complexity so the application code does not have to.

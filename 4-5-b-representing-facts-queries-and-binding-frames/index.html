Chapter 4/4.5 Unification

4.5.2 Representing Facts, Queries, and Binding Frames

◷ approx. 14 min read · 1 worked example

You have seen, on paper, how a query like (?x c ?x) can match a fact like ((a b) c (a b)) by binding ?x to (a b). That was all done by hand. This lesson is where matching stops being a paper exercise and becomes something a Python program actually does.

By the end you will be able to do four things. You will import the provided logic interpreter. You will turn a line of logic-language text into the Python data the interpreter reasons about, using read_line. You will read a binding frame and say what it stores. And you will call unify and explain the one fact that trips up almost everyone: unify does not hand the answer back to you, it writes the answer into a frame you passed in, and only returns whether the match succeeded.

The whole lesson rests on a single mental separation: the expression data (the shape being matched) lives in one place, and the bindings (what the variables turned out to mean) live in a different place. Keep those two apart and the rest follows.

The Two Languages in This Room

There are two languages in play, and confusing them is the first trap.

The logic language is the parenthesized, Lisp-like notation with question-mark variables. Lines like (parent abraham barack) and (?x c ?x) are written in it. It is not Python. You never run it at the Python prompt directly; you hand it to the interpreter as a string.

Python is the language the interpreter itself is written in. read_line, Pair, nil, Frame, and unify are all Python names. When you type at the >>> prompt, you are talking to Python, and Python is running the logic interpreter for you.

Throughout this lesson, logic-language snippets are shown plainly, while Python sessions are shown with the >>> prompt. Watch which one you are looking at.

An Analogy: A Form and a Scratchpad

Imagine a clerk processing a paper form. The form has blanks on it, and the blanks have labels like "applicant" and "guarantor." The clerk's job is to check whether this form can be made to agree with a record already on file.

The clerk cannot reason about ink. So the first thing they do is transcribe the form into a structured slip they can manipulate: each field in its own box, boxes linked in order. That transcription is what read_line does — it turns a line of logic text into linked Python boxes.

Separately, the clerk keeps a scratchpad. Every time matching the form against the record forces a blank to take a particular value ("applicant must be Abraham"), the clerk writes that on the scratchpad. The scratchpad is not the form and it is not the record; it is the running list of what the blanks have been forced to mean. That scratchpad is the binding frame.

At the end, the clerk can answer two different questions. "Did the form and the record turn out compatible?" is a yes-or-no question. "What did the blanks have to mean for that to work?" is answered by reading the scratchpad. unify answers the first with True or False; the frame holds the answer to the second. Holding these two apart is the entire idea.

Representing Logic Text as Pair Data

Both queries and facts are represented as Scheme lists in the logic language, using the same Pair class and nil object from the previous chapter. There is nothing new to invent: a logic expression is just a linked list of boxes, exactly like the Scheme lists you already built.

Take the query expression:

(?x c ?x)

As Python data, this is a chain of Pair boxes. Each box holds one element and a link to the rest, and the chain ends in nil:

Pair box: first = '?x',  rest -> next box
Pair box: first = 'c',   rest -> next box
Pair box: first = '?x',  rest -> nil

The first of each box is the current element. The rest is the remainder of the list. The final nil means "no more elements." This is the same two-field shape you used for Scheme lists:

Pair(first, rest)

The important consequence: a logic expression is ordinary nested data. The interpreter walks it with the same list operations you already know.

Importing the Logic System

The official examples assume the provided logic interpreter has been imported. At the Python prompt:

>>> from logic import *

This is an interactive Python session, so it carries the >>> prompt. After this line runs, you can use the interpreter's names — including read_line, Pair, nil, Frame, and unify — directly. For the rest of this lesson, assume this import has already happened.

Turning Text into Pair Data: read_line

read_line takes a string written in the logic language and returns the nested Pair data that represents it. It is the transcription step from the analogy: text in, linked boxes out.

Here is the official transcript:

>>> read_line("(?x c ?x)")
Pair('?x', Pair('c', Pair('?x', nil)))

Read the call one token at a time. The line being analyzed is read_line("(?x c ?x)"):

read_line("(?x c ?x)")
├─ read_line ........... the interpreter function that parses logic text
├─ ( ................... begins the one argument
├─ "(?x c ?x)" ......... a Python string holding one line of logic language
├─ ) ................... ends the argument
└─ returns ............. Pair('?x', Pair('c', Pair('?x', nil)))

Now read the value it returned. The line being analyzed is Pair('?x', Pair('c', Pair('?x', nil))):

Pair('?x', Pair('c', Pair('?x', nil)))
├─ Pair ................ the outermost box
├─ '?x' ................ its first: the symbol ?x
├─ Pair('c', ...) ...... its rest: a box whose first is 'c'
├─ Pair('?x', nil) ..... that box's rest: a box whose first is '?x'
├─ nil ................. the innermost rest: the empty list, end of the chain
└─ represents .......... the logic expression (?x c ?x) as Python data

Notice the symmetry: three elements in the logic text, three Pair boxes, ending in nil. The string on the left and the Pair structure on the right are the same list in two forms.

The Binding Frame

In the previous chapter, environment frames bound names to values so Scheme could be evaluated. Here, the same idea stores logic-variable bindings.

An environment that binds symbols to values is an instance of the Frame class, which has an attribute called bindings. The bindings attribute is a dictionary mapping each bound variable name to the value it stands for. This is the scratchpad: a small table that grows as matching forces variables to take values.

A Frame is created with a parent. For a single, isolated match with no earlier assumptions to inherit, the parent is None:

>>> env = Frame(None)

Conceptually, a frame that has bound ?x to the list (a b) looks like this:

Frame
  bindings: {'?x': Pair('a', Pair('b', nil))}
  parent:   None

The parent is None here because there is no outer frame to consult; this is the first and only environment in the example. The variable name '?x' is a key in the dictionary, and the value is the Pair data it was bound to.

Calling unify

The unify function performs pattern matching in the logic language. It takes two inputs, e and f, as well as an environment env that records the bindings of variables to values. Its return value is a Boolean: True means the pattern f was able to match the expression e. Here e and f are just the two argument names the source uses; in this example e is the plain expression ((a b) c (a b)) and f is the pattern (?x c ?x) that carries the variable. The match itself — what the variables had to mean — is recorded in the bindings inside env, not returned.

Read the call shape one token at a time. The line being analyzed is unify(e, f, env):

unify(e, f, env)
├─ unify ............... the matching function
├─ ( ................... begins the inputs
├─ e ................... the first expression
├─ , ................... separates inputs
├─ f ................... the second expression (the pattern, which may hold variables)
├─ , ................... separates inputs
├─ env ................. the frame where new bindings are recorded
├─ ) ................... ends the inputs
└─ returns ............. True or False, and mutates env with any new bindings

That final row is the crux. The arrow gives you a Boolean. The bindings are a side effect written into env. Two different outputs, two different places.

A Warm-Up: Two Outputs, One Call

Before the official transcript, here is the smallest version of the idea. Suppose a match forces the single variable ?x to become the list (a b). Two separate things are now true:

the call succeeded        -> a Boolean: True
?x was forced to mean     -> the data (a b), stored in the frame

If you only looked at True, you would know the match worked but not what it concluded. If you only looked at the frame, you would know the conclusion but not whether the match was even valid. You need both, which is exactly why unify splits them. Hold this picture as you read the real example.

A Bridge: Reading the Pieces Before Running Them

The official example matches ((a b) c (a b)) against (?x c ?x). Before running it, line up the pieces by hand so the transcript holds no surprises.

The expression e is ((a b) c (a b)): a three-element list whose first and third elements are both the sublist (a b).

The pattern f is (?x c ?x): a three-element list whose first and third elements are the same variable ?x, with the symbol c in the middle.

position 1:  e has (a b)   |  f has ?x
position 2:  e has c       |  f has c
position 3:  e has (a b)   |  f has ?x

For the lists to agree, ?x must equal (a b) (from position 1), the constant c must equal c (position 2, trivially true), and ?x must again equal (a b) (position 3). Because both positions demand the same thing, there is no conflict. The single binding ?x -> (a b) makes the two lists identical. That is the whole match; the transcript below just performs it in Python.

The Official Transcript

Here is the official session in full, exactly as it appears in the source:

>>> e = read_line("((a b) c (a b))")
>>> f = read_line("(?x c ?x)")
>>> env = Frame(None)
>>> unify(e, f, env)
True
>>> env.bindings
{'?x': Pair('a', Pair('b', nil))}
>>> print(env.lookup('?x'))
(a b)

Walk it line by line.

The first two lines use read_line to turn the logic text into Pair data, binding the Python names e and f to the two expressions. The third line creates a fresh, empty frame with no parent. The fourth line calls unify(e, f, env) and Python prints True: the pattern f matched the expression e.

The fifth line asks for env.bindings and gets back the dictionary {'?x': Pair('a', Pair('b', nil))}. This is the scratchpad after the match: ?x is bound to the Pair data for (a b). The match did not return this; it wrote it into the frame.

The last line, print(env.lookup('?x')), asks the frame what ?x currently means and prints (a b). The method env.lookup('?x') looks the variable up in the frame's bindings and returns its value. Printing a Pair list displays it in the readable logic-language form (a b) rather than as raw Pair(...) boxes.

A Step-by-Step Trace of the Match

Here is what unify does internally as it compares the two lists, position by position. The frame starts empty.

Step Comparing Frame action Bindings after
Start — frame is empty {}
1 e first (a b) vs f first ?x ?x is unbound, so bind it to (a b) {'?x': (a b)}
2 e second c vs f second c two equal constants, nothing to record {'?x': (a b)}
3 e third (a b) vs f third ?x ?x already bound to (a b); the value matches, so this is consistent {'?x': (a b)}
End all positions agreed return True {'?x': (a b)}

The key moment is step 3. The second time ?x appears, it is no longer free to take any value — it was already pinned to (a b) in step 1. The interpreter checks the existing binding against (a b) and finds they agree, so the match holds. Had position 3 demanded a different value, step 3 would have failed and unify would return False. This is why the frame must be carried through the whole match: it is what makes a repeated variable mean the same thing each time.

Why the Frame Is Separate from the Return Value

It can feel roundabout to record the answer in a frame instead of just returning it. The reason is that a real query rarely produces one binding in isolation. A match may bind several variables at once, and it often needs to extend a frame that already holds earlier assumptions rather than start blank.

The following is a constructed illustration, not an official transcript, just to show the shape of "already-present plus newly-added" bindings in one frame:

already in the frame before this match:
  ?a -> some-value
added by the current match:
  ?child -> another-value

The interpreter needs one shared place to hold every current assumption so they stay mutually consistent. That place is the frame. The Boolean return value answers a narrow question — did this match succeed? — while the frame answers the richer question of what every variable had to mean for it to succeed. Different questions, different outputs.

⚠Common Beginner Traps

  • Thinking read_line("(?x c ?x)") returns a string. It returns nested Pair data; the string is only the input.
  • Thinking unify returns the bindings. It returns True or False. The bindings are a side effect written into the frame you passed as env.
  • Forgetting to look at env.bindings (or env.lookup) after a successful match. The True alone tells you nothing about what the variables mean.
  • Confusing the logic variable ?x with a Python name x. The ?x is a string symbol living inside Pair data, not a Python variable.
  • Expecting nil to be Python's None. The nil is the empty Scheme list object that ends a Pair chain; None is the parent of an isolated Frame. They are different things that happen to both mean "nothing here."
  • Reading the logic language as Python. Lines like (?x c ?x) are logic text you pass as a string; they are not Python expressions you can run.

§Summary

Facts and queries in the logic language are represented as Scheme lists built from the same Pair class and nil object as the previous chapter, so a logic expression is just nested Python data. The read_line function turns a line of logic text into that Pair data. A Frame holds an environment whose bindings attribute is a dictionary mapping variable names to values. The unify function takes two expressions e and f plus an environment env; it returns True when f matches e, and records the resulting bindings inside env rather than returning them. After a successful match you read the conclusion from the frame, using env.bindings or env.lookup. The lasting idea is the separation: expression data in one place, the bindings that satisfy a match in another, with the Boolean answering only whether a match was possible.

This checkpoint uses a small, self-contained sketch of Pair, nil, and a Frame so you can watch the expression data and the binding appear as separate structures. Stepping through it draws the environment diagram that keeps these two apart: the Pair("a", Pair("b", nil)) chain and the Frame object sit as separate boxes on the heap, and env.define makes the frame's bindings dict gain a "?x" key that points at that same chain rather than copying it, so answer, value, and the dict entry are arrows into one structure. Before you step through the block below, write down three predictions: how many Pair objects the line building value creates, whether the frame's "?x" entry will hold a copy of that list or point at the same objects, and what the two print lines will output.

Step through it one line at a time and check each prediction as the structures appear. Two Pair objects form the chain, and define adds one entry to the frame's bindings dictionary that points at that same chain rather than a copy; lookup returns the same object, so answer, value, and the dictionary entry all lead to one structure. The frame never contains the whole expression; it holds a single binding from a variable name to a data structure. The two print lines then walk into the chain — answer.first is "a" and answer.rest.first is "b" — which is exactly how env.lookup('?x') recovers the list (a b) after a real match.

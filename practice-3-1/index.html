3.1 · Programs, Languages, and Interpreters — Coding Practice

◷ approx. 9 min read · 32 worked examples

These problems build a feel for the chapter's central machinery: representing a program as data (an expression tree made of tuples and strings), and writing the two cooperating jobs — evaluation and application — that give that data meaning. You will inspect trees, evaluate them, and extend a tiny evaluator with new operations and special forms.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Build an addition tree

Input:  build_sum(10, -4)
Output: ('+', 10, -4)
Explanation: operands are stored verbatim, not added.

Solutions

Setup

These problems share the section's tiny evaluator model. Every snippet below assumes this block has already run:

Problems 9 and 10 also use these tables:

1. Build an addition tree

Approach — programs as data. This section's first move is that a program can be represented as data before any interpreter gives it meaning. The expression 2 + 3 becomes the tuple ("+", 2, 3). Building the tree is pure construction — no evaluation.

def build_sum(a, b):
    return ("+", a, b)

Why it works. It assembles exactly the tuple shape the evaluator expects: operator name first, then the two operands, stored verbatim.

Complexity. Time O(1), Space O(1). One fixed-size tuple is built.

assert build_sum(2, 3) == ("+", 2, 3)
assert build_sum(10, -4) == ("+", 10, -4)
assert evaluate(build_sum(2, 3), {}) == 5

2. Is it a literal?

Approach — the evaluator's first dispatch. The evaluator decides what an expression is before deciding what it means. Its very first test, isinstance(expression, int), recognizes a literal that evaluates to itself.

def is_literal(expression):
    return isinstance(expression, int)

Why it works. Only integers are self-evaluating literals in this model; names are strings and calls are tuples, both of which return False.

Complexity. Time O(1), Space O(1). A single type check.

assert is_literal(7) is True
assert is_literal(("+", 1, 2)) is False
assert is_literal("x") is False

3. Read the operator

Approach — destructuring the call tuple. Every call expression stores its operator in slot 0, exactly as the evaluator reads it with expression[0]. Inspecting the tree means indexing into it.

def operator_of(expression):
    return expression[0]

Why it works. The tuple layout (operator, left, right) puts the operator name first, regardless of how the operands are nested.

Complexity. Time O(1), Space O(1). One index access.

assert operator_of(("+", 2, 3)) == "+"
assert operator_of(("*", 4, ("+", 1, 1))) == "*"
assert operator_of(("-", 9, 5)) == "-"

4. Apply one flat operation

Approach — application without evaluation. When the operands are already values, no evaluation step is needed — this is pure application. Look the operator up in OPERATORS and call it on the two operands.

def eval_flat(expression):
    op = expression[0]
    left = expression[1]
    right = expression[2]
    return OPERATORS[op](left, right)

Why it works. OPERATORS maps each operator name to a two-argument function, so the lookup-then-call is exactly the applicator's job for a built-in.

Complexity. Time O(1), Space O(1). One lookup and one call.

assert eval_flat(("+", 2, 3)) == 5
assert eval_flat(("*", 6, 7)) == 42
assert eval_flat(("-", 10, 4)) == 6

5. Evaluate with names

Approach — the three-way evaluator. The evaluator dispatches on what the expression is: a literal evaluates to itself, a name is looked up in the environment, and a call recursively evaluates its operands and then applies the operator. This is the evaluate-apply loop in miniature.

def eval_with_names(expression, env):
    if isinstance(expression, int):
        return expression
    if isinstance(expression, str):
        return env[expression]
    op = expression[0]
    left = eval_with_names(expression[1], env)
    right = eval_with_names(expression[2], env)
    return OPERATORS[op](left, right)

Why it works. Recursion handles arbitrary nesting: each operand is fully evaluated to a number before the operator is applied, and names resolve through env.

Complexity. Time O(n), Space O(d) for a tree of n nodes and depth d (recursion stack).

assert eval_with_names(("+", "x", "y"), {"x": 4, "y": 6}) == 10
assert eval_with_names(("*", "x", 3), {"x": 5}) == 15
assert eval_with_names(("-", ("+", "a", 1), "b"), {"a": 7, "b": 2}) == 6

6. Count operations

Approach — structural recursion over the tree. Because an expression is data, you can walk its structure. A literal or name is a leaf contributing 0; a call contributes 1 plus whatever its operands contribute.

def count_operations(expression):
    if isinstance(expression, int) or isinstance(expression, str):
        return 0
    total = 1
    for operand in expression[1:]:
        total = total + count_operations(operand)
    return total

Why it works. Every operator call is counted exactly once, and the recursion reaches every nested call through the operands.

Complexity. Time O(n), Space O(d) for n nodes and depth d.

assert count_operations(("+", 2, 3)) == 1
assert count_operations(("*", ("+", 1, 2), 4)) == 2
assert count_operations(("-", ("+", "x", 1), ("*", 2, 3))) == 3

7. Extend the evaluator with max

Approach — growing the language. Adding "max" is exactly how an interpreter gains vocabulary: keep the evaluate-then-apply shape, but add one more branch in the application step. Both operands are evaluated first, then the new operator is applied.

def eval_extended(expression, env):
    if isinstance(expression, int):
        return expression
    if isinstance(expression, str):
        return env[expression]
    op = expression[0]
    left = eval_extended(expression[1], env)
    right = eval_extended(expression[2], env)
    if op == "+":
        return left + right
    if op == "-":
        return left - right
    if op == "*":
        return left * right
    if op == "max":
        if left > right:
            return left
        return right
    raise ValueError(op)

Why it works. "max" behaves like any other strict operator — its operands are fully evaluated before the comparison runs — so it slots cleanly into the same dispatch.

Complexity. Time O(n), Space O(d) for n nodes and depth d.

assert eval_extended(("max", 2, 9), {}) == 9
assert eval_extended(("max", ("+", 3, 4), 5), {}) == 7
assert eval_extended(("+", ("max", 1, 8), ("max", 2, 2)), {}) == 10

8. Depth of an expression tree

Approach — recursion that tracks the maximum. Depth is another structural property of the program-as-data. A leaf has depth 0; a call is one level deeper than its deepest operand.

def tree_depth(expression):
    if isinstance(expression, int) or isinstance(expression, str):
        return 0
    best = 0
    for operand in expression[1:]:
        d = tree_depth(operand)
        if d > best:
            best = d
    return best + 1

Why it works. Taking the maximum over operands captures the longest path; adding 1 accounts for the current call.

Complexity. Time O(n), Space O(d) for n nodes and depth d.

assert tree_depth(5) == 0
assert tree_depth(("+", 2, 3)) == 1
assert tree_depth(("*", ("+", 1, ("-", 4, 2)), 7)) == 3

9. Apply user-defined functions

Approach — the mutual recursion of evaluate and apply. Applying a user-defined function is the move the whole chapter is built on: bind the parameter in a fresh environment, then hand the function's body back to the evaluator. Evaluation calls application (for the operator), and application of a defined function calls evaluation again (for the body).

def eval_user(expression, env):
    if isinstance(expression, int):
        return expression
    if isinstance(expression, str):
        return env[expression]
    op = expression[0]
    values = [eval_user(operand, env) for operand in expression[1:]]
    if op in OPERATORS:
        return OPERATORS[op](values[0], values[1])
    if op in DEFINITIONS:
        new_env = {PARAM: values[0]}
        return eval_user(DEFINITIONS[op], new_env)
    raise ValueError(op)

Why it works. A built-in is applied directly; a defined function builds {PARAM: value} and evaluates its body there, so each application runs the body in its own environment. Nested calls like double(double(6)) resolve from the inside out.

Complexity. Time O(n), Space O(d) where n counts the nodes evaluated across all bodies and d is the nesting depth of active calls.

assert eval_user(("double", 5), {}) == 10
assert eval_user(("square", ("inc", 3)), {}) == 16
assert eval_user(("+", ("double", 4), ("square", 2)), {}) == 12
assert eval_user(("double", ("double", 6)), {}) == 24

10. Add an if special form

Approach — a special form skips evaluation. Unlike an operator, "if" must not evaluate all of its parts. It evaluates the test, then evaluates only the selected branch. Handling "if" before the arithmetic branches is what keeps the unused branch from running.

def eval_if(expression, env):
    if isinstance(expression, int):
        return expression
    if isinstance(expression, str):
        return env[expression]
    op = expression[0]
    if op == "if":
        condition = eval_if(expression[1], env)
        if condition != 0:
            return eval_if(expression[2], env)
        return eval_if(expression[3], env)
    left = eval_if(expression[1], env)
    right = eval_if(expression[2], env)
    if op == "+":
        return left + right
    if op == "-":
        return left - right
    if op == "*":
        return left * right
    raise ValueError(op)

Why it works. The "if" branch evaluates the test, then returns the value of exactly one sub-expression, leaving the other untouched. The arithmetic operators keep their strict, evaluate-both behavior.

Complexity. Time O(n), Space O(d) for the nodes actually evaluated and the call depth d.

assert eval_if(("if", 1, 10, 20), {}) == 10
assert eval_if(("if", 0, 10, 20), {}) == 20
assert eval_if(("if", ("-", "x", "x"), ("+", 1, 1), ("*", 3, 3)), {"x": 5}) == 9
assert eval_if(("+", ("if", 1, 4, 99), 6), {}) == 10

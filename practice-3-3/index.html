3.3 · Exceptions — Coding Practice

◷ approx. 14 min read · 30 worked examples

These problems build fluency with Python's exception machinery: raising the right exception with raise, checking assumptions with assert, protecting code with try/except, binding a caught object with as to read its class and attributes, recovering with fallbacks, aggregating failures across many inputs, and defining custom exception classes that carry structured information out to a handler with the context to use it.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

Output: 0.0
Explanation: 0 ** 0.5 is 0.0; the try block succeeds.

  • Compute the root with x ** 0.5 inside a try block.
  • Catch only TypeError; do not use a bare except.

10. Safe Pipeline

A pipeline is a list of one-argument functions (stages) applied in order, each fed the previous stage's output. Define StageError(Exception) storing self.stage_number (1-based) and self.value (the input that stage received). Write run_pipeline(stages, start) that threads start through every stage; if a stage raises any exception, it raises StageError naming which stage (counting from 1) failed and the value it was given. Then write run_pipeline_safe(stages, start) that returns the final value on success, or — catching StageError — the tuple (stage_number, value).

Examples

Input:  run_pipeline_safe([lambda x: x + 1, lambda x: x * 2], 3)
Output: 8
Explanation: (3 + 1) then (* 2) gives 8; no stage fails.

Input:  run_pipeline_safe([lambda x: x + 1, lambda x: 1 / 0, lambda x: x * 2], 3)

Explanation: stage 2 receives 4 and divides by zero, so StageError carries stage 2 and value 4.

Output: 7
Explanation: an empty pipeline returns the start value unchanged.

  • Number stages from 1; the StageError must record the stage number and the value passed into the failing stage.
  • run_pipeline catches any exception from a stage and re-raises it as a StageError; run_pipeline_safe catches only StageError.
  • Detect the failure where it happens (inside run_pipeline), but decide the recovery shape in run_pipeline_safe.

Solutions

1. Safe Square Root

Approach — protect one expression, return a fallback. This is the smallest try/except shape from the lesson's safe_reciprocal: the risky computation x ** 0.5 goes in the try, and a single except TypeError clause supplies a fallback when the operand has no power operation. A number flows straight through the try; a non-number triggers the handler.

def safe_sqrt(x):
    try:
        return x ** 0.5
    except TypeError:
        return 'not a number'

Why it works. When x is numeric, x ** 0.5 evaluates and the return fires before the except is ever consulted. When x is a string or list, ** 0.5 raises TypeError, Python abandons the rest of the try block, and control jumps to the matching handler, which returns the fallback string. Catching TypeError specifically — not a bare except — leaves genuinely unexpected failures free to surface.

Complexity. Time O(1), Space O(1) — a single arithmetic attempt.

assert safe_sqrt(9) == 3.0
assert safe_sqrt(2) == 2 ** 0.5
assert safe_sqrt('nine') == 'not a number'
assert safe_sqrt(0) == 0.0
assert safe_sqrt([1, 2]) == 'not a number'

2. Validate Age

Approach — raise on expected invalid input. Validation is the lesson's reason to use raise rather than assert: a caller may pass a bad age, and that is a recoverable condition the program should signal with a real exception. Each failing case is detected with an if and reported with raise ValueError(message); only a value that survives both checks reaches return age.

def validate_age(age):
    if age < 0:
        raise ValueError('age must not be negative')
    if age > 150:
        raise ValueError('age is too large')
    return age

Why it works. A raise stops the function immediately, so the first failing check ends the call before the next line runs — the negative case never reaches the too-large check or the return. A valid age skips both raise statements and returns unchanged. ValueError is the conventional class for an argument whose type is fine but whose value is out of range.

Complexity. Time O(1), Space O(1) — two comparisons.

assert validate_age(0) == 0
assert validate_age(30) == 30
assert validate_age(150) == 150
try:
    validate_age(-1)
    raised = False
except ValueError as error:
    raised = True
    message = str(error)
assert raised is True
assert message == 'age must not be negative'
try:
    validate_age(200)
    big_raised = False
except ValueError:
    big_raised = True
assert big_raised is True

3. Lookup With Fallback

Approach — catch the lookup failure, supply a default. Indexing a dictionary with a missing key raises KeyError; the try/except shape from the lesson turns that crash into a graceful fallback. The successful index returns inside the try; a missing key sends control to the except KeyError clause, which returns default.

def lookup(prices, item, default):
    try:
        return prices[item]
    except KeyError:
        return default

Why it works. A present key — even one whose stored value is 0 — makes prices[item] succeed, so the try's return fires and the default is never reached. That is why catching the exception is more faithful than testing truthiness: it distinguishes "absent" from "present but falsy." A missing key raises KeyError, the only class the handler catches, so the fallback is returned.

Complexity. Time O(1) average for the dictionary lookup, Space O(1).

assert lookup({'pen': 2, 'ink': 5}, 'pen', 0) == 2
assert lookup({'pen': 2, 'ink': 5}, 'glue', 0) == 0
assert lookup({}, 'pen', -1) == -1
assert lookup({'a': 0}, 'a', 99) == 0

4. Check Even

Approach — assert an assumption, then proceed. assert <condition>, <message> is the lesson's tool for checking that an assumption holds. When n is even the condition n % 2 == 0 is true, so the assertion is silent and the function returns n // 2. When n is odd the condition is false and Python raises AssertionError carrying the message built from str(n).

def check_even(n):
    assert n % 2 == 0, str(n) + ' is not even'
    return n // 2

Why it works. A passing assertion does nothing visible — execution simply falls through to the return. A failing one raises AssertionError before the return can run, and the message after the comma becomes the exception's text, so a handler that binds the exception with as error reads exactly '7 is not even'. Building the message with str(n) puts the offending value into the report.

Complexity. Time O(1), Space O(1) — one modulus check.

assert check_even(8) == 4
assert check_even(0) == 0
try:
    check_even(7)
    odd_raised = False
except AssertionError as error:
    odd_raised = True
    odd_message = str(error)
assert odd_raised is True
assert odd_message == '7 is not even'

5. Retry Until Valid

Approach — loop, catch, and continue. Each candidate is tried in turn; the lesson's pattern of catching a specific exception lets a failure skip to the next iteration instead of stopping the search. A try/except ValueError wraps the check; on success the candidate is returned, and on ValueError the loop continues. If the loop ends without a success, None is returned.

def first_valid(check, candidates):
    for candidate in candidates:
        try:
            check(candidate)
            return candidate
        except ValueError:
            continue
    return None

Why it works. When check(candidate) raises ValueError, the return candidate on the next line is skipped — a raised exception abandons the rest of the try block — and continue advances to the next candidate. When check returns normally, no exception interrupts the path, so return candidate fires. Reaching the end of the loop means every candidate failed, so the function falls through to return None.

Complexity. Time O(n) for n candidates (each tried once), Space O(1).

def positive(n):
    if n <= 0:
        raise ValueError('not positive')
    return n


assert first_valid(positive, [-3, -1, 4, 9]) == 4
assert first_valid(positive, [0, -5, -2]) is None
assert first_valid(positive, [7, 8]) == 7
assert first_valid(positive, []) is None

6. Divide All

Approach — recover per element, keep going. The point of catching the exception inside the loop is that one bad denominator must not abort the whole computation. Each division sits in its own try/except ZeroDivisionError; a good denominator appends the quotient, a zero appends None, and the loop proceeds to the next either way.

def divide_all(numerator, denominators):
    results = []
    for d in denominators:
        try:
            results.append(numerator / d)
        except ZeroDivisionError:
            results.append(None)
    return results

Why it works. Because the try surrounds a single division, a ZeroDivisionError only abandons that one iteration's try body — the handler appends None and the for loop moves on. Each input therefore contributes exactly one slot to the output, in order, so the result lines up position-for-position with the denominators.

Complexity. Time O(n) for n denominators, Space O(n) for the result list.

assert divide_all(12, [2, 3, 4]) == [6.0, 4.0, 3.0]
assert divide_all(12, [2, 0, 6]) == [6.0, None, 2.0]
assert divide_all(10, [0, 0]) == [None, None]
assert divide_all(8, []) == []

7. Count Failures

Approach — aggregate any failure with a broad handler. Here we genuinely want to count every kind of failure, not respond to one specific class, so a broad except Exception is the right tool — the rare case the lesson warns is otherwise a trap. Each call runs in a try; a normal return adds nothing, and any raised exception bumps the failure counter.

def count_failures(action, inputs):
    failures = 0
    for value in inputs:
        try:
            action(value)
        except Exception:
            failures = failures + 1
    return failures

Why it works. When action(value) returns normally, control falls past the except and the counter is untouched. When it raises, the matching except Exception catches it — Exception is a base class of ZeroDivisionError, ValueError, and the rest — and increments failures. Because the handler swallows the exception, the loop continues to the next input rather than stopping at the first failure.

Complexity. Time O(n) for n inputs, Space O(1) — a single counter.

def reciprocal(x):
    return 1 / x


assert count_failures(reciprocal, [1, 2, 4]) == 0
assert count_failures(reciprocal, [1, 0, 2, 0]) == 2
assert count_failures(reciprocal, [0, 0, 0]) == 3
assert count_failures(reciprocal, []) == 0

8. Withdraw With Custom Exception

Approach — a custom class that carries a value. Following the lesson's IterImproveError, the exception is built from a class that stores extra information — here the shortfall. withdraw detects the overdraft and raises InsufficientFunds(amount - balance), packing the missing amount into the object. withdraw_or_zero is the handler with context: it catches the exception and reads error.shortfall back out.

class InsufficientFunds(Exception):
    def __init__(self, shortfall):
        self.shortfall = shortfall


def withdraw(balance, amount):
    if amount > balance:
        raise InsufficientFunds(amount - balance)
    return balance - amount


def withdraw_or_zero(balance, amount):
    try:
        return withdraw(balance, amount)
    except InsufficientFunds as error:
        return error.shortfall

Why it works. Because InsufficientFunds inherits from Exception, an instance is a full exception object that can be raised and caught, and its __init__ saves self.shortfall so the value travels with the exception. The as error binding gives withdraw_or_zero the object itself, so it recovers the structured shortfall rather than parsing a text message. This is the lesson's design split: detect the problem deep down, decide the response where the context is.

Complexity. Time O(1), Space O(1) — one comparison and one attribute.

assert withdraw(100, 40) == 60
assert withdraw_or_zero(100, 40) == 60
assert withdraw_or_zero(100, 150) == 50
try:
    withdraw(50, 80)
    over_raised = False
except InsufficientFunds as error:
    over_raised = True
    short = error.shortfall
assert over_raised is True
assert short == 30
assert isinstance(InsufficientFunds(10), Exception) is True

9. Parse Or Report

Approach — translate a low-level error into a richer one. A bare ValueError from int(token) knows only that a token failed, not which. So parse_ints catches that low-level error and re-raises a ParseError carrying the index and the offending tokens[index] — the lesson's idea of an exception that carries information. parse_report is the outer layer with the context to format a message, so it catches ParseError and reads its attributes.

Why it works. index advances only after a token parses, so when to_int raises ValueError mid-loop, index still points at the token that failed and tokens[index] is that token. The except ValueError in parse_ints re-raises this as a ParseError holding both facts, which propagates past to_int straight to parse_report. There the as error binding exposes error.bad_index and error.bad_token, and repr(...) formats the token with quotes — 'oops' — matching the report. The error is detected low and reported high, exactly the lesson's separation.

Complexity. Time O(n) for n tokens, Space O(n) for the parsed list.

assert parse_ints(['3', '14', '9']) == [3, 14, 9]
assert parse_report(['3', '14', '9']) == [3, 14, 9]
assert parse_report(['3', 'oops', '9']) == "bad token 'oops' at index 1"
assert parse_report(['x']) == "bad token 'x' at index 0"
try:
    parse_ints(['1', '2', 'z'])
    pe_raised = False
except ParseError as error:
    pe_raised = True
    pe_index = error.bad_index
    pe_token = error.bad_token
assert pe_raised is True
assert pe_index == 2
assert pe_token == 'z'

10. Safe Pipeline

Approach — capture the failing stage's number and input. As run_pipeline threads the value through the stages, it tracks a 1-based number and the current value. A broad except Exception catches whatever a stage throws and re-raises a StageError carrying both the stage number and the value that stage received — the most useful snapshot at the moment of failure, in the spirit of IterImproveError.last_guess. run_pipeline_safe catches only that StageError and returns the recorded pair.

class StageError(Exception):
    def __init__(self, stage_number, value):
        self.stage_number = stage_number
        self.value = value


def run_pipeline(stages, start):
    value = start
    number = 0
    for stage in stages:
        number = number + 1
        try:
            value = stage(value)
        except Exception:
            raise StageError(number, value)
    return value


def run_pipeline_safe(stages, start):
    try:
        return run_pipeline(stages, start)
    except StageError as error:
        return (error.stage_number, error.value)

Why it works. number is bumped at the top of each iteration, so it names the stage about to run; value is only overwritten after a stage succeeds, so at the moment a stage raises, value still holds that stage's input. The assignment value = stage(value) never completes on failure — a raised exception abandons it — which is why the captured value is the input, not a half-built output. run_pipeline wraps each stage in its own try, re-raising any exception as a StageError; run_pipeline_safe catches that single class and returns (stage_number, value). An empty pipeline runs no stages and returns start unchanged.

Complexity. Time O(n) for n stages (up to the failing one), Space O(1) beyond the threaded value.

assert run_pipeline([lambda x: x + 1, lambda x: x * 2], 3) == 8
assert run_pipeline_safe([lambda x: x + 1, lambda x: x * 2], 3) == 8
assert run_pipeline_safe([lambda x: x + 1, lambda x: 1 / 0, lambda x: x * 2], 3) == (2, 4)
assert run_pipeline_safe([lambda x: 1 / 0], 5) == (1, 5)
assert run_pipeline_safe([], 7) == 7
try:
    run_pipeline([lambda x: x - 1, lambda x: x[0]], 4)
    se_raised = False
except StageError as error:
    se_raised = True
    se_stage = error.stage_number
    se_value = error.value
assert se_raised is True
assert se_stage == 2
assert se_value == 3

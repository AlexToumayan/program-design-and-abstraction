Chapter 4/4.2 Implicit Sequences

4.2.2-4.2.4 Iterable Processing and For Statements

◷ approx. 15 min read · 7 worked examples · best read in two passes

You already know that an iterator is a one-at-a-time value dispenser: you call next on it and it hands back the next value, remembering where it left off. This lesson zooms out one step. By the end of it you will be able to do four things: say what makes a value iterable, iterate over a dictionary and its values, use the built-in functions that build iterators lazily (map, filter, zip, and reversed), and explain exactly what a for statement does behind the scenes.

The big payoff is the for statement. You have almost certainly seen it before. After this lesson you will not just use it; you will know the hidden machinery it runs every single time it loops, because you will have written that machinery out by hand.

A Vending Machine and Its Button

Imagine a vending machine full of snacks.

The machine itself is like an iterable: it holds a collection of items and knows how to hand you a way to dispense them. The button-and-slot mechanism is like an iterator: each press gives you one item and advances to the next.

vending machine -> dispenser -> snack, snack, snack
iterable        -> iterator  -> value, value, value

The key word is "produce." The vending machine does not dispense snacks directly; it produces a dispenser, and the dispenser does the work. In Python the iterable does not produce values directly; you ask it for an iterator with iter, and the iterator produces the values with next. Your program never needs to know how the iterable stores its contents. It only needs a way to request them one at a time.

iterable object -> iter(...) -> iterator -> next(...) -> values

What Makes a Value Iterable

An iterable value is any value that can produce an iterator. In Python, that means any value you can pass to the built-in iter function. This is a wide net. Sequence values such as strings and tuples are iterable. Other containers such as sets and dictionaries are iterable. And iterators are themselves iterable, because you can pass an iterator to iter too.

A list is iterable. A range is iterable. A dictionary is iterable. The for statement leans on this every time it loops over a collection.

There is a subtle point about unordered collections. A dictionary or a set does not let you control the order of its contents, so we call them unordered. But the moment one of them produces an iterator, it must commit to some ordering, because an iterator dispenses values one after another and that is inherently a sequence. Python does not promise you control over that order, but it does guarantee certain properties about it in the language specification.

Iterating Over a Dictionary

The official source demonstrates that dictionaries are iterable. Here is the original interactive session:

>>> d = {'one': 1, 'two': 2, 'three': 3}
>>> d
{'one': 1, 'three': 3, 'two': 2}
>>> k = iter(d)
>>> next(k)
'one'
>>> next(k)
'three'
>>> v = iter(d.values())
>>> next(v)
1
>>> next(v)
3

Two things are happening here. Calling iter(d) produces an iterator over the dictionary's keys: each next call returns one key, like 'one' and then 'three'. Calling iter(d.values()) produces a separate iterator over the dictionary's values: each next call returns one value, like 1 and then 3. The key iterator and the value iterator are independent. They walk through different views of the same single dictionary, and each remembers its own position.

One note on order. The transcript above comes from an older version of Python whose dictionaries did not preserve insertion order, which is why you see 'one', then 'three', then 'two'. Modern Python preserves the order in which you inserted the keys, so when you run the executable version below you will see keys come out in the order one, two, three. The conceptual point is unchanged: iterating a dictionary gives keys, and iterating d.values() gives values.

Here is a runnable version. Watch the two iterators advance separately:

one
two

The line iter(d) deserves a token-by-token look, because the call quietly chooses keys for you:

iter(d)
├─ iter ........ the built-in that asks a value for an iterator
├─ ( ........... begins the input
├─ d ........... the dictionary to iterate over
├─ ) ........... ends the input
└─ evaluates to  an iterator over the keys of d ('one', 'two', 'three')

And the value iterator differs by one method call, d.values():

iter(d.values())
├─ iter ........ the built-in that asks a value for an iterator
├─ ( ........... begins the input
├─ d.values() .. a view of the dictionary's values
├─ ) ........... ends the input
└─ evaluates to  an iterator over the values of d (1, 2, 3)

The dictionary is one data object. The key iterator and the value iterator are two separate ways to walk through views of that data.

Changing a Dictionary While You Iterate

A dictionary that changes structure while an iterator is walking through it invalidates that iterator. Adding a key or removing a key counts as changing the structure. Once that happens, existing iterators become invalid, and future iterators may show arbitrary changes to their order. There is one safe exception: changing the value stored at an existing key does not change the ordering and does not invalidate iterators, because the set of keys is untouched.

Here is the official transcript. It removes a key with d.pop('two') (which returns the removed value, 2) and then asks the key iterator for its next value:

>>> d.pop('two')
2
>>> next(k)
   
RuntimeError: dictionary changed size during iteration
Traceback (most recent call last):

Here is the same idea as a safe, self-contained program. It catches the error so the program finishes cleanly and prints the error's type name instead of crashing:

Expected output:

one
RuntimeError

Why does Python object so loudly? The iterator holds an internal plan for walking through the dictionary, including a record of how many items the dictionary has. When you remove a key, the dictionary's size no longer matches what the iterator recorded. Rather than silently dispense wrong or duplicated values, Python detects the mismatch and raises RuntimeError: dictionary changed size during iteration. An error you can see beats a bug you cannot.

Built-In Functions That Build Iterators

Several built-in functions take an iterable as an argument and return an iterator. They are used heavily for lazy sequence processing, which means they do not compute their results up front. They wait until you ask.

The clearest demonstration is map. Calling map does not perform the work of computing its result elements. Instead it creates an iterator that will compute each result only when you query it with next. The official example proves this by mapping a function that prints a message every time it runs, so you can literally see when the work happens:

>>> def double_and_print(x):
        print('***', x, '=>', 2*x, '***')
        return 2*x
>>> s = range(3, 7)
>>> doubled = map(double_and_print, s)  # double_and_print not yet called
>>> next(doubled)                       # double_and_print called once
*** 3 => 6 ***
6
>>> next(doubled)                       # double_and_print called again
*** 4 => 8 ***
8
>>> list(doubled)                       # double_and_print called twice more
*** 5 => 10 ***
*** 6 => 12 ***
[10, 12]

Read the comments alongside the output. Building the doubled iterator prints nothing, because double_and_print has not run yet. The first next(doubled) runs the function on exactly one value, 3, and you see one line of *** output. The second next(doubled) runs it on 4. Then list(doubled) forces every remaining value at once, running the function on 5 and 6, which is why two *** lines appear before the final list [10, 12].

Here is the runnable version:

Expected output:

*** 3 => 6 ***
6
*** 4 => 8 ***
8
*** 5 => 10 ***
*** 6 => 12 ***
[10, 12]

The line that creates the lazy iterator is worth a token-by-token look. Notice that the final row is "an iterator," not a list of results:

doubled = map(double_and_print, s)
├─ doubled ........... the name that will hold the lazy iterator
├─ = ................. binds the name on the left to the value on the right
├─ map ............... the built-in that maps a function over an iterable
├─ ( ................. begins the inputs
├─ double_and_print .. the function to apply to each element (not called yet)
├─ , ................. separates the inputs
├─ s ................. the iterable to draw elements from, range(3, 7)
├─ ) ................. ends the inputs
└─ performs          binds doubled to a lazy iterator; no element computed yet

The print statements inside double_and_print are not decoration. They are how you can watch laziness happen: the work is delayed until each element is requested.

Filter, Zip, and Reversed

The official source names three more built-in functions that return iterators: filter, zip, and reversed. The original text introduces them by name but leaves their demonstrations unwritten, so the runnable examples here are added to show each one concretely. Each builds an iterator, and each serves a different purpose.

Expected output:

2
[4, 6]
[('a', 10), ('b', 20), ('c', 30)]
6
5

Here is what each one does. The filter function keeps only the values for which a predicate function returns a true result; here the predicate lambda x: x % 2 == 0 keeps the even numbers, so the first next(evens) gives 2 and forcing the rest gives [4, 6]. The zip function pairs up values from several iterables position by position, producing tuples like ('a', 10). The reversed function walks a reversible sequence from its end back to its beginning, so the first two values out of backward are 6 then 5. All three are lazy iterators that support one-at-a-time processing.

What a For Statement Actually Does

The for statement operates on iterators. A value is iterable if it has an __iter__ method that returns an iterator, and any iterable value may appear as the <expression> in a for statement header. The __iter__ method is simply what the iter function calls for you behind the scenes: iter(x) runs x.__iter__(), and likewise next(x) runs x.__next__(). The double-underscore names are the methods; iter and next are the friendly built-ins that invoke them. The shape is:

for <name> in <expression>:
    <suite>

To execute a for statement, Python does the following. It evaluates the header <expression>, which must produce an iterable value. It invokes that value's __iter__ method to obtain an iterator. Then, until a StopIteration exception is raised, it repeatedly invokes the iterator's __next__ method, binds each returned value to <name>, and executes the <suite> once for that value. When StopIteration is raised, the loop ends.

Here is the official example. The list counts provides the iterator, and the body prints each item:

>>> counts = [1, 2, 3]
>>> for item in counts:
        print(item)
1
2
3

In this example, the counts list returns an iterator from its __iter__() method. The for statement then calls that iterator's __next__() method repeatedly, assigning each returned value to item. The process continues until the iterator raises StopIteration, at which point the for statement is done.

Here is the runnable version, with the result also accumulated so you can see the loop variable being used:

Expected output:

1
2
3

Because a for statement is a statement, not an expression, its anatomy ends with the action it performs rather than a value it evaluates to:

for item in counts:
├─ for ......... begins a for statement
├─ item ........ the loop variable, rebound on each pass
├─ in .......... separates the loop variable from the iterable
├─ counts ...... the iterable; its __iter__ method supplies an iterator
├─ : ........... ends the header; the suite below runs once per value
└─ performs      iterate: bind item to each value of counts in turn, run the suite

The Hidden Iterator, Written by Hand

With everything you now know about iterators, you can implement the rule for a for statement yourself, using only while, assignment, and try statements. Here is the official equivalent of the loop above. It calls counts.__iter__() to get the iterator, then loops forever calling items.__next__() until StopIteration is caught:

>>> items = counts.__iter__()
>>> try:
        while True:
            item = items.__next__()
            print(item)
    except StopIteration:
        pass
1
2
3

The iterator returned by counts.__iter__() is bound to the name items so it can be queried for each element in turn. The handler for StopIteration does nothing on purpose; its only job is to provide a clean way to exit the while loop when the iterator runs out. This is exactly what a for statement does internally, made visible.

One detail makes this work for any iterator: to use an iterator in a for loop, the iterator must itself have an __iter__ method. The Python documentation suggests that an iterator's __iter__ method return the iterator itself, so that every iterator is also iterable. That is why you can write for x in some_iterator directly.

The for statement is a convenient interface over four operations: get an iterator with __iter__ (which iter calls for you), pull values with __next__ (which next calls for you), bind each value to the loop variable, and stop on StopIteration.

Step-by-Step Trace of the Hidden Loop

It helps to walk the iterator's state through the hand-written loop in (6). The iterator items carries one hidden piece of state: a position marker pointing at the next list element to return.

Start:           items position -> before 1
__next__() #1 -> returns 1, items position -> before 2, prints 1
__next__() #2 -> returns 2, items position -> before 3, prints 2
__next__() #3 -> returns 3, items position -> past end, prints 3
__next__() #4 -> raises StopIteration; the except clause runs pass; loop ends

The loop variable item is not a growing collection. It is a single name, rebound to 1, then 2, then 3, then never again. The position marker lives inside the iterator, not inside item, which is why the loop knows where it is even though item only ever holds the current value.

⚠Common Beginner Traps

One trap is thinking map immediately processes every value. The official double_and_print example proves otherwise: the function runs only when you pull a value with next or force the rest with list.

Another trap is changing a dictionary's structure while iterating over it. Adding or removing a key invalidates active iterators, and Python raises RuntimeError: dictionary changed size during iteration. Changing the value at an existing key is fine, because the set of keys does not change.

A third trap is believing the loop variable stores all the values it has seen. In a for loop the loop variable is rebound on each pass; it normally holds only the current item.

A fourth trap is forgetting that list(iterator) consumes the remaining values. After you force an iterator with list, it is usually exhausted, and pulling from it again yields nothing.

A fifth trap is assuming iter(d) gives you key-value pairs. It gives you keys only. For values, iterate d.values(); for pairs, iterate d.items().

§Summary

An iterable is any value you can pass to iter to get an iterator, and iterators dispense values one at a time. Strings, tuples, lists, ranges, sets, and dictionaries are all iterable, and so are iterators themselves. Iterating a dictionary yields its keys, while d.values() yields its values; changing a dictionary's structure mid-iteration invalidates its iterators. The built-in functions map, filter, zip, and reversed return lazy iterators that delay their work until you request each value. The for statement is built directly on the iterator protocol: it calls __iter__ to get an iterator, calls __next__ repeatedly, binds the loop variable to each value, runs the body, and stops when the iterator raises StopIteration — exactly the while-and-try loop you can write by hand.

▶Step-Through Checkpoint

This program does the work of a for statement over counts by hand, with the iterator written out so you can watch its state, and it accumulates a running total so a value persists across the passes. Stepping through it draws the environment diagram behind the loop: a single iterator object on the heap holding its hidden position, with the global names item and total rebound as each next call advances that one object. Before you step through the block below, write down three predictions: how many local frames will open over the whole run, what item and total each hold once the middle pair of lines has finished, and, counting the calls to next against the number of values in counts, what the final two lines will do.

Step through it one line at a time and check each prediction. No local frame ever opens, because iter and next are built-ins, so every binding lands in the global frame, which holds counts, items, total, and item. You should have seen the second call to next bind its value, 2, directly to total, with the addition then leaving total at 3; after the middle pair of lines, item holds 3 and total holds 6. The fourth call to next asks a three-value iterator for a value it does not have, so it raises StopIteration and the last line never runs. Throughout, the position marker stays hidden inside the iterator that items names; the global names are simply rebound as values arrive. A for statement performs this same sequence of iter, repeated next, and rebinding, but it also catches StopIteration for you, which is exactly the safety net this hand-written version lacks.

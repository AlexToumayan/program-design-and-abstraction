Chapter 2/2.6 Implementing Classes and Objects

2.6 Instances, Classes, and Bound Methods

◷ approx. 17 min read · 11 worked examples · best read in two passes

In the previous section you used Python's built-in class statement: you wrote class Account, defined some methods, and Python handled the rest. The dot in account.balance and the method call in account.deposit(20) just worked. By the end of this lesson you will know exactly what "the rest" was, because you will build it yourself.

You will construct a small but complete object system using only tools you already have: functions, dictionaries, local frames, and closures. You will be able to do four things: explain where an object keeps its own data, explain how an object borrows shared behavior from its class, explain the rule that turns a plain function into a method that knows its object, and trace the full pipeline that runs when a class is asked to make a new instance.

Nothing here requires new syntax. The whole point is that classes are not magic; they are a pattern of lookups that you can write out by hand.

A School Office With Folders and a Handbook

Picture a school office that has no computer. For each student there is a paper folder, and the folder holds facts that belong to that one student.

folder for Maya
  name: "Maya"
  grade: 10
  credits: 42

The office also keeps a single handbook on the shelf. The handbook holds rules and procedures that apply to every student, not to any one of them.

student handbook
  can_graduate: a procedure every student follows
  add_credit:   a procedure every student follows

When a clerk needs an answer about Maya, the clerk checks Maya's folder first. If the folder does not contain the answer, the clerk turns to the handbook. The folder is specific; the handbook is shared; the folder wins when both could answer.

That ordering is the whole object model in one sentence: look in the instance first, then fall back to the class. The folder is an instance, the handbook is a class, and "check the folder, then the handbook" is attribute lookup.

There is one extra wrinkle for procedures. A handbook rule like "add one credit" is written for any student in general. To actually run it for Maya, the clerk has to apply it to Maya's specific folder. Turning a general procedure from the handbook into one that acts on a particular folder is exactly what method binding will do.

Making the Machinery Visible

Python's dot notation hides this folder-then-handbook machinery. These two lines look simple:

account.balance
account.deposit(20)

To expose the moving parts, our hand-built system will not use dots. Instead, every object will be a dictionary that responds to string messages like 'get' and 'set'. The same two ideas become:

account['get']('balance')
account['get']('deposit')(20)

The second form is uglier, and that is the point. The dictionary lookup account['get'] is the act of asking the object to fetch something, and the trailing (20) is the act of calling what comes back. Writing it this way forces the lookup and the call to become separate, visible steps. This is the message-passing idea from earlier in the chapter, now used to build objects from the ground up.

A Warm-Up: One Attribute Box

Before instances and classes, build the smallest useful piece: a dictionary that hides a private store of attributes behind two operations, get and set.

# (1)
def make_box():
    contents = {}

    def get(name):
        return contents[name]

    def set(name, value):
        contents[name] = value

    return {'get': get, 'set': set}

The dictionary named contents is private: nothing outside make_box is handed a reference to it. The only way to reach it is through the get and set functions, and those two functions keep working after make_box returns because each one closes over the frame where contents lives. That closure behavior is the engine of this entire lesson.

Use the box:

# (2)
box = make_box()
box['set']('color', 'blue')
box['get']('color')

The last line evaluates to:

'blue'

This warm-up has no class, no methods, and no inheritance. It teaches exactly one brick: a dispatch dictionary can hide a private attribute dictionary and expose it only through named operations.

Reading the Return Line Token by Token

The return value in (1) is where the dispatch dictionary is born. Walk that one line from left to right, one token at a time.

return {'get': get, 'set': set}
├─ return ...... hand the dictionary back to whoever called make_box
├─ { ........... begin a dictionary literal
├─ 'get' ....... a string key naming the fetch operation
├─ : ........... separates this key from its value
├─ get ......... the local get function, stored as the value for 'get'
├─ , ........... separates one key-value entry from the next
├─ 'set' ....... a string key naming the store operation
├─ : ........... separates this key from its value
├─ set ......... the local set function, stored as the value for 'set'
├─ } ........... end the dictionary literal
└─ evaluates to  a dispatch dictionary {'get': <function>, 'set': <function>}

Reading a Message Send Token by Token

Now look at how that dictionary is used. The line box['set']('color', 'blue') is two operations stuck together, so read its tokens carefully.

box['set']('color', 'blue')
├─ box ......... the dispatch dictionary returned by make_box
├─ ['set'] ..... fetch the function stored under the key 'set'
├─ ( ........... begin a call to that fetched function
├─ 'color' ..... first argument: the attribute name to store
├─ , ........... separates the arguments
├─ 'blue' ...... second argument: the value to store under that name
├─ ) ........... end the call
└─ evaluates to  None, but as a side effect contents['color'] becomes 'blue'

The lesson of this anatomy is that box['set'] and ('color', 'blue') are genuinely separate: first a dictionary lookup produces a function, then that function is called. Every method call in the rest of this lesson has this same two-part shape.

From Boxes to Instances

An instance is a box with one added rule, the folder-then-handbook rule:

> If the instance does not have an attribute of its own, ask its class for it.

We can write that rule as a definition. Let the instance's own dictionary of attributes be checked first, and the class be consulted only on a miss. The big brace below is just a compact way to write "if this, use that; otherwise use the other." It says the same thing as the sentence above, nothing new.

lookup_instance(name)={the instance’s own attribute named name,the class attribute named name,​if the instance has itotherwise​

This local-first rule is what lets every account carry its own balance while sharing one deposit procedure with every other account of the same class.

The Instance Factory

Here is the official function that manufactures one instance. It is a dispatch dictionary, just like the box, but its get knows how to fall back to the class.

# (3)
def make_instance(cls):
    """Return a new object instance, which is a dispatch dictionary."""
    def get_value(name):
        if name in attributes:
            return attributes[name]
        else:
            value = cls['get'](name)
            return bind_method(value, instance)
    def set_value(name, value):
        attributes[name] = value
    attributes = {}
    instance = {'get': get_value, 'set': set_value}
    return instance

The parameter cls is the class for this new instance. We have not built classes yet, so for now read cls as "the handbook this folder can consult when it does not know something itself." It is another dispatch dictionary, and cls['get'](name) is how the instance asks the handbook.

What Each Name Holds

A call to make_instance(cls) builds a local frame. By the time the function returns, that frame holds these bindings:

make_instance frame
  cls        -> the class dispatch dictionary passed in
  get_value  -> a function (the instance's get operation)
  set_value  -> a function (the instance's set operation)
  attributes -> {}   the instance's private, initially empty store
  instance   -> {'get': get_value, 'set': set_value}

The crucial fact is that get_value and set_value were defined inside this frame, so they close over it. Even after make_instance returns and you are holding only the instance dictionary, those two functions can still reach attributes. The private store survives because the functions remember where it lives.

The Local-First Lookup, One Line at a Time

The heart of make_instance is the body of get_value. The branch that decides where to look is a single line, so give it the token treatment.

if name in attributes:
├─ if .......... begin a conditional test
├─ name ........ the attribute name being requested, a string
├─ in .......... membership test: is this key present?
├─ attributes .. the instance's own private dictionary
└─ performs ...  takes the local branch when the instance owns name, else the fallback branch

When that test is true, the instance answers from its own store:

return attributes[name]
├─ return ...... hand a value back to the caller
├─ attributes .. the instance's private dictionary
├─ [name] ...... look up the requested key in it
└─ evaluates to  the instance's own value for name

When the test is false, control goes to the else branch, which consults the class. Read its first line:

value = cls['get'](name)
├─ value ....... a new local name for what the class returns
├─ = ........... assignment
├─ cls ......... the class dispatch dictionary
├─ ['get'] ..... fetch the class's get operation
├─ (name) ...... call it, asking the class for this attribute
└─ performs ...  binds value to whatever the class supplies for name

The very next line decides whether that class value needs to become a method:

return bind_method(value, instance)
├─ return ........... hand the result back to the caller
├─ bind_method ...... the helper that turns a function into a bound method
├─ ( ................ begin the call
├─ value ............ the value the class supplied
├─ , ................ separates the arguments
├─ instance ......... this very instance, to be wired in as self
├─ ) ................ end the call
└─ evaluates to  value unchanged if it is data, or a method bound to instance if it is callable

A First Lookup Trace

Make the rule concrete. Suppose an instance has just one local attribute, and its class has two.

instance attributes
  'balance' -> 20

class attributes
  'interest' -> 0.02
  'deposit'  -> a function

Now compare three different lookups against that picture:

Expression Where the name is found Result
instance['get']('balance') the instance's own attributes 20
instance['get']('interest') the class attributes 0.02
instance['get']('deposit') the class attributes a bound method

The first lookup succeeds locally and stops. The second misses locally and falls back to the class, returning plain data. The third also falls back to the class, but because the class value is a function, the instance hands it to bind_method first, so what comes back is a method that already knows its instance.

Why a Method Must Be Bound

A method is more than a function. It is a function that already knows which object it is acting on. In built-in Python, a method call like this:

account.deposit(20)

is carried out as if you had written:

Account.deposit(account, 20)

The object sitting before the dot is quietly slipped in as the first argument, the one almost always named self. Our hand-built system gets no help from Python here, so it must recreate that slipping-in by hand. That is the job of bind_method.

The Binding Helper

# (4)
def bind_method(value, instance):
    """Return a bound method if value is callable, or value otherwise."""
    if callable(value):
        def method(*args):
            return value(instance, *args)
        return method
    else:
        return value

The contract is short. If value is not a function, hand it back untouched, because plain data like 0.02 is not a method and needs no self. If value is a function, build and return a brand-new function that calls the original with instance wedged in as the first argument.

Reading the Bound Method, Token by Token

First, the test that decides whether binding is even needed:

if callable(value):
├─ if .......... begin a conditional test
├─ callable .... a built-in that reports whether its argument can be called
├─ ( ........... begin the call to callable
├─ value ....... the value fetched from the class
├─ ) ........... end the call
└─ performs ...  binds value as a method when it is a function, else returns it as plain data

Now the line inside the new method that actually does the work. This is the line that inserts self, so read every token.

return value(instance, *args)
├─ return ....... hand the result back to whoever called the bound method
├─ value ........ the original function found in the class
├─ ( ............ begin the call to that original function
├─ instance ..... wedged in as the first argument, the implicit self
├─ , ............ separates the first argument from the rest
├─ *args ........ unpack every other argument the caller supplied
├─ ) ............ end the call
└─ evaluates to  whatever the original class function returns

The single name args does two jobs depending on where the star sits, and confusing them is a classic mistake. This table pins down both uses:

Where the star appears What it does
def method(*args): gathers every argument the caller passes into one tuple named args
value(instance, *args) spreads that tuple back out as separate arguments after instance

The payoff: if the class function expects (self, amount), the bound method only needs amount. The caller supplies amount, *args collects it, and value(instance, *args) supplies self automatically before spreading amount back in.

A Bridge You Can Verify by Hand

Before trusting bind_method inside the full machine, watch it work in isolation on a function you can read at a glance. Here is a function that genuinely expects self:

# (5)
def describe(self):
    return self['get']('name')

If describe were stored on a class, you would not want users to pass self themselves. Wire it up through the machinery and let binding supply self:

# (6)
MiniClass = {'get': lambda name: describe}
obj = make_instance(MiniClass)
obj['set']('name', 'Ada')
bound_describe = obj['get']('describe')
bound_describe()

Trace it by hand. MiniClass is a stand-in class whose get ignores the requested name and always returns the describe function. The call obj['get']('describe') misses obj's local attributes, so it asks MiniClass, gets describe, and passes it to bind_method. Because describe is callable, bind_method returns a wrapper that will call describe(obj). Calling that wrapper with no arguments runs describe(obj), which fetches obj's own 'name'. The final line evaluates to:

'Ada'

Without binding, calling describe() with no arguments would fail, because self would be missing. Binding is precisely what supplies it.

Building the Class Side

Instances can ask a class for attributes, so now we must build something for them to ask. In Python a class is itself an object, and in our system a class is itself another dispatch dictionary. (We make one simplifying choice: our implemented classes do not themselves have classes, so the chain of "what is the class of this thing" stops at the class.)

A class must answer three messages:

Message Meaning
'get' look up an attribute on the class
'set' store an attribute on the class
'new' build and return a new initialized instance

Here is the official class factory:

# (7)
def make_class(attributes, base_class=None):
    """Return a new class, which is a dispatch dictionary."""
    def get_value(name):
        if name in attributes:
            return attributes[name]
        elif base_class is not None:
            return base_class['get'](name)
    def set_value(name, value):
        attributes[name] = value
    def new(*args):
        return init_instance(cls, *args)
    cls = {'get': get_value, 'set': set_value, 'new': new}
    return cls

The parameter attributes is the class's own dictionary of names. The parameter base_class is either another class dictionary, when this class inherits from a parent, or None, when this class has no parent in our small system. Inheritance, one class building on another parent class, is the subject of a later section; here we only leave a hook (base_class) for it. If a class has no parent, base_class is just None and that branch never runs, so you can safely read past it for now. Notice that a class's get_value returns class values directly and does no binding; binding only happens on the instance side, because only an instance has a self to supply.

The Class Lookup Rule

A class checks its own attributes first, then falls back to its base class if there is one. If neither has the name, the lookup yields None.

lookup_class(name)=⎩⎨⎧​the class attribute named name,the base-class attribute named name,None,​if the class has itif there is a base classotherwise​

The line that handles the fallback uses an identity test, which is worth reading closely.

elif base_class is not None:
├─ elif ........... try this branch only if the first test failed
├─ base_class ..... the parent class dictionary, or None
├─ is not ......... identity test: are these literally different objects?
├─ None ........... Python's single sentinel object meaning "nothing here"
└─ performs ...... delegates the lookup to base_class when a parent exists

The test uses is not rather than != because None is a unique singleton object. The right question is not "does base_class equal some value?" but "is base_class literally the object None?" When a base class does exist, the lookup continues upward into it, which is how inheritance is expressed in this system.

Reading the new Message, Token by Token

The new function is short, but it closes over cls, which is the subtle part. Read its one meaningful line.

return init_instance(cls, *args)
├─ return .......... hand the finished instance back to the caller
├─ init_instance ... the helper that creates and initializes an instance
├─ ( ............... begin the call
├─ cls ............. this very class dictionary, captured by closure
├─ , ............... separates the arguments
├─ *args ........... spread the constructor arguments the caller passed
├─ ) ............... end the call
└─ evaluates to  a new, initialized instance of cls

The name cls appears in the body of new, yet new is defined on the line before cls is assigned. This is fine: a function body is not evaluated when the function is defined, only when it is called. By the time anyone sends the 'new' message, the line cls = {...} has already run, so cls is bound.

This is the dictionary-message version of calling a class. Where built-in Python writes Account('Kirk'), our system writes Account['new']('Kirk'): send the class the 'new' message to fetch its constructor function, then call that function with the constructor arguments.

Initializing a New Instance

The last official piece ties creation and construction together.

# (8)
def init_instance(cls, *args):
    """Return a new object with type cls, initialized with args."""
    instance = make_instance(cls)
    init = cls['get']('__init__')
    if init:
        init(instance, *args)
    return instance

This function does three things in order. First it creates a fresh, empty instance of cls. Then it asks the class for its constructor, the attribute named __init__. Finally, if such a constructor exists, it calls that constructor on the new instance, passing the instance as the first argument and spreading the rest. The initialized instance is returned.

The Full Creation Pipeline

When you evaluate an expression like SomeClass['new']('A'), the object system runs this exact sequence. Each numbered line is one step in a single chain, not a separate construct:

  1. Look up the class's new function with the 'new' message.
  2. Call that function as new('A').
  3. new calls init_instance(SomeClass, 'A').
  4. init_instance calls make_instance(SomeClass) to get a fresh instance.
  5. make_instance returns a new, empty instance dispatch dictionary.
  6. init_instance asks the class for __init__ with SomeClass['get']('__init__').
  7. If __init__ exists, it runs as init(instance, 'A'), filling in the instance.
  8. init_instance returns the now-initialized instance.

Every one of these steps is hidden behind a single ordinary SomeClass('A') in real Python. You have now seen all of it.

A Tiny Complete Class

Banking logic would bury the object-system mechanics, so test the whole machine on a deliberately small class first: a visitor badge that knows a name and a fixed label.

# (9)
def make_badge_class():
    label = 'visitor'
    def __init__(self, name):
        self['set']('name', name)
    def display(self):
        return self['get']('name') + ' - ' + self['get']('label')
    return make_class(locals())

The trick in the last line is locals(), a built-in that returns a dictionary of every name bound in the current frame. At that point the frame holds three names, so make_class receives exactly the class attributes we want:

{
  'label': 'visitor',
  '__init__': <function __init__>,
  'display': <function display>
}

Create the class and use it:

# (10)
Badge = make_badge_class()
maya = Badge['new']('Maya')
maya['get']('display')()

The final line evaluates to:

'Maya - visitor'

Tracing the Method Dispatch

Walk the last line as the machine does. The dispatch splits into the lookup half and the call half.

maya['get']('display')
  'display' is not among maya's own attributes
  so maya asks Badge for 'display'
  Badge has 'display', and returns the display function
  maya's get_value passes that function to bind_method
  bind_method wraps it into a method bound to maya
  the result is a bound method

maya['get']('display')()
  call that bound method with no extra arguments
  the bound method runs display(maya)
  display asks maya for 'name', found locally, gives 'Maya'
  display asks maya for 'label', missing locally, so Badge supplies 'visitor'
  the two strings are joined into 'Maya - visitor'

The split is the lesson: 'name' is the instance's own attribute, stored when __init__ ran, while 'label' lives on the class and is reached only by fallback. One method body, display, transparently reads from both the folder and the handbook.

What the Environment Looks Like

Step the badge block above and watch the global frame fill in. It holds the four machine functions plus the two objects you built:

Global frame
  make_instance    -> function
  bind_method      -> function
  make_class       -> function
  init_instance    -> function
  make_badge_class -> function
  Badge -> the class dispatch dictionary
  maya  -> the instance dispatch dictionary

The frame created by make_badge_class holds label, __init__, and display, and the Badge dictionary keeps reaching those names through its captured get_value. The frame created by make_instance(Badge) holds maya's private attributes dictionary, where 'name' lives. And while maya['get']('display')() is running, there is one more short-lived frame for the bound method that bind_method produced, the frame that supplies maya as self.

⚠Common Beginner Traps

Almost every mistake in this section comes from losing track of which dictionary owns which name.

Mistake Why it happens How to repair the mental model
Looking for every attribute on the instance Forgetting that the class is the fallback Lookup is local first, then the class; the folder, then the handbook
Treating a class function as ready to call Forgetting that methods must be bound A function from the class still needs an instance inserted as self
Writing value(*args) inside method Dropping the instance The correct call is value(instance, *args), with the instance first
Reaching for attributes.name Borrowing built-in dot notation A dict has no such attribute, so this raises AttributeError; this system uses dictionary messages: attributes[name]
Expecting set to search the class Confusing storing with looking up set always stores on whichever dictionary received the message, never the class above it
Worrying that new refers to cls too early Forgetting when a body runs A function body runs only when called, by which time cls is already assigned

✓Check Your Understanding

These check the machine model rather than memorization. Answer them before moving on to the account example in the next packet.

  1. If an instance has no local attribute named 'display', where does instance['get']('display') look next?
  2. Why does bind_method build an inner function named method instead of returning the class function directly?
  3. In def method(*args):, what does the star in *args collect, and what does the star do later in value(instance, *args)?
  4. When init_instance runs init(instance, *args), why is instance passed separately from the rest of the arguments?
  5. After maya is created, if maya['set']('label', 'guest') runs, which dictionary receives the new value: the instance dictionary or the class dictionary?
Show the answers
  1. It asks the class with cls['get']('display'). The instance does not hold the method itself; it delegates to its class, and then the object system binds the returned function to the instance.
  2. The class function still needs an object as its first argument, its self; the inner method exists to insert the specific instance automatically so the caller does not have to.
  3. In the definition, the star gathers every argument the caller passes into one tuple named args. In the call value(instance, *args), the star spreads that tuple back out as separate arguments, placed after instance.
  4. instance becomes the self argument that the constructor expects first; the remaining items in *args are the ordinary constructor arguments the programmer supplied, so they must follow self, not replace it.
  5. The instance dictionary receives the value, because set always stores on the dictionary that got the message. The instance's 'label' now shadows the class's 'label' for that one object.

§Summary

An object system needs nothing exotic: functions, dictionaries, local frames, and closures are enough. In this lesson an instance is a dispatch dictionary with a private store of attributes, and a class is a dispatch dictionary with shared attributes plus a 'new' message for building instances. Attribute lookup checks the instance first and falls back to the class, the folder-then-handbook rule. Method lookup adds one step: when the value found on the class is callable, bind_method wraps it so the instance is inserted as self. Constructing an object runs a fixed pipeline, new to init_instance to make_instance and then __init__, which is exactly the work that Python's class statement performs for you behind a single dot.

▶Step-Through Checkpoint

This is the lesson's capstone: the complete object system, then three lines that build Badge and maya and dispatch one method. Watching it draws the environment diagram that makes the hand-built object model visible: maya and Badge are dispatch dictionaries on the heap, each instance's and class's get_value (and set_value) is a closure whose parent pointer reaches back into the make_instance or make_class frame that holds its private attributes, while bind_method is a plain global helper whose returned method wrapper captures maya as the self it wedges in front of every call. Before you step through the block below, write down three predictions: how many local frames open while maya = Badge['new']('Maya') runs, how many get_value frames open during the last line, and how many times bind_method's frame opens during that same line.

# (11)
def make_instance(cls):
    def get_value(name):
        if name in attributes:
            return attributes[name]
        else:
            value = cls['get'](name)
            return bind_method(value, instance)
    def set_value(name, value):
        attributes[name] = value
    attributes = {}
    instance = {'get': get_value, 'set': set_value}
    return instance

def bind_method(value, instance):
    if callable(value):
        def method(*args):
            return value(instance, *args)
        return method
    else:
        return value

def make_class(attributes, base_class=None):
    def get_value(name):
        if name in attributes:
            return attributes[name]
        elif base_class is not None:
            return base_class['get'](name)
    def set_value(name, value):
        attributes[name] = value
    def new(*args):
        return init_instance(cls, *args)
    cls = {'get': get_value, 'set': set_value, 'new': new}
    return cls

def init_instance(cls, *args):
    instance = make_instance(cls)
    init = cls['get']('__init__')
    if init:
        init(instance, *args)
    return instance

def make_badge_class():
    label = 'visitor'
    def __init__(self, name):
        self['set']('name', name)
    def display(self):
        return self['get']('name') + ' - ' + self['get']('label')
    return make_class(locals())

Badge = make_badge_class()
maya = Badge['new']('Maya')
answer = maya['get']('display')()

Step through it one line at a time and check each prediction as the frames open. The creation line opens six local frames: new, init_instance, make_instance, the class's get_value fetching __init__, then __init__ and the set_value that stores 'name' in maya's formerly empty folder. The last line opens get_value five times, three on the instance side for 'display', 'name', and 'label' and two on the class side, where Badge's get_value reaches the badge attributes through its closure. And bind_method's frame opens twice, not once: once to wrap display with maya wedged in as the hidden first argument, and once on 'visitor', where the callable test fails and the string passes through untouched. Every value borrowed from the class makes that trip; only functions come back changed. When the last line finishes, answer holds 'Maya - visitor' in the global frame.

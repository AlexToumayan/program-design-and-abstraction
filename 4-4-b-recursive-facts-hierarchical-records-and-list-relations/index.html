Chapter 4/4.4 Logic Programming

4.4.2 Recursive Facts, Hierarchical Records, and List Relations

◷ approx. 20 min read · 1 worked example · best read in two passes

In the previous packet you learned to state simple facts and compound facts in the logic language, and to ask the query interpreter to fill in ?-variables. Everything there could be answered by looking one step away: a parent, or a child defined directly from a parent.

This lesson is about reach. By the end of it you will be able to do four things. You will write a recursive fact, where a relation is defined partly in terms of itself, so the interpreter can follow a chain as long as it needs to. You will run compound queries that force several conditions to hold at once. You will store and query hierarchical facts, where a record contains nested lists instead of a flat row of symbols. And you will write a relation like append-to-form and run it in more than one direction, asking it both "what is the result?" and "what inputs would produce this result?"

The one habit to carry in: a fact in logic is not an instruction that runs. It is a statement that is either true or not, and the interpreter's job is to search for a way to make your query true using the facts it has. Recursion here is not a function calling itself; it is a relation that can be proven by appealing to a smaller version of the same relation.

A Chain Is Not a Single Link

Picture a family tree drawn as arrows, each arrow pointing from a parent down to a child.

fillmore ──▶ delano ──▶ herbert

If you only know the word "parent," you can read exactly one arrow at a time. fillmore is the parent of delano. delano is the parent of herbert. But there is no single arrow from fillmore straight to herbert, so the word "parent" cannot connect them.

The word ancestor is what lets you walk the whole chain. An ancestor is a parent, or a parent of a parent, or a parent of a parent of a parent, and so on for any length. That phrase "and so on" is the entire idea of recursion in one breath: you do not list every possible length, you say "one step, then the same thing again on what remains."

This is the same move you already know from recursive functions, where a base case handles the smallest input and a recursive case reduces a larger input to a smaller one. Here the base case is "a direct parent already counts as an ancestor," and the recursive case is "take one parent step, then ask the ancestor question again about a shorter chain." The logic interpreter will do the walking; you only describe what one step looks like and where to stop.

Recursive Facts: Defining ancestor

The logic language allows recursive facts. That means the conclusion of a fact may depend on a hypothesis that mentions the same relation name. The official text defines the ancestor relation with exactly two facts. In words: some ?a is an ancestor of ?y if it is a parent of ?y, or if it is the parent of an ancestor of ?y.

(fact (ancestor ?a ?y) (parent ?a ?y))
(fact (ancestor ?a ?y) (parent ?a ?z) (ancestor ?z ?y))

The first fact is the base case. Read it as a compound fact: the conclusion (ancestor ?a ?y) holds provided the single hypothesis (parent ?a ?y) holds. So a direct parent is, by itself, enough to be an ancestor.

The second fact is the recursive case. Its conclusion is again (ancestor ?a ?y), but now there are two hypotheses, and both must hold. First (parent ?a ?z): ?a is the parent of some intermediate dog ?z. Then (ancestor ?z ?y): that intermediate ?z is itself an ancestor of ?y. The relation name ancestor appears in its own hypothesis, which is what makes the fact recursive.

Walk the first fact one token at a time. Every row annotates a single piece of the line; the final row says what the whole fact asserts.

(fact (ancestor ?a ?y) (parent ?a ?y))
├─ fact ............... declares a new fact (not a procedure call)
├─ (ancestor ?a ?y) ... the conclusion: ?a is an ancestor of ?y
├─ (parent ?a ?y) ..... the one hypothesis that must hold
└─ asserts ........... a direct parent already counts as an ancestor

Now the second fact. It is the same conclusion, but with two hypotheses chained by a shared intermediate variable ?z.

(fact (ancestor ?a ?y) (parent ?a ?z) (ancestor ?z ?y))
├─ fact ............... declares a new fact
├─ (ancestor ?a ?y) ... the conclusion: ?a is an ancestor of ?y
├─ (parent ?a ?z) ..... hypothesis 1: ?a is the parent of some ?z
├─ (ancestor ?z ?y) ... hypothesis 2: that ?z is an ancestor of ?y
└─ asserts ........... one parent step plus a smaller ancestor problem proves the whole

The intermediate variable ?z is the hinge. It does not appear in the conclusion at all; it exists only to connect "the parent step" to "the smaller ancestor problem." The interpreter is free to try every dog in the database as a candidate ?z.

A Warm-Up Before the Recursion

Recursion is easier to trust once you have watched the base case work on its own. Suppose, for a moment, the database held just this parent fact:

(parent delano herbert)

Then the base-case fact alone answers "is delano an ancestor of herbert?" The conclusion (ancestor delano herbert) needs only the hypothesis (parent delano herbert), and that fact is right there. No recursion is involved yet. This is the smallest possible ancestor proof, and it is the floor that every longer chain eventually lands on.

Querying Recursive Facts

With both ancestor facts in place, a single query can list every ancestor of a dog. The official query asks for all ancestors of herbert:

(query (ancestor ?a herbert))

The interpreter reports success and offers every value of ?a it can prove. To see why there is more than one answer, recall the parent facts that involve this branch of the family. From the previous packet's database, the relevant links are (parent delano herbert), (parent fillmore delano), and (parent eisenhower fillmore).

Here is the reasoning the interpreter carries out, written as a trace. This is a derivation built from the official facts, not a printed transcript from the page; the page does not print output for this query.

?a = delano      base case:      (parent delano herbert) holds directly
?a = fillmore    recursive case: (parent fillmore delano) and delano is an ancestor of herbert
?a = eisenhower  recursive case: (parent eisenhower fillmore) and fillmore is an ancestor of herbert

Each successful ?a is one rung higher than the last. delano is found by the base case. fillmore is found by taking one parent step to delano, then reusing the fact that delano is already an ancestor of herbert. eisenhower is found by taking one parent step to fillmore, then reusing the fact that fillmore is an ancestor of herbert. A single recursive definition reaches the entire chain.

Compound Queries

A query may contain more than one subexpression. When it does, all of the subexpressions must be satisfied at the same time by one assignment of values to variables. The rule that makes this powerful: if a variable appears more than once in a query, it must take the same value in every place it appears.

The official query finds the dogs that are ancestors of both herbert and barack:

(query (ancestor ?a barack) (ancestor ?a herbert))

Because ?a appears in both subexpressions, the interpreter is not looking for two independent answers. It is looking for a single dog that is simultaneously an ancestor of barack and an ancestor of herbert.

Walk the query token by token.

(query (ancestor ?a barack) (ancestor ?a herbert))
├─ query ................. asks the interpreter to find satisfying values
├─ (ancestor ?a barack) .. subexpression 1: ?a must be an ancestor of barack
├─ (ancestor ?a herbert) . subexpression 2: the SAME ?a must be an ancestor of herbert
└─ finds ................. every ?a for which both hold at once

To see which dogs qualify, compare the two ancestor chains. The ancestors of barack run abraham, then fillmore, then eisenhower. The ancestors of herbert run delano, then fillmore, then eisenhower. The dogs that appear in both lists are fillmore and eisenhower. Those are the satisfying values of ?a. You can read the supporting parent steps as two short chains for each answer:

fillmore   -> abraham  -> barack        and   fillmore   -> delano -> herbert
eisenhower -> fillmore -> abraham -> barack   eisenhower -> fillmore -> delano -> herbert

abraham, by contrast, is an ancestor of barack but not of herbert, so it is excluded — the shared variable forces both conditions on one value.

A Step-by-Step Proof: (ancestor fillmore herbert)

Recursive facts may require long chains of inference to connect a query to the simple facts already in the database. The official text shows the chain the interpreter must establish to prove (ancestor fillmore herbert). Each line is proven in turn, and later lines lean on earlier ones.

(parent delano herbert)       ; (1), a simple fact
(ancestor delano herbert)     ; (2), from (1) and the 1st ancestor fact
(parent fillmore delano)      ; (3), a simple fact
(ancestor fillmore herbert)   ; (4), from (2), (3), & the 2nd ancestor fact

Read it from the top. Line (1) is a simple fact that is already in the database, no proof required. Line (2) applies the base-case ancestor fact to line (1): since delano is a parent of herbert, delano is an ancestor of herbert. Line (3) is another simple fact straight from the database. Line (4) applies the recursive ancestor fact, using line (3) as the parent step (parent fillmore delano) and line (2) as the smaller ancestor problem (ancestor delano herbert); together they prove (ancestor fillmore herbert).

This is the heart of how the interpreter "thinks." It does not run a hand-written loop that climbs the family tree. It searches for a chain of facts that supports the query, applying the ancestor facts as rules of inference. In this way a single fact can imply a large number of further facts — possibly infinitely many — as long as the interpreter can discover them.

Hierarchical Facts: Records With Nested Lists

So far every fact and query has been a flat list of symbols, like (parent abraham barack). But a fact list is allowed to contain other lists. That gives you a way to represent hierarchical data: structure nested inside structure. The official text stores the color of each dog alongside its name as one richer record:

(fact (dog (name abraham) (color white)))
(fact (dog (name barack) (color tan)))
(fact (dog (name clinton) (color white)))
(fact (dog (name delano) (color white)))
(fact (dog (name eisenhower) (color tan)))
(fact (dog (name fillmore) (color brown)))
(fact (dog (name grover) (color tan)))
(fact (dog (name herbert) (color brown)))

Look at one record closely. The fact (dog (name clinton) (color white)) is not three loose symbols; it is a dog whose body is two sub-lists. One sub-list, (name clinton), holds the name. Another, (color white), holds the color. The nesting is the point: the record carries labeled fields rather than relying on position alone.

Walk the structure of a single hierarchical fact.

(fact (dog (name clinton) (color white)))
├─ fact ................. declares a new fact
├─ (dog ...) ............ the record is a dog with a nested body
├─ (name clinton) ....... nested sub-list: the name field
├─ (color white) ........ nested sub-list: the color field
└─ stores .............. one dog record with two labeled, nested fields

This extra structure is not decoration. It gives the query interpreter more shape to match against, which is exactly what the next queries exploit.

Querying Inside a Hierarchical Fact

A query can spell out the full structure of a hierarchical fact, or it can let a variable stand in for a whole nested list. The official text shows both.

First, a query that drills into the structure and pulls out one field. It fixes the name to clinton and leaves the color open as ?color:

(query (dog (name clinton) (color ?color)))

The interpreter matches this pattern against the records and finds the dog whose name is clinton. That record is (dog (name clinton) (color white)), so the only way to satisfy the pattern is to bind ?color to white. The query succeeds with ?color taking the value white.

Second, a query that puts a single variable where a whole sub-list sits. Here ?info is not inside (color ...); it stands in for everything after the name:

(query (dog (name clinton) ?info))

Now ?info matches the entire remaining list of the record. For clinton, the part after (name clinton) is (color white), so the query succeeds with ?info bound to the whole list (color white). Walk the difference between the two queries:

(query (dog (name clinton) ?info))
├─ query ............. find satisfying values
├─ (dog ...) ......... match a dog record
├─ (name clinton) .... fix the name field to clinton
├─ ?info ............. one variable in place of the WHOLE rest of the record
└─ binds ............ ?info to the entire nested list (color white)

This is a sharp break from ordinary table rows, where a column holds a single flat value. A logic query can match against nested structure directly, binding a variable to a single symbol or to an entire sub-list depending on where you place it.

Joining Several Kinds of Facts in One Query

Much of a database's power is the interpreter's ability to join different kinds of facts inside one query. The official text combines the dog records with the ancestor relation to find every pair of dogs where one is an ancestor of the other and they share a color:

(query (dog (name ?name) (color ?color))
       (ancestor ?ancestor ?name)
       (dog (name ?ancestor) (color ?color)))

Read the three subexpressions as three conditions that must all hold for one assignment. Find a dog named ?name with color ?color. Find an ?ancestor of that dog. Then require that the ancestor is itself a dog whose color is ?color — the same ?color as the first dog, because the variable repeats.

(query (dog (name ?name) (color ?color))
       (ancestor ?ancestor ?name)
       (dog (name ?ancestor) (color ?color)))
├─ (dog (name ?name) (color ?color)) ..... a dog with some name and color
├─ (ancestor ?ancestor ?name) ............ ?ancestor is an ancestor of that dog
├─ (dog (name ?ancestor) (color ?color)) . the ancestor is a dog of the SAME color
└─ finds ................................. descendant/ancestor pairs that share a color

The repeated ?color is what ties the two dog records together: the color found for the descendant must equal the color found for the ancestor. Tracing the data, the satisfying combinations are these four:

name: barack    color: tan     ancestor: eisenhower
name: clinton   color: white   ancestor: abraham
name: grover    color: tan     ancestor: eisenhower
name: herbert   color: brown   ancestor: fillmore

For instance, herbert is brown and has the ancestor fillmore, who is also brown, so that pair qualifies. The programmer never wrote a loop over all pairs of dogs. The interpreter searched for substitutions that make all three subexpressions true at once. (As with the earlier queries, the page prints no output for this query; the rows above are derived from the official facts to show what the join yields.)

Dotted Lists and the Rest of a Fact

Variables can refer to nested lists, as you just saw with ?info. They can also grab the tail of a list using dot notation. The dot splits a list into its first element and the rest: (a b c) can also be written (a . (b c)), and a pattern like (?first . ?rest) matches any non-empty list, binding ?first to the head and ?rest to everything after it. A variable following a dot matches the rest of the list of a fact, and dotted lists are allowed in either facts or queries. In a pattern, putting a variable after the dot lets that variable capture however many elements are left over.

The official text uses dotted lists to construct pedigrees — a dog's chain of ancestry — observing that young barack follows a venerable line of presidential pups:

(fact (pedigree ?name) (dog (name ?name) . ?details))
(fact (pedigree ?child ?parent . ?rest)
      (parent ?parent ?child)
      (pedigree ?parent . ?rest))

The first fact is the base case. Its conclusion (pedigree ?name) is a one-element pedigree containing just the dog itself. Its hypothesis (dog (name ?name) . ?details) simply requires that ?name is some dog on record; the dotted ?details soaks up the rest of that dog record (its color), which the pedigree does not otherwise need.

The second fact is the recursive case. Walk its conclusion, which is itself a dotted list:

(pedigree ?child ?parent . ?rest)
├─ pedigree .... the relation being concluded
├─ ?child ...... the first element of the pedigree
├─ ?parent ..... the second element of the pedigree
├─ . ?rest ..... a variable capturing the remaining elements of the pedigree
└─ means ...... a pedigree that starts ?child, ?parent, then whatever ?rest holds

Its two hypotheses say: ?parent is the parent of ?child, and ?parent already has a pedigree whose tail is ?rest. Put together, if ?parent is the parent of ?child, and ?parent has a pedigree, then ?child has a longer pedigree that begins with ?child and ?parent and continues with ?parent's pedigree tail.

The official query asks for barack's pedigree, capturing the tail after barack in ?lineage:

(query (pedigree barack . ?lineage))

Each successful answer extends the chain one ancestor further. Following barack's parent links — barack's parent is abraham, whose parent is fillmore, whose parent is eisenhower — the lineages the interpreter can prove are:

?lineage = ()
?lineage = (abraham)
?lineage = (abraham fillmore)
?lineage = (abraham fillmore eisenhower)

The empty ?lineage comes from the base case (a pedigree of just barack himself); each longer list comes from one more application of the recursive pedigree fact. (These tails are derived from the official parent facts; the page itself prints no output for this query.)

A Relational Version of Append

Declarative programming can express relationships among facts with striking economy. As the capstone, the official text shows how to describe appending two lists — forming a longer list with the elements of the first followed by the elements of the second — using two facts.

First, the base case. Appending an empty list to any list gives that same list back:

(fact (append-to-form () ?x ?x))

Read the three arguments positionally as left list, right list, and whole list. The left list is (), the right list is ?x, and the whole is also ?x. So () appended to ?x is ?x for any ?x.

Second, the recursive fact. A list whose first element is ?a and whose rest is ?r, appended to a list ?y, forms a list whose first element is ?a and whose rest is some ?z. For that to hold, ?r and ?y must themselves append to form ?z:

(fact (append-to-form (?a . ?r) ?y (?a . ?z)) (append-to-form ?r ?y ?z))

Walk the conclusion of this recursive fact one argument at a time. The dotted lists are doing the structural work.

(append-to-form (?a . ?r) ?y (?a . ?z))
├─ append-to-form .... the relation: left, right, whole
├─ (?a . ?r) ......... left list: first element ?a, rest ?r
├─ ?y ................ right list, unchanged
├─ (?a . ?z) ......... whole list: same first element ?a, rest ?z
└─ holds when ....... ?r and ?y append to form ?z  (the single hypothesis)

The shared first element ?a is the key: appending does not touch the head of the left list, it just carries that head straight through to the result and recurses on the shorter rest ?r. The base case stops the recursion once the left list is empty.

Computing an Append Result

Using these two facts, the interpreter can compute the result of appending two lists. The official forward query supplies both inputs and leaves the whole as ?result:

(query (append-to-form (a b c) (d e) ?result))

The interpreter peels one element off the left list at each recursive step — a, then b, then c — until the left list is empty and the base case attaches (d e). The answer it computes is ?result bound to (a b c d e). This looks like calling an append function, but notice the framing is different. You did not say "run append on two lists." You said "find a ?result such that the append-to-form relation holds."

Running the Relation Backward

Because append-to-form is a relation and not a one-way function, the very same two facts can solve for the inputs. The official backward query fixes the whole list and leaves both ?left and ?right open:

(query (append-to-form ?left ?right (a b c d e)))

This asks for every pair of lists that append to form (a b c d e). There is one such pair for each place you could split the list, and the interpreter finds them all:

?left = ()           ?right = (a b c d e)
?left = (a)          ?right = (b c d e)
?left = (a b)        ?right = (c d e)
?left = (a b c)      ?right = (d e)
?left = (a b c d)    ?right = (e)
?left = (a b c d e)  ?right = ()

That is the distinctive strength of logic programming: a single relationship can be used in multiple directions. The same two facts that compute a result can also enumerate the inputs, because the interpreter is not executing a procedure — it is searching for assignments that make the relation true. (The six splits above are derived from the two official facts; the page itself prints no output for this query.)

Although this query interpreter can look quite intelligent, you will see in the next section that it finds all of these combinations through one simple operation repeated many times: matching two lists that contain variables within an environment. That operation is called unification, and it is the subject of the next packet.

⚠Common Beginner Traps

  • Treating recursive facts as recursive Python functions that call themselves. A fact does not run or call anything. The interpreter uses the facts as rules while it searches for a proof of your query.
  • Dropping the base case. Without (fact (ancestor ?a ?y) (parent ?a ?y)), the recursive fact has nothing to bottom out on, and a direct parent would never be recognized as an ancestor.
  • Forgetting that a repeated variable forces one shared value. In (query (ancestor ?a barack) (ancestor ?a herbert)), both ?a's must be the same dog; that constraint is the whole reason the answer is just fillmore and eisenhower.
  • Thinking nested lists are cosmetic. In (dog (name clinton) (color white)) the nesting is real structure, and a query can match it directly — even binding one variable to an entire sub-list.
  • Misreading the dot. In (?a . ?r) the dot separates the first element from the rest; ?r captures everything after ?a, which may be many elements or none.
  • Assuming (append-to-form ?left ?right (a b c d e)) can only fill in the third argument. A relation can solve for whichever parts you leave as variables, so this query returns every way to split the list.

✓Check Your Understanding

  1. Of the two ancestor facts, which is the base case, and why can it succeed without any recursive step?
  2. In (query (ancestor ?a barack) (ancestor ?a herbert)), why must ?a take the same value in both subexpressions, and what answers does that leave?
  3. What does the query (query (dog (name clinton) ?info)) bind ?info to, and how does that differ from (query (dog (name clinton) (color ?color)))?
  4. In the recursive append fact (fact (append-to-form (?a . ?r) ?y (?a . ?z)) (append-to-form ?r ?y ?z)), what role does the shared symbol ?a play?
  5. Run the backward relation on a new list: how many ?left/?right pairs satisfy (query (append-to-form ?left ?right (a b c))), and what are they?
Show the answers
  1. The first fact, (fact (ancestor ?a ?y) (parent ?a ?y)), is the base case. Its only hypothesis is a single parent fact, which can be found directly in the database, so no smaller ancestor problem has to be solved.
  2. The variable ?a appears in both subexpressions, and a repeated variable must take one shared value across the whole query. That forces ?a to be an ancestor of barack and of herbert simultaneously, leaving fillmore and eisenhower.
  3. It binds ?info to the whole remaining sub-list of the record, (color white). The color query instead reaches inside that sub-list and binds ?color to the single symbol white.
  4. ?a is the first element of both the left list and the whole list. It carries the head of the left list straight through to the result unchanged, while the hypothesis recurses on the shorter rest ?r.
  5. Four pairs, one for each place the list can be split: ?left = () with ?right = (a b c); ?left = (a) with ?right = (b c); ?left = (a b) with ?right = (c); and ?left = (a b c) with ?right = ().

§Summary

Recursive facts let a logic program derive relationships that need a chain of evidence rather than a single link. The ancestor relation uses a base case for a direct parent and a recursive case that takes one parent step and then asks the ancestor question again about a shorter chain, and the interpreter proves a query by searching for a chain of facts that supports it. Compound queries push several conditions at once, with repeated variables forced to share one value. Hierarchical facts store nested sub-lists, and queries can match that structure directly — binding a variable to a single field or to a whole sub-list, and using dot notation to capture the rest of a list. Relations like append-to-form show why logic programming is distinctive: one declaration can compute a result or infer the inputs, because the interpreter is not running a procedure but matching lists that contain variables in an environment — the unification operation explored next.

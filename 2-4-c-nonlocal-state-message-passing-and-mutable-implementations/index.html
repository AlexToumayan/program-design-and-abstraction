Chapter 2/2.4 Mutable Data

2.4 Nonlocal State, Message Passing, and Mutable Implementations

◷ approx. 20 min read · 17 worked examples · best read in two passes

You already know that an inner function can keep a private value alive across calls by rebinding it with nonlocal. This lesson asks what that ability buys you, what it costs you, and how far it can take you. By the end you will be able to explain why two functions built from the same constructor have completely separate memories, why giving one of those functions a second name is dangerous, and how a single function can act like a whole object by reading a message and choosing its own behavior. You will then use exactly that idea to build a mutable linked list, a dictionary, and a bank account, all out of nothing but functions and local state, before this course ever shows you the class keyword.

The thread running through all of it is one habit: when state is involved, stop reading code as a list of values and start tracking which frame each name lives in. A value that changes over time has an identity that outlives its current contents, and identity is the thing you have to keep your eye on.

Separate Tellers, Separate Drawers

Picture a bank with a row of tellers. Each teller has a private cash drawer that only they can open. When you walk up to a teller and ask to withdraw money, that teller reaches into their own drawer, hands you cash, and the drawer now holds less. The teller two windows down is completely unaffected; their drawer never moved.

That is exactly the situation a constructor function creates. A function that builds and returns another function is like a hiring office that, each time it runs, sets up a fresh teller with a fresh drawer. Two tellers hired on two separate visits share nothing. But if you simply write a second name on the same teller's nameplate, you have not hired anyone new, you have just given one teller two names, and both names reach into the same drawer.

Later in the lesson the metaphor shifts. A dispatch function is less like a teller and more like a front desk that takes a written request, reads the first word to decide what you want, and carries out the matching task. That single front desk can answer many different kinds of request, which is the whole point of organizing a program around messages.

A State-Keeping Function, Reviewed

The constructor we keep returning to builds a withdraw function around a private balance. Here it is exactly as the original text writes it, including its docstring:

# (1)
def make_withdraw(balance):
    """Return a withdraw function that draws down balance with each call."""
    def withdraw(amount):
        nonlocal balance
        if amount > balance:
            return 'Insufficient funds'
        balance = balance - amount
        return balance
    return withdraw

Each time you call make_withdraw, Python builds a new frame for that call, and the balance parameter lives in that frame. The inner withdraw function carries a reference back to whichever frame it was born in. So the saved balance is not stored on the withdraw function as a public attribute; it sits in the parent frame, reachable by withdraw and by nothing else in the program.

Calling the returned function repeatedly walks the balance down, and because the same call returns a different number the second time, this is not a pure function:

>>> withdraw = make_withdraw(100)
>>> withdraw(25)
75
>>> withdraw(25)
50
>>> withdraw(60)
'Insufficient funds'
>>> withdraw(15)
35

The nonlocal balance line is the hinge that makes this work. Read it on its own:

nonlocal balance
├─ nonlocal ........ a declaration about the name that follows
├─ balance ......... the name being declared
└─ states that      assignments to balance rebind the balance in the
                    enclosing make_withdraw frame, instead of creating
                    a new local balance inside withdraw

Without that declaration, the assignment balance = balance - amount would make balance a brand-new local name, and the persistent drawer would never change. With it, the assignment reaches outward into the frame where the money is actually kept.

The Benefit: Separate Calls Build Separate State

Here is the payoff of all that frame machinery. Because every call to make_withdraw builds its own frame, two functions made from two separate calls maintain two separate balances. The original text extends the example to make this concrete, binding a second result to wd2:

The two mutating calls at the bottom produce, in order:

1
12

The reason is that wd and wd2 have different parent frames. The name wd is bound to a function whose drawer started at 20; the name wd2 is bound to a different function whose drawer started at 7.

wd  -> withdraw function whose parent frame has balance = 20
wd2 -> withdraw function whose parent frame has balance = 7

Calling wd2(6) lowers wd2's private balance from 7 to 1 and returns 1. That change touches nothing belonging to wd. So when wd(8) runs next, it works against the untouched balance of 20, lowers it to 12, and returns 12. Each instance of withdraw manages its own internal state, hidden from every other function in the program. That is the abstraction of a bank account: a thing that changes over time according to its own private history.

Tracing the Two Drawers by Hand

You can verify the two return values without a computer by tracking each drawer separately:

Step 1  make_withdraw(20) builds frame f1 with balance = 20; wd points to its withdraw.
Step 2  make_withdraw(7) builds a separate frame f2 with balance = 7; wd2 points to its withdraw.
Step 3  wd2(6): amount is 6, 6 is not greater than 7, so f2's balance drops 7 -> 1; returns 1.
Step 4  wd(8): amount is 8, 8 is not greater than 20, so f1's balance drops 20 -> 12; returns 12.

Nothing in Step 3 can reach f1, and nothing in Step 4 can reach f2. Separate calls, separate frames, separate memories.

The Cost: A Second Name Is Not a Second Account

Now change one line and the meaning changes completely. Instead of calling the constructor a second time, the next program copies the function into a second name:

The two mutating calls now produce:

11
10

The assignment wd2 = wd does not run make_withdraw again, so no new frame is born. It writes a second name on the existing function. Both names point at one withdraw, and that one withdraw reaches into one drawer.

wd  ----+
        |
        v
     one withdraw function whose parent frame has balance = 12
        ^
        |
wd2 ----+

So wd2(1) lowers the shared balance from 12 to 11, and the very next call wd(1) sees that already-lowered 11 and brings it to 10. Calling the function through either name changes the single value that both names observe.

Why This Breaks Easy Reasoning

Before mutation entered the picture, you could replace an expression with its value and the program meant the same thing. An expression built only from pure calls is referentially transparent: substituting any subexpression with the value it computes does not change anything. Rebinding operations break that property, because they do more than return a value, they change the environment.

Concretely, suppose wd = make_withdraw(12). The expression wd(1) returns 11 the first time you evaluate it, but it also lowers the hidden balance, so the same expression returns 10 the next time. You can no longer treat wd(1) as a fixed stand-in for 11. The key rule for analyzing code like this is short: only evaluating a call expression (a function call) introduces a new frame, and assignment always changes a binding in an existing frame. Unless make_withdraw was actually called twice, there is exactly one balance to talk about.

Identity Outlives the Pieces

These subtleties force a question that pure values let you ignore: what does it mean for two things to be "the same"? With unchanging values, equal is as good as identical:

# (4)
a = 4
b = 4

It does not matter whether a and b point at "the same 4" or two different fours, because 4 will never change. You can treat a compound value built only from immutable pieces as nothing more than the totality of those pieces; a rational number simply is its numerator and denominator.

State breaks that view. Once a value can change, it gains an identity separate from its current contents:

# (5)
first_account = make_withdraw(10)
second_account = make_withdraw(10)
alias = first_account

Here first_account and second_account start with equal balances, but they are different accounts; lowering one leaves the other alone. Meanwhile alias and first_account are two names for the same account, so a withdrawal through either one is visible through the other. A bank account stays "the same" account after you change its balance, and two accounts can share a balance yet remain distinct objects. Holding identity and contents apart in your head is the mental shift that the rest of this lesson depends on.

Building Mutable Data Out of Functions

Despite the care it demands, nonlocal assignment is powerful: different parts of a program, living in different frames, can evolve independently. In fact, functions with local state are enough to implement mutable data types that behave like the built-in list and dict. Python does not expose how lists are implemented internally, only the sequence abstraction and the mutation methods, so to see how mutation could work underneath, we will build a mutable linked list ourselves.

We need the linked-list toolkit from the sequences chapter. None of this is new; it is the substrate the mutable list will store and operate on:

A mutable linked list needs an identity, like any mutable value. We cannot use None for an empty mutable list, because two empty lists must be distinguishable (appending to one must not append to the other), yet None is None. Two different functions that each hold empty as their local state solve this: they are distinct objects even when both are logically empty.

The Dispatch Function

If a mutable list is going to be a single function, what arguments does it take? The answer is a pattern you will reuse constantly. The function is a dispatch function: its first argument is a message, a string naming what to do, followed by any extra arguments that the chosen behavior needs. One dispatch function is effectively many functions bundled together, with the message selecting which one runs.

Before the official version, build the intuition on a tiny dispatch function you can evaluate in your head. This one keeps a private counter and understands two messages:

# (7)
def make_counter():
    count = 0
    def dispatch(message):
        nonlocal count
        if message == 'increment':
            count = count + 1
            return count
        elif message == 'reset':
            count = 0
    return dispatch

Trace it: c = make_counter() hands back the inner dispatch. Then c('increment') returns 1, c('increment') returns 2, c('reset') returns None (and quietly sets the count back to 0), and a following c('increment') returns 1 again. The string you pass in steers the behavior; the private count survives between calls. That is the entire idea, just smaller.

Now the official mutable linked list. It answers five messages: len, getitem, push_first, pop_first, and str. Reproduced verbatim:

# (8)
def mutable_link():
    """Return a functional implementation of a mutable linked list."""
    contents = empty
    def dispatch(message, value=None):
        nonlocal contents
        if message == 'len':
            return len_link(contents)
        elif message == 'getitem':
            return getitem_link(contents, value)
        elif message == 'push_first':
            contents = link(value, contents)
        elif message == 'pop_first':
            f = first(contents)
            contents = rest(contents)
            return f
        elif message == 'str':
            return join_link(contents, ", ")
    return dispatch

The local name contents is the hidden state, and it starts as empty. The header line that makes this a dispatch function deserves a careful reading:

def dispatch(message, value=None):
├─ def ............. begins a function definition
├─ dispatch ........ the name of the inner function, the one that gets returned
├─ ( ............... begins the parameter list
├─ message ......... required first parameter: the string that selects a behavior
├─ , ............... separates the parameters
├─ value=None ...... optional second parameter; defaults to None when no extra
│                    argument is supplied, so messages like 'len' need no value
├─ ) ............... ends the parameter list
└─ defines          a single function that performs different work depending on
                    the message it receives

The five-way if/elif chain is the menu. Two messages (len, getitem) implement the sequence abstraction; two (push_first, pop_first) add or remove the front element; the last (str) renders the whole list as text. Here is the menu as a quick reference:

'len'         return the length
'getitem'     return the element at an index
'push_first'  add a value to the front (mutates; returns None)
'pop_first'   remove and return the first value (mutates)
'str'         return a readable string

Building a Mutable Link From a Sequence

A convenience constructor turns any built-in sequence into one of these functional lists by pushing each element onto the front in reverse order. The reversed function gives back the elements of source in reverse order, last element first, which is why the loop visits them back to front:

# (9)
def to_mutable_link(source):
    """Return a functional list with the same contents as source."""
    s = mutable_link()
    for element in reversed(source):
        s('push_first', element)
    return s

The reversal is the subtle part. Because push_first always adds to the front, pushing the elements in their original order would flip them. Walking source from its last element to its first lays them down so that the final list reads in the original order.

Using the Mutable Link

Now construct one and interact with it by sending messages. The linked list itself is a function, which type confirms:

# (10)
suits = ['heart', 'diamond', 'spade', 'club']
s = to_mutable_link(suits)
type(s)
print(s('str'))

The results, in order:

<class 'function'>
heart, diamond, spade, club

You can also pass messages that change the contents. Removing the first element returns it and shortens the list:

# (11)
s('pop_first')
print(s('str'))

The results:

'heart'
diamond, spade, club

Notice what did not happen: s is the same function object it always was. The pop_first message did not replace s; it changed the hidden contents that s carries. In principle push_first and pop_first alone are enough to make any change at all, since you could empty the list and rebuild it.

Reading a Message-Passing Call

The expression s('getitem', 1) is easy to skim past, so slow it down with a follow-the-line reading:

s('getitem', 1)
├─ s ............... evaluates to the dispatch function that to_mutable_link returned
├─ ( ............... begins the call, which runs the dispatch body
├─ 'getitem' ....... the message: tells dispatch which behavior to run
├─ , ............... separates the message from the extra argument
├─ 1 ............... the extra information for this message, here the index
├─ ) ............... ends the call
└─ evaluates to     the element stored at index 1 of contents

The punctuation is load-bearing: the parenthesis launches a call, the first argument is the message, the comma separates the message from the index, and the closing parenthesis tells Python to run the body now. Inside dispatch, the clause elif message == 'getitem': ends in a colon because it opens a controlled block; the indented return getitem_link(contents, value) belongs to that branch and runs only when the message matches.

Tracing the Call, Frame by Frame

When the message arrives, Python does the ordinary thing it does for any call, and the dispatch logic falls out of it:

Step 1  Look up the name s; it is the dispatch function object.
Step 2  Open a new local frame for dispatch.
Step 3  Bind message to 'getitem' in that frame.
Step 4  Bind value to 1 in that frame.
Step 5  Run the if/elif chain; the 'getitem' branch matches.
Step 6  Return getitem_link(contents, 1), reading the hidden contents.

This is a preview of object-oriented programming. Soon, class-based objects will reach their behaviors through dot notation, but the underlying idea is the one you are using now: a value receives a request and decides how to respond.

Message Passing as a Discipline

The approach you just used has a name. Message passing is the discipline of putting all the operations on a data value inside one function that responds to different messages, and organizing a program by passing those message strings as the first argument. Had you wanted more list operations like extend or insert, you had two honest choices: write them as outside functions that lean on push_first and pop_first, or add more elif clauses inside dispatch that change contents directly. Either way, the operations live with the data they operate on. Here is the message vocabulary of s, summarized:

s('len')              ask for a length
s('getitem', 1)       ask for an item by index
s('push_first', 'x')  request a mutation
s('pop_first')        request a mutation and get the removed value back
s('str')              ask for a string representation

Implementing a Dictionary by Message Passing

The same technique implements something that behaves like a dictionary. The contents are stored as a list of key-value records, each record a two-element list. Before the official version, here is the data shape it maintains, so the filtering logic reads clearly:

records = [[3, 9], [4, 16]]

The official implementation supports two messages, getitem and setitem, reproduced verbatim:

# (12)
def dictionary():
    """Return a functional implementation of a dictionary."""
    records = []
    def getitem(key):
        matches = [r for r in records if r[0] == key]
        if len(matches) == 1:
            key, value = matches[0]
            return value
    def setitem(key, value):
        nonlocal records
        non_matches = [r for r in records if r[0] != key]
        records = non_matches + [[key, value]]
    def dispatch(message, key=None, value=None):
        if message == 'getitem':
            return getitem(key)
        elif message == 'setitem':
            setitem(key, value)
    return dispatch

Storing a value filters out any existing record that already uses that key, then appends one fresh record, which guarantees each key appears at most once. Looking up a value filters for the single record whose key matches and returns its value. Both helpers share the hidden records through the enclosing frame, and setitem declares nonlocal records because it rebinds the list to a brand-new list rather than mutating it in place.

Using the Functional Dictionary

Store two entries and read them back:

# (13)
d = dictionary()
d('setitem', 3, 9)
d('setitem', 4, 16)
d('getitem', 3)
d('getitem', 4)

The two lookups at the end produce:

9
16

The three-part setitem call follows the same message-passing shape as before, now carrying both a key and a value:

d('setitem', 3, 9)
├─ d ............... evaluates to the dispatch function returned by dictionary()
├─ ( ............... begins the call
├─ 'setitem' ....... the message: store a value under a key
├─ 3 ............... the key to store it under
├─ 9 ............... the value to store
├─ ) ............... ends the call
└─ performs         the setitem behavior, replacing records with a list that
                    holds the record [3, 9] and no other record keyed 3

It helps to picture the frame that each call builds. When d('setitem', 3, 9) runs, the dispatch frame binds:

message -> 'setitem'
key     -> 3
value   -> 9

That frame selects the setitem branch, which rebuilds the hidden records. Later, when d('getitem', 3) runs, the call supplies no third argument, so the frame binds:

message -> 'getitem'
key     -> 3
value   -> None

The value parameter falls back to None precisely because the call left it out, which is why def dispatch(message, key=None, value=None): gives the extra parameters defaults: not every message needs the same number of inputs. This dictionary is not built for speed; every lookup filters through all records, while the built-in dict is far more efficient. Ours is for understanding the mechanism, not for performance.

Dispatch Dictionaries

So far, dispatch has compared the message against a fixed set of strings using if and elif. But Python already has a general tool for looking up a value by a key: the dictionary. So instead of a conditional chain, you can dispatch by storing the behaviors in a dictionary whose keys are message names. Strikingly, the same dictionary can also hold the object's state right next to its behaviors.

The original text implements a bank account this way. The constructor account builds a dictionary dispatch that maps 'deposit' and 'withdraw' to functions and maps 'balance' to a number. Note that there are two functions named deposit here: the inner one built inside account, and the outer helper just below. They share a name but are different functions; the same is true of withdraw. Reproduced verbatim:

The account object is just a dictionary, holding two functions and one number side by side:

{
    'deposit':  deposit function,
    'withdraw': withdraw function,
    'balance':  current number
}

Notice that deposit and withdraw mention dispatch even though dispatch is defined a few lines below them. That is fine: the bodies of deposit and withdraw do not run when account is defined, only later when they are called, and by then dispatch exists. Because deposit and withdraw close over the name dispatch, they can both read and rewrite dispatch['balance']. Storing the balance inside the dictionary, rather than as a plain name in the account frame, is what lets these functions skip the nonlocal statement: they never rebind a local name called balance, they mutate a dictionary entry.

Reading a Lookup-Then-Call

The helper withdraw(account, amount) packs a call expression inside a dictionary lookup, and unpacking that is the key to dispatch dictionaries:

return account['withdraw'](amount)
├─ account ............. the account dictionary passed into the helper
├─ ['withdraw'] ........ a dictionary lookup using the key 'withdraw', returning the stored withdraw function
├─ (amount) ............ calls that returned function on amount
└─ evaluates to         whatever the stored withdraw function returns

Python runs this in two phases, and seeing the seam between them is the whole lesson:

Step 1  Evaluate account['withdraw']; the lookup returns the withdraw function in the dictionary.
Step 2  Call that returned function on amount.
Step 3  Inside the stored withdraw function, read and update dispatch['balance'].
Step 4  Return the updated balance.

This is what makes dispatch dictionaries powerful: a dictionary entry can hold a plain number like the balance, but it can equally hold a function. Looking up a function-valued entry hands Python something it can immediately call, so a single lookup picks the behavior and the very next pair of parentheses runs it.

Using the Account

Drive the account through its helper functions:

# (15)
a = account(20)
deposit(a, 5)
withdraw(a, 17)
check_balance(a)

The return values, in order:

25
8
8

deposit(a, 5) raises the stored balance from 20 to 25 and returns it. withdraw(a, 17) checks that 17 does not exceed 25, lowers the balance to 8, and returns it. check_balance(a) simply reads a['balance'], which is now 8. No line here needed nonlocal, because the changing value never lived in a frame as a bare name; it lived in a mutable dictionary the whole time.

Augmented Assignment

The lines inside deposit and withdraw used += and -=, which are Python shorthand (shared by many languages) for a combined lookup and re-assignment. The last two statements below are equivalent:

# (16)
a = 2
a = a + 1
a += 1

After all three run, a is:

4

Walk a += 1 one token at a time:

a += 1
├─ a ............... the name whose current value is read, then re-bound
├─ += .............. add-and-reassign: combine the left value with the right, store back into a
├─ 1 ............... the value added to a's current value
└─ re-binds         a to its old value plus 1, here 3 plus 1, giving 4

For a mutable container, the lookup half and the write half can both touch the same object, as in dispatch['balance'] += amount: Python reads dispatch['balance'], adds amount, and writes the sum back into that same dictionary entry, all in one statement. That compactness is exactly why the dispatch-dictionary account reads so cleanly.

⚠Common Beginner Traps

The first trap is thinking wd2 = wd opens a new account. It does not. It writes a second name on the same withdraw function, and both names share one balance.

The second trap is treating a dispatch function as mysterious machinery. It is just a function that checks its message argument and picks a branch; the counter in code (7) is the entire idea in miniature.

The third trap is forgetting that a mutation message such as s('push_first', value) changes hidden state and returns None. An operation that exists to change something often has no useful value to hand back.

The fourth trap is assuming the functional dictionary is fast. Every lookup filters the whole list of records, so it slows down as the dictionary grows; the built-in dict does not work this way.

The fifth trap is expecting nonlocal everywhere state changes. The dispatch-dictionary account changes its balance with no nonlocal at all, because it mutates a dictionary entry instead of rebinding a local name. nonlocal is only required when you reassign a name that lives in an enclosing frame.

Show the answers
  1. Each call to make_withdraw builds a fresh parent frame with its own balance, and the returned function remembers that frame. Two calls means two frames means two balances.
  2. wd2 = wd copies the function reference, not the saved balance. Both names now point at the same withdraw function and therefore the same hidden state.
  3. An expression is referentially transparent if you can replace any subexpression with the value it computes without changing the program. make_withdraw breaks it because calling the returned function changes the hidden balance, so the same call expression can return a different value next time.
  4. It keeps the current contents of the linked list in an enclosed name that its dispatch function reads or replaces. to_mutable_link reverses the source because push_first adds to the front, so laying elements down back to front leaves the list in the original order.

The lookup for key 7 finds no matching record, so getitem's len(matches) == 1 guard is false and the function falls off the end, returning None.

§Summary

Nonlocal assignment lets a function keep private state, and that single ability carries you a long way. Its benefit is independence: separate calls to a constructor build separate frames, so two functions made the same way evolve separately. Its cost is that identity now matters; copying a stateful function into a second name shares one hidden value through both names, and the program is no longer referentially transparent. A dispatch function turns this state into an object by reading a message and choosing a behavior, which is the discipline of message passing. Using only functions, local state, and dispatch, you implemented a mutable linked list, a dictionary, and a bank account, the last as a dispatch dictionary that stores its behaviors and its balance together. Every one of these is something you will soon rebuild with classes, but you now understand the machinery that classes are built on top of.

Before you step through it, write down two predictions: how many local frames will open over the whole run, and how many frames named deposit will be open at the same moment partway through the line deposit(a, 5).

Check both as the frames appear. Six local frames open in all, and partway through deposit(a, 5) two frames named deposit are open at once: the helper, and inside it the stored function that the lookup account['deposit'] returned.

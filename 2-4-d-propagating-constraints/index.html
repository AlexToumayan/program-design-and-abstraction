Chapter 2/2.4 Mutable Data

2.4 Propagating Constraints

◷ approx. 20 min read · 16 worked examples · best read in two passes

Every program you have written so far computes in one direction: arguments go in, a value comes out. A function that turns Celsius into Fahrenheit cannot run backward to turn Fahrenheit into Celsius. In this lesson you will build something different — a small system that stores a relationship between quantities and can run it in any direction. Give it the Celsius temperature and it produces the Fahrenheit; give it the Fahrenheit instead and the same network produces the Celsius.

By the end you will be able to do five things: explain why a relation is more flexible than a one-way function; read the network as a wiring diagram of connectors and constraints; trace how a single value set by a user propagates through the wires until the whole network is filled in; read the unusual connector['set_val'](source, value) message-send syntax with confidence; and follow the implementation, which is built entirely out of tools you already have — mutable dictionaries, local functions, and nonlocal state.

This is the last stop before objects. The connectors and constraints you build here are values that carry their own data and their own behavior together, which is exactly what an object is. You are about to invent the idea by hand.

A Shared Measurement Board

Picture a whiteboard in a science lab, divided into labeled boxes. One box is for the Celsius temperature, one is for Fahrenheit, and a few more hold intermediate numbers. Standing at the board is a small team. Each person watches only a couple of boxes and knows exactly one rule connecting them — "this box is always nine times that one," or "this box is always that one plus thirty-two."

Nobody on the team knows the whole conversion formula. Yet the moment someone writes a number in the Celsius box and announces it, the people who watch that box can fill in the boxes next to it, and announce those, and the announcement ripples down the line until Fahrenheit appears. Erase Celsius later and the ripple runs the other way: every value that depended on Celsius gets erased too, because the support for it is gone.

A constraint system gives a program that same disciplined style of communication. The labeled boxes become connectors. The people with their one local rule become constraints. The announcements become messages passed between them. No piece understands the whole problem, but the right answer still emerges because each piece does its small job and notifies its neighbors.

Relations Instead of One-Way Functions

Most functions point in a single direction. This one computes Fahrenheit from Celsius and nothing else:

# (1)
def fahrenheit_from_celsius(c):
    return 9 * c / 5 + 32

Hand it 25 and you get 77.0. But there is no way to hand it 77.0 and recover 25. To go the other way you would have to write a second, separate function. The knowledge that Celsius and Fahrenheit are related lives twice, once in each direction, and the two copies can drift apart.

A constraint system stores the relationship a single time, as an equation rather than a recipe:

9⋅c=5⋅(f−32)

An equation does not prefer a direction. It is equally a statement about c in terms of f and about f in terms of c. If you know c, the equation determines f. If you know f, the same equation determines c. The system we build will honor exactly that even-handedness.

This kind of two-way relationship is everywhere in science. The ideal gas law,

p⋅V=n⋅k⋅T

relates pressure, volume, amount of gas, Boltzmann's constant, and temperature. Knowing any four of those five quantities pins down the fifth, and which four you happen to know is not fixed in advance. A one-way function would force you to pick one unknown and commit to it. A constraint network refuses to commit, which is the whole point.

A Network of Connectors and Constraints

A single equation with multiplication and subtraction is more than one primitive operation can handle, so we break the temperature relation into a chain of simpler steps, each of which is a single multiply or add. Reading 9⋅c=5⋅(f−32) as a sequence of small facts gives us three of them:

c times 9 equals u            (a product we will call u)
v times 5 equals u            (so v is u divided by 5)
v plus 32 equals f            (so f is v plus 32)

The shared name u is what ties the two multiplications together: the product c * 9 and the product v * 5 must be the same number. Drawing each operation as a box and each shared quantity as a wire gives the network its shape:

celsius --[ * 9 ]--> u <--[ * 5 ]-- v --[ + 32 ]--> fahrenheit

Two vocabulary words name the two kinds of thing in that diagram.

A connector is a wire. It is an object that can hold one value and that remembers which constraints are attached to it, so it can notify them whenever its value appears or disappears. In the diagram, celsius, u, v, and fahrenheit are connectors.

A constraint is a box. It is an object that enforces one relationship among the connectors attached to it. In the diagram, each [ * 9 ], [ * 5 ], and [ + 32 ] is a constraint. The defining feature of a constraint is that it works in every direction at once. An adder for a+b=c will compute c when it learns a and b, but it will equally compute b when it learns a and c, by subtracting. Multidirectional behavior like that is what separates a constraint from an ordinary call expression (a function call).

Building and Using the Network

Before looking inside the machinery, watch it work from the outside. A user first creates two named connectors, one for each end of the conversion:

# (2)
celsius = connector('Celsius')
fahrenheit = connector('Fahrenheit')

The string argument is just a label for printing; a connector with a name announces itself whenever its value changes, which is how we will watch propagation happen. The connector object that comes back is a dispatch dictionary — a dictionary whose entries are mostly functions — and we will build it later in the lesson.

The intermediate connectors and all the constraints are assembled by a single function. It is worth reading slowly because it is the wiring diagram written as code. The diagram showed two working wires, u and v, but the code also needs three more connectors to hold the fixed numbers 9, 5, and 32 — those are w, x, and y. So converter creates five helper connectors in all, which is why the first line unpacks five names at once:

# (3)
def converter(c, f):
    """Connect c to f with constraints to convert from Celsius to Fahrenheit."""
    u, v, w, x, y = [connector() for _ in range(5)]
    multiplier(c, w, u)
    multiplier(v, x, u)
    adder(v, y, f)
    constant(w, 9)
    constant(x, 5)
    constant(y, 32)

The first line creates five unnamed helper connectors in one stroke. Three of them, w, x, and y, will be pinned to the fixed numbers 9, 5, and 32 by the constant calls at the bottom. The other two, u and v, are the working wires from the diagram. Here is what each connector stands for:

c   the Celsius end (passed in)
f   the Fahrenheit end (passed in)
w   the constant 9
x   the constant 5
y   the constant 32
u   the shared product, equal to both c * 9 and v * 5
v   the intermediate value f - 32

The three middle lines wire the constraints: multiplier(c, w, u) enforces c * w == u, the second multiplier(v, x, u) enforces v * x == u, and adder(v, y, f) enforces v + y == f. The three constant lines drop the fixed numbers onto w, x, and y. With that, the network exists. We attach it to our two named connectors:

# (4)
converter(celsius, fahrenheit)

Reading a Connector Message

Now the user puts a value into the network by sending the Celsius connector a message:

# (5)
celsius['set_val']('user', 25)

That line has an unfamiliar shape because it does two separate things back to back. The square brackets look up an entry in a dictionary, and the parentheses immediately call whatever that lookup returned. Follow it one piece at a time:

celsius['set_val']('user', 25)
├─ celsius .......... the connector dictionary for Celsius
├─ ['set_val'] ...... selects the connector's set_value function from the dictionary
├─ ('user', 25) ..... calls that function: source is 'user', requested value is 25
└─ performs the action: stores 25 in the connector and notifies its constraints

The brackets do selection — they pick one entry out of the dictionary. The parentheses do calling — they run the function that selection produced. It is the exact two-step pattern you met as account['withdraw'](amount) in the dispatch-dictionary lesson: reach into the object, find the operation by name, then invoke it. The first argument, 'user', records who is asking; that source tag matters later, when values need to be forgotten by whoever set them.

After that single line runs, the network prints two lines on its own:

Celsius = 25
Fahrenheit = 77.0

The user supplied one number. The network discovered the other.

How the Value Propagates

The Fahrenheit reading did not appear by magic, and it did not appear because some function knew the whole formula. It appeared because each connector, on learning a value, woke up its attached constraints, and each constraint, on finding two of its three connectors known, computed the third. Following the ripple step by step:

1. celsius receives 25, stores it, and notifies the multiplier wired to (celsius, w, u).
2. that multiplier sees celsius = 25 and w = 9, both known, so it sets u = 25 * 9 = 225.
3. u, now known, notifies the second multiplier wired to (v, x, u).
4. that multiplier sees u = 225 and x = 5, both known, so it sets v = 225 / 5 = 45.
5. v, now known, notifies the adder wired to (v, y, f).
6. that adder sees v = 45 and y = 32, both known, so it sets fahrenheit = 45 + 32 = 77.

No connector ever held the equation 9⋅c=5⋅(f−32). Each one knew only its own value and the list of constraints to poke. The conversion is an emergent result of disciplined local notifications, like the lab team filling boxes one announcement at a time.

Contradiction

The network believes the values it holds. If you now try to set Fahrenheit to a value that disagrees with the 77.0 it already worked out, it refuses:

# (6)
fahrenheit['set_val']('user', 212)

Contradiction detected: 77.0 vs 212

The Fahrenheit connector already has the value 77.0, supplied by the adder. A request to make it 212 does not overwrite the old value; the connector compares the two, finds they disagree, and reports the conflict instead of silently corrupting the network.

Forgetting, and Running the Other Way

To reuse the network for a fresh conversion, the original source of a value asks the connector to forget it:

# (7)
celsius['forget']('user')

Celsius is forgotten
Fahrenheit is forgotten

Forgetting propagates just as setting did. Once Celsius drops its 25, the products and sums that were derived from it lose their support, so they forget in turn, all the way out to Fahrenheit. The network empties itself back to a blank board.

Now the same wires can run backward. Set Fahrenheit and watch Celsius appear:

# (8)
fahrenheit['set_val']('user', 212)

Fahrenheit = 212
Celsius = 100.0

This is the payoff promised at the start. The very same converter network that turned 25 °C into 77.0 °F now turns 212 °F into 100.0 °C, with no second function and no rewiring. The adder ran its subtraction to recover v, each multiplier ran its division to recover the wire feeding it, and Celsius emerged at the far end.

The Two Message Protocols

Everything above worked through messages. It helps to name the complete set before reading the implementation, because the implementation is just these messages made real.

A connector understands five messages:

connector['set_val'](source, value)   request that the connector take a value
connector['has_val']()                ask whether the connector currently has a value
connector['val']                      the current value itself (data, not a function)
connector['forget'](source)           request that the connector forget its value
connector['connect'](source)          attach a new constraint to this connector

A constraint understands two messages:

constraint['new_val']()   tells the constraint that a connected connector has a new value
constraint['forget']()    tells the constraint that a connected connector forgot its value

That is the whole vocabulary. Connectors and constraints talk only by sending each other entries out of these dictionaries. This is message passing, built by hand from plain dictionaries.

The Primitive Constraints

We need three kinds of constraint box: an adder, a multiplier, and a constant. The adder and multiplier are nearly identical, so a single helper, make_ternary_constraint, does the shared work, and adder and multiplier are thin wrappers that pass it the right arithmetic.

Adder

An adder enforces a+b=c among three connectors:

# (9)
from operator import add, sub

def adder(a, b, c):
    """The constraint that a + b = c."""
    return make_ternary_constraint(a, b, c, add, sub, sub)

The three functions handed to the helper are the three directions the relation can run. Reading them in order:

make_ternary_constraint(a, b, c, add, sub, sub)
├─ a, b, c ...... the three connectors the constraint ties together
├─ add .......... given a and b, compute c as a + b
├─ sub .......... given c and a, compute b as c - a
├─ sub .......... given c and b, compute a as c - b
└─ returns a constraint object that enforces a + b = c in all three directions

The forward direction uses addition; both backward directions use subtraction, because subtraction is how you undo a sum. If a is 10 and c is 17, the constraint recovers b as 17 - 10, which is 7.

Ternary Constraints

Here is the shared machine. It connects three connectors and knows how to compute any one of them from the other two:

# (10)
def make_ternary_constraint(a, b, c, ab, ca, cb):
    """The constraint that ab(a,b)=c and ca(c,a)=b and cb(c,b)=a."""
    def new_value():
        av, bv, cv = [connector['has_val']() for connector in (a, b, c)]
        if av and bv:
            c['set_val'](constraint, ab(a['val'], b['val']))
        elif av and cv:
            b['set_val'](constraint, ca(c['val'], a['val']))
        elif bv and cv:
            a['set_val'](constraint, cb(c['val'], b['val']))
    def forget_value():
        for connector in (a, b, c):
            connector['forget'](constraint)
    constraint = {'new_val': new_value, 'forget': forget_value}
    for connector in (a, b, c):
        connector['connect'](constraint)
    return constraint

The two inner functions are exactly the 'new_val' and 'forget' messages from the constraint protocol. When a connector announces a new value, it calls new_value, which first asks all three connectors whether they have values:

av, bv, cv = [connector['has_val']() for connector in (a, b, c)]
├─ for each connector in (a, b, c) ...... loop over the three wires
├─ connector['has_val']() ............... ask that wire whether it holds a value
├─ [ ... ] .............................. collect the three answers into a list
└─ av, bv, cv = ......................... unpack them: True/False for a, b, c

With those three booleans in hand, new_value finds the one connector that is still unknown and computes it. If a and b are known, it computes c using the forward function ab. If a and c are known, it recovers b using ca. If b and c are known, it recovers a using cb. The single line that fills in a connector is worth its own breakdown:

c['set_val'](constraint, ab(a['val'], b['val']))
├─ c ................ the connector to fill in
├─ ['set_val'] ...... select its set_value function
├─ constraint ....... the source argument: this constraint is who is supplying the value
├─ ab(a['val'], b['val']) ... compute the value from a's and b's current values
└─ performs the action: gives c the computed value, tagging this constraint as the source

Passing constraint as the source is what makes forgetting work cleanly later: the connector remembers that this constraint is the reason it holds its value. The bottom three lines of make_ternary_constraint run once at construction time. They build the constraint dictionary and then attach it to all three connectors with the 'connect' message, so each wire knows to notify this box when its value changes.

Multiplier and Constant

A multiplier is the same machine with multiplication forward and division backward:

# (11)
from operator import mul, truediv

def multiplier(a, b, c):
    """The constraint that a * b = c."""
    return make_ternary_constraint(a, b, c, mul, truediv, truediv)

A constant is the simplest constraint of all. It sets one connector once, at construction, and then has nothing further to do:

# (12)
def constant(connector, value):
    """The constraint that connector = value."""
    constraint = {}
    connector['set_val'](constraint, value)
    return constraint

Its dictionary is empty because it never responds to 'new_val' or 'forget'; it has no behavior after it drops its value onto the wire. Notice that it, too, passes itself as the source, so the value it sets is anchored to it and will not be forgotten by anyone else.

The Connector

The connector is where the state lives. A connector must remember three things: its current value, the source that supplied that value (called the informant), and the list of constraints attached to it. Here is the implementation:

# (13)
def connector(name=None):
    """A connector between constraints."""
    informant = None
    constraints = []
    def set_value(source, value):
        nonlocal informant
        val = connector['val']
        if val is None:
            informant, connector['val'] = source, value
            if name is not None:
                print(name, '=', value)
            inform_all_except(source, 'new_val', constraints)
        else:
            if val != value:
                print('Contradiction detected:', val, 'vs', value)
    def forget_value(source):
        nonlocal informant
        if informant == source:
            informant, connector['val'] = None, None
            if name is not None:
                print(name, 'is forgotten')
            inform_all_except(source, 'forget', constraints)
    connector = {
        'val': None,
        'set_val': set_value,
        'forget': forget_value,
        'has_val': lambda: connector['val'] is not None,
        'connect': lambda source: constraints.append(source)
    }
    return connector

This is a dispatch dictionary, like the ones from the dispatch-dictionary lesson, but mixed: most entries are functions, while the 'val' entry is the current value itself, plain data sitting alongside the behavior.

The set_value function is the heart of the logic. When asked to take a value, it first reads its current value. If the connector is empty (val is None), it records the new value and its source, prints an announcement if it was given a name, and then calls inform_all_except to notify every attached constraint other than the one that just supplied the value. If the connector already holds a value, it does nothing unless the new value disagrees, in which case it reports a contradiction. That is exactly the two behaviors you saw from the outside: a fresh value propagates; a conflicting value is rejected.

The forget_value function is guarded by a single careful check: if informant == source. A connector forgets its value only when the very source that supplied it asks. A constraint cannot make a connector forget a value that the user set, and vice versa. When the check passes, the connector clears its value and informant and propagates the forget outward, which is what unwound the whole network in example (7).

The two lambdas finish the dictionary. 'has_val' reports whether a value is present by testing connector['val'] is not None. 'connect' attaches a constraint by appending it to the constraints list.

Informing the Neighbors

One small helper carries every notification. When a connector changes, it tells every attached constraint about it — except the one that caused the change, to avoid bouncing the message straight back to its sender:

# (14)
def inform_all_except(source, message, constraints):
    """Inform all constraints of the message, except source."""
    for c in constraints:
        if c != source:
            c[message]()

The message argument is a dictionary key, either 'new_val' or 'forget', and c[message]() looks that entry up on each constraint and calls it. Skipping source is what prevents a constraint from being told about a value it itself just supplied, which would echo forever.

A Bridge You Can Check by Hand

The full temperature network has six connectors and three constraints, which is a lot to hold in your head on a first read. Before trusting it, build the smallest possible constraint network and verify it by hand: a single adder enforcing a+b=c, with a pinned to 10 and c pinned to 17, leaving the adder to discover b.

Trace it by hand. constant(a, 10) sets a to 10 and notifies the adder; the adder checks and finds only a known, so it can do nothing yet. constant(c, 17) sets c to 17 and notifies the adder again; now a and c are both known, so the adder recovers b using its cb direction, which is sub, computing 17 - 10. The program prints a = 10, then c = 17, then b = 7, and the final b['has_val']() returns True. You can confirm the arithmetic without running anything: b=c−a=17−10=7. This is the same engine as the temperature converter, scaled down until it fits in one glance.

An Object Trace of the Converter

The bridge shows a constraint recovering one value. The full converter is the same idea three constraints deep, and the order in which the connectors get filled is the part worth seeing explicitly. Picture each connector as a small box that starts empty and gets stamped with a value during propagation. Setting Celsius to 25 stamps the boxes in this order:

event                                    celsius  w   x   y    u    v   fahrenheit
--------------------------------------   -------  --  --  --   ---  --  ----------
start (after constants w, x, y set)        -      9   5   32    -    -      -
celsius['set_val']('user', 25)            25      9   5   32    -    -      -
  multiplier(c, w, u): c=25, w=9 -> u      25      9   5   32   225   -      -
  multiplier(v, x, u): u=225, x=5 -> v     25      9   5   32   225  45      -
  adder(v, y, f): v=45, y=32 -> f          25      9   5   32   225  45     77.0

Each row is one constraint firing. The constraint reads the two boxes it already has and stamps the third, which immediately wakes the next constraint down the line. When the dust settles, every box holds a value, and fahrenheit reads 77.0. The forget in example (7) runs this table in reverse: clearing celsius removes the support for u, which removes the support for v, which removes the support for fahrenheit, blanking the boxes back out one constraint at a time.

The deeper point is that connectors and constraints behave like objects. Each connector has identity (it is a specific dictionary in memory), state (the value currently stamped in its box), and behavior (the functions stored under its keys). The whole network is a little society of such objects passing messages. That is precisely the picture the next section formalizes with classes.

⚠Common Beginner Traps

Expecting a fixed direction. A constraint network is not a call expression with designated inputs and outputs. Which connectors are inputs depends on which ones you happen to set. Setting Celsius makes Fahrenheit the output; setting Fahrenheit makes Celsius the output. Do not look for a single forward direction — there isn't one.

Forgetting who the source is. A connector forgets its value only when the same source that supplied it asks. That is why set_val and forget both take a source argument, and why the constants and constraints pass themselves as the source. Lose track of the source and you lose the ability to unwind the network correctly.

Trying to mutate connectors directly. A connector's value is reached and changed only through messages: 'set_val', 'forget', 'has_val', and 'connect'. Reaching in and reassigning connector['val'] yourself skips the notifications, and the rest of the network never finds out the value changed.

Confusing selection with calling. The line celsius['set_val']('user', 25) is two operations: ['set_val'] selects a function out of the dictionary, and ('user', 25) calls it. Reading it as a single mysterious blob is what makes it look hard. It is the same select-then-call shape as account['withdraw'](amount).

Underestimating what this is. It is tempting to file this away as a temperature gadget. It is bigger than that: connectors and constraints are values with identity, state, and behavior, communicating by messages. You have just built objects out of dictionaries and functions, one lesson before objects get a name.

✓Check Your Understanding

  1. Why is the relation 9⋅c=5⋅(f−32) more flexible than a one-way function like fahrenheit_from_celsius?
  2. Suppose you build a fresh converter network and run celsius['set_val']('user', 0). What two lines does the network print, and in what order?
  3. Why does adder(a, b, c) hand make_ternary_constraint two subtraction functions in addition to one addition function?
  4. What happens when a connector that already holds a value is asked to take a different one, and why does the system behave that way?
  5. Why does inform_all_except deliberately skip the constraint named by source?
Show the answers
  1. An equation does not prefer a direction. If Celsius is known the relation determines Fahrenheit; if Fahrenheit is known the same relation determines Celsius. A one-way function bakes in a single direction, so reversing it requires writing and maintaining a second function.
  2. Celsius = 0 first, then Fahrenheit = 32.0. Setting Celsius to 0 propagates through the network: u becomes 0 * 9 = 0, v becomes 0 / 5 = 0, and fahrenheit becomes 0 + 32 = 32.0, which prints because it is the second named connector to receive a value.
  3. Because the constraint must be able to compute any one of a, b, c from the other two. The forward direction a + b gives c, but recovering b from c and a, or a from c and b, requires the inverse operation, subtraction.
  4. The connector compares the requested value with the one it already holds; if they differ it prints a contradiction and refuses to change. The system does this so that inconsistent information is reported rather than silently overwriting a value that other parts of the network already depend on.
  5. To avoid echoing a notification straight back to the constraint that caused it. Without that guard, the constraint that just set a connector would be told about the new value and could re-fire endlessly.

§Summary

A constraint system organizes computation as relationships among changing quantities rather than as one-way functions. The quantities are connectors — wires that hold a value, remember the source that set it, and keep a list of attached constraints. The relationships are constraints — boxes such as the adder, multiplier, and constant that enforce one equation and can compute any unknown connector from the known ones. Setting a single connector triggers a wave of message passing: each connector notifies its constraints, each constraint fills in a connector it can now determine, and the wave continues until the network is consistent. Forgetting a value runs the same wave in reverse. Because each connector carries its own state and its own behavior and communicates only by messages, the whole network is built out of objects in everything but name, which is exactly the idea the next section makes explicit. Underneath it all sit the tools you already know: mutable dictionaries, local functions, nonlocal state, and the select-then-call dispatch pattern.

When celsius['set_val']('user', 25) fires, the set_value frames nest one inside another as the ripple runs, each rebinding its own informant via nonlocal and stamping a value into its dictionary's 'val' entry while the unfilled connectors wait below. You do not need to predict all of this perfectly — the stepper is what shows it to you. Before you step through it, write down three predictions: how many connector frames open over the run, how many set_value frames are open at once when Fahrenheit = 77.0 prints, and whether informant in the frames that built celsius and u ends up holding 'user' or a constraint dictionary.

Step through it one line at a time and check each prediction as the frames appear. Seven connector frames open, five inside converter, and all seven persist because the dispatch dictionaries' functions still point into them. When Fahrenheit = 77.0 prints, four set_value frames are open at once: the ripple is one stack of nested message sends, not calls that finish in turn.

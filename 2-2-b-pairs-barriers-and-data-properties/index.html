Chapter 2/2.2 Data Abstraction

2.2 Pairs, Abstraction Barriers, and the Properties of Data

◷ approx. 20 min read · 10 worked examples · best read in two passes

In the previous lesson you designed a complete set of rational-number operations before you decided how a rational number would actually be stored. You wrote add_rationals, mul_rationals, print_rational, and rationals_are_equal, and every one of them talked to a rational number only through the three names rational, numer, and denom. That left a hole on purpose: those three names were never implemented.

This lesson fills the hole, and then it does something more interesting. By the end you will be able to do four things: build a rational number out of a pair (a container that holds two values), explain why the line between "code that uses a value" and "code that builds a value" is called an abstraction barrier, improve the representation without touching any of the operations that use it, and finally see the surprising claim at the center of the whole chapter: a piece of data is defined by how it behaves, not by what it is physically made of. To drive that last point home, you will build a working pair out of nothing but a function.

A Pair Is a Two-Slot Container

Start with the everyday version of the idea. Think of an egg carton built for exactly two eggs. The carton itself is one object you can pick up and carry, but it holds two separate things, and each thing has a fixed spot: a left cup and a right cup. If a friend asks for "the egg on the left," there is no ambiguity. The carton bundles two values into one value while still letting you reach in and pull out either piece on demand.

That is precisely what we need for a rational number. A rational number is one value, but it is built from two integers: a numerator and a denominator. We want a single container that holds both, keeps them in a fixed order, and lets us pull out the first or the second whenever we ask.

The simplest such container in Python is a list with two elements. A list is written by surrounding comma-separated values with square brackets. Here is the smallest one that matters to us, typed directly at the interpreter:

>>> [10, 20]
[10, 20]

You typed a list of two elements, and Python echoed the same list back as its value. Nothing is stored yet, because we never named it. Give it a name so we can come back to it:

>>> pair = [10, 20]
>>> pair
[10, 20]

Picture the value bound to pair as a two-slot tray:

index:   0    1
       +----+----+
pair = | 10 | 20 |
       +----+----+

The two facts to hold onto are that the list keeps its elements in order, and that the positions are numbered starting from 0, not from 1. This zero-based counting trips up almost everyone at first, so we will keep returning to it.

Follow the Line: A List Literal

Here is the list literal broken into its parts. Read the left column as "the piece of code," and the right column as "what that piece does."

[10, 20]
├─ [ ........... open the list literal; everything until the matching ] is an element
├─ 10 .......... first element; it lands at index 0
├─ , ........... separator that marks the boundary between elements
├─ 20 .......... second element; it lands at index 1
├─ ] ........... close the list literal
└─ evaluates to  a two-element list [10, 20]

The comma is not decoration. It is the token that tells Python there is more than one element, and it is what fixes which value is "first" and which is "second."

Pulling a Value Back Out

A container is only useful if you can reach into it. To select an element from a list, write the list, then square brackets, then the index you want:

>>> pair[0]
10
>>> pair[1]
20

pair[0] is the value in the first slot, 10, and pair[1] is the value in the second slot, 20. The square brackets here mean something different from the square brackets that built the list. When brackets sit immediately after a value, they mean "select the element at this index."

Follow the Line: Element Selection

pair[0]
├─ pair ........ the sequence to select from
├─ [ ........... begin an index selection
├─ 0 ........... the index to select; 0 is the first position
├─ ] ........... end the index selection
└─ evaluates to  10, the value stored at index 0 of pair

The single most common beginner mistake is reading pair[0] as "the zeroth thing, which surely means nothing useful" or as "the first thing I'd point at, which is the 10... or is it the 20?" Anchor it: index 0 is the first slot, index 1 is the second slot.

Selection Is Just a Call Expression

Square-bracket selection looks like special punctuation, but it is not magic. The same operation has an ordinary call expression form (a function call) named getitem, which lives in the operator module. Importing it and calling it does exactly what the brackets do:

>>> from operator import getitem
>>> getitem(pair, 0)
10
>>> getitem(pair, 1)
20

getitem(pair, 0) returns the same value as pair[0]. This is worth a moment of attention because it reveals the shape of the idea: selecting an element is an operation that takes a compound value and an index and returns one part. The brackets are convenient surface syntax for that operation. Seeing the function form makes it concrete that selection is a normal computation, not a built-in spelling you simply have to memorize.

Unpacking Both Values at Once

Often you want both elements in named variables. Python lets you bind two names from a two-element sequence in a single statement, which is called multiple assignment (or sequence unpacking):

>>> x, y = pair
>>> x
10
>>> y
20

After this statement, x is 10 and y is 20. The list itself is unchanged; x and y are two new names that point at the values that were inside it.

Follow the Line: Multiple Assignment

x, y = pair
├─ x ........... first target name on the left
├─ , ........... separates the target names
├─ y ........... second target name on the left
├─ = ........... assignment; bind the targets to the elements on the right
├─ pair ........ a sequence on the right, expected to have exactly two elements
└─ performs      binds x to pair's element 0 and y to its element 1

Because this is a statement and not an expression, it prints nothing on its own; it performs the action of creating two bindings. Here is the machine model, step by step:

1. Evaluate the right side: look up pair, the list [10, 20].
2. Check that the sequence has exactly two elements, matching the two targets.
3. Bind x to the element at index 0, which is 10.
4. Bind y to the element at index 1, which is 20.

After all four steps, the current frame holds three bindings:

+------+----------+
| pair | [10, 20] |
| x    | 10       |
| y    | 20       |
+------+----------+

If the number of names on the left does not match the number of elements on the right, Python raises an error rather than guessing. Two names require exactly two elements.

Representing Rational Numbers with Pairs

Now we have everything needed to implement the three names the previous lesson left undefined. A rational number will be a two-element list: numerator at index 0, denominator at index 1. The constructor builds that list, and each selector pulls out one element.

# (1)
def rational(n, d):
    return [n, d]

def numer(x):
    return x[0]

def denom(x):
    return x[1]

rational(n, d) constructs a rational number by packaging n and d into a list. numer(x) selects the element at index 0, the numerator. denom(x) selects the element at index 1, the denominator. These three functions are the only code in the entire program that is allowed to know that a rational number is secretly a list. Everything else goes through them.

Follow the Line: Building the Pair

The body of the constructor is the line that does the packaging:

return [n, d]
├─ return ...... hand a value back to whoever called rational
├─ [ ........... begin a list literal
├─ n ........... first element expression; the numerator lands at index 0
├─ , ........... separator between the two elements
├─ d ........... second element expression; the denominator lands at index 1
├─ ] ........... end the list literal
└─ evaluates to  a two-element list whose value is returned, e.g. [1, 2]

Trying the Representation

With the constructor and selectors defined, every operation from the previous lesson springs to life, because each of them was written against rational, numer, and denom. Here is the official text's check at the interpreter:

>>> half = rational(1, 2)
>>> print_rational(half)
1 / 2
>>> third = rational(1, 3)
>>> print_rational(mul_rationals(half, third))
1 / 6
>>> print_rational(add_rationals(third, third))
6 / 9

The first two results are exactly what you expect. The third deserves a long look. Mathematically the answer is correct, but it is not in lowest terms:

31​+31​=32​

yet the program displays 6 / 9. The addition rule built a numerator of 1⋅3+1⋅3=6 and a denominator of 3⋅3=9, and nothing in the program ever reduced the fraction. We would rather see 2 / 3. Fixing this is the moment that shows why the boundary we are about to name is so valuable.

The Abstraction Barrier

An abstraction barrier is a boundary in your program with a rule attached: code on one side may touch a value only through an agreed-upon set of functions, never by reaching into its representation directly. For rational numbers, the agreed-upon interface is exactly three functions:

rational(n, d)    construct a rational number
numer(x)          select its numerator
denom(x)          select its denominator

The user-level operations are supposed to use a rational number only through this interface. They should never assume a rational number is a list, a tuple, a function, or anything else in particular. Drawn as layers, the program looks like this:

Operations that USE rational numbers
add_rationals, mul_rationals, rationals_are_equal, print_rational
        |
        |  may use only rational, numer, denom
        v
The interface (the abstraction barrier)
rational, numer, denom
        |
        |  may know the storage format
        v
The representation
the two-element list [n, d]

The official text lays out these same layers as a table. Each row is one kind of code, the middle column says what that code is allowed to treat a rational number as, and the right column lists the only tools that code is permitted to use:

Parts of the program that... Treat rationals as... Using only...
Use rational numbers to perform computation whole data values add_rational, mul_rational, rationals_are_equal, print_rational
Create rationals or implement rational operations numerators and denominators rational, numer, denom
Implement selectors and constructor for rationals two-element lists list literals and element selection

Read the middle column from top to bottom: the same rational number is seen as a single whole data value by the code that computes with it, as a numerator and denominator by the code that implements the operations, and only as a two-element list at the very bottom, where the constructor and selectors are written with list literals and element selection. Each layer knows less about the representation than the one below it, and that shrinking knowledge is exactly what makes the layers independent.

The discipline reads as three short rules:

  • add_rationals may call numer(x) to get a numerator.
  • add_rationals must not write x[0], because that reaches past the barrier into the representation.
  • numer may write x[0], because numer lives in the representation layer; touching the list is its whole job.

The official text makes the cost of crossing the barrier concrete with three versions of a function that squares a rational number. All three compute the right answer today, but they differ in how many layers they reach through:

# (2)
def square_rational(x):
    return mul_rational(x, x)

def square_rational_violating_once(x):
    return rational(numer(x) * numer(x), denom(x) * denom(x))

def square_rational_violating_twice(x):
    return [x[0] * x[0], x[1] * x[1]]

The first version, square_rational, treats a rational number as a whole and reuses the existing multiplication operation; it respects every barrier. (Note that the official text writes the multiply call as mul_rational(x, x) here, singular, while the operation defined in the previous section is named mul_rationals. That is a small naming slip in the source; the runnable form would call mul_rationals(x, x).) The second version, square_rational_violating_once, reaches through one layer: it knows a rational number has a numerator and a denominator. The third version, square_rational_violating_twice, reaches through two layers: it assumes the exact list shape with x[0] and x[1]. The more layers a function reaches through, the more code you must rewrite if the representation ever changes. The third version is the most fragile of the three.

Improving the Constructor, and Nothing Else

Here is the payoff of the barrier. Because every rational number in the program is born through rational, we can make all of them reduce to lowest terms by changing that one function. We use the greatest common divisor: divide both the numerator and the denominator by gcd(n, d).

This is what the original text wrote:

>>> from fractions import gcd
>>> def rational(n, d):
        g = gcd(n, d)
        return (n//g, d//g)

Two notes on that original snippet. First, it returns a tuple (n//g, d//g) rather than a list; a tuple is another two-element sequence, and selection with x[0] and x[1] works on it identically, so the selectors do not need to change. Second, gcd no longer lives in the fractions module in modern Python; it was moved, and from fractions import gcd will fail on current versions. The runnable, up-to-date form imports gcd from the math module instead:

# (3)
from math import gcd

def rational(n, d):
    g = gcd(n, d)
    return [n // g, d // g]

The operator // is floor division, which divides and discards any fractional part to return an integer. It is the right choice here because gcd(n, d) divides both n and d evenly, so each quotient is already a whole number; floor division simply keeps the result an int rather than a float. With the new constructor in place, the troublesome sum reduces on its own:

>>> print_rational(add_rationals(third, third))
2 / 3

Read that result against what changed. We did not touch add_rationals. We did not touch numer, denom, mul_rationals, or print_rational. We changed rational alone, and every rational number in the entire program is now reduced. That is exactly what the abstraction barrier buys you: behavior on one side of the boundary can be improved without disturbing anything on the other side.

A Careful Trace of Reduction

To see the new constructor work, trace a single call by hand:

# (4)
rational(6, 9)

The machine steps inside the call are:

1. Open a new frame for rational; bind n to 6 and d to 9.
2. Evaluate gcd(6, 9), which is 3.
3. Bind g to 3.
4. Evaluate 6 // 3, which is 2.
5. Evaluate 9 // 3, which is 3.
6. Build and return the list [2, 3].

The frame for the call, at the moment just before the return, holds:

rational frame
+---+---+
| n | 6 |
| d | 9 |
| g | 3 |
+---+---+

The returned list [2, 3] represents 32​, the reduced form of 96​. Notice that the caller never sees g; it lives only inside this frame and disappears when the call returns.

What Must Be True for This to Count as Data?

We have now used two different representations for a rational number in this single lesson: first a plain list [n, d], then a reduced list (or, in the original, a reduced tuple). The operations did not care which one was in play. That raises a sharper question than it first appears: what actually makes a representation a valid rational number? What is the contract?

For rational numbers, the constructor and selectors must satisfy one promise. If you build a rational number from a numerator and a denominator, the selectors must give back values consistent with that numerator and denominator:

If r = rational(n, d), then numer(r) and denom(r) describe the same
rational number as n / d (possibly reduced to lowest terms).

This is more fundamental than any storage choice. As long as that promise holds, the rest of the program works, and the representation underneath is free to be a list, a tuple, or something stranger.

We can state the same idea for the more basic building block, the pair itself. A pair is defined entirely by the relationship between its constructor and its selector. Whatever a "pair" is made of, it must satisfy: if you build a pair from x and y, then selecting position 0 gives back x, and selecting position 1 gives back y. Writing the constructor as pair and the selector as select:

select(pair(x,y),0)=x
select(pair(x,y),1)=y

If a construction-and-selection pair of functions satisfies those two equations, it is a working pair. That is the whole specification. Nothing in it mentions lists, brackets, or memory. This is the key that unlocks the next idea.

A Pair Built From a Function

Here is the part of the lesson that changes how you think about data. We will build something that satisfies the pair specification above without using a list at all. The pair will be a function.

The trick rests on something you already know from Chapter 1: a function defined inside another function can remember the values from the call that created it. We construct a pair by capturing x and y inside an inner function, and we return that inner function as the pair. To select from the pair, we call it with the index we want.

This is the official implementation, with the inner function named get:

# (5)
def pair(x, y):
    """Return a function that represents a pair."""
    def get(index):
        if index == 0:
            return x
        elif index == 1:
            return y
    return get

def select(p, i):
    """Return the element at index i of pair p."""
    return p(i)

Look closely at what pair returns. It does not return x, and it does not return y. It returns get, the inner function itself. That returned function carries x and y with it, because they live in the frame of the pair call that produced it. The function is the pair. To read one of its values, select(p, i) simply calls p(i), which runs get with the chosen index and hands back the matching captured value.

Follow the Line: Returning a Function

The single most important line in that definition is the last line of pair. It is also the easiest to get wrong.

return get
├─ return ...... hand a value back to whoever called pair
├─ get ......... the inner function object itself, NOT a call to it
└─ evaluates to  the function get, which still remembers x and y

Contrast it with what you must not write:

return get      hands back the function object (correct: this is the pair)
return get(0)   calls get immediately and hands back only x (wrong: not a pair)

The absence of parentheses after get is the entire difference. With parentheses you would run the function and return a single value; without them you return the function, which is the thing we want, because the function carries both values and can be asked for either one later.

Environment Trace: A Functional Pair

Trace the construction and one selection:

# (6)
p = pair(20, 14)
select(p, 0)

After the first line runs, the frames look like this:

Global frame
+--------+-------------------------------+
| pair   | function pair                 |
| select | function select               |
| p      | function get [parent: f1]     |
+--------+-------------------------------+

f1: the frame from calling pair(20, 14)
+---+----+
| x | 20 |
| y | 14 |
+---+----+

The crucial structural fact is that p is bound to the get function, and that get remembers frame f1 as its parent. Even though the pair call has returned, frame f1 does not vanish, because get still points back to it. That is how x and y survive.

Now evaluate select(p, 0). Step the block below and watch the get frame open: notice that it holds only index, never an x of its own, so when return x runs the lookup has to follow get's parent link back to f1 to find x as 20. The same structure answers select(p, 1) with 14. So the functional pair satisfies the specification exactly:

>>> p = pair(20, 14)
>>> select(p, 0)
20
>>> select(p, 1)
14

Compared to the two equations from the previous section, this implementation makes both of them true: select(pair(20, 14), 0) is 20, and select(pair(20, 14), 1) is 14. There is not a list anywhere in sight, yet by every test that matters, this is a pair.

Why This Matters

The point is not that you should store data inside functions from now on. Lists are simpler and faster, and you will use them constantly. The point is what the functional pair proves: the meaning of a compound data type comes entirely from the behavior its constructor and selectors guarantee, not from any particular physical shape in memory. If the interface keeps its promises, the implementation underneath can be swapped freely. That single idea, that a stable interface lets the representation change without breaking the code that depends on it, is one of the most powerful tools in all of programming, and you just watched it hold even in the extreme case where the representation is a closure (the name for a function that carries remembered values, as our pair does) rather than a container.

⚠Common Beginner Traps

The first trap is off-by-one indexing. The first element of a list is at index 0, so pair[1] is the second element, not the first:

# (7)
pair = [10, 20]
pair[1]   # 20, the second element, not 10

The second trap is reaching through the abstraction barrier. The following function works only as long as a rational number happens to be a list, because it indexes with x[0] and y[1] instead of calling the selectors:

# (8)
def bad_add_rationals(x, y):
    return [x[0] * y[1] + y[0] * x[1], x[1] * y[1]]

If the representation later changes to a tuple, a function, or anything else, this breaks, while the version that uses numer, denom, and rational keeps working.

The third trap is the parentheses on a returned function. Returning get(0) instead of get turns a pair constructor into something that hands back a single value:

# (9)
def broken_pair(x, y):
    def get(index):
        if index == 0:
            return x
        elif index == 1:
            return y
    return get(0)

broken_pair(20, 14) returns 20, an integer, not a function you can select from. The whole point of returning the function is lost.

The fourth trap is confusing the two meanings of square brackets. In [n, d] the brackets build a new list, while in x[0] the brackets select an element from an existing sequence. Same symbol, opposite directions: one packs values in, the other pulls a value out.

✓Check Your Understanding

  1. After running fresh = [7, 4] and then a, b = fresh, what are a and b, and why?
  2. Which functions in the rational-number program are allowed to write x[0], and which are not?
  3. You changed rational to reduce fractions with gcd, and add_rationals started returning reduced results without being edited. Why?
  4. In the functional pair, why does get still know x and y after the call to pair has already returned?
  5. State the two equations that any working pair, no matter how it is built, must satisfy.
  6. Why does return get build a usable pair while return get(0) does not?
Show the answers
  1. a is 7 and b is 4. Multiple assignment binds the target names in order to the elements of the sequence on the right, so a takes the element at index 0 and b takes the element at index 1.
  2. Only the representation-layer functions rational, numer, and denom may write x[0]. The user-level operations add_rationals, mul_rationals, rationals_are_equal, and print_rational must go through the interface and never index directly.
  3. Every rational number is created by rational, so reducing inside rational reduces all of them. add_rationals builds its result by calling rational, which now reduces, so its output is reduced even though its own code is unchanged.
  4. get is a function defined inside pair, and it remembers the frame of the pair call as its parent. That frame holds x and y, and it stays alive as long as get refers to it, so the values are still reachable when get is later called.
  5. select(pair(x, y), 0) must equal x, and select(pair(x, y), 1) must equal y.
  6. return get hands back the function object, which carries both values and can be asked for either one later, so it behaves as a pair. return get(0) calls the function immediately and hands back only the first value, which is a single integer, not a selectable pair.

§Summary

A pair is a container that holds two values in order, and the simplest one in Python is a two-element list, selected with pair[0] and pair[1] (or, equivalently, with getitem) and unpackable with x, y = pair. Using a pair, you implemented the rational-number interface that the previous lesson left undefined: rational packs a numerator and denominator into a list, and numer and denom pull them back out. The larger lesson is the abstraction barrier: user-level operations must touch a rational number only through rational, numer, and denom, never through its list representation. Because of that barrier, improving the constructor to reduce fractions changed every rational number in the program without editing a single operation. Finally, the functional pair, built with an inner get function and selected through select, proves the deepest claim of the section: data is defined by the behavior its constructor and selectors promise, not by any one physical representation, so a stable interface lets the implementation change freely underneath.

▶Step-Through Checkpoint

This program is the lesson's capstone: it defines the functional pair and select, builds a pair, and then selects both values, without a list anywhere in the program. In an environment diagram, p points at the get function object, and the pair frame that held x as 20 and y as 14 stays alive as that function's parent; the diagram makes the closure visible, because you can watch each later get frame reach back along its parent link to read those saved values rather than storing any of its own. Before you step through the block below, write down three predictions: how many local frames will open over the whole run, which names will appear inside each get frame, and what index holds and which return line runs during the final selection.

Step through it one line at a time and check each prediction as the frames appear. You should have seen five local frames in all: one for pair, then a select frame and a get frame for each of the two selections. After p = pair(20, 14) runs, p is bound to a function, not a list, and the frame holding x as 20 and y as 14 stays on screen even though pair has returned. Neither get frame ever holds an x or y of its own; each holds only index, and the lookup follows the parent link back to that one saved frame, where the final selection's index == 1 test sends 14 back out through return y. When the last line finishes, the global frame holds first_value as 20 and second_value as 14. That parent link from get back to the pair frame is the entire mechanism that lets a function behave like data.

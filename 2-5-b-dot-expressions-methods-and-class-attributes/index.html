Chapter 2/2.5 Object-Oriented Programming

2.5 Dot Expressions, Methods, and Class Attributes

◷ approx. 18 min read · 12 worked examples · best read in two passes

In the previous lesson you built account objects. Each account had its own balance and holder, and the methods deposit and withdraw changed that one account. You already wrote dot expressions like spock_account.deposit(90) without slowing down to ask what the dot actually does.

This lesson slows down the dot. By the end you will be able to do four things. You will read any dot expression by breaking it into its parts and predicting the value. You will explain why Account.deposit and spock_account.deposit print as two different kinds of value, even though they run the same body. You will store a value once, on a class, and have every instance share it. And you will trace, step by step, the exact order Python uses to resolve a name after a dot, so that you can predict which value wins when an instance and its class both define the same name.

The whole lesson rests on one habit you have been building since day one: when code looks dense, separate it into the smallest meaningful pieces and evaluate one piece at a time.

The Dot Is a Request to an Object

Picture the front desk of a large building. You do not wander the halls yourself. You walk up to one specific desk, say a name, and the desk either hands you a piece of information it is holding or performs a service on your behalf. The desk is the object. The name you say is the attribute. The dot is the act of walking up to that particular desk and asking.

This matters because the same name can mean different things at different desks. If you ask the third-floor desk for "the schedule," you get the third floor's schedule, not the first floor's. In the same way, spock_account.balance asks one specific account for its balance, and a different account would answer with a different number. The object on the left of the dot decides whose answer you get.

In the message-passing system from Section 2.4, you sent a request to a value by indexing it with a string key:

account['withdraw'](amount)

With classes, Python gives you syntax for the same idea:

account.withdraw(amount)

The two lines express the same intent: send a named request to a value. The difference is that the message name is no longer an arbitrary string typed into brackets. It is a name that belongs to the object's class, and the language gives it real syntax of its own: a dot, not a string in brackets. Because the dot is built into Python's grammar, it works smoothly with the rest of the language. You do not need one message to read an attribute and a separate message to change it the way a hand-built dispatch dictionary did; an ordinary assignment statement can target an attribute directly.

Rebuilding the Account Class

This document stands on its own, so here is the account class from the previous lesson again. It is the official Account class, reproduced exactly:

# (1)
class Account:
    def __init__(self, account_holder):
        self.balance = 0
        self.holder = account_holder

    def deposit(self, amount):
        self.balance = self.balance + amount
        return self.balance

    def withdraw(self, amount):
        if amount > self.balance:
            return 'Insufficient funds'
        self.balance = self.balance - amount
        return self.balance

Make one account and move some money through it, so that we have a concrete object to ask questions of:

# (2)
spock_account = Account('Spock')
spock_account.deposit(100)
spock_account.withdraw(90)

After these three lines, the balance of spock_account is 10. Hold on to that number; the rest of this lesson keeps asking spock_account about itself.

Dot Expressions

The code fragment:

spock_account.deposit

is called a dot expression. A dot expression is built from three parts: an expression, a dot, and a name. Its general form is:

<expression> . <name>

The <expression> on the left can be any valid Python expression, as long as it evaluates to an object. The <name> on the right must be a simple name. It is not a string, and it is not another expression that produces a name. A dot expression evaluates to the value of the attribute called <name> on whatever object the <expression> produced.

Follow the line one token at a time:

spock_account.deposit
├─ spock_account ... expression on the left; evaluates to an Account object
├─ . ............... the attribute-lookup operator; "ask that object for a name"
├─ deposit ......... the simple name to look up on that object
└─ evaluates to .... a bound method that will run deposit with self = spock_account

The dot does real work in two stages. First it forces Python to evaluate the left side all the way down to an actual object. Only then does it search that object for an attribute named deposit. Reading an attribute that simply stores data hands you the stored value:

>>> spock_account.balance
10
>>> spock_account.holder
'Spock'

Reading an attribute that is a method does something more interesting, which the next sections take apart.

getattr and hasattr

Dot notation requires you to type the name literally into your code. Sometimes you instead have the name sitting in a string, computed at runtime. The built-in function getattr is the function equivalent of the dot: it returns an attribute of an object by name, where the name is supplied as a string. This is exactly the kind of lookup you did with a dispatch dictionary, where the message was a string key.

# (3)
getattr(spock_account, 'balance')

The result is:

10

Walk through the call:

getattr(spock_account, 'balance')
├─ getattr ......... built-in: look up an attribute by string name
├─ ( .............. begins the inputs
├─ spock_account .. the object to search
├─ , .............. separates the inputs
├─ 'balance' ...... the attribute name, given as a string
├─ ) .............. ends the inputs
└─ evaluates to ... 10, the value stored under the attribute named balance

You can also ask whether an object even has an attribute of a given name, without retrieving it, using hasattr:

# (4)
hasattr(spock_account, 'deposit')

The result is:

True

The token walk is the same shape:

hasattr(spock_account, 'deposit')
├─ hasattr ........ built-in: test for an attribute by string name
├─ ( .............. begins the inputs
├─ spock_account .. the object to search
├─ , .............. separates the inputs
├─ 'deposit' ...... the attribute name to check for, as a string
├─ ) .............. ends the inputs
└─ evaluates to ... True, because spock_account does have a deposit attribute

The attributes Python finds for an object include all of its instance attributes, such as balance and holder, together with everything defined in its class, including the methods deposit and withdraw. Methods are attributes of the class, but they need special handling when you reach them through an instance. That special handling is the subject of the next two sections.

Methods Versus Functions

When you call a method on an object, that object is passed automatically as the first argument to the method. The object on the left of the dot becomes the value bound to the parameter self inside the method body. You have relied on this since the previous lesson; now we name the mechanism.

To make automatic self binding possible, Python keeps two distinct kinds of value. A function is the kind of value you have been creating with def since the very beginning of the course. A bound method couples together a function and the particular object the method will run on. A bound method already carries its first argument with it: the instance it was looked up from, which will be named self when the method runs.

A pre-addressed, stamped envelope is a useful image. A plain function is a blank envelope: before you can send it, you must write in every address yourself, including who it is from. A bound method is an envelope that already has the "from" line filled in. You only supply what is left.

You can see the two kinds of value directly by calling type on the result of a dot expression. Reached as an attribute of the class, a method is just a function. Reached as an attribute of an instance, it is a bound method:

# (5)
type(Account.deposit)
type(spock_account.deposit)

The results are:

<class 'function'>
<class 'method'>

These two results differ in exactly one way. Account.deposit is the ordinary two-argument function, with parameters self and amount. spock_account.deposit is a one-argument method: the name self will be bound to spock_account automatically when the method is called, and the parameter amount will be bound to whatever argument you pass. Both values, the function and the bound method, run the same deposit function body.

Follow the difference line by line. First, the lookup on the class:

Account.deposit
├─ Account ......... the class object
├─ . ............... attribute lookup on the class
├─ deposit ......... the name to look up
└─ evaluates to .... a plain function with parameters self and amount

Now the lookup on the instance:

spock_account.deposit
├─ spock_account ... an Account instance
├─ . ............... attribute lookup on the instance
├─ deposit ......... the name to look up; found on the class, then bound
└─ evaluates to .... a bound method with self already fixed to spock_account

Calling the Same Body Two Ways

Because the function and the bound method share one body, you can run deposit in two ways. As a function, you must supply the self argument yourself. As a bound method, self is filled in for you. The official text shows both, with the same comments it uses:

# (6)
Account.deposit(spock_account, 1001)  # The deposit function takes 2 arguments
spock_account.deposit(1000)           # The deposit method takes 1 argument

The results are:

1011
2011

The first call goes through the class, so nothing is filled in automatically:

Account.deposit(spock_account, 1001)
├─ Account.deposit ... a plain function: needs self and amount
├─ spock_account ..... supplied by hand as the first argument -> self
├─ 1001 ............. supplied as the second argument -> amount
└─ performs .......... self.balance goes 10 -> 1011, returns 1011

The second call goes through the instance, so self is already attached:

spock_account.deposit(1000)
├─ spock_account.deposit ... a bound method: self is already spock_account
├─ 1000 ................... the only argument you supply -> amount
└─ performs ............... self.balance goes 1011 -> 2011, returns 2011

The body executed is identical in both calls. The only difference is how self gets its value: handed in by you, or filled in by the bound method. This is why a single balance keeps climbing across the two calls: both calls operate on the same spock_account.

There is one more consequence worth stating now, because it connects back to getattr. The function getattr behaves exactly like dot notation here. If its first argument is an instance and the name is a method defined in the class, getattr returns a bound method. If its first argument is a class, getattr returns the attribute value directly, which is a plain function. For example, getattr(spock_account, 'deposit') returns the same bound method that spock_account.deposit does, while getattr(Account, 'deposit') returns the plain function, exactly the <class 'method'> versus <class 'function'> split you saw in code (5). The same instance-versus-class distinction governs both spellings of the lookup.

A Method-Dispatch Trace

It is worth watching one bound-method call resolve all the way through, because every method call you write follows this path. Trace spock_account.deposit(1000) from code (6):

spock_account.deposit(1000)

step 1  Evaluate the dot expression on the left of the call: spock_account.deposit
step 2    Evaluate spock_account -> the Account instance for 'Spock'
step 3    Look for an instance attribute named deposit on that object -> none
step 4    Look for deposit in its class Account -> found, a function
step 5    The class attribute is a function, so build a bound method:
              function  = the deposit function body
              self      = spock_account   (already attached)
step 6  Now apply that bound method to the argument list (1000):
              self   -> spock_account     (carried by the bound method)
              amount -> 1000              (the argument you supplied)
step 7  Run the body:
              self.balance = self.balance + amount   # 1011 + 1000
              return self.balance                    # 2011
step 8  The call evaluates to 2011, and spock_account.balance is now 2011.

The thing to notice is step 5. The bound method is created at the moment of lookup, before any argument is applied. That is why self never appears in your argument list when you call through an instance: it was attached one step earlier.

Class Attributes

Some information does not belong to any one object. It belongs to the whole class. Suppose a bank pays interest on every account at one fixed rate. That rate might change over time, but at any moment it is a single value shared across all accounts. Storing a separate copy of it on every account would be wasteful and error-prone.

A value like that is stored as a class attribute. You create a class attribute with an assignment statement in the body of a class statement, outside of any method definition. Other communities call these class variables or static variables. Here is the official class statement that gives Account a class attribute named interest, reproduced with its own comments:

# (7)
class Account:
    interest = 0.02            # A class attribute
    def __init__(self, account_holder):
        self.balance = 0
        self.holder = account_holder
    # Additional methods would be defined here

The line that creates the class attribute is:

interest = 0.02
├─ interest ........ the attribute name, written with no self in front
├─ = ............... assignment
├─ 0.02 ............ the value
└─ performs ........ binds interest as an attribute of the class Account itself

The position of the line is what makes it a class attribute. It sits inside the class body but outside every method, so the name is assigned on the class object rather than on any instance. Compare it to self.balance = 0, which lives inside __init__ and therefore attaches balance to one particular instance.

Reading a Class Attribute Through Instances

A class attribute can be read from any instance of the class. Make two accounts and ask each one for the interest rate:

# (8)
spock_account = Account('Spock')
kirk_account = Account('Kirk')
spock_account.interest
kirk_account.interest

The results are:

0.02
0.02

Neither account stores its own interest yet. When Python evaluates spock_account.interest, it first searches the instance, finds nothing named interest, and then searches the class, where it finds 0.02. Both accounts reach the same shared value the same way.

Follow the lookup for one of them:

spock_account.interest
├─ spock_account ... the Account instance for 'Spock'
├─ . ............... attribute lookup
├─ interest ........ not an instance attribute here, so search the class
└─ evaluates to .... 0.02, the class attribute on Account

Changing a Class Attribute

Because the value lives on the class, one assignment to the class attribute changes what every instance sees, as long as that instance has not stored its own attribute of the same name.

# (9)
Account.interest = 0.04
spock_account.interest
kirk_account.interest

The results are:

0.04
0.04

The left-hand side of the assignment is the dot expression Account.interest, and the object there is the class itself, so the assignment updates the attribute on the class. Both instances were reading the shared class value, so both now report 0.04.

Account.interest = 0.04
├─ Account ......... the object on the left is the class
├─ .interest ....... the attribute being assigned
├─ = 0.04 .......... the new value
└─ performs ........ updates the class attribute, seen by every plain instance

Attribute Lookup Order

You now have enough machinery that a name after a dot could match either an instance attribute or a class attribute. Python needs a definite rule for which one wins. Recall the form of a dot expression:

<expression> . <name>

To evaluate a dot expression, Python follows these steps in order:

1. Evaluate the <expression> to the left of the dot to get the object.
2. Match <name> against the object's instance attributes; if found, return that value.
3. If <name> is not an instance attribute, look it up in the object's class.
4. Return that class value, unless it is a function, in which case return a bound method.

Instance attributes are found before class attributes. This is the same priority rule you already know from environments, where a local name takes precedence over a global one. Step 4 is exactly the bound-method behavior from earlier in this lesson: a function found on the class is combined with the object on the left to form a bound method. The class-lookup part of step 3 gains more steps once inheritance appears, but the instance-first ordering will not change.

The parallel is worth stating plainly:

environment lookup:   local frame first, then enclosing frames
attribute lookup:     instance first, then the class

There is a matching rule for assignment. Any assignment statement whose left-hand side is a dot expression sets an attribute on the object of that dot expression. If that object is an instance, the assignment sets an instance attribute. If that object is a class, it sets a class attribute. A consequence of this rule is that assigning to an attribute of an instance can never reach back and change the class. The next section shows that consequence in action.

Instance Attributes Can Shadow Class Attributes

Give only Kirk's account its own interest rate:

# (10)
kirk_account.interest = 0.08
kirk_account.interest
spock_account.interest

The results are:

0.08
0.04

The assignment kirk_account.interest = 0.08 has an instance on its left-hand side, so it creates a new instance attribute on kirk_account that happens to share the name interest with the class attribute. From now on, a dot lookup on kirk_account finds that instance attribute at step 2 and stops, returning 0.08. The class attribute is untouched and still reads 0.04 for spock_account, which has no instance attribute of that name.

kirk_account.interest = 0.08
├─ kirk_account .... the object on the left is an instance
├─ .interest ....... the attribute being assigned
├─ = 0.08 .......... the new value
└─ performs ........ creates an instance attribute that shadows the class attribute

Now change the class attribute one more time and watch the two accounts diverge:

# (11)
Account.interest = 0.05
spock_account.interest
kirk_account.interest

The results are:

0.05
0.08

spock_account has no instance attribute named interest, so its lookup falls through to the class and reports the new 0.05. kirk_account still has its own instance attribute, found first at step 2, so it keeps reporting 0.08 and never consults the class. The instance value shadows the class value for that one object.

An Attribute-Lookup Trace

Put the whole shadowing story in one picture. After running code (7) through (11), the objects look like this:

Account class
+----------+------+
| interest | 0.05 |
+----------+------+

spock_account instance
+---------+---------+
| balance | 0       |
| holder  | 'Spock' |
+---------+---------+      (no interest attribute of its own)

kirk_account instance
+----------+--------+
| balance  | 0      |
| holder   | 'Kirk' |
| interest | 0.08   |
+----------+--------+

Trace each lookup against that picture:

spock_account.interest
  step 1  object = spock_account
  step 2  instance attributes: balance, holder -> no interest
  step 3  look in class Account -> interest = 0.05
  result  0.05

kirk_account.interest
  step 1  object = kirk_account
  step 2  instance attributes: balance, holder, interest -> interest = 0.08
  result  0.08   (class is never consulted)

The single most important detail is the absence of an interest row on spock_account. That absence is the reason its lookup keeps falling through to the class, and it is the reason the two accounts now report different rates from the same starting class.

Naming Conventions

Python programmers signal intent through naming conventions. None of these are enforced by the language; they are agreements that make code readable to other people.

Class names use the CapWords convention, also called CamelCase because the capital letters in the middle of a name look like humps. Examples are Account and CheckingAccount. Method names follow the usual convention for functions: lowercased words separated by underscores, such as deposit or withdraw.

Some attributes and methods exist only to keep an object internally consistent and are not meant for users of the class to touch. They are part of the implementation, not part of the abstraction the class offers. Python's convention is that an attribute whose name begins with an underscore should be accessed only within the methods of the class itself, not by outside code. The underscore does not lock the attribute; you can still reach it. It is a message to other programmers that the name is not part of the public interface and may change without warning.

⚠Common Beginner Traps

The first trap is treating Account.deposit and spock_account.deposit as the same kind of value. The first is a plain function with parameters self and amount. The second is a bound method that has already attached spock_account as self.

The second trap is supplying self twice. Writing spock_account.deposit(spock_account, 100) fails, because the bound method has already filled in self, so the call now has one argument too many. Through an instance you pass only amount. Through the class you pass both, as in Account.deposit(spock_account, 100).

The third trap is believing that kirk_account.interest = 0.08 changes the rate for every account. It does not. An assignment with an instance on the left only ever creates or updates an attribute on that one instance, leaving the class attribute alone.

The fourth trap is mistaking the right side of the dot for a string. In spock_account.deposit, the name deposit is a simple name, not 'deposit'; writing spock_account.'deposit' raises a SyntaxError because a string literal cannot follow a dot. When you genuinely need to look up an attribute from a string, that is what getattr is for.

The fifth trap is treating a leading underscore as a security feature. It is a convention, not a lock. Nothing stops the code from reading a _-prefixed name; the underscore only signals that it should not.

  4. After Account.interest = 0.05, why does kirk_account.interest remain 0.08 while spock_account.interest becomes 0.05?
  5. What does getattr(spock_account, 'balance') do, and how is it like a dispatch-dictionary lookup?
Show the answers
  1. spock_account becomes self. It is attached when the dot expression spock_account.deposit is evaluated and a bound method is created, which happens before the argument 1000 is applied.
  2. Account.deposit is looked up on the class, so it is the plain two-argument function with parameters self and amount. spock_account.deposit is looked up on an instance, so Python returns a bound method that already carries spock_account as self and takes only amount.
  3. 0.08 is wrong; the answer is 0.05. jean_account is a fresh instance, so it has no instance attribute named interest. Its lookup finds nothing at the instance, falls through to the class Account, and returns the current class value 0.05. Kirk's 0.08 lives only on the kirk_account instance and is invisible to any other account.
  4. kirk_account has its own instance attribute named interest, found first during lookup, so it shadows the class attribute and the class change is never seen. spock_account has no such instance attribute, so its lookup falls through to the updated class value 0.05.
  5. It looks up the attribute named by the string 'balance' on spock_account and returns 10. Like a dispatch dictionary, the name is supplied as data, a string, and used to perform the lookup, rather than being written literally in the code.

§Summary

Dot expressions are Python's built-in syntax for sending a named request to an object. A dot expression has the form <expression> . <name>: Python evaluates the expression on the left to an object, then looks up the simple name on the right. The function getattr performs the same lookup from a string, and hasattr tests whether the name exists. A function defined in a class is a plain function when reached through the class, but reaching it through an instance produces a bound method that has already attached the instance as self, which is why a method call through an instance supplies only the remaining arguments. Class attributes are assigned in the class body outside any method and are shared by every instance through the lookup order: instance attributes are checked first, then the class. Because an assignment with an instance on the left only ever sets an instance attribute, an instance can shadow a class attribute of the same name without changing the class for anyone else.

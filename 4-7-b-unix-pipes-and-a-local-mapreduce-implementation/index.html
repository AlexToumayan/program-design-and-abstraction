Chapter 4/4.7 Distributed Data Processing

4.7.2 Unix Pipes and a Local MapReduce Implementation

◷ approx. 16 min read · 4 worked examples · best read in two passes

You already know the shape of MapReduce: a map function turns input into key-value pairs, the framework groups those pairs by key, and a reduce function combines the values for each key. That description assumes a framework exists to do the grouping and coordination. This lesson builds the smallest such framework that actually runs, using nothing but tools that come with the Unix operating system.

By the end you will be able to do five things: explain what standard input and standard output are, read a shebang line and say why it turns a Python file into a runnable program, describe what the pipe symbol does between two programs, read the local MapReduce command cat input | ./mapper.py | sort | ./reducer.py as an assembly line, and trace the vowel-counting pipeline from a three-line haiku all the way to its final counts.

Nothing here requires you to have used a Unix shell before. It does require one habit: when a pipeline looks like a wall of symbols, you slow down and read it one program at a time, left to right.

An Assembly Line of Workers

Picture a small factory assembly line. Raw material arrives at the first station. A worker there does one job and slides the result to the next station. The second worker does one job and passes it on. No single worker understands the whole product. Each one only needs to know how to take what arrives, transform it, and hand off what leaves.

Two things make this work. First, every station agrees on the same kind of conveyor belt, so any worker's output physically fits as the next worker's input. Second, each worker is replaceable: as long as a new worker reads from the belt and writes to the belt in the same format, you can swap it in without touching the others.

Unix is built on exactly this idea. Each program is a worker. The conveyor belt is a stream of text lines. The power of the system, as you will see, comes less from any one program than from how easily you can chain them together.

The Unix Abstraction Barrier

The Unix operating system sits between your programs and the raw hardware of the machine. One of the services it provides is a way for programs to communicate, in particular by letting one program consume the output of another. This is the conveyor belt made real.

In their classic book on Unix programming, Kernighan and Pike put the point this way:

"The power of a system comes more from the relationships among programs
than from the programs themselves."

That sentence is the whole design philosophy of this lesson. You will write two small, dull programs, a mapper and a reducer, and the interesting behavior will come entirely from how you connect them.

Standard Input and Standard Output

To act as a station on the assembly line, every Unix program is given two streams to work with.

standard input:  the stream of text a program reads from
standard output: the stream of text a program writes to

In Python these streams are ordinary objects. Standard input is an iterable object accessed as sys.stdin; iterating over it yields the input one line at a time, each line a string. Standard output is accessed as sys.stdout, and the built-in print function writes a line of text to it.

This is a data-processing abstraction barrier. A program does not need to know whether its input arrived from a file on disk, from your keyboard, or from another program upstream. It just reads lines from standard input and writes lines to standard output. That ignorance is the whole point: it is what makes any program freely swappable into any pipeline.

Turning a Python File Into a Unix Program: The Shebang

A Python source file is not, by itself, a program the operating system knows how to launch. You can hand it to the interpreter by hand, but to make the file itself executable you add one special first line, called a shebang:

#!/usr/bin/env python3

The name comes from the two characters that open it: the "hash" # and the "bang" !. When the operating system is asked to run the file directly, it reads this first line to learn which interpreter should run the rest. Follow the line one token at a time:

#!/usr/bin/env python3
├─ #! ............... marks the line as an instruction for which interpreter to use
├─ /usr/bin/env ..... a helper that locates a program on the system
├─ python3 .......... the program to locate: the Python 3 interpreter
└─ result ........... run the rest of this file with python3

Everything below the shebang is still ordinary Python. The shebang itself begins with #, so to Python it is just a comment and is ignored. It speaks only to the operating system, and only about how to execute the file.

A Reversing Program

Here is the official first example: a complete Unix program that reads each line of its input and writes that line back out reversed.

# (1)
#!/usr/bin/env python3

import sys

for line in sys.stdin:
    print(line.strip('\n')[::-1])

The source saves this to a file named rev.py. Read the body as a loop over the input stream: for line in sys.stdin walks the input one line at a time, and the single line inside the loop does all the work. Follow that inner line token by token:

print(line.strip('\n')[::-1])
├─ line ............ one line of text read from standard input
├─ .strip('\n') .... a copy of that line with the trailing newline removed
├─ [::-1] .......... a reversed copy of the resulting string
├─ print(...) ...... writes the reversed string to standard output
└─ action .......... emit the input line, reversed, as one line of output

The .strip('\n') matters: each line that arrives from standard input still carries the newline character that ended it, and reversing without stripping it would put that newline at the front. The slice [::-1] is Python's idiom for "same string, back to front." Then print sends the result down the conveyor belt as one line of output.

Making the File Executable

Saving the file is not enough. Before the operating system will run rev.py as a program, you must mark the file executable. The source does this with chmod:

$ chmod u+x rev.py

Read this as "change the file mode: for the user who owns the file, add execute permission to rev.py." After this command, the shell will let you launch the file directly as ./rev.py. The leading ./ simply means "the program named rev.py in the current directory."

Pipes: Connecting Two Programs

A single program reading from standard input and writing to standard output is one station. To build the assembly line, you connect stations so that one program's output becomes the next program's input. Unix writes that connection with the pipe symbol, |.

$ first_program | second_program

The pipe channels the standard output of the program on its left into the standard input of the program on its right. Neither program is changed; the pipe just wires the belt from one to the next.

The Official Pipe Examples

The source first pipes the program nslookup, which reports the host name of an IP address (here, an address belonging to the New York Times), into rev.py:

$ nslookup 170.149.172.130 | ./rev.py
moc.semityn.www

This is fenced as plain shell text, not Python, because it is a shell session and because its first stage depends on a live network lookup. The stable lesson is the wiring: whatever host name nslookup prints, rev.py receives it as input and prints it reversed. Reading moc.semityn.www backwards spells out the host name nslookup produced.

The second example pipes cat, a program that prints the contents of files, into rev.py. Here the file cat prints is rev.py itself, so the program ends up reversing its own source:

$ cat rev.py | ./rev.py
3nohtyp vne/nib/rsu/!#

sys tropmi

:nidts.sys ni enil rof
)]1-::[)'n\'(pirts.enil(tnirp

cat emits the five lines of rev.py, blank lines and all, and rev.py reverses each line it receives. The blank lines in the source file stay blank when reversed, which is why blank lines appear in the output too.

A Local MapReduce Pipeline

Those two ideas, a program that reads and writes line-by-line and a pipe that connects programs, are already enough to build a minimal MapReduce framework. This version is deliberately tiny: a single map task and a single reduce task, each a Unix program written in Python. The source runs an entire MapReduce application with one command:

$ cat input | ./mapper.py | sort | ./reducer.py

Read it as a four-station assembly line:

cat input       emits the lines of the input file
mapper.py       turns each line into key-value pairs
sort            puts lines with equal keys next to each other
reducer.py      combines the values that share a key

The sort stage is the framework's grouping phase in miniature. sort orders the mapper's output lines, and because the key is at the front of each line, sorting lands all the pairs for a given key in one contiguous run. That adjacency is exactly what the reducer needs to process one key's values together. This local version is not yet distributed, but it has the same conceptual shape as the full framework: map, group, reduce.

The Vowel Mapper

To run the vowel-counting application from the previous lesson, the source supplies the map function as a Unix program named count_vowels_mapper.py:

# (2)
#!/usr/bin/env python3

import sys
from mr import emit

def count_vowels(line):
    """A map function that counts the vowels in a line."""
    for vowel in 'aeiou':
        count = line.count(vowel)
        if count > 0:
            emit(vowel, count)

for line in sys.stdin:
    count_vowels(line)

The function emit is imported from the companion mr module; conceptually it writes one key-value pair to standard output in the format the reducer expects. Inside count_vowels, the loop checks each of the five vowels and emits a pair only when that vowel actually appears. Follow the emit call token by token:

emit(vowel, count)
├─ emit ............ the helper that writes one key-value pair to output
├─ vowel ........... the key: the vowel being counted, such as 'a'
├─ count ........... the value: how many times that vowel appeared in this line
└─ action .......... write one (vowel, count) pair to standard output

The guard if count > 0 is doing real work: a vowel that never appears in a line emits nothing at all. The map function does not produce a row of zeros; it produces pairs only for vowels that are present. The last two lines are the bridge from Unix to MapReduce:

for line in sys.stdin:
    count_vowels(line)

Every line read from standard input becomes exactly one call to the map function. The framework feeds lines in; the mapper turns each one into zero or more key-value pairs.

The Sum Reducer

The reduce function is supplied as a second Unix program named sum_reducer.py:

# (3)
#!/usr/bin/env python3

import sys
from mr import values_by_key, emit

for key, value_iterator in values_by_key(sys.stdin):
    emit(key, sum(value_iterator))

This reducer is almost entirely about reading grouped input. The helper values_by_key consumes the sorted mapper output from standard input and yields one key at a time, paired with an iterator over all of that key's values. The reducer then emits the key together with the built-in sum of those values. Follow the loop header token by token:

for key, value_iterator in values_by_key(sys.stdin):
├─ key ................. the next distinct key from the sorted input
├─ value_iterator ...... an iterator over every value recorded for that key
├─ values_by_key(...) .. reads sorted mapper output and groups it by key
├─ sys.stdin ........... the grouped key-value stream arriving from sort
└─ action .............. give the loop body one key and all of its values

Notice what the reducer does not do: it never counts a vowel itself. By the time data reaches the reducer, the counting is finished and the framework has grouped the pairs; the reducer only sums the values that already share a key. The built-in sum is the entire reduction.

The reducer relies on a helper named values_by_key from the mr module. Whatever it is called, its job is simple: it reads the sorted stream and hands the reducer one key at a time along with that key's values. We use the name that appears in the runnable code, values_by_key, throughout this lesson.

Running the Haiku Example

To exercise the pipeline end to end, the source provides a three-line input file named haiku.txt:

Google MapReduce
Is a Big Data framework
For batch processing

Running the local pipeline on this file produces the count of each vowel across the whole haiku:

$ cat haiku.txt | ./count_vowels_mapper.py | sort | ./sum_reducer.py
'a'   6
'e'   5
'i'   2
'o'   5
'u'   1

Each output line is one final key-value result. The key is a vowel, printed in quotes, and the value is the total number of times that vowel appears across all three input lines.

Tracing the Pipeline Stage by Stage

To see why those totals come out, watch the data change shape at each station. The trace below is worked out directly from the haiku and the mapper code above; the source prints only the first and last stages, so the intermediate stages here are derived, not quoted.

First, cat haiku.txt emits the three input lines unchanged. Then each line goes to count_vowels_mapper.py. The mapper checks each vowel against the raw line and emits a pair only when the count is above zero. Counting by hand, line by line:

Input line Pairs emitted by the mapper
Google MapReduce ('a', 1), ('e', 3), ('o', 2), ('u', 1)
Is a Big Data framework ('a', 4), ('e', 1), ('i', 1), ('o', 1)
For batch processing ('a', 1), ('e', 1), ('i', 1), ('o', 2)

The mapper emits one pair per vowel per line, where the value is that vowel's count within the line, not one pair per individual letter. The counts are case sensitive: the loop tests only the lowercase vowels in 'aeiou', so the capital letters that begin words contribute nothing. That is why the capital I in Is and the capital O in Google and For are not counted.

Next, sort orders all of those emitted lines so that equal keys sit together. Grouped by key, the values lined up for each vowel are:

Key Values arriving at the reducer, grouped by sort
'a' 1, 4, 1
'e' 3, 1, 1
'i' 1, 1
'o' 2, 1, 2
'u' 1

Finally, sum_reducer.py sums each key's values and emits one line per key:

Key Sum of its values Total
'a' 1 + 4 + 1 6
'e' 3 + 1 + 1 5
'i' 1 + 1 2
'o' 2 + 1 + 2 5
'u' 1 1

Those five totals are exactly the published output, 'a' 6, 'e' 5, 'i' 2, 'o' 5, 'u' 1. The pipeline turned a stream of text lines into key-value pairs, sorted them so equal keys were adjacent, and summed each group, the map, group, reduce shape, built from cat, a mapper, sort, and a reducer.

⚠Common Beginner Traps

  • Thinking a pipe sends Python objects. A pipe carries a stream of text lines, not live Python lists, tuples, or dictionaries. Every stage reads strings and writes strings.
  • Confusing the Unix cat with concatenation. Here cat is a Unix program that prints the contents of a file to standard output. It is not Python string concatenation.
  • Forgetting executable permission. A file can have a correct shebang and still refuse to run as ./rev.py until you mark it executable with chmod u+x.
  • Believing sort is optional. The reducer assumes that all values for one key arrive together. Remove sort and that assumption breaks, so the reducer can split one key across several partial sums.
  • Expecting the mapper to emit zeros. The guard if count > 0 means a vowel that never appears in a line produces no pair at all. Absent keys are simply missing, not present with a value of zero.
  • Assuming the count is case insensitive. The mapper tests only the lowercase letters in 'aeiou', so capital vowels are not counted.
  • Mixing up the two jobs. The mapper produces partial, per-line facts; the reducer combines already-grouped values. The mapper never sums across lines, and the reducer never counts vowels.

§Summary

Unix pipes give you a working, local version of the MapReduce idea built entirely from operating-system tools. Programs read from standard input (sys.stdin) and write to standard output (sys.stdout), and a shebang line plus chmod u+x turns a Python file into a launchable program. The pipe symbol connects programs so that one's output becomes the next one's input, which is the conveyor belt that makes the assembly line possible, and as Kernighan and Pike note, the system's power lives in those relationships more than in any single program. The local vowel-counting pipeline wires four stations together, cat | ./count_vowels_mapper.py | sort | ./sum_reducer.py: the mapper turns each line into per-line vowel counts, sort groups equal keys, and the reducer sums each group, producing the final totals 'a' 6, 'e' 5, 'i' 2, 'o' 5, 'u' 1 for the haiku.

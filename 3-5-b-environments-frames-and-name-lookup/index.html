Chapter 3/3.5 Interpreters for Languages with Abstraction

3.5.2 Environments, Frames, and Name Lookup

◷ approx. 17 min read · 2 worked examples · best read in two passes

In the previous packet you built the heart of the evaluator. Its entry point takes two inputs:

scheme_eval(expr, env)

You spent that packet on the first input, expr — the expression being evaluated. This packet is about the second input, env. The environment is where names get their meanings. When the interpreter sees the symbol factorial or n, the environment is the thing it consults to find out what that symbol stands for.

By the end of this lesson you will be able to do four things: describe the two pieces of data that make up a frame, explain what the define operation does to a frame, trace a name lookup as a walk up a chain of parent frames, and follow the recursive factorial example to see why every call gets its own copy of n while every call can still reach the same global procedure.

You already know more about this than it might feel like. The environment model was introduced back in Chapter 1 to explain how Python finds the value of a name. The new idea here is that you are now building the machinery that implements that model, rather than just using it.

A Classroom With Name Tags

Imagine a classroom where every desk has a name tag taped to it, and each tag pairs a name with a value:

x      -> 10
y      -> 4
square -> a procedure

If someone walks in and asks "what is x?", you scan the name tags in the room and find 10. The room is a lookup table: a name comes in, a value goes out.

Now imagine one student steps into a smaller breakout room for a tutoring session. That room has its own name tags taped to its own desks:

n -> 5

Two things are now true at once. If the student asks for n, the breakout room can answer directly, because n is on a tag in that room. But if the student asks for square, the breakout room has no tag for it. Rather than give up, the student walks back out to the main classroom and looks there, where square is found.

That walk — answer locally if you can, otherwise step out to the larger room and look there — is the entire environment model. The breakout room is connected to the classroom by a door, and that door is what makes the walk possible.

An environment is a chain of these rooms. In the interpreter, each room is called a frame. Each frame stores its own name tags, and each frame may have a door leading to a parent frame.

Frames and the Environment Model

The official source states it directly. Each Frame instance represents an environment in which symbols are bound to values. A frame has a dictionary of bindings, as well as a parent frame that is None for the global frame.

So a frame is exactly two things:

  • a dictionary of bindings, mapping each symbol to the value it stands for
  • a parent frame, which is None only for the outermost frame, the global frame

The picture of a single frame:

Frame
  bindings: {symbol -> value}
  parent: another Frame, or None

The global frame is special because it sits at the bottom of every chain. It has no enclosing room to walk out to, so its parent is None. Every other frame you create during evaluation will have a real parent.

Bindings are not accessed directly. Instead they are reached through two Frame methods:

  • lookup: find the value bound to a symbol, searching outward through parent frames if necessary
  • define: bind a symbol to a value in the current frame

Routing all access through these two operations is what keeps the environment rules in one place. Nobody reaches into the bindings dictionary by hand; everybody asks lookup or define, and those two methods enforce the rules described below.

Defining a Name

The define method always binds a symbol to a value in the current frame. It does not search. It writes the binding right here, in the frame you call it on.

Suppose the current frame is the global frame and it currently holds one binding:

Global frame
  x -> 10

The interpreter now evaluates this Scheme expression:

(define y 4)

After define runs, the same frame holds an additional binding:

Global frame
  x -> 10
  y -> 4

Notice what did not happen. define did not walk up to any parent frame to check whether y already existed somewhere. There is no searching in define. It binds the name in the current frame and stops. (Looking outward through parents is the job of lookup, which is the opposite operation, covered next.)

Looking Up a Name

The lookup method implements the look-up procedure of the environment model of computation described in Chapter 1. Unlike define, this operation does search.

A symbol is matched against the bindings of the current frame. If it is found, the value to which it is bound is returned. If it is not found, look-up proceeds to the parent frame. There the symbol is matched again, and again either it is found and returned, or look-up proceeds one more frame outward. The walk continues until either the symbol is found or there is no parent frame left to check.

In sequence:

Start in the current frame. Is the symbol in this frame's bindings?
  If yes -> return the value, stop.
  If no  -> move to the parent frame and ask the same question.

Repeat until the symbol is found, or until a frame has no parent.

If the walk reaches a frame with no parent and the symbol still has not been found, the name does not exist in this environment, and the interpreter reports an unknown-identifier error.

The order matters. The current frame always gets the first chance to answer. A parent frame is consulted only when the current frame has no binding for the symbol. This is exactly why a local name can shadow a global one (that is, hide it, so the global one is never reached): if both the current frame and a parent frame bind square, lookup finds the local binding first and never reaches the parent.

Following the Line: a Lookup That Walks

Walk through one lookup call token by token. The interpreter is asked to look up the symbol square starting in a local frame that does not contain it. Below, each token of the call is one branch row; the final row is the result the call produces.

lookup(square)   [current frame: local, bindings: n -> 5]
├─ lookup ........ the Frame method that searches for a binding
├─ square ........ the symbol to resolve; missing here, so the search walks to the parent
└─ evaluates to  the procedure value bound to square in the parent global frame

The branch rows name the two tokens of the call: the method lookup and the symbol square. The symbol misses in the current frame, so the search follows the parent link outward and matches square in the global frame. The final row is the value handed back. Compare that with a define, which performs an action rather than producing a searched-for value:

define(y, 4)     [current frame: global]
├─ define ........ the Frame method that writes a binding
├─ y ............. the symbol to bind, written into THIS frame only
├─ 4 ............. the value to bind it to
└─ action ....... adds y -> 4 to the current frame; no search, no walk

Read these two blocks side by side and the division of labor is clear. lookup is a walk outward that returns a value. define is a single write into the current frame that performs an action.

A Small Executable Frame Model

The source leaves the implementation of lookup and define as exercises, so no Frame class code appears in this subsection of the text. To make the rules concrete, here is a compact, runnable Python model that follows the official description: a frame with a bindings dictionary and a parent, a define that writes locally, and a lookup that walks outward.

Running it produces:

5
10
procedure object

Look closely at the last two lines of output. local_frame.lookup("x") and local_frame.lookup("square") both succeed even though neither x nor square is in local_frame. Each lookup misses locally, follows self.parent to global_frame, and finds the binding there. That is the walk, written out in code: the first if checks the current frame, and the recursive call to self.parent.lookup(symbol) steps one frame outward. The final raise NameError(symbol) is what happens when the walk runs out of parents.

Lookup Is a Walk, Not a Guess

When students first meet environments, it is tempting to imagine that Python or Scheme "just knows" where a name lives, the way you might know a friend's phone number. The interpreter knows nothing in advance. It follows the rule, every single time.

For the model above, evaluating local_frame.lookup("square") proceeds exactly like this:

current frame: local_frame
bindings here: n -> 5
question: is square here?
answer: no

follow parent link to global_frame
bindings here: x -> 10, square -> procedure object
question: is square here?
answer: yes

return: procedure object

The current frame gets the first chance to answer. The parent is checked only because the current frame came up empty. And this is the precise mechanism behind shadowing: if local_frame had its own binding for square, the very first question — "is square here?" — would answer "yes," lookup would return the local value, and the parent's binding for square would never be consulted.

The Official Factorial Program

The source illustrates lookup and define together with this example Scheme program:

(define (factorial n)
  (if (= n 0) 1 (* n (factorial (- n 1)))))

(factorial 5)

Note the two forms of define at work. The (define y 4) you saw earlier binds a name to a value, while (define (factorial n) ...) is shorthand that names a procedure and lists its parameter n in one step. This short program is well chosen because it exercises both operations and both phases of the interpreter. The first input is a definition; the second is a call. Defining factorial uses define to create a binding. Calling (factorial 5) uses lookup repeatedly, and it also creates new frames. Walk through each phase in turn.

Phase One: Defining the Function

The first input expression is a define special form, evaluated by the do_define_form Python function. Defining a function has several steps:

  1. Check the format of the expression to ensure that it is a well-formed Scheme list with at least two elements following the keyword define.
  2. Analyze the first element, in this case a Pair, to find the function name factorial and formal parameter list (n).
  3. Create a LambdaProcedure with the supplied formal parameters, body, and parent environment.
  4. Bind the symbol factorial to this function, in the first frame of the current environment. In this case, the environment consists only of the global frame.

So after the definition runs, the global frame gains one binding:

Global frame
  factorial -> LambdaProcedure(formals=(n), body=(if ...), env=global)

Step 3 is the one to slow down on. The procedure is built with three pieces, and the third — the parent environment — is the environment in which the define is being evaluated. Here that is the global frame. The procedure remembers it.

That remembered environment is the door the procedure will walk through later. When factorial is eventually called, the call will create a new frame whose parent is this saved environment. That is how the body of factorial, running in some fresh call frame, will still be able to find the global names it needs — including factorial itself.

Phase Two: Applying the Function

The second input is a call expression. Here the interpreter does several things in order, and each is worth naming because each one is either a lookup, an evaluation, or a frame creation.

First the evaluator recognizes a call expression with an operator and an operand:

operator: factorial
operand: 5

It looks up factorial in the global frame — a lookup — and finds the LambdaProcedure that was just bound there. It evaluates the operand 5, which simply produces 5.

Now application proper begins. The procedure passed to scheme_apply is the LambdaProcedure just created and bound to the symbol factorial. The args passed is a one-element Scheme list (5). To apply the procedure, a new frame is created that extends the global frame — the parent environment of the factorial procedure. In this frame, the symbol n is bound to the value 5:

factorial-frame-1
  n -> 5
  parent -> Global frame

Then, the body of factorial is evaluated in that environment, and its value is returned. The body is:

(if (= n 0) 1 (* n (factorial (- n 1))))

and it is evaluated with factorial-frame-1 as the current frame.

Following the Line: Applying (factorial 5)

Here is the whole application as a single follow-the-line walk. Each token of the call expression is a branch row; the final row states the action that application performs.

(factorial 5)
(factorial 5)
├─ factorial ..... the operator; looked up in the global frame, finds the LambdaProcedure
├─ 5 ............. the operand; evaluates to the value 5
└─ action ....... scheme_apply makes a new frame extending the global frame, binds n -> 5, and evaluates the body (if ...) there

Watch which names resolve where while the body runs in factorial-frame-1. When the body evaluates the subexpression:

(= n 0)

the lookup of n matches immediately in factorial-frame-1, because the new frame bound n to 5. But when the body needs:

factorial

to make the recursive call, lookup misses in factorial-frame-1 — there is no factorial tag in that room — so it walks the parent link to the global frame and finds the procedure there. Each call has its own n locally, while every call reaches the one shared factorial through its parent link. That single fact is how recursion works in the environment model.

The Recursive Stack of Frames

The recursive call does not stop at one frame. Each time the body evaluates (factorial (- n 1)), application happens again: a fresh frame is created, extending the global frame, with n bound to the new, smaller value. The full computation of (factorial 5) therefore creates a chain of call frames:

factorial-frame-1: n -> 5, parent -> global
factorial-frame-2: n -> 4, parent -> global
factorial-frame-3: n -> 3, parent -> global
factorial-frame-4: n -> 2, parent -> global
factorial-frame-5: n -> 1, parent -> global
factorial-frame-6: n -> 0, parent -> global

Each call gets its own frame, and these frames do not overwrite one another. factorial-frame-1 still holds n -> 5 while factorial-frame-6 holds n -> 0. This is exactly why recursion can remember many different values of n at the same time: each value lives in its own frame.

There is one subtlety worth stating plainly, because it is the most common point of confusion. The parent link is not a link to the previous recursive call. Every factorial call frame points back to the global frame, because factorial was defined globally and each call frame extends the procedure's saved environment, which is the global frame. The chain of active calls — frame 1 waiting on frame 2 waiting on frame 3 — records the order of pending work. The parent links record where names are looked up. These are two different structures that happen to be active at the same time, and keeping them separate is the key to reading the diagram correctly.

⚠Common Beginner Traps

Trap 1: Thinking an environment is one big dictionary. An environment is a chain of frames, each with its own bindings, not one flat storage space. Lookup walks that chain.

Trap 2: Thinking define searches upward. define always writes into the current frame and never searches. It is lookup that walks outward through parents.

Trap 3: Thinking recursive calls share one parameter binding. Each procedure call creates a new frame with its own binding for n. Six calls means six separate bindings of n, alive at once. If they did share one binding, every frame would read the final value n -> 0, the multiplication (* n (factorial ...)) would use the wrong n at each level, and (factorial 5) would not produce 120.

Trap 4: Forgetting the parent link. The parent link is the door that lets a local frame still reach global names. Without it, the body of factorial could never find factorial to recur.

Trap 5: Confusing the parent link with the call stack. In this example every call frame's parent is the global frame, not the calling frame. The parent link is about where to look up names; the stack of pending calls is about the order of unfinished work.

Trap 6: Confusing defining a procedure with calling it. Evaluating the define form stores a LambdaProcedure in the global frame. It does not run the body. The body runs only when the procedure is applied, which creates a frame and evaluates the body in it.

  2. What is the difference between define and lookup?
  3. Why does the global frame have parent None?
  4. After the define special form is evaluated, where is factorial bound, and what is it bound to?
  5. When (factorial 5) is applied, what new frame is created and what does it bind?
  6. While the body of factorial runs, why can it find both n and factorial, and where is each one found?
  7. In the factorial example, does each call frame's parent point to the previous factorial call frame or to the global frame? Why?
Show the answers
  1. local_frame.lookup("x") returns 99, consulting only one frame: the new local binding for x is found immediately and shadows the global x, so the parent is never checked. local_frame.lookup("square") returns "procedure object", consulting two frames: it misses in local_frame, follows the parent link to global_frame, and finds square there.
  2. define binds a symbol to a value in the current frame and never searches. lookup searches for a binding, starting in the current frame and walking outward to parent frames until it finds the symbol or runs out of parents.
  3. The global frame is the outermost frame, so there is no enclosing frame to walk out to; its parent is None.
  4. factorial is bound in the first frame of the current environment, which here is the global frame. It is bound to a LambdaProcedure built from the formal parameters (n), the body, and the parent environment (the global frame).
  5. A new frame is created that extends the global frame, and in it the symbol n is bound to the value 5. The body of factorial is then evaluated in that frame.
  6. n is found in the current call frame, because application bound it there. factorial is not in the call frame, so lookup follows the parent link to the global frame and finds it there.
  7. Each call frame's parent points to the global frame, because each call extends the factorial procedure's saved parent environment, which is the global frame where factorial was defined. The chain of pending recursive calls is a separate structure from the environment parent links.

§Summary

An environment is a chain of frames. Each frame holds a dictionary of bindings and a parent frame, which is None only for the global frame. The two operations that touch a frame are define, which always binds a symbol in the current frame, and lookup, which searches the current frame and then walks outward through parent frames until the symbol is found or no parent remains. The official factorial program exercises both: evaluating its define form builds a LambdaProcedure and binds factorial in the global frame, and applying (factorial 5) creates a new frame that extends the global frame with n bound to 5, then evaluates the body there. Because each call gets its own frame while every call reaches the shared global factorial through its parent link, recursion can keep many values of n alive at once. The stack of pending calls and the chain of parent links are two distinct structures that operate side by side.

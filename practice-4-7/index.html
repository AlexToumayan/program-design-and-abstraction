4.7 · MapReduce — Coding Practice

◷ approx. 14 min read · 30 worked examples

These problems drill the core moves of Section 4.7: writing pure mapper functions that emit (key, value) pairs, grouping pairs by key, writing reducers that combine the values for one key, and wiring the whole map → group → reduce flow together — including the fault-tolerance idea that a pure task is always safe to recompute.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Tag Positive Amounts

A mapper reads a list of (category, amount) records and emits one (category, amount) pair for every record whose amount is strictly greater than 0. Records with a zero or negative amount emit nothing at all — this is the "zero or more pairs per input" rule from the map stage. Return the list of emitted pairs, in input order.

Examples

Input:  tag_amounts([('food', 12), ('rent', 0), ('food', 8), ('toys', 5)])
Output: [('food', 12), ('food', 8), ('toys', 5)]
Explanation: the ('rent', 0) record is dropped because its amount is not > 0.
Input:  tag_amounts([('a', -3), ('b', 4)])
Output: [('b', 4)]
Explanation: a negative amount emits no pair; only ('b', 4) survives.
Input:  tag_amounts([('x', 0), ('x', 0)])

  • The input is a list of (category, amount) pairs; amount is a number.
  • A mapper is a pure function: build and return a fresh list, do not read or write any outside state.
  • Emit a pair only when amount > 0.

Solutions

1. Tag Positive Amounts

Approach — a pure mapper with an emit guard. This is the map stage from 4.7-A: read each record, and emit a pair only when it qualifies. The if amount > 0: guard is the same shape as the vowel mapper's if count > 0: — an input that has nothing interesting to say stays silent, so the mapper emits zero or more pairs per input, never a forced one.

def tag_amounts(records):
    pairs = []
    for category, amount in records:
        if amount > 0:
            pairs.append((category, amount))
    return pairs

Why it works. The function reads only its argument and builds a fresh list, so it is pure. Each record contributes a pair exactly when amount > 0, and the input order is preserved because we append as we go.

Complexity. Time O(n), Space O(n) for n records. One pass; the output holds at most one pair per input.

assert tag_amounts([('food', 12), ('rent', 0), ('food', 8), ('toys', 5)]) == \
    [('food', 12), ('food', 8), ('toys', 5)]
assert tag_amounts([]) == []
assert tag_amounts([('a', -3), ('b', 4)]) == [('b', 4)]
assert tag_amounts([('x', 0), ('x', 0)]) == []

2. Reduce One Key's Values

Approach — the built-in sum reducer. As 4.7-A and 4.7-B stress, the reducer for counting is just Python's sum applied to one key's grouped values. By the time the reducer runs, grouping is already done, so it only ever sees one key's pile and folds it into a total.

def reduce_sum(key, values):
    return (key, sum(values))

Why it works. sum(values) drains the iterable — a list or the iterator the group stage hands over — into one number, and we pair it back with key to form the final (key, value) result. An empty pile sums to 0.

Complexity. Time O(m) for m values, Space O(1). sum makes a single pass and keeps one running total.

assert reduce_sum('a', [1, 4, 1]) == ('a', 6)
assert reduce_sum('u', [1]) == ('u', 1)
assert reduce_sum('x', []) == ('x', 0)
assert reduce_sum('e', iter([3, 1, 1])) == ('e', 5)

3. Group Pairs by Key

Approach — the group/sort stage as a dict. This is the framework's middle stage from 4.7-B, written in plain Python: collect every (key, value) slip into the bin named by its key. A dict from key to list of values is the natural "bins" structure.

def group_by_key(pairs):
    groups = {}
    for key, value in pairs:
        if key not in groups:
            groups[key] = []
        groups[key].append(value)
    return groups

Why it works. The first time a key appears we open an empty list for it; every value for that key is then appended in emit order. The result hands each reducer exactly one key's values, already collected.

Complexity. Time O(n), Space O(n) for n pairs. Each pair is placed once; the bins together hold every value.

assert group_by_key([('a', 1), ('b', 2), ('a', 3)]) == {'a': [1, 3], 'b': [2]}
assert group_by_key([]) == {}
assert group_by_key([('o', 2), ('o', 1), ('o', 2)]) == {'o': [2, 1, 2]}
assert group_by_key([('k', 9)]) == {'k': [9]}

4. A Pure Task Runs Twice the Same

Approach — purity made visible. This mirrors the fault-tolerance warm-up in 4.7-C: a pure task is just a function whose output depends only on its input, so running it twice yields the same answer twice. count_char reads only its arguments, so the two runs must agree.

def count_char(chunk, target):
    return chunk.count(target)

def run_twice(chunk, target):
    first = count_char(chunk, target)
    second = count_char(chunk, target)
    return (first, second, first == second)

Why it works. str.count has no hidden state; given the same chunk and target it returns the same number. So first and second are always equal, which is exactly the property that makes "run the failed task again" a guarantee rather than a gamble.

Complexity. Time O(k) for a chunk of length k (two scans), Space O(1). Constant extra storage.

assert run_twice('banana', 'a') == (3, 3, True)
assert run_twice('mississippi', 's') == (4, 4, True)
assert run_twice('xyz', 'a') == (0, 0, True)

5. Word Count, End to End

Approach — map → group → reduce in one function. This is the canonical MapReduce job, with all three stages from 4.7-B inlined. The map stage emits (word, 1) per token, the group stage buckets the 1s by word, and the reduce stage sums each bucket — the same sum reducer, now counting words instead of vowels.

def word_count(lines):
    pairs = []
    for line in lines:
        for word in line.split():
            pairs.append((word, 1))
    groups = {}
    for word, one in pairs:
        if word not in groups:
            groups[word] = []
        groups[word].append(one)
    totals = {}
    for word in groups:
        totals[word] = sum(groups[word])
    return totals

Why it works. Each token contributes one 1 slip under its word; grouping collects all of a word's 1s; summing a list of c ones gives the word's count c. Because split() is case- and whitespace-exact, identical tokens land in the same bin.

Complexity. Time O(N), Space O(N) for N total tokens. Every token is emitted, grouped, and summed once.

assert word_count(['the cat sat', 'the dog ran', 'the cat ran']) == \
    {'the': 3, 'cat': 2, 'sat': 1, 'dog': 1, 'ran': 2}
assert word_count(['hello']) == {'hello': 1}
assert word_count([]) == {}
assert word_count(['a a a', 'a']) == {'a': 4}

6. Emit Only the Qualifying Records

Approach — a filtering mapper. Same map-stage shape as problem 1, with the guard now a numeric threshold. The map function emits zero or more pairs per input; here it stays silent for any round under 100, so the standout scores flow downstream and the rest vanish.

def emit_high_scores(rounds):
    pairs = []
    for player, score in rounds:
        if score >= 100:
            pairs.append((player, score))
    return pairs

Why it works. The >= 100 test keeps the boundary value 100 and drops everything below it. Appending inside the guard preserves input order among the survivors, and the function touches no outside state, so it is pure.

Complexity. Time O(n), Space O(n) for n rounds. One pass; the output holds at most one pair per round.

assert emit_high_scores([('ann', 120), ('bob', 80), ('ann', 100), ('cy', 99)]) == \
    [('ann', 120), ('ann', 100)]
assert emit_high_scores([('x', 50)]) == []
assert emit_high_scores([]) == []
assert emit_high_scores([('z', 100)]) == [('z', 100)]

7. Reduce to the Maximum per Key

Approach — a fold reducer that is not sum. The reduce stage combines a pile of values for one key into a final result; it does not have to be addition. Here each key's list is folded by keeping the running best, the same "combine a pile" job the section describes, with max as the combiner.

def reduce_max(groups):
    result = {}
    for key in groups:
        best = groups[key][0]
        for value in groups[key]:
            if value > best:
                best = value
        result[key] = best
    return result

Why it works. Seeding best with the first value (the list is non-empty) and replacing it whenever a larger value appears leaves best holding the maximum after one scan. Each key is reduced independently, exactly as reducers see one key at a time.

Complexity. Time O(N), Space O(K) for N total values across K keys. Each value is examined once; the result holds one entry per key.

assert reduce_max({'ann': [120, 100], 'bob': [80, 95]}) == {'ann': 120, 'bob': 95}
assert reduce_max({'solo': [42]}) == {'solo': 42}
assert reduce_max({}) == {}
assert reduce_max({'t': [-5, -9, -1]}) == {'t': -1}

8. Recompute Failed Tasks, Then Combine

Approach — fault tolerance via purity. This is the recovery model from 4.7-C made runnable: a failed record carries no result, but it still knows its chunk, so the framework reruns the pure mapper on that chunk to recover. Because count_a depends only on its input, the recomputed value matches what the failed task would have produced.

def count_a(chunk):
    return chunk.count('a')

def run_with_recovery(records):
    for record in records:
        if record['status'] == 'failed':
            record['result'] = count_a(record['chunk'])
            record['status'] = 'done'
    total = 0
    for record in records:
        total = total + record['result']
    return total

Why it works. The first loop repairs every failed record in place by recomputing count_a(chunk) and marking it done; the second loop sums all results. Re-running a pure function on the same chunk is a guarantee, not a gamble, which is the whole reason a distributed framework can retry lost tasks safely.

Complexity. Time O(R+C) for R records of total chunk length C, Space O(1) beyond the input. Two passes plus the recomputation work.

assert run_with_recovery([
    {'task': 't1', 'chunk': 'banana', 'status': 'done', 'result': 3},
    {'task': 't2', 'chunk': 'alpaca', 'status': 'failed', 'result': None},
    {'task': 't3', 'chunk': 'data', 'status': 'done', 'result': 2},
]) == 8
assert run_with_recovery([
    {'task': 'x', 'chunk': 'aaa', 'status': 'failed', 'result': None},
]) == 3
assert run_with_recovery([
    {'task': 'y', 'chunk': 'no vowels here', 'status': 'done', 'result': 0},
]) == 0

9. A Local MapReduce Engine

Approach — the framework as a higher-order function. This packages the entire map → group/sort → reduce flow as one reusable engine, taking the mapper and reducer as arguments — the separation of concerns from 4.7-A in code. The engine owns coordination (flatten, group, iterate keys in order); the supplied functions own the computation.

def map_reduce(inputs, mapper, reducer):
    pairs = []
    for item in inputs:
        for pair in mapper(item):
            pairs.append(pair)
    groups = {}
    for key, value in pairs:
        if key not in groups:
            groups[key] = []
        groups[key].append(value)
    result = {}
    for key in sorted(groups):
        result[key] = reducer(groups[key])
    return result


def vowel_mapper(line):
    pairs = []
    for vowel in 'aeiou':
        count = line.count(vowel)
        if count > 0:
            pairs.append((vowel, count))
    return pairs


def length_mapper(word):
    return [(len(word), 1)]

Why it works. The first phase flattens every mapper's output into one pair list, the second groups values by key, and the third reduces each key in sorted order for deterministic output. Swapping vowel_mapper for length_mapper (and any reducer) changes the computation without touching the engine — the abstraction barrier the section is built around.

Complexity. Time O(P+KlogK) for P emitted pairs across K distinct keys, Space O(P). Flattening and grouping are linear; sorting the keys adds the log factor.

assert map_reduce(['open data tools', 'cloud queue'], vowel_mapper, sum) == \
    {'a': 2, 'e': 3, 'o': 4, 'u': 3}
assert map_reduce(['hi', 'yo', 'cat', 'dog', 'a'], length_mapper, sum) == \
    {1: 1, 2: 2, 3: 2}
assert map_reduce([], vowel_mapper, sum) == {}
assert map_reduce(['xyz'], vowel_mapper, sum) == {}

10. Build an Inverted Index

Approach — a map-reduce with a multi-value reduce. A search index is a MapReduce job: map each document to (word, doc_id) pairs, group by word, and reduce each word's bag of ids. Unlike sum, this reducer emits a list — sorted(set(...)) — showing that a reducer can output zero or more combined values, not just a single number.

def inverted_index(docs):
    pairs = []
    for doc_id, text in docs:
        for word in text.split():
            pairs.append((word, doc_id))
    groups = {}
    for word, doc_id in pairs:
        if word not in groups:
            groups[word] = []
        groups[word].append(doc_id)
    index = {}
    for word in sorted(groups):
        index[word] = sorted(set(groups[word]))
    return index

Why it works. Every word occurrence emits its document id; grouping collects all ids that mentioned each word; set deduplicates a word repeated within one document, and sorted gives a stable id list. Iterating keys in sorted order makes the index itself deterministic.

Complexity. Time O(PlogP) for P word occurrences (dominated by the per-key sorts), Space O(P). Each occurrence is emitted and grouped once, then each bag is sorted.

assert inverted_index([(1, 'map reduce'), (2, 'reduce data'), (3, 'map data map')]) == \
    {'data': [2, 3], 'map': [1, 3], 'reduce': [1, 2]}
assert inverted_index([(7, 'solo')]) == {'solo': [7]}
assert inverted_index([]) == {}
assert inverted_index([(5, 'a a a')]) == {'a': [5]}

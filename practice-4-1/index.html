4.1 · Data and Processing Pipelines — Coding Practice

◷ approx. 9 min read · 30 worked examples

These problems build the core data-processing habit from this section: treating a list of records as a stream and moving it through small pipeline steps — filter the records that matter, transform each one to the field you want, and accumulate a summary (count, sum, average, tally, or per-group best).

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Count Matching Records

A list of room-entry events is given, where each event is a dictionary with a "time" and a "kind". Return how many events have a kind equal to the given kind. This is the simplest pipeline: read each record, test it, and accumulate a count.

Examples

Input:  count_kind([{"time": "9:00", "kind": "student"}, {"time": "9:01", "kind": "teacher"}, {"time": "9:03", "kind": "student"}], "student")
Output: 2
Explanation: two of the three records have kind "student".

Input:  count_kind([{"time": "9:00", "kind": "student"}], "visitor")
Output: 0
Explanation: no record matches "visitor".

Input:  count_kind([], "student")
Output: 0
Explanation: an empty stream produces a count of 0.

Constraints

  • events is a list of dictionaries, each with at least a "kind" key (possibly empty).
  • Walk the list once with a for loop and keep one running counter.

Starter

def count_kind(events, kind):
    pass
Hint

Start a counter at 0, look up event["kind"] for each record, and add 1 only when it equals kind.

  • Build a fresh list; do not mutate the input records.

Solutions

1. Count Matching Records

Approach — filter-and-count. This is the smallest pipeline from the section: read each record one at a time, test its "kind", and accumulate a count. Nothing more than a counter is held in memory.

def count_kind(events, kind):
    total = 0
    for event in events:
        if event["kind"] == kind:
            total = total + 1
    return total

Why it works. The loop visits every record exactly once and increments total only on the records that pass the test, so the final value is the number of matches.

Complexity. Time O(n), Space O(1). One pass over n records, with a single running counter.

assert count_kind([{"time": "9:00", "kind": "student"},
                   {"time": "9:01", "kind": "teacher"},
                   {"time": "9:03", "kind": "student"}], "student") == 2
assert count_kind([{"time": "9:00", "kind": "student"}], "visitor") == 0
assert count_kind([], "student") == 0

2. Extract One Field

Approach — transform stage. Reduce each full record to a single field. This is the "transform" station of a pipeline: same number of items out as in, but each item is smaller.

def field_values(records, key):
    values = []
    for record in records:
        values.append(record[key])
    return values

Why it works. Looking up record[key] selects one field from each dictionary; appending each in turn preserves order and builds a fresh list without touching the inputs.

Complexity. Time O(n), Space O(n). One pass, and the result holds one value per record.

assert field_values([{"name": "amp", "watts": 30},
                     {"name": "fan", "watts": 55}], "name") == ["amp", "fan"]
assert field_values([{"name": "amp", "watts": 30},
                     {"name": "fan", "watts": 55}], "watts") == [30, 55]
assert field_values([], "name") == []

3. Filter by Threshold

Approach — filter stage. Keep only the items that pass a test and let the rest fall away, exactly like the worker who pulls boxes off the belt.

def above(readings, limit):
    kept = []
    for reading in readings:
        if reading > limit:
            kept.append(reading)
    return kept

Why it works. Each reading is appended only when reading > limit, so the result contains exactly the surviving readings in their original order.

Complexity. Time O(n), Space O(n). One pass; the kept list is at most as long as the input.

assert above([12, 4, 19, 7, 25], 10) == [12, 19, 25]
assert above([1, 2, 3], 10) == []
assert above([], 0) == []

4. Scale a Column

Approach — map with a comprehension. Applying one rule to every value is a map. A list comprehension expresses "for each price, produce price * factor" directly.

def scale(prices, factor):
    return [price * factor for price in prices]

Why it works. The comprehension walks the list and emits the transformed value for each element, building a new list of the same length.

Complexity. Time O(n), Space O(n). One value computed and stored per input element.

assert scale([2, 5, 10], 3) == [6, 15, 30]
assert scale([100], 0) == [0]
assert scale([], 2) == []

5. Filter-then-Transform Pipeline

Approach — two stages in one pass. This is the section's headline pipeline: filter to the student events, then transform each to its time. Doing both inside one loop avoids building an intermediate list of full records.

def student_times(events):
    times = []
    for event in events:
        if event["kind"] == "student":
            times.append(event["time"])
    return times

Why it works. The if performs the filter; the append(event["time"]) performs the transform, but only on records that pass. The order of survivors is preserved because the loop reads left to right.

Complexity. Time O(n), Space O(k) for the k student events kept. One pass, no intermediate record list.

assert student_times([{"time": "10:00", "kind": "teacher"},
                      {"time": "10:02", "kind": "student"},
                      {"time": "10:05", "kind": "visitor"},
                      {"time": "10:06", "kind": "student"}]) == ["10:02", "10:06"]
assert student_times([{"time": "9:00", "kind": "visitor"}]) == []
assert student_times([]) == []

6. Sum a Field Over Matching Records

Approach — filter-and-accumulate. Same shape as counting, but the accumulator grows by a record's "duration" instead of by 1 — the aggregate station of the pipeline.

def total_duration(events, kind):
    total = 0
    for event in events:
        if event["kind"] == kind:
            total = total + event["duration"]
    return total

Why it works. total starts at 0 and absorbs the duration of each matching event; non-matching events leave it unchanged, so the result is the summed duration over the filtered stream.

Complexity. Time O(n), Space O(1). One pass with a single running sum.

assert total_duration([{"kind": "call", "duration": 3},
                       {"kind": "move", "duration": 1},
                       {"kind": "call", "duration": 5}], "call") == 8
assert total_duration([{"kind": "call", "duration": 3},
                       {"kind": "move", "duration": 1}], "move") == 1
assert total_duration([], "call") == 0

7. Tally by Category

Approach — dictionary as accumulator. To summarize how often each value occurs, carry a dictionary that maps each seen value to its running count.

def tally(items):
    counts = {}
    for item in items:
        if item in counts:
            counts[item] = counts[item] + 1
        else:
            counts[item] = 1
    return counts

Why it works. The first time an item appears, its count is set to 1; every later appearance bumps the existing count. After one pass each key holds its total occurrences.

Complexity. Time O(n), Space O(d) for d distinct values. Dictionary lookups and updates are O(1) on average.

assert tally(["student", "teacher", "student", "visitor", "student"]) == {
    "student": 3, "teacher": 1, "visitor": 1}
assert tally(["a"]) == {"a": 1}
assert tally([]) == {}

8. Three-Stage Aggregate Pipeline

Approach — filter twice, then aggregate. Collect the totals of orders that pass both the validity and the size test, then average that collected list — guarding the empty case so there is no division by zero.

def average_large_order(orders):
    totals = []
    for order in orders:
        if order["valid"]:
            if order["total"] >= 100:
                totals.append(order["total"])
    if len(totals) == 0:
        return 0
    return sum(totals) / len(totals)

Why it works. The nested ifs keep only valid orders of at least 100, so totals holds exactly the qualifying amounts. The guard returns 0 when nothing qualifies; otherwise sum / len is their average.

Complexity. Time O(n), Space O(k) for the k qualifying totals. One pass to collect, one to sum.

assert average_large_order([{"total": 150, "valid": True},
                            {"total": 40, "valid": True},
                            {"total": 250, "valid": True},
                            {"total": 999, "valid": False}]) == 200.0
assert average_large_order([{"total": 100, "valid": True}]) == 100.0
assert average_large_order([{"total": 50, "valid": True},
                            {"total": 80, "valid": True}]) == 0

9. Group Field Values by Key

Approach — dictionary of lists. Route each record into a bucket named by its group_key, appending its value_key to that bucket's list. This is the grouping pattern that powers report-style summaries.

def group_by(records, group_key, value_key):
    groups = {}
    for record in records:
        bucket = record[group_key]
        value = record[value_key]
        if bucket in groups:
            groups[bucket].append(value)
        else:
            groups[bucket] = [value]
    return groups

Why it works. A new bucket starts as a one-element list; existing buckets are appended to. Because the loop reads records in order, each group's list preserves arrival order.

Complexity. Time O(n), Space O(n). Each record contributes exactly one value across all buckets.

assert group_by([{"dept": "eng", "name": "ada"},
                 {"dept": "art", "name": "leo"},
                 {"dept": "eng", "name": "grace"},
                 {"dept": "eng", "name": "alan"}],
                "dept", "name") == {"eng": ["ada", "grace", "alan"],
                                    "art": ["leo"]}
assert group_by([{"dept": "eng", "name": "ada"}], "dept", "name") == {
    "eng": ["ada"]}
assert group_by([], "dept", "name") == {}

10. Top Aggregate per Group

Approach — group then reduce. Carry a dictionary from team to its best score so far; each record either starts a new team or improves the stored maximum. Finally turn the dictionary into a sorted list of pairs.

def best_per_group(records):
    best = {}
    for record in records:
        team = record["team"]
        score = record["score"]
        if team not in best or score > best[team]:
            best[team] = score
    pairs = []
    for team in best:
        pairs.append((team, best[team]))
    return sorted(pairs)

Why it works. The or puts the new-team case first, so an absent team is added immediately; otherwise the stored score is replaced only when the current score is larger. After the pass each team maps to its maximum, and sorted orders the pairs by team name.

Complexity. Time O(n+glogg) for g groups, Space O(g). One pass to aggregate, then a sort of the g pairs.

assert best_per_group([{"team": "red", "score": 7},
                       {"team": "blue", "score": 4},
                       {"team": "red", "score": 9},
                       {"team": "blue", "score": 4},
                       {"team": "green", "score": 12}]) == [
    ("blue", 4), ("green", 12), ("red", 9)]
assert best_per_group([{"team": "solo", "score": 3}]) == [("solo", 3)]
assert best_per_group([]) == []

1.6 · Higher-Order Functions — Coding Practice

◷ approx. 11 min read · 30 worked examples

These problems drill the core moves of Section 1.6: passing functions as arguments, the summation/general-method pattern, nested definitions, functions that return functions, closures, lambda, currying, and decorators.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Sum of Transformed Terms

Write sum_terms(n, term) that adds up term(1) + term(2) + ... + term(n), where term is a function applied to each counter value. This is the summation machine from the lesson: one fixed loop with a swappable term head. If n is 0, the sum is 0.

Examples

Input:  sum_terms(3, lambda k: k * k)
Output: 14
Explanation: 1 + 4 + 9 = 14.
Input:  sum_terms(4, lambda k: k)
Output: 10
Explanation: 1 + 2 + 3 + 4 = 10 (identity as the term).
Input:  sum_terms(5, lambda k: 2 * k)
Output: 30
Explanation: 2 + 4 + 6 + 8 + 10 = 30.

Constraints

  • n is an integer with n >= 0.
  • term is a one-argument function from an integer to a number.
  • Use term as a higher-order argument; do not hard-code any specific term rule.

Starter

def sum_terms(n, term):
    pass
Hint

Keep the loop skeleton fixed (total starts at 0, k counts from 1) and let term(k) decide each value you add.

Hard

9. Iterative Improvement to a Fixed Point

Build a general improver, then use it. First write iter_improve(update, done, guess=1) that repeatedly replaces guess with update(guess) until done(guess) is true, then returns guess. Provide approx_eq(x, y, tolerance=1e-9) for closeness tests. Then write sqrt_two(), which uses iter_improve with the averaging update x -> (x + 2/x)/2 and a done test that checks x * x is approximately 2. Also write fixed_point(f, start=1.0) that finds an x with f(x) approximately equal to x.

Examples

Input:  sqrt_two()
Output: 1.4142135623746899
Explanation: iterating the averaging update from guess 1 converges to the square root of 2.
Input:  fixed_point(lambda x: (x + 4 / x) / 2)
Output: 2.0
Explanation: 2 is the value left unchanged by averaging x with 4/x, i.e. the square root of 4.
Input:  fixed_point(lambda x: 1 + 1 / x)
Output: 1.6180339882053252
Explanation: the golden ratio is the fixed point of x -> 1 + 1/x.

Constraints

  • update, done, and f are one-argument functions.
  • Use approx_eq (not ==) for floating-point closeness; exact equality may loop forever.
  • sqrt_two and fixed_point must define their update/done helpers as nested functions and delegate to iter_improve.

10. Compose Functions Given by Index

Write pipe(get, n) that returns a function. get is a function where get(k) returns the function to apply at step k. Calling the returned function on x should apply get(0), then get(1), on up through get(n-1), threading the result through each. If n is 0, the returned function leaves its input unchanged (no functions applied).

Examples

Input:  pipe(lambda k: (lambda x: x + k), 3)(10)
Output: 13
Explanation: applies +0, then +1, then +2 in order: 10 + 0 + 1 + 2 = 13.
Input:  pipe(lambda k: (lambda x: x + k), 0)(10)
Output: 10
Explanation: with n = 0 no functions are applied, so the input is returned unchanged.
Input:  pipe(alt, 3)(3)   where alt(k) squares when k is even and increments when odd
Output: 100
Explanation: square -> 9, increment -> 10, square -> 100.

Constraints

  • get is a function from an index to a one-argument function; n is an integer with n >= 0.
  • pipe must return a function (a closure over get and n).
  • Apply the functions strictly in index order 0, 1, ..., n-1.

Starter

def pipe(get, n):
    pass
Hint

Inside the returned function, loop k from 0 to n-1, fetch f = get(k), and replace the running result with f(result).

Solutions

1. Sum of Transformed Terms

Approach — the summation machine. This is the summation pattern from 1.6-A: a fixed loop skeleton with the per-term rule supplied as a higher-order argument term. The loop never knows which rule it holds; it just calls term(k) each pass.

def sum_terms(n, term):
    total, k = 0, 1
    while k <= n:
        total, k = total + term(k), k + 1
    return total

Why it works. total starts at 0 and k counts from 1 to n; each pass adds term(k), so the result is term(1) + ... + term(n). When n is 0 the loop never runs and total stays 0.

Complexity. Time O(n), Space O(1). One pass over the counter, plus a constant number of local names.

assert sum_terms(3, lambda k: k * k) == 14
assert sum_terms(4, lambda k: k) == 10
assert sum_terms(5, lambda k: 2 * k) == 30
assert sum_terms(0, lambda k: k) == 0

2. Apply Twice

Approach — functions as arguments. f arrives as a first-class function value. We call it, feed its result back into it, and return that. The whole point is that apply_twice never knows what f does.

def apply_twice(f, x):
    return f(f(x))

Why it works. The inner f(x) is evaluated first, producing a value; the outer f(...) is then called on that value, giving f applied twice.

Complexity. Time O(1) calls to f (two of them), Space O(1). The cost beyond f itself is constant.

assert apply_twice(lambda x: x + 3, 10) == 16
assert apply_twice(lambda x: x * x, 2) == 16
assert apply_twice(lambda x: x - 1, 5) == 3

3. Make Multiplier

Approach — returned function / closure. Following make_adder from 1.6-B, we define an inner function that references the outer parameter n and return it. The returned function remembers n through its parent frame.

def make_multiplier(n):
    def multiplier(k):
        return k * n
    return multiplier

Why it works. When make_multiplier(3) runs, it builds a multiplier whose parent frame binds n -> 3. Calling that function later looks up n in the remembered frame, so multiplier(5) computes 5 * 3.

Complexity. Time O(1), Space O(1). Building and calling the closure are constant-time.

triple = make_multiplier(3)
assert triple(5) == 15
assert make_multiplier(4)(6) == 24
assert make_multiplier(0)(99) == 0
assert make_multiplier(-2)(7) == -14

4. Count Hits

Approach — predicate as argument. A close cousin of summation: instead of adding term(k), we add 1 whenever the higher-order predicate pred(k) is true. The decision of what counts as a hit lives entirely in pred.

def count_hits(n, pred):
    total, k = 0, 1
    while k <= n:
        if pred(k):
            total = total + 1
        k = k + 1
    return total

Why it works. Each integer from 1 to n is tested once; total rises by 1 exactly for the values where pred returns true.

Complexity. Time O(n), Space O(1). One pass, constant extra storage.

assert count_hits(10, lambda k: k % 2 == 0) == 5
assert count_hits(10, lambda k: k % 3 == 0) == 3
assert count_hits(5, lambda k: k > 10) == 0
assert count_hits(7, lambda k: True) == 7

5. Product of Terms

Approach — the general-method sibling of summation. Same skeleton as summation, but the accumulator starts at 1 and combines with *. factorial is then a one-liner that passes the identity term, exactly as sum_naturals passed identity to summation.

def product(n, term):
    total, k = 1, 1
    while k <= n:
        total, k = total * term(k), k + 1
    return total

def factorial(n):
    return product(n, lambda k: k)

Why it works. Starting total at 1 is the multiplicative identity, so an empty product (n == 0) is 1; each pass multiplies in term(k). factorial(n) multiplies 1 * 2 * ... * n.

Complexity. Time O(n), Space O(1). One pass over the counter.

assert product(4, lambda k: k) == 24
assert product(3, lambda k: 2) == 8
assert product(5, lambda k: k * k) == 14400
assert factorial(0) == 1
assert factorial(5) == 120

6. Repeated Application

Approach — returned function that loops. repeated is a higher-order function that both takes a function (f) and returns one. The returned closure remembers f and n, and when called it applies f to its input n times.

def repeated(f, n):
    def repeat(x):
        result, k = x, 0
        while k < n:
            result, k = f(result), k + 1
        return result
    return repeat

Why it works. Inside repeat, result starts as the input and is overwritten with f(result) on each of n passes. When n is 0 the loop never runs, so the input is returned unchanged.

Complexity. Time O(n) calls to f, Space O(1). The loop runs n times with constant extra storage.

increment = lambda x: x + 1
assert repeated(increment, 5)(0) == 5
assert repeated(lambda x: x * 2, 3)(1) == 8
assert repeated(lambda x: x * x, 2)(3) == 81
assert repeated(increment, 0)(42) == 42

7. Curried Add

Approach — currying by hand. Like curried_pow from 1.6-C, the outer function captures the first argument x and returns an inner function still waiting for y. Two single-argument calls placed next to each other do the work of one two-argument call.

def curried_add(x):
    def adder(y):
        return x + y
    return adder

Why it works. curried_add(3) returns an adder whose parent frame remembers x -> 3; calling that with 4 runs 3 + 4. The first pair of parentheses produces a function; the second pair calls it.

Complexity. Time O(1), Space O(1). Both calls are constant-time.

assert curried_add(3)(4) == 7
assert curried_add(10)(-2) == 8
add_five = curried_add(5)
assert add_five(0) == 5
assert add_five(100) == 105

8. Counting Decorator

Approach — decorator with closure state. Like trace from 1.6-C, count_calls takes a function and returns a wrapped replacement. The wrapper adds bookkeeping (a call tally) and still delegates to the original with return f(x). The count lives in a dict in the enclosing frame so the inner function can mutate it without nonlocal.

def count_calls(f):
    state = {'count': 0}
    def wrapped(x):
        state['count'] = state['count'] + 1
        return f(x)
    wrapped.calls = state
    return wrapped

Why it works. wrapped closes over state; each call bumps state['count'] and then returns the original f(x), so results are preserved while the tally accumulates. Exposing wrapped.calls lets a caller read the count back.

Complexity. Time O(1) overhead per call (plus f's own cost), Space O(1). One dict held in the closure.

@count_calls
def square(x):
    return x * x

assert square(3) == 9
assert square(4) == 16
assert square.calls['count'] == 2
assert square(5) == 25
assert square.calls['count'] == 3

9. Iterative Improvement to a Fixed Point

Approach — the general improve method. iter_improve is the improve general method from 1.6-A/B: refine a guess with update until done says it is good enough. sqrt_two and fixed_point fill the two slots with nested helpers (closures), so the loop machinery never mentions the specific problem.

Why it works. Each update contracts the guess toward the target; done uses approx_eq (not ==) so floating-point rounding does not trap the loop. For fixed_point, the update is f itself and done checks whether f(x) already equals x within tolerance.

Complexity. Time O(i) where i is the number of iterations to reach the tolerance, Space O(1). Each pass does constant work; the iteration count depends on how fast the update converges.

assert approx_eq(sqrt_two(), 1.4142135623730951)
assert approx_eq(fixed_point(lambda x: (x + 4 / x) / 2), 2.0)
assert approx_eq(fixed_point(lambda x: 1 + 1 / x), 1.618033988749895)

10. Compose Functions Given by Index

Approach — first-class functions threaded in order. pipe returns a closure over get and n. When called, it fetches each step's function with get(k) and threads the running result through them in index order — a generalization of compose1 to a sequence of functions selected by index.

def pipe(get, n):
    def piped(x):
        result, k = x, 0
        while k < n:
            f = get(k)
            result, k = f(result), k + 1
        return result
    return piped

Why it works. result starts as the input; pass k replaces it with get(k) applied to it, for k from 0 up to n-1. With n == 0 the loop never runs, so the input passes through unchanged.

Complexity. Time O(n) calls to the fetched functions, Space O(1). One pass with constant extra storage (beyond whatever get and the step functions cost).

adders = lambda k: (lambda x: x + k)
assert pipe(adders, 3)(10) == 13
assert pipe(adders, 1)(10) == 10
assert pipe(adders, 0)(10) == 10

def alt(k):
    if k % 2 == 0:
        return lambda x: x * x
    return lambda x: x + 1

assert pipe(alt, 3)(3) == 100

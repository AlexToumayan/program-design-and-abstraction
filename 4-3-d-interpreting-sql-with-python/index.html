Chapter 4/4.3 Declarative Programming

4.3.4 Interpreting SQL with Python

◷ approx. 14 min read · 9 worked examples

By the end of this lesson you will be able to read a small Python program that behaves like a SQL interpreter, and explain why such a program is possible at all. You will see five ideas, each one a concrete piece of Python: a database row modeled as a Python object, projection modeled as map, a parsed select statement modeled as a Select object, the construction of output rows from a column description, and a join modeled as the product of rows from each table.

The reason this matters is bigger than SQL. SQL is a language, which means SQL text has to be read and carried out by some other program before it can produce a table. That other program is an interpreter, and an interpreter is itself just code. In this lesson you write (and read) that code.

SQL Is Text That Something Has to Run

When you type a SQL query, nothing happens by magic. The query is a string of characters. Some program has to look at those characters, work out what they ask for, and produce the rows. You already met this shape in Chapter 3, where you studied interpreters for a small programming language:

program text  ->  interpreter  ->  result

For SQL the same shape holds, with table results at the end instead of numbers:

SQL query text  ->  SQL interpreter  ->  table result

The powerful idea is that the interpreter in the middle is also a program. In this lesson we build a small but real Python model of a SQL interpreter. To do that we need three parts: a way to represent tables, a parser that turns query text into structured data, and an evaluator that runs the parsed query. This lesson focuses on the representation and the evaluation; the parser is mentioned but lives in the full example program.

A useful real-world analogy: think of a restaurant. The SQL query is the customer's order written on a ticket ("two eggs, no toast, side of fruit"). The order describes the result the customer wants, not the steps to make it. The kitchen is the interpreter. The cooks read the ticket, decide a procedure, and hand back a plate. Two kitchens can fulfill the same ticket with different procedures, and the customer does not care which, as long as the plate is right. Everything in this lesson is the kitchen.

A Database Row as a Python Object

In this model, every table has its own class, and each row of the table is an instance of that class. A row has one attribute per column, and a table is simply a sequence (a list) of rows.

table  ->  list of rows
row    ->  object with one attribute per column

To make a row class quickly, the source uses the namedtuple function from the collections package of the Python standard library. A namedtuple returns a new subclass of tuple that also gives a name to each position. That means a row behaves like a tuple, but you can also read its fields by name.

Recall the cities table from the previous section. In SQL it was created like this:

create table cities as
   select 38 as latitude, 122 as longitude, "Berkeley" as name union
   select 42,             71,               "Cambridge"        union
   select 45,             93,               "Minneapolis";

The following Python statements construct a representation for this table.

# (1)
>>> from collections import namedtuple
>>> CitiesRow = namedtuple("Row", ["latitude", "longitude", "name"])
>>> cities = [CitiesRow(38, 122, "Berkeley"),
              CitiesRow(42,  71, "Cambridge"),
              CitiesRow(43,  93, "Minneapolis")]

Read the call that builds the row class one token at a time. This is the central new construct, so slow down on it.

CitiesRow = namedtuple("Row", ["latitude", "longitude", "name"])
├─ CitiesRow ............ the name we bind the new row class to
├─ = ................... assignment: bind the class to that name
├─ namedtuple .......... the factory that builds a tuple subclass
├─ ( ................... begins the arguments to namedtuple
├─ "Row" ............... the type name stored on the class (shows up in printing)
├─ , ................... separates the two arguments
├─ ["latitude", "longitude", "name"] ... the field names, one per column
├─ ) ................... ends the arguments
└─ builds a class whose instances are tuples with named fields latitude, longitude, name

Now read one row construction the same way.

CitiesRow(38, 122, "Berkeley")
├─ CitiesRow ........... the row class we just built
├─ ( ................... begins the field values
├─ 38 ................. the latitude field
├─ , ................... separates fields
├─ 122 ................ the longitude field
├─ , ................... separates fields
├─ "Berkeley" ......... the name field
├─ ) ................... ends the field values
└─ evaluates to a row that is both a 3-tuple (38, 122, "Berkeley") and an object with .latitude, .longitude, .name

A note on parity that is easy to miss but worth stating plainly. The SQL create table cities above lists Minneapolis with latitude 45. The Python representation lists Minneapolis with latitude 43. That difference is in the original source, not a typo here, and it is why later the Python trace prints a distance of 300 for Minneapolis while the SQL transcript prints 420. The teaching point is the interpreter's structure, not the geography, so do not let the number mismatch distract you.

Projection as map

A select statement that computes new columns from old ones is called a projection: it produces a new view of each row. Recall the distances table, which is built from the name and latitude columns of cities.

sqlite> create table distances as
   ...>   select name, 60*abs(latitude-38) as distance from cities;
sqlite> select distance/5, name from distances;
0|Berkeley
48|Cambridge
84|Minneapolis

In Python, producing one output row for each input row is exactly what map does. We write a function that turns a CitiesRow into a DistancesRow, then map it over the table.

Here is an easier warm-up first, so the shape is clear before the official names appear. Given a list of pairs, map a function that keeps the name and computes a number from the other field:

# (2)
>>> pairs = [(38, "Berkeley"), (42, "Cambridge"), (43, "Minneapolis")]
>>> def shrink(pair):
        latitude, name = pair
        return (name, 60 * abs(latitude - 38))
>>> list(map(shrink, pairs))
[('Berkeley', 0), ('Cambridge', 240), ('Minneapolis', 300)]

That is the whole idea: one function, applied to every input row, yields every output row. Now the official version, which uses a named output row class so each result prints with its column names.

# (3)
>>> DistancesRow = namedtuple("Row", ["name", "distance"])
>>> def select(cities_row):
        latitude, longitude, name = cities_row
        return DistancesRow(name, 60*abs(latitude-38))
>>> distances = list(map(select, cities))
>>> for row in distances:
        print(row)
Row(name='Berkeley', distance=0)
Row(name='Cambridge', distance=240)
Row(name='Minneapolis', distance=300)

The Python function named select is not SQL. It is the Python expression of the projection: an input row goes in, an output row comes out. The first line of the body unpacks the three fields of the incoming row, and the body returns a fresh DistancesRow built from name and the computed distance. Mapping that function over cities reproduces the distances table as a list of rows.

Representing a Parsed select Statement

An interpreter cannot work on raw query text directly; it works on structured data that stands for the query. The design generalizes the projection above. A select statement is represented as an instance of a class named Select, built from the clauses of the query. It calls two helper functions, create_make_row and join; both are defined in the sections that follow, so it is fine to take them on faith for now.

# (4)
>>> class Select:
        """select [columns] from [tables] where [condition] order by [order]."""
        def __init__(self, columns, tables, condition, order):
            self.columns = columns
            self.tables = tables
            self.condition = condition
            self.order = order
            self.make_row = create_make_row(self.columns)
        def execute(self, env):
            """Join, filter, sort, and map rows from tables to columns."""
            from_rows = join(self.tables, env)
            filtered_rows = filter(self.filter, from_rows)
            ordered_rows = self.sort(filtered_rows)
            return map(self.make_row, ordered_rows)
        def filter(self, row):
            if self.condition:
                return eval(self.condition, row)
            else:
                return True
        def sort(self, rows):
            if self.order:
                return sorted(rows, key=lambda r: eval(self.order, r))
            else:
                return rows

Two helpers used here, create_make_row and join, plus the two-argument form of eval (run a string of code, looking names up in the dictionary you pass as the second argument), are all explained in the sections just below; read this class for its shape first, then fill in the pieces.

The heart of this class is execute. It reads top to bottom as a clean pipeline, and each stage names a SQL clause you already know.

from_rows = join(self.tables, env)
├─ join ................ build every combination of rows from the listed tables
├─ self.tables ......... the comma-separated table names, e.g. "cities, distances"
├─ env ................. the dictionary that maps table names to their lists of rows
└─ produces the raw input rows: this is the SQL `from` clause
filtered_rows = filter(self.filter, from_rows)
├─ filter .............. keep only rows for which the predicate is true
├─ self.filter ......... the method that evaluates the condition on one row
├─ from_rows ........... every joined row from the previous stage
└─ keeps the surviving rows: this is the SQL `where` clause
ordered_rows = self.sort(filtered_rows)
├─ self.sort ........... sort the surviving rows by the order expression
├─ filtered_rows ....... the rows that passed the condition
└─ returns the rows in order: this is the SQL `order by` clause
return map(self.make_row, ordered_rows)
├─ map ................. apply make_row to every ordered row
├─ self.make_row ....... the projection function built from the column description
├─ ordered_rows ........ the sorted, filtered rows
└─ returns the final output rows: this is the SQL column projection (the `select` list)

So the whole interpreter pipeline is just four sequence operations in a fixed order:

join input tables  ->  filter rows  ->  sort rows  ->  project columns

The make_row function is created once, in the constructor, by calling create_make_row on the column description. The filter method evaluates the stored condition string against a row (and keeps the row when there is no condition); the sort method sorts by evaluating the stored order string against each row (and leaves the rows untouched when there is no order). The SQL text described the result; this code chooses one procedure for producing it.

Building Output Rows

The projection function is produced by create_make_row, a higher-order function. It receives a column description string and returns a function. That returned function maps an input environment dictionary to an output row. (A version with more error handling and special cases appears in the full sql program.)

# (5)
>>> def create_make_row(description):
        """Return a function from an input environment (dict) to an output row.
        description -- a comma-separated list of [expression] as [column name]
        """
        columns = description.split(", ")
        expressions, names = [], []
        for column in columns:
            if " as " in column:
                expression, name = column.split(" as ")
            else:
                expression, name = column, column
            expressions.append(expression)
            names.append(name)
        row = namedtuple("Row", names)
        return lambda env: row(*[eval(e, env) for e in expressions])

The function builds two parallel lists as it scans the description. The list expressions holds the strings to compute, and the list names holds the output column names. When a column contains the word as, the left side is the expression and the right side is the name; otherwise the column text serves as both. Then it makes a row class from the names and returns a function that, given an environment, evaluates each expression and packs the results into a row.

Read the returned function's body one token at a time, because the eval is the key to how a string column like "60*abs(latitude-38) as distance" becomes a number.

lambda env: row(*[eval(e, env) for e in expressions])
├─ lambda env: ........ a function taking one argument, the row environment dict
├─ row ................ the namedtuple class built from the column names
├─ ( .................. begins the field values for the new row
├─ * .................. unpacks the list that follows into separate arguments
├─ [eval(e, env) for e in expressions] ... evaluate every column expression in this environment
├─ ) .................. ends the field values
└─ returns one output row whose fields are the evaluated column expressions

The call eval(e, env) means: evaluate the expression string e using env as the namespace, so a name like latitude inside the string is looked up in the row's environment. This is a deliberate teaching simplification. Real SQL interpreters do not call Python's eval on query fragments; doing so on untrusted input would be unsafe. Here it keeps the model short.

Joining Tables as a Product of Rows

Finally we need join, which creates the input rows for the pipeline. Given an env dictionary that holds the existing tables (lists of rows) keyed by name, join groups together every combination of rows from the listed tables using the product function from the itertools package. It maps a function named make_env over those combinations, turning each combination into a dictionary so expressions can be evaluated against it. (A version with more error handling and special cases appears in the full sql program.)

# (6)
>>> from itertools import product
>>> def join(tables, env):
        """Return an iterator over dictionaries from names to values in a row.
        tables -- a comma-separate sequences of table names
        env    -- a dictionary from global names to tables
        """
        names = tables.split(", ")
        joined_rows = product(*[env[name] for name in names])
        return map(lambda rows: make_env(rows, names), joined_rows)
>>> def make_env(rows, names):
        """Create an environment of names bound to values."""
        env = dict(zip(names, rows))
        for row in rows:
            for name in row._fields:
                env[name] = getattr(row, name)
        return env

The decisive line inside join is the one that forms every combination of rows.

joined_rows = product(*[env[name] for name in names])
├─ product ............ itertools product: every combination, one row from each table
├─ * .................. unpacks the list of tables into separate arguments to product
├─ [env[name] for name in names] ... look up each named table's list of rows in env
└─ produces every combination of rows across the input tables (the raw join)

Then make_env turns one combination of rows into a dictionary. It starts by binding each table name to its row, then walks each row and binds every column name to that column's value. The expression row._fields evaluates to the column names of the table containing the row; the _fields attribute exists because the row's type is a namedtuple class. After make_env, an expression string can look up plain names like latitude, name, longitude, or qualified names like cities.name (because the table name is also a key).

Executing the Official Queries

The interpreter is now complete enough to run select statements. As a first example, compute the latitude distance from Berkeley for all other cities, ordered by their longitude.

# (7)
>>> env = {"cities": cities}
>>> select = Select("name, 60*abs(latitude-38) as distance",
                    "cities", "name != 'Berkeley'", "-longitude")
>>> for row in select.execute(env):
        print(row)
Row(name='Minneapolis', distance=300)
Row(name='Cambridge', distance=240)

Walk through what happened, stage by stage, so the pipeline becomes concrete:

  • join: with only one table, every "combination" is just one cities row, turned into an environment dictionary by make_env.
  • filter: the condition string "name != 'Berkeley'" is evaluated against each row's environment. Berkeley fails and is dropped; Cambridge and Minneapolis pass.
  • sort: the order string "-longitude" is evaluated per row, so rows are sorted by negative longitude. Minneapolis (longitude 93) comes before Cambridge (longitude 71).
  • project: make_row evaluates name and 60*abs(latitude-38) for each surviving row, producing the two output rows shown.

The example above is equivalent to the following SQL statement. Note again the 420 versus 300: the SQL transcript uses the earlier table where Minneapolis has latitude 45.

sqlite> select name, 60*abs(latitude-38) as distance
   ...>        from cities where name != "Berkeley" order by -longitude;
Minneapolis|420
Cambridge|240

We can also store this resulting table in the environment and join it with the cities table, retrieving the longitude for each city. This is where the join machinery earns its keep: two tables, matched on a shared name.

# (8)
>>> env["distances"] = list(select.execute(env))
>>> joined = Select("cities.name as name, distance, longitude", "cities, distances",
                    "cities.name == distances.name", None)
>>> for row in joined.execute(env):
        print(row)
Row(name='Cambridge', distance=240, longitude=71)
Row(name='Minneapolis', distance=300, longitude=93)

Here join forms every combination of a cities row with a distances row, make_env binds both cities.name and distances.name (along with the bare column names), and the condition "cities.name == distances.name" keeps only the combinations whose names agree. The projection then reads cities.name, distance, and longitude from each surviving environment.

The example above is equivalent to the following SQL statement.

sqlite> select cities.name as name, distance, longitude
   ...>        from cities, distances where cities.name = distances.name;
Cambridge|240|71
Minneapolis|420|93

The full sql example program also contains a simple parser for select statements, plus execute methods for create table and union. That interpreter can correctly execute every SQL statement shown in the text so far. This model implements only a small slice of the full Structured Query Language, but its structure makes the relationship between sequence-processing operations and query languages clear: projection is map, filtering is filter, ordering is sorted, and joining is product.

Query Plans

Declarative languages describe the form of a result, but do not say how to compute it. This interpreter always follows one fixed plan:

join  ->  filter  ->  order  ->  project

That is a correct way to compute the result, but it is not the only way, and often not the fastest. A query interpreter is free to choose among procedures that produce the same rows. Choosing efficient procedures is a core feature of real database systems.

Consider the final select above. Rather than computing the full join of cities and distances and then filtering, a database could first sort both tables by the name column, then join only the rows that share a name in a single linear pass through the sorted tables. When tables are large, the efficiency gains from choosing a good query plan can be substantial. This is the deeper payoff of declarative programming: the programmer describes the result, and the interpreter is free to choose an efficient process.

⚠Common Beginner Traps

  • Thinking SQL needs no interpreter because it looks nothing like Python. SQL text still has to be read and run by a program; that program is the interpreter, and here it is Python.
  • Forgetting that a row environment in this model is just a dictionary mapping names to values. latitude inside a condition string is not magic; it is a key looked up in that dictionary.
  • Believing eval is a reasonable way to run real queries. Here it is a small educational shortcut; production interpreters never eval user-supplied query text.
  • Confusing the declarative query with the interpreter's chosen plan. The query says what rows to return; the plan is one procedure for returning them, and a different plan can return the same rows faster.
  • Expecting the Python distance for Minneapolis to be 420. The Python cities list uses latitude 43, so the Python trace prints 300; only the SQL transcript, built from the earlier 45 table, prints 420.

Sort orders those by latitude ascending, so Cambridge (42) precedes Minneapolis (43). Project reads name and longitude, giving Row(name='Cambridge', longitude=71) then Row(name='Minneapolis', longitude=93).
  6. Because SQL describes only the form of the result, not the procedure. Any plan that returns the same rows is valid, so the interpreter may reorder or optimize the work, for example by sorting both tables and matching names in a single linear pass.

§Summary

A SQL interpreter can be modeled in Python out of four ordinary ingredients: rows as namedtuple objects, parsed queries as Select objects, environments as dictionaries, and sequence operations as the engine. Projection is map, filtering is filter, ordering is sorted, and joining is itertools.product. The Select.execute method strings these together in a fixed pipeline that joins, filters, sorts, and projects. The teaching create_make_row and join helpers show how column descriptions and table names become real rows, using eval against a row environment as a deliberate shortcut. The lasting lesson is the separation at the center of declarative programming: a query describes the result, and the interpreter is free to choose how to compute it, which is exactly what lets real databases pick efficient query plans.

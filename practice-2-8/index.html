Setup

from math import sqrt

1. Count the Calls

Approach — the count wrapper from 2.8-A. count is a higher-order function that returns a counted wrapper sharing one call_count attribute on the wrapper object. Because the count lives on the function object, every call to that wrapper bumps the same number; a single top-level call therefore reads back as exactly 1, no matter how much looping collatz_steps does inside.

def count(f):
    def counted(*args):
        counted.call_count += 1
        return f(*args)
    counted.call_count = 0
    return counted

def collatz_steps(n):
    steps = 0
    while n != 1:
        if n % 2 == 0:
            n = n // 2
        else:
            n = 3 * n + 1
        steps += 1
    return steps

def count_collatz_calls(n):
    counted = count(collatz_steps)
    counted(n)
    return counted.call_count

Why it works. counted.call_count starts at 0 and rises by one each time the wrapper runs. count_collatz_calls wraps a fresh counter, calls it once, and reads the attribute back, so the result is the number of calls, never the number of internal loop iterations.

Complexity. Time O(s) where s is the number of internal Collatz steps for n (the wrapper itself adds O(1) overhead per call), Space O(1). The counter is a single shared attribute.

assert count_collatz_calls(1) == 1
assert count_collatz_calls(6) == 1
assert count_collatz_calls(27) == 1

2. Slow to Fast: Count Divisors

Approach — stop at the square root. Factors pair up so that one partner is below n​ and the other above it, so checking candidates only up to n​ and counting each pair with += 2 finds them all. This is the Θ(n​) trick of count_factors from 2.8-C, a dramatic improvement over the Θ(n) slow scan.

def count_divisors(n):
    sqrt_n = sqrt(n)
    k, divisors = 1, 0
    while k < sqrt_n:
        if n % k == 0:
            divisors += 2
        k += 1
    if k * k == n:
        divisors += 1
    return divisors

Why it works. Each k below n​ that divides n contributes both k and its partner n // k, hence += 2. A perfect square's middle factor is its own partner, so the loop stops just before it and the final if k * k == n counts it once.

Complexity. Time O(n​), Space O(1). The loop runs about n​ times instead of n times.

assert count_divisors(12) == 6
assert count_divisors(1) == 1
assert count_divisors(49) == 3
assert count_divisors(576) == 21
assert count_divisors(13) == 2

4. A Cache by Hand

Approach — the manual answer key from 2.8-B. Before the memo wrapper, the lesson builds a cache directly: a dict guarded by if n not in cache:. The dict lives in the frame created by make_cached_squarer, so it persists across calls to the returned cached_square and is shared by all of them.

def make_cached_squarer():
    cache = {}
    def cached_square(n):
        if n not in cache:
            cache[n] = n * n
        return cache[n]
    return cached_square, cache

Why it works. On a miss, the body computes n * n and stores it; on a hit, the body is skipped and the stored value is returned. Because cache belongs to the enclosing frame, it remembers across calls; a second factory call makes a brand-new, independent cache.

Complexity. Time O(1) per call (amortized, after the first compute), Space O(d) for d distinct inputs cached.

sq, store = make_cached_squarer()
assert sq(5) == 25
assert sq(5) == 25
assert store == {5: 25}
assert sq(3) == 9
assert store == {5: 25, 3: 9}
sq2, store2 = make_cached_squarer()
assert store2 == {}
assert sq2(7) == 49
assert store2 == {7: 49}
assert store == {5: 25, 3: 9}

5. Memoize a Tree-Recursive Function

Approach — the memo higher-order function from 2.8-B. memo packages the manual cache: it owns one cache dict and returns a memoized wrapper that computes f(n) only on a cache miss. Rebinding trib = memo(trib) means the recursive calls trib(n-1), trib(n-2), trib(n-3) inside the body all reach the cache, so each distinct input is computed exactly once and the exploding tree collapses.

Why it works. The cache born inside memo stays alive because memoized still refers to it. After the rebinding, the name trib is the wrapper, so the body's recursive calls are served from the cache; only the first request for each input falls through to the real computation.

Complexity. Time O(n) (each of inputs 0..n computed once, then constant-time reads), Space O(n) for the cache and recursion depth. Without memo the same trib is exponential.

assert trib(10) == 81
assert trib(0) == 0
assert trib(2) == 1
assert trib(15) == 1705
assert trib(30) == 15902591

6. Fast Exponentiation

Approach — repeated squaring from 2.8-D. Instead of n multiplications, build the power by doubling the exponent. An even n squares the half-power; an odd n peels off one b, turning the exponent even for the next step. Each step roughly halves the exponent, so the count of multiplications grows like logn.

def square(x):
    return x * x

def fast_power(b, n):
    if n == 0:
        return 1
    if n % 2 == 0:
        return square(fast_power(b, n // 2))
    else:
        return b * fast_power(b, n - 1)

Why it works. b2k=(bk)2 (the even case) and b2k+1=b⋅b2k (the odd case), with b0=1 as the base. Floor division // keeps the exponent an integer. The answer is identical to the slow method; only the path is shorter.

Complexity. Time O(logn) multiplications, Space O(logn) for the recursion depth. The exponent is cut down, not stepped down.

assert fast_power(2, 5) == 32
assert fast_power(2, 10) == 1024
assert fast_power(3, 0) == 1
assert fast_power(2, 100) == 1267650600228229401496703205376
assert fast_power(5, 3) == 125
for base in range(0, 6):
    for e in range(0, 13):
        assert fast_power(base, e) == base ** e

7. Count the Multiplications

Approach — measure the work, not just the answer (2.8-D). Thread a multiplication tally up through the same squaring recursion. Each even step squares (one multiply) and each odd step multiplies by b (one multiply), so the returned count is the exact number of multiplications, which stays logarithmic in n.

def power_mults(b, n):
    if n == 0:
        return 1, 0
    if n % 2 == 0:
        half, m = power_mults(b, n // 2)
        return half * half, m + 1          # one multiply to square
    else:
        rest, m = power_mults(b, n - 1)
        return b * rest, m + 1             # one multiply for the peeled factor

Why it works. The base case does zero multiplications. Each recursive step performs exactly one multiplication on the way back up and adds 1 to the count it received, so the total tracks the length of the halving-and-peeling chain, not the size of n.

Complexity. Time O(logn), Space O(logn). The chain of exponents has logarithmic length, so both the multiply count and the recursion depth are logarithmic.

assert power_mults(2, 10)[0] == 1024
assert power_mults(3, 4)[0] == 81
assert power_mults(2, 10)[1] == 5
assert power_mults(2, 8)[1] == 4
assert power_mults(2, 1)[1] == 1
assert power_mults(7, 0)[1] == 0
v, m = power_mults(2, 1000)
assert v == 2 ** 1000
assert m < 1000
assert m == 15

8. De-quadratic a Duplicate Finder

Approach — replace a nested scan with a dictionary (2.8-E). The slow version nests two loops, an Θ(n2) all-pairs comparison. Recording seen values in a dictionary turns the inner search into a constant-time in check, so one linear pass replaces the quadratic one: nesting is gone, and the product collapses to a sum.

def has_duplicate(values):
    seen = {}
    for v in values:
        if v in seen:
            return True
        seen[v] = True
    return False

Why it works. A duplicate exists exactly when some value is encountered while already in seen. Dictionary membership and insertion are O(1) on average, so the single pass detects the first repeat in linear time.

Complexity. Time O(n) average (versus O(n2) for the nested-loop version), Space O(n) for the dictionary of seen values.

assert has_duplicate([1, 2, 3, 4]) is False
assert has_duplicate([1, 2, 3, 2]) is True
assert has_duplicate([]) is False
assert has_duplicate([7]) is False
assert has_duplicate([5, 5]) is True

9. Memoize a Two-Argument Grid Walk

Approach — a tuple-keyed cache (2.8-B with two arguments). The recursion has two parameters, so the cache key is the immutable tuple (r, c) — exactly the case the lesson flags, where the key must be immutable. With each interior cell stored on first computation, the duplicated subtrees of the plain recursion collapse into single dictionary reads.

def grid_paths(rows, cols):
    cache = {}
    def paths(r, c):
        if r == 0 or c == 0:
            return 1
        key = (r, c)
        if key not in cache:
            cache[key] = paths(r - 1, c) + paths(r, c - 1)
        return cache[key]
    return paths(rows, cols)

Why it works. A boundary cell has exactly one path, and an interior cell's path count is the sum of the cell above and the cell to its left. Caching by (r, c) ensures each interior cell is expanded once; later requests for the same cell are constant-time reads.

Complexity. Time O(rows⋅cols) (each interior cell computed once), Space O(rows⋅cols) for the cache. The un-memoized version is exponential.

def grid_paths_slow(rows, cols):
    if rows == 0 or cols == 0:
        return 1
    return grid_paths_slow(rows - 1, cols) + grid_paths_slow(rows, cols - 1)

assert grid_paths(2, 2) == 6
assert grid_paths(2, 2) == grid_paths_slow(2, 2)
assert grid_paths(3, 3) == 20
assert grid_paths(0, 5) == 1
assert grid_paths(4, 4) == grid_paths_slow(4, 4)
assert grid_paths(1, 7) == 8

10. Report the Order of Growth

Approach — count the real steps for each shape (2.8-C/D/E). Rather than guess, run the described loop and tally its body executions, then attach the category label. The step counts make the separation concrete: at n = 10000 the sqrt process does 100 steps, linear does 10000, and quadratic does a hundred million.

def analyze(loop_kind, n):
    """Return (number_of_basic_steps, growth_label) for a described process.

    loop_kind is one of:
      'constant'  : a fixed amount of work, ignores n
      'sqrt'      : a loop that runs about floor(sqrt(n)) times
      'linear'    : a loop that runs n times
      'log'       : repeatedly halve n until it reaches 1 (about log2 n steps)
      'quadratic' : a loop of n, each pass doing n inner steps
    """
    if loop_kind == 'constant':
        return 1, 'constant'
    if loop_kind == 'sqrt':
        steps = 0
        k = 1
        while k * k <= n:
            steps += 1
            k += 1
        return steps, 'sqrt'
    if loop_kind == 'linear':
        steps = 0
        for _ in range(n):
            steps += 1
        return steps, 'linear'
    if loop_kind == 'log':
        steps = 0
        m = n
        while m > 1:
            m = m // 2
            steps += 1
        return steps, 'logarithmic'
    if loop_kind == 'quadratic':
        steps = 0
        i = 0
        while i < n:
            j = 0
            while j < n:
                steps += 1
                j += 1
            i += 1
        return steps, 'quadratic'

Why it works. The sqrt loop runs while k * k <= n, so k goes from 1 to floor(sqrt(n)); the log loop halves m until it hits 1, about log2 n times; the quadratic loop fires its inner body n * n times. Each counter is the exact step total for that shape, and the label names the order of growth.

Complexity. Time proportional to the steps it counts (O(1), O(n​), O(n), O(logn), or O(n2) depending on loop_kind), Space O(1). The counters use only a fixed number of names.

assert analyze('constant', 1000) == (1, 'constant')
assert analyze('sqrt', 100) == (10, 'sqrt')
assert analyze('sqrt', 99) == (9, 'sqrt')
assert analyze('linear', 50) == (50, 'linear')
assert analyze('log', 16) == (4, 'logarithmic')
assert analyze('log', 1024) == (10, 'logarithmic')
assert analyze('quadratic', 8) == (64, 'quadratic')
assert analyze('sqrt', 10000)[0] == 100
assert analyze('linear', 10000)[0] == 10000
assert analyze('log', 10000)[0] == 13

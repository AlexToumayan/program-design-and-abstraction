4.6 · Distributed Computing — Coding Practice

◷ approx. 13 min read · 30 worked examples

These problems build fluency with the deterministic Python models from this section: cutting a message into numbered packets and reassembling them the way IP and TCP do, parsing delimited and fixed-format messages against an agreed protocol, answering client requests over a server-state dict, and routing a message neighbor-to-neighbor across a peer network represented as a dict. No real sockets or threads — every model is plain Python you can run and trace.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Parse a Framed Packet

Before a packet travels, the sender wraps each chunk in a small header so the receiver can place it: a sequence number, the total packet count, and the chunk itself. Here that frame is a string "seq/total:payload" — the header is everything before the first :, and inside it the sequence number and total are separated by /. Write parse_packet(packet) that returns {"seq": ..., "total": ..., "payload": ...}, converting seq and total to integers and leaving payload as a string. The payload may be empty, and it may itself contain : characters.

Examples

Input:  parse_packet("2/5:hello")
Output: {'seq': 2, 'total': 5, 'payload': 'hello'}
Explanation: header "2/5" gives seq 2 of 5; everything after the first ':' is the payload.

Input:  parse_packet("3/4:")
Output: {'seq': 3, 'total': 4, 'payload': ''}
Explanation: nothing after the ':' means an empty payload.

Input:  parse_packet("0/2:time: 9:30")
Output: {'seq': 0, 'total': 2, 'payload': 'time: 9:30'}
Explanation: only the first ':' ends the header, so the payload keeps its own colons.

Constraints

  • packet is a string of the form "seq/total:payload" with one / in the header.
  • Split off the header on the first : only; the payload may contain more :.
  • Convert seq and total with int(...); keep payload a string.

5. Reassemble or Report the Gap

Before TCP can hand a stream to the application, it must do two things: put the out-of-order, possibly duplicated packets back in order, and confirm that none are missing. Write reassemble(packets) where packets is a list of (seq, payload) tuples — seq is a 0-based sequence number, and packets may arrive out of order and may be duplicated. If every sequence number from 0 to the highest one present is accounted for, return the payloads joined in sequence order (a duplicate carries the same payload, so collapse it). If any sequence number in that range is missing, the stream is incomplete: return the sorted list of the missing sequence numbers instead. An empty input returns the empty string.

Examples

Input:  reassemble([(2, "recv"), (0, "send"), (2, "recv"), (1, "and")])
Output: 'sendandrecv'
Explanation: seqs 0,1,2 are all present; order them, drop the duplicate 2, and join.

Input:  reassemble([(0, "send"), (2, "recv")])
Output: [1]
Explanation: the highest seq is 2 but seq 1 never arrived, so report the gap.

Input:  reassemble([(4, "d"), (0, "a"), (2, "c")])
Output: [1, 3]
Explanation: seqs 1 and 3 are both missing below the highest seq 4 — return them sorted.

Constraints

  • packets is a list of (seq, payload) tuples; seq is a non-negative int, payload a string.
  • Sequence numbers are 0-based; "complete" means every seq from 0 to max(seq) is present.
  • On a complete stream return a joined string; on a gap return the sorted list of missing seqs.

6. Parse a Fixed-Format Message

A fixed-format protocol marks fields by position, not by a delimiter. Each message is a list of byte values: byte 0 is the destination, byte 1 is the sender, byte 2 is a length field, and the next length bytes are the data. Any bytes past the data are ignored. Write parse_fixed(packet_bytes) returning {"to": ..., "from": ..., "data": [...]}.

Examples

Input:  parse_fixed([2, 1, 2, 72, 73])
Output: {'to': 2, 'from': 1, 'data': [72, 73]}
Explanation: length is 2, so the two data bytes 72 and 73 are read.

Input:  parse_fixed([5, 9, 0])
Output: {'to': 5, 'from': 9, 'data': []}
Explanation: a length of 0 means no data bytes follow.

Input:  parse_fixed([7, 3, 1, 65, 66, 67])
Output: {'to': 7, 'from': 3, 'data': [65]}
Explanation: length is 1, so only byte 65 is data; 66 and 67 are ignored.

Constraints

  • packet_bytes is a list of integers, length at least 3.
  • The length field at index 2 controls how many data bytes are read.
  • Use slicing for the data field; do not assume bytes past the data exist beyond the length.

Solutions

1. Parse a Framed Packet

Approach — split the header off, then split inside it. A framed packet agrees its layout in advance: a header ends at the first :, and within the header a / separates the sequence number from the total. Cut on the first : to separate header from payload, split the header on /, and convert the two numeric fields with int.

def parse_packet(packet):
    header, payload = packet.split(":", 1)
    seq_text, total_text = header.split("/")
    return {"seq": int(seq_text), "total": int(total_text), "payload": payload}

Why it works. Passing 1 as the second argument to split(":", 1) stops after the first colon, so a payload containing its own : stays intact. The header is then a clean "seq/total", which splits into exactly two parts; int turns each into a number while the payload remains the raw string — empty when nothing follows the colon.

Complexity. Time O(n), Space O(n) in the length of the packet string — a constant number of scans over it.

assert parse_packet("2/5:hello") == {"seq": 2, "total": 5, "payload": "hello"}
assert parse_packet("1/1:done") == {"seq": 1, "total": 1, "payload": "done"}
assert parse_packet("3/4:") == {"seq": 3, "total": 4, "payload": ""}
assert parse_packet("0/2:time: 9:30") == {"seq": 0, "total": 2, "payload": "time: 9:30"}
assert parse_packet("10/12:x")["seq"] == 10
assert parse_packet("10/12:x")["total"] == 12

2. Parse a Delimited Message

Approach — split on the agreed delimiter. A delimited protocol fixes one separator character; the receiver recovers the fields by splitting on it. Splitting "a|b|c" on "|" gives the three parts directly.

def parse_delimited(message):
    parts = message.split("|")
    return {"to": parts[0], "from": parts[1], "body": parts[2]}

Why it works. With exactly two delimiters, str.split("|") yields exactly three fields in order, and an empty trailing field becomes "" — the body can legitimately be empty.

Complexity. Time O(n), Space O(n) — one scan to split the string into parts.

assert parse_delimited("alice|bob|meet at 4") == {
    "to": "alice",
    "from": "bob",
    "body": "meet at 4",
}
assert parse_delimited("server|client|200 OK") == {
    "to": "server",
    "from": "client",
    "body": "200 OK",
}
assert parse_delimited("a|b|") == {"to": "a", "from": "b", "body": ""}

3. HTTP Status Text

Approach — fixed codes, fixed meanings. A protocol assigns each common message a fixed numeric code with a shared meaning, so the receiver branches on the number. A small chain of if checks maps each known code to its phrase, with a fallback for the rest.

def status_text(code):
    if code == 200:
        return "OK"
    if code == 404:
        return "Not Found"
    if code == 500:
        return "Server Error"
    return "Unknown"

Why it works. Each return exits as soon as its code matches, and any code outside the agreed set falls through to the final "Unknown".

Complexity. Time O(1), Space O(1) — a constant number of comparisons.

assert status_text(200) == "OK"
assert status_text(404) == "Not Found"
assert status_text(500) == "Server Error"
assert status_text(302) == "Unknown"

4. Does a Peer Know a Neighbor?

Approach — membership in the neighbor list. In the peer-network dict, network[peer] is the small set of computers peer knows directly. Whether it knows target is just whether target appears in that list.

def knows_neighbor(network, peer, target):
    return target in network[peer]

Why it works. The in operator tests list membership, which is precisely "does this peer know that neighbor directly" — no relaying involved.

Complexity. Time O(k), Space O(1), where k is the number of neighbors peer has.

assert knows_neighbor({"Ada": ["Ben", "Cora"], "Ben": ["Ada"]}, "Ada", "Cora") is True
assert knows_neighbor({"Ada": ["Ben", "Cora"], "Ben": ["Ada"]}, "Ada", "Eli") is False
assert knows_neighbor({"Ada": ["Ben"], "Ben": ["Ada"]}, "Ben", "Ada") is True

5. Reassemble or Report the Gap

Approach — index by seq, then check the range before joining. TCP cannot deliver a stream until it is both ordered and complete. Collect the packets into a dict keyed by sequence number (duplicates overwrite with the identical payload). The highest key tells you how long the stream should be; walk range(highest + 1) once to find any sequence number that never arrived. If none are missing, join the payloads in seq order; if some are, return them sorted instead.

def reassemble(packets):
    by_seq = {}
    for seq, payload in packets:
        by_seq[seq] = payload
    if not by_seq:
        return ""
    highest = max(by_seq)
    missing = []
    for seq in range(highest + 1):
        if seq not in by_seq:
            missing.append(seq)
    if missing:
        return missing
    pieces = []
    for seq in range(highest + 1):
        pieces.append(by_seq[seq])
    return "".join(pieces)

Why it works. Keying by sequence number collapses duplicates and lets range(highest + 1) step through the seqs the stream should contain. Because that range is built in order, the missing list comes out sorted for free, and on a complete stream the same range drives the join — so the payloads are concatenated in exactly seq order regardless of arrival order. The empty-input guard returns "" before max is ever asked for an empty dict.

Complexity. Time O(n), Space O(n) — one pass to index the packets and one pass over the sequence range, both linear in the number of packets.

assert reassemble([(0, "send"), (1, "and"), (2, "recv")]) == "sendandrecv"
assert reassemble([(2, "recv"), (0, "send"), (2, "recv"), (1, "and")]) == "sendandrecv"
assert reassemble([(0, "send"), (2, "recv")]) == [1]
assert reassemble([(4, "d"), (0, "a"), (2, "c")]) == [1, 3]
assert reassemble([(0, "only")]) == "only"
assert reassemble([]) == ""

6. Parse a Fixed-Format Message

Approach — count positions, read the length field. A fixed format pins each field to a known position. Read the to/from bytes at indices 0 and 1, read the length at index 2, then slice exactly length data bytes starting at index 3.

def parse_fixed(packet_bytes):
    to = packet_bytes[0]
    sender = packet_bytes[1]
    length = packet_bytes[2]
    data = packet_bytes[3:3 + length]
    return {"to": to, "from": sender, "data": data}

Why it works. The protocol agrees the layout in advance, so parsing is arithmetic on positions: the length field says how many data bytes follow, and the slice [3:3 + length] takes exactly that many, ignoring anything after.

Complexity. Time O(L), Space O(L) in the data length L — the data slice is the only non-constant work.

assert parse_fixed([2, 1, 2, 72, 73]) == {"to": 2, "from": 1, "data": [72, 73]}
assert parse_fixed([5, 9, 0]) == {"to": 5, "from": 9, "data": []}
assert parse_fixed([7, 3, 1, 65, 66, 67]) == {"to": 7, "from": 3, "data": [65]}

7. A Read/Write Request Handler

Approach — a request handler over server state. This extends the handle_request model from 4.6-B with a write method. "GET" looks the path up in the server's state dict and returns 200 or 404; "PUT" mutates the state and echoes the body; anything else is a 405.

def handle_request(server_data, method, path, body):
    if method == "GET":
        if path in server_data:
            return {"status": 200, "body": server_data[path]}
        return {"status": 404, "body": "Not Found"}
    if method == "PUT":
        server_data[path] = body
        return {"status": 200, "body": body}
    return {"status": 405, "body": "Method Not Allowed"}

Why it works. The protocol fixes the meaning of each status code, and the handler branches on the method just as a real server does. Because "PUT" assigns into server_data, the server's state changes, so a later "GET" on the same path now finds the key and returns 200.

Complexity. Time O(1), Space O(1) — a dict lookup or assignment per request.

store = {"/": "Home", "/about": "About"}
assert handle_request(store, "GET", "/about", None) == {"status": 200, "body": "About"}
assert handle_request(store, "GET", "/missing", None) == {"status": 404, "body": "Not Found"}
assert handle_request(store, "PUT", "/contact", "Email us") == {"status": 200, "body": "Email us"}
assert handle_request(store, "GET", "/contact", None) == {"status": 200, "body": "Email us"}
assert handle_request(store, "DELETE", "/about", None) == {"status": 405, "body": "Method Not Allowed"}

8. How Many Packets?

Approach — integer ceiling division. The number of packets is the message length divided by capacity, rounded up. Compute the full packets with //, then add one more only if there is a non-zero remainder for the leftover bytes.

def count_packets_needed(length, capacity):
    if length == 0:
        return 0
    full = length // capacity
    if length % capacity == 0:
        return full
    return full + 1

Why it works. length // capacity counts the fully packed packets; length % capacity is the leftover, which needs exactly one extra short packet whenever it is non-zero. The length == 0 guard returns 0 directly, since an empty message needs no packets.

Complexity. Time O(1), Space O(1) — pure arithmetic, no imports.

assert count_packets_needed(11, 4) == 3
assert count_packets_needed(10, 3) == 4
assert count_packets_needed(8, 4) == 2
assert count_packets_needed(0, 5) == 0
assert count_packets_needed(1, 65535) == 1
assert count_packets_needed(65535, 65535) == 1
assert count_packets_needed(65536, 65535) == 2

9. Route a Message Through the Network

Approach — recursive neighbor forwarding with a visited set. This builds on can_reach from 4.6-C, but returns the actual hop path. Mark start as visited, try each unvisited neighbor in order, and when a recursive call returns a route, prepend start to it. The shared visited list stops the search from looping back through peers that know each other.

def find_route(network, start, target, visited):
    if start == target:
        return [start]
    visited.append(start)
    for neighbor in network[start]:
        if neighbor not in visited:
            rest = find_route(network, neighbor, target, visited)
            if rest is not None:
                return [start] + rest
    return None

Why it works. A message reaches a non-neighbor by composition of single hops. Each call either is the target (base case, route [start]), finds a neighbor whose sub-route succeeds (prepend start), or exhausts its neighbors and reports None. Appending to visited before recursing guarantees the search ends even when neighbors point back at each other.

Complexity. Time O(V+E), Space O(V) — each peer is visited once, every neighbor link is examined once, and the recursion depth and visited list grow with the number of peers V.

10. Replica Health Across Peers

Approach — count online copies per record. Peer-to-peer storage keeps multiple copies on purpose so data survives a lost peer. For each record, count how many of its storing peers are still online, then map the count to a status: two or more is safe, exactly one is at risk, zero is lost.

def replica_health(storage, online):
    report = {}
    for record in storage:
        copies = 0
        for peer in storage[record]:
            if peer in online:
                copies = copies + 1
        if copies == 0:
            report[record] = "lost"
        elif copies == 1:
            report[record] = "at risk"
        else:
            report[record] = "safe"
    return report

Why it works. Only online peers can serve a copy, so counting the storing peers that are also in online is exactly the number of live replicas. The thresholds capture the recovery story from 4.6-C: with two or more copies a single failure is survivable, with one copy the record is one failure from lost, and with none it is already gone.

Complexity. Time O(R⋅P), Space O(R) — for each of the R records, scan its peer list and test each against online; the report holds one entry per record.

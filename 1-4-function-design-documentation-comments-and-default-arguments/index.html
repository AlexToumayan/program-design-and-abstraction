Chapter 1/1.4 Designing Functions

1.4 Designing Functions: Documentation, Comments, and Default Arguments

◷ approx. 14 min read · 6 worked examples

You already know how to define a function. This lesson is about defining a function well. By the end of it you will be able to name what separates a good function from a working-but-messy one, write a docstring that tells a future reader how to use your function, leave comments that explain a single tricky line, and give a parameter a default argument value so the common case is easy to call. You will also learn to tell apart the two jobs the = symbol does, which trips up almost everyone at first.

The thread running through all of it is one idea: a function is an abstraction. It hides a chunk of work behind a name, and the better you design it, the less anyone, including future you, has to think about what is inside.

A Good Function Is a Small, Honest Promise

Think about the tools in a kitchen drawer. A whisk does one thing, and you can tell what it does by looking at it. Nobody has to read a manual to use a whisk. Now imagine a single gadget that was a whisk, a can opener, a meat thermometer, and a flashlight fused together. Even if it technically worked, you would dread reaching for it, because you would have to relearn it every time.

Functions are the same. A well-designed function is like the whisk: it does one recognizable job, it announces what it expects, and you can trust it without taking it apart. A badly designed function is the fused gadget: it forces the reader to hold every internal detail in their head just to use it once.

That is the whole motivation for the rules in this lesson. They are not etiquette. They are what lets a reader say, "I know what this promises, I know what to feed it, I know what comes back, and I do not need to re-read its body every single time."

Three Guidelines for Designing Functions

Composing Programs gives three concrete guidelines for what makes a function good. They all reinforce the same point: functions are abstractions.

Each function should have exactly one job. That job should be identifiable with a short name and characterizable in a single line of text. If a function performs several jobs in sequence, that is a signal to split it into several functions, each named for its own job.

Do not repeat yourself. This is so central to software engineering that it has a name, the DRY principle: multiple fragments of code should not describe redundant logic. Instead, write the logic once, give it a name, and apply that name wherever you need it. The practical tell is simple. If you find yourself copying and pasting a block of code, you have almost certainly found a place that wants to become a function.

Functions should be defined generally. A function that solves a slightly broader problem is usually more useful than one nailed to a single special case. The classic example: there is no square function in the Python Library, precisely because squaring is just a special case of pow, which raises numbers to arbitrary powers. One general tool beats a pile of narrow ones.

These guidelines make code more readable, reduce errors, and often shrink the total amount of code you write. Breaking a complex task into clean, concise functions is a skill that takes real practice. The good news is that Python gives you several features built to support exactly this, and the rest of this lesson is about three of them.

Documentation: Writing for Future Readers

Code is written once but read many times, often by someone who has completely forgotten the details, including you in three weeks. Documentation is how a function explains itself to that future reader.

A function definition will often include documentation describing it, called a docstring. A docstring is a string that sits as the first thing inside the function body, indented along with the rest of the body. By convention docstrings are triple quoted, which lets them span multiple lines. The first line states the function's job in one sentence. The lines after it can describe the arguments and clarify how the function behaves.

Here is the official example. It computes the pressure of an ideal gas, and its docstring documents the job and each parameter.

# (1)
def pressure(v, t, n):
    """Compute the pressure in pascals of an ideal gas.

    Applies the ideal gas law: http://en.wikipedia.org/wiki/Ideal_gas_law

    v -- volume of gas, in cubic meters
    t -- absolute temperature in degrees kelvin
    n -- particles of gas
    """
    k = 1.38e-23  # Boltzmann's constant
    return n * k * t / v

The science behind the return value is the ideal gas law:

PV=NkT

Solving that for the pressure P gives:

P=VNkT​

The function body computes n * k * t / v, which is exactly that fraction. Physics writes the quantities with capital letters because mathematics names things abstractly; Python needs concrete identifiers, so the parameters are lowercase. The docstring is the bridge between the two worlds for a human reader. Without it, someone reading the call pressure(v, t, n) sees three bare letters and has to guess. With it, the function states the role of each input up front.

Follow the Line: the Docstring

The triple-quoted string is what makes this a docstring rather than just a comment. Read it part by part.

"""Compute the pressure in pascals of an ideal gas.  ...  """
├─ """ ............... opens a triple-quoted string that may span many lines
├─ Compute the ...... first line: states the function's single job in one sentence
├─ (blank line) ..... separates the summary from the details below it
├─ v -- ............. documents the parameter v and what it represents
├─ t -- ............. documents the parameter t and what it represents
├─ n -- ............. documents the parameter n and what it represents
├─ """ ............... closes the triple-quoted string
└─ becomes the docstring because it is the first statement in the body

Two details carry the meaning. First, the triple quotes let the string run across many lines, which a normal single-quote string cannot do. Second, the indentation matters: because the string is indented inside the body and is the first statement there, Python attaches it to the function as its official documentation. Move it out of the body and it stops being a docstring.

Reading a Function with help

Because Python stores the docstring with the function, you can ask for it later. When you call help with a function's name as the argument, Python shows you that function's docstring. Type q to quit the help screen and return to the prompt.

>>> help(pressure)

The point is the link between writing and reading. You wrote the docstring once, inside the def. Anyone, at any later time, can run help(pressure) and get that explanation back without opening your source file. When you write Python programs, include docstrings for all but the simplest functions. The Python documentation even publishes docstring style guidelines so that conventions stay consistent across different projects.

Comments Are for Humans Alone

A comment is a different kind of note. In Python, a comment can be attached to the end of a line following the # symbol. Everything from the # to the end of the line is ignored by the interpreter; it is never evaluated, never stored, and never shown by help. Comments exist for humans alone.

You already saw one in pressure. The comment after the k assignment describes what that mysterious small number is.

# (2)
k = 1.38e-23  # Boltzmann's constant

Follow the Line: a Commented Assignment

k = 1.38e-23  # Boltzmann's constant
├─ k ................. the local name being assigned
├─ = ................. assignment: bind the name on the left to the value on the right
├─ 1.38e-23 ......... scientific notation for a very small float
├─ # ................. begins a comment; everything after it on this line is ignored
├─ Boltzmann's constant ... a human note explaining why this number is here
└─ binds k to 1.38e-23, and the comment runs nowhere

The 1.38e-23 is scientific notation, Python's way of writing a number that would be clumsy with all its zeros:

1.38×10−23

The e notation shows up constantly in programming because both enormous and tiny values are painful to type out in full.

Docstring Versus Comment

These two tools look similar but answer different reader questions, and keeping them straight will save you confusion later.

Docstring Comment
Where it lives First statement inside the function body After # on any line
Who reads it A caller asking how to use the function A reader of the source asking why a line is the way it is
Does help show it Yes, it is stored as documentation No, the interpreter ignores it entirely
Typical question it answers What does pressure do, and what inputs does it want? Why is this tiny number named k?

A strong function often has both. The docstring is the public face that tells you how to call it correctly; the comment is a private note about one local implementation detail. Use the docstring to document the whole function and the comment to explain a single confusing line.

Default Argument Values

Defining functions generally tends to add parameters, and functions with many arguments can be awkward to call and harder to read. Python's answer is the default argument value. You can give an argument a default in the def header. When someone calls the function, any argument that has a default becomes optional: if they leave it out, Python binds the default value to that parameter name instead.

Suppose an application almost always computes pressure for one mole of particles. You can make that the default for n.

# (3)
def pressure(v, t, n=6.022e23):
    """Compute the pressure in pascals of an ideal gas.

    v -- volume of gas, in cubic meters
    t -- absolute temperature in degrees kelvin
    n -- particles of gas (default: one mole)
    """
    k = 1.38e-23  # Boltzmann's constant
    return n * k * t / v

The number is Avogadro's number, the count of particles in one mole:

6.022×1023

Follow the Line: a Default in the Header

def pressure(v, t, n=6.022e23):
├─ def .............. begins a function definition
├─ pressure ......... the name the function will be bound to
├─ ( ................ opens the parameter list
├─ v ................ a required parameter: the caller must supply it
├─ t ................ a required parameter: the caller must supply it
├─ n=6.022e23 ....... an optional parameter: if omitted, use 6.022e23
├─ ) ................ closes the parameter list
├─ : ................ ends the header; the indented body follows
└─ defines pressure with two required parameters and one optional one

So the header now mixes two kinds of parameters. v and t are required: the caller has to provide them. n is optional: Python can fill it in with 6.022e23 if the caller omits it. The default is part of the recipe card itself, decided when the function is defined, so the common case becomes easy to write while a caller who needs a different amount can still pass their own value.

The = Symbol Means Two Different Things

This is the part that catches people. In the example above, the = symbol does two completely different jobs depending on where it sits.

In the def statement header, = does not perform assignment. It indicates a default value to use when pressure is called.

# (4)
def pressure(v, t, n=6.022e23):
    ...

By contrast, the assignment statement in the body of the function binds the name k to an approximation of Boltzmann's constant. That is ordinary assignment, the action you already know.

# (5)
k = 1.38e-23

Same symbol, different grammatical location, different meaning. In a header it sets a default; in the body it binds a name.

Calling With the Default, Then Overriding It

pressure is defined to take three arguments, but you can call it with only two. When you do, the value for n is taken from the def statement default. If you provide a third argument, the default is ignored.

>>> pressure(1, 273.15)
2269.974834
>>> pressure(1, 273.15, 3 * 6.022e23)
6809.924502

In the first call, Python binds v to 1 and t to 273.15, and since no third argument is given, n takes the default 6.022e23. The result is about 2269.974834. In the second call, the third argument 3 * 6.022e23 is supplied, so the default is ignored and n is three times as large. Because pressure scales directly with n in the formula, the answer is three times as large too, about 6809.924502. The math backs this up: tripling n while holding everything else fixed gives

v3nkt​=3⋅vnkt​

A Guideline on Where Values Should Live

This leads to a useful rule of thumb for design. Most data values used inside a function's body should be expressed as default values to named arguments, so they are easy to inspect and easy for the caller to change. Some values that never change, like the fundamental constant k, can instead be bound in the function body or in the global frame. The distinction is about flexibility: things a caller might reasonably want to vary become defaulted arguments; things that are truly fixed stay tucked inside.

A Step-by-Step Trace of the Two Calls

Walking the two calls through frames makes the default mechanism concrete. The definition in (3) binds pressure in the global frame. Each call then builds its own local frame for the parameters.

The first call, pressure(1, 273.15), omits the third argument, so n is filled from the default:

Global frame
pressure -> function pressure(v, t, n)

    pressure local frame  (call 1)
    v -> 1
    t -> 273.15
    n -> 6.022e23        (filled in from the def default, since omitted)
    k -> 1.38e-23        (bound by the assignment in the body)
    returns n * k * t / v  ==>  2269.974834

The second call, pressure(1, 273.15, 3 * 6.022e23), supplies all three arguments, so the default is never consulted:

    pressure local frame  (call 2)
    v -> 1
    t -> 273.15
    n -> 3 * 6.022e23     (supplied by the caller; default ignored)
    k -> 1.38e-23
    returns n * k * t / v  ==>  6809.924502

The shape of the body is identical in both calls. The only difference is where n came from, and that single difference is the entire idea of default arguments.

⚠Common Beginner Traps

  • Writing one function that does several jobs in sequence. If you cannot describe it in a single line, it probably wants to be split into several functions, each with its own name.
  • Copying and pasting a block of code instead of naming it once. Repeated logic is a signal to make a function, by the DRY principle.
  • Confusing a docstring with a comment. A docstring is the first string statement in the body and help can show it; a comment starts with # and the interpreter ignores it. If you expect help to print your # note, you will be surprised when it does not.
  • Reading n=6.022e23 in a header as if it runs an assignment every call. In a header, = sets a default value; in the body, = performs assignment. Same symbol, different job.
  • Leaving a bare constant like 1.38e-23 with no comment. A reader cannot connect it to the formula without a note saying what it is.
  • Thinking a defaulted parameter is forbidden when you do want to pass it. The default is just a fallback; supply your own value and it wins.

Show the answers
  1. The third argument is omitted, so n is filled from the header default 6.022e23. With v bound to 2 and t to 273.15, the call returns about 1134.987417.
  2. Python stores a docstring as the function's documentation, so help can display it, while comments after # are ignored entirely by the interpreter and never stored.

§Summary

Designing a function well means treating it as an abstraction that keeps one honest promise. The three guidelines, one job per function, do not repeat yourself, and define generally, all push toward code that is readable, less error-prone, and shorter overall. Python gives you concrete tools to support that. A docstring, a triple-quoted string as the first statement in the body, documents the function's job and arguments and can be retrieved later with help. A comment, anything after #, explains a single local line and is ignored by the interpreter. A default argument value, written as name=value in the def header, makes a parameter optional so the common case is easy to call while a caller can still override it; and the = in a header sets a default rather than performing the assignment it would in the body.

Practice/Chapter 4 · Coding Practice

4.3 · SQL Queries — Coding Practice

◷ approx. 11 min read

These problems build fluency with SQL as a declarative query language: projecting and computing columns with select and as, filtering rows with where, sorting with order by, combining tables with joins and self-joins, summarizing with the aggregate functions and group by / having, and generating sequences with recursive with clauses.

> Work each problem from the schema in the Setup block first. Write a query that produces the rows shown in the Examples, then open the matching solution below to check your SQL and the idea behind it.

All problems run against the same small library database, defined once in the Setup block at the top of the Solutions section. Each Example shows the query's output the way the sqlite> prompt prints it: one row per line, with column values separated by a vertical bar.

Easy

1. List Titles by Year

Write a query that returns each book's title and year, sorted from the earliest publication year to the latest.

Examples

Query:  select title, year from books order by year;
Output:
Emma|1815
Persuasion|1817
The Hobbit|1937
I, Robot|1950
Foundation|1951
Dune|1965
The Silmarillion|1977
Explanation: every row of books, projected to two columns and sorted by year ascending.

Constraints

  • Use only select, from, and order by.
  • Project exactly the title and year columns, in that order.

Task

-- Return: title, year  (one row per book, ordered by year ascending)
Hint

order by sorts in increasing order of the expression you give it, so a bare column name sorts ascending.

2. Long Books

Write a query that returns the title and pages of every book longer than 300 pages, sorted alphabetically by title.

Examples

Query:  select title, pages from books where pages > 300 order by title;
Output:
Dune|412
Emma|474
The Hobbit|310
The Silmarillion|365
Explanation: where keeps only rows whose pages exceed 300; order by title sorts them by name.

Constraints

  • Use a where clause to keep only the qualifying rows.
  • The comparison is strict: a 300-page book is not longer than 300.

Task

-- Return: title, pages  (only books with pages > 300, ordered by title)
Hint

where filters rows; the column list after select chooses which columns appear.

3. Book Ages

Treating the current year as 2024, write a query that returns the title and the age (2024 - year, named age) of every science-fiction book (genre = "scifi"), sorted from newest to oldest.

Examples

Query:  select title, 2024 - year as age from books where genre = "scifi" order by age;
Output:
Dune|59
Foundation|73
I, Robot|74
Explanation: age is computed per row as 2024 - year; sorting by age ascending puts the smallest age (newest book) first.

Constraints

  • Compute the age as a column expression and name it with as.
  • Filter to genre = "scifi" (SQL uses a single = for equality).

Task

-- Return: title, age  where age = 2024 - year, scifi only, ordered by age
Hint

A column description can be any expression, not just a stored column; as gives the computed column a name.

4. Heaviest First

Write a query that returns the title and pages of every book, sorted from the most pages to the fewest.

Examples

Query:  select title, pages from books order by -pages;
Output:
Emma|474
Dune|412
The Silmarillion|365
The Hobbit|310
Persuasion|277
I, Robot|253
Foundation|244
Explanation: order by sorts ascending, so negating pages puts the largest page count first.

Constraints

  • Sort with a single order by clause.
  • Do not assume a desc keyword; use the negation trick from the lesson.

Task

-- Return: title, pages  (every book, most pages first)
Hint

order by -pages makes the largest page count the smallest (most negative) sort key.

Medium

5. Who Borrowed What

The loans table records a title, a patron, and the loan length in days, but it does not store the author. Write a query that joins loans to books on the title and returns the patron, the book's author, and the loan days. Sort by patron, then by days, then by author.

Examples

Query:  select patron, books.author, days
               from books, loans where books.title = loans.title
               order by patron, days, books.author;
Output:
Liang|Herbert|21
Liang|Austen|30
Maria|Herbert|7
Maria|Asimov|14
Maria|Tolkien|14
Priya|Asimov|10
Priya|Asimov|14
Explanation: every loan paired with the matching book row, keeping only pairs whose titles agree.

Constraints

  • Join books and loans by listing both in the from clause.
  • Match the related rows with where; use a dot expression where a column name could be ambiguous.

Task

-- Return: patron, author, days  (each loan with its book's author),
-- ordered by patron, then days, then author
Hint

A raw join pairs every loan with every book; the where condition keeps only the pairs whose title columns match.

6. Genre Summary

Write a query that, for each genre, returns the genre, the number of books in it (count(*)), and the average page count (avg(pages)). Sort the result alphabetically by genre.

Examples

Query:  select genre, count(*), avg(pages) from books group by genre order by genre;
Output:
fantasy|2|337.5
romance|2|375.5
scifi|3|303.0
Explanation: group by genre forms one pile per genre; the aggregates summarize each pile independently.

Constraints

  • Use group by so the aggregates apply per genre, not over the whole table.
  • Include the grouping column in the select list so you can tell which row is which.

Task

-- Return: genre, count(*), avg(pages)  (one row per genre, ordered by genre)
Hint

Aggregates like count(*) and avg(...) collapse a group of rows into a single summary row.

7. Frequently Borrowed Titles

Write a query that returns each title that has been borrowed more than once, together with how many times it was borrowed (named times). Sort alphabetically by title.

Examples

Query:  select title, count(*) as times from loans group by title having count(*) > 1 order by title;
Output:
Dune|2
I, Robot|2
Explanation: group by title counts the loans per book; having keeps only the groups whose count exceeds one.

Constraints

  • Group the loans rows by title.
  • The "more than once" test depends on a group's count, so it belongs in having, not where.

Task

-- Return: title, times  (titles borrowed more than once, ordered by title)
Hint

count(*) > 1 cannot be checked for a single row; it can only be tested after a group is formed, which is exactly what having is for.

8. Same-Author Pairs

Write a query that returns every pair of distinct books written by the same author, as two title columns a.title and b.title. List each pair only once, with the alphabetically earlier title first. Sort by the first title, then the second.

Examples

Query:  select a.title, b.title from books as a, books as b
               where a.author = b.author and a.title < b.title
               order by a.title, b.title;
Output:
Emma|Persuasion
Foundation|I, Robot
The Hobbit|The Silmarillion
Explanation: self-join books with itself, keep same-author pairs, and a.title < b.title removes self-pairs and mirror duplicates.

Constraints

  • Join books to a second copy of itself using table aliases (as a, as b).
  • Use a dot expression to say which copy each column comes from.
  • The condition a.title < b.title should keep exactly one ordering of each pair.

Task

-- Return: a.title, b.title  (distinct same-author book pairs, earlier title first)
Hint

Two aliases turn one table into two independently named copies; comparing the titles alphabetically fixes a single order per pair.

Hard

9. Powers of Two

Using a recursive with clause, build a temporary table of the powers of two from 2^0 up to 2^8, then return each value, sorted in increasing order.

Examples

Query:  with powers(n, value) as (
          select 0, 1 union
          select n + 1, value * 2 from powers where n < 8
        )
        select value from powers order by value;
Output:
1
2
4
8
16
32
64
128
256
Explanation: the base row is (0, 1); each recursive row doubles value and increments n, stopping once n reaches 8.

Constraints

  • Define the sequence with a recursive with clause (a base row unioned with a recursive select that reads from the table being defined).
  • The exponent column n is the loop counter; stop extending once n is no longer below 8.

Task

-- Build powers(n, value): n from 0..8, value = 2^n, then return value sorted ascending
Hint

Each row carries enough state to make the next: increment n and double value, and let where n < 8 halt the growth.

10. Management Chain

The staff table pairs each employee name with their direct manager. Starting from Eve, write a recursive with clause that walks up the chain of managers, recording each person's depth (Eve is 0, her manager is 1, and so on), and stops before reaching the top boss Reed. Return one sentence per person of the form "<name> is level <depth>", ordered by depth.

Examples

Query:  with chain(name, depth) as (
          select name, 0 from staff where name = "Eve" union
          select staff.manager, chain.depth + 1
                 from staff, chain where staff.name = chain.name and staff.manager != "Reed"
        )
        select name || " is level " || depth from chain order by depth;
Output:
Eve is level 0
Di is level 1
Bo is level 2
Ada is level 3
Explanation: each recursive step joins the current person to their staff row to find the next manager up, adding one to depth, until the manager would be Reed.

Constraints

  • The base row is Eve at depth 0; each recursive row climbs one level toward the top.
  • Join the table being built (chain) to staff to look up the current person's manager.
  • Use || to build each output sentence, and != to stop before Reed.

Task

-- Build chain(name, depth) from Eve upward (excluding Reed),
-- then return "<name> is level <depth>" ordered by depth
Hint

Each recursive row reads a chain row, finds that person's staff entry, and emits their manager at depth + 1; the != guard stops the climb before the top boss.

Solutions

Setup

Every query runs against this in-memory SQLite database. Create the three tables once, then each solution below is a single query against them:

create table books as
  select "The Hobbit"        as title, "Tolkien"   as author, "fantasy"  as genre, 310 as pages, 1937 as year union
  select "Dune"                       , "Herbert"             , "scifi"             , 412         , 1965         union
  select "Foundation"                 , "Asimov"              , "scifi"             , 244         , 1951         union
  select "Emma"                       , "Austen"              , "romance"           , 474         , 1815         union
  select "Persuasion"                 , "Austen"              , "romance"           , 277         , 1817         union
  select "I, Robot"                   , "Asimov"              , "scifi"             , 253         , 1950         union
  select "The Silmarillion"           , "Tolkien"             , "fantasy"           , 365         , 1977;

create table loans as
  select "The Hobbit"  as title, "Maria" as patron, 14 as days union
  select "Dune"                 , "Liang"          , 21         union
  select "Dune"                 , "Maria"          , 7          union
  select "Foundation"           , "Priya"          , 14         union
  select "Emma"                 , "Liang"          , 30         union
  select "I, Robot"             , "Priya"          , 10         union
  select "I, Robot"             , "Maria"          , 14;

create table staff as
  select "Ada"   as name, "Reed"  as manager union
  select "Bo"           , "Ada"             union
  select "Cy"           , "Ada"             union
  select "Di"           , "Bo"              union
  select "Eve"          , "Di";

The verification harness loads exactly these tables with Python's sqlite3 module, runs each query below, and asserts the returned rows match the Examples.

1. List Titles by Year

Approach — projection plus order. A select describes a new table built from books: name the two columns you want, and let order by year sort the rows in increasing year. No row is filtered, so all seven books appear.

select title, year from books order by year;

Why it works. Each book row is projected to its title and year, and order by sorts the result by the year expression in ascending order.

Complexity. A full scan of the table plus a sort: time O(nlogn), space O(n) for the result rows.

Emma|1815
Persuasion|1817
The Hobbit|1937
I, Robot|1950
Foundation|1951
Dune|1965
The Silmarillion|1977

2. Long Books

Approach — filter then project. A where clause keeps only the rows whose pages value exceeds 300; the column list projects title and pages, and order by title sorts the survivors.

select title, pages from books where pages > 300 order by title;

Why it works. The filtering expression pages > 300 is evaluated once per row; only rows for which it is true are projected. Four of the seven books qualify.

Complexity. One pass to filter plus a sort of the survivors: time O(nlogn), space O(n).

Dune|412
Emma|474
The Hobbit|310
The Silmarillion|365

3. Book Ages

Approach — computed column. The output column 2024 - year is an expression evaluated per row, named age with as so order by age can refer to it. The where clause restricts the rows to science fiction first.

select title, 2024 - year as age from books where genre = "scifi" order by age;

Why it works. A column description can be any expression; 2024 - year produces a fresh number for each row, and naming it age lets the order by clause sort by that computed value. Smaller age means a newer book, so the newest scifi book sorts first.

Complexity. Filter plus sort over the scifi rows: time O(nlogn), space O(n).

Dune|59
Foundation|73
I, Robot|74

4. Heaviest First

Approach — negated sort key. order by always sorts in increasing order, so negating pages turns the largest page count into the smallest key, placing the longest book first.

select title, pages from books order by -pages;

Why it works. For two books with pages p1 > p2, the keys satisfy -p1 < -p2, so the heavier book sorts ahead. Every book appears, longest first.

Complexity. A full scan plus a sort: time O(nlogn), space O(n).

Emma|474
Dune|412
The Silmarillion|365
The Hobbit|310
Persuasion|277
I, Robot|253
Foundation|244

5. Who Borrowed What

Approach — join on a shared key. Listing books and loans in the from clause pairs every book with every loan; the where condition books.title = loans.title keeps only the pairs that describe the same book, so each loan picks up its book's author.

select patron, books.author, days
       from books, loans where books.title = loans.title
       order by patron, days, books.author;

Why it works. A raw join builds every combination of a books row and a loans row; matching the two title columns with = keeps exactly the meaningful pairs. The dot expression books.author says which table the author comes from, and the three-key order by makes the result fully determined.

Complexity. The raw join considers m⋅n pairs before filtering, then sorts the survivors: time O(mnlog(mn)), space O(mn) in the worst case.

Liang|Herbert|21
Liang|Austen|30
Maria|Herbert|7
Maria|Asimov|14
Maria|Tolkien|14
Priya|Asimov|10
Priya|Asimov|14

6. Genre Summary

Approach — group then aggregate. group by genre partitions the books into one pile per genre, and the aggregate functions count(*) and avg(pages) summarize each pile independently into a single row.

select genre, count(*), avg(pages) from books group by genre order by genre;

Why it works. Without group by, the aggregates would fold the whole table into one row; grouping by genre makes them apply per group, so each genre reports its own count and average page count.

Complexity. One pass to assign rows to groups plus a sort of the groups: time O(nlogn), space O(g) for the g groups.

fantasy|2|337.5
romance|2|375.5
scifi|3|303.0

7. Frequently Borrowed Titles

Approach — having on a group count. Group the loans by title, then keep only the groups whose row count exceeds one. The test count(*) > 1 depends on a whole group, so it lives in having, evaluated after the groups are formed.

select title, count(*) as times from loans group by title having count(*) > 1 order by title;

Why it works. group by title collects all loans of a book into one group; having count(*) > 1 discards the single-loan groups, leaving only titles borrowed more than once.

Complexity. One grouping pass plus a sort of the surviving groups: time O(nlogn), space O(g).

Dune|2
I, Robot|2

8. Same-Author Pairs

Approach — self-join with aliases. Two aliases a and b make books appear twice in one query. Pairing every a row with every b row and keeping a.author = b.author finds books sharing an author; a.title < b.title keeps a single, ordered copy of each pair and drops self-pairs.

select a.title, b.title from books as a, books as b
       where a.author = b.author and a.title < b.title
       order by a.title, b.title;

Why it works. Without the a.title < b.title guard, each unordered pair would appear twice (and every book would pair with itself). Demanding the first title sort before the second leaves exactly one row per distinct same-author pair.

Complexity. The self-join forms n2 pairs before filtering, then sorts the survivors: time O(n2logn), space O(n2) in the worst case.

Emma|Persuasion
Foundation|I, Robot
The Hobbit|The Silmarillion

9. Powers of Two

Approach — recursive with clause. A with clause defines a query-local table powers with a base row (0, 1) and a recursive row that reads from powers itself, incrementing the exponent n and doubling value. The guard where n < 8 stops the growth.

with powers(n, value) as (
  select 0, 1 union
  select n + 1, value * 2 from powers where n < 8
)
select value from powers order by value;

Why it works. Each row carries both the exponent n and the current power value, which is exactly the state needed to compute the next row. The recursion halts once n reaches 8, yielding 2^0 through 2^8.

Complexity. The recursion produces k+1 rows for exponents 0..k, each in constant work, then a sort: time O(klogk), space O(k).

10. Management Chain

Approach — recursive join up the hierarchy. The base row places Eve at depth 0. Each recursive row joins the current chain row to its staff entry to find that person's manager, emitting the manager at depth + 1, and the != guard stops the climb before the top boss Reed. The final || expression formats each sentence.

with chain(name, depth) as (
  select name, 0 from staff where name = "Eve" union
  select staff.manager, chain.depth + 1
         from staff, chain where staff.name = chain.name and staff.manager != "Reed"
)
select name || " is level " || depth from chain order by depth;

Why it works. Joining chain to staff on staff.name = chain.name looks up the current person's row, whose manager becomes the next name one level higher. The condition staff.manager != "Reed" prevents adding the row that would climb to Reed, so the recursion terminates at Ada.

Complexity. The chain has one row per level of the hierarchy; each step does a constant lookup, so for a chain of length d the cost is time O(dlogd) with the final sort, space O(d).

Eve is level 0
Di is level 1
Bo is level 2
Ada is level 3

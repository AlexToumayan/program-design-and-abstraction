3.5 · The Scheme Interpreter — Coding Practice

◷ approx. 11 min read · 31 worked examples

These problems build fluency with the Scheme interpreter you study in this section: representing Scheme programs as Pair data, looking names up through a chain of Frame environments, and extending scheme_eval with new special forms so the eval/apply engine can interpret more of the language. Every problem operates on the interpreter machinery reproduced once in the Setup block at the top of the Solutions section.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

The interpreter machinery — Pair, nil, scheme_list, Frame, LambdaProcedure, scheme_eval, scheme_apply, and make_global — is reproduced once in the Setup block at the top of the Solutions section. Copy that block into your interpreter before running any solution, then paste the solution under it.

1. Bind and Read a Name

Build a fresh Frame, define the binding symbol -> value in it, then return what lookup finds for that symbol. This is the round trip every name in a Scheme program makes: a name is written into a frame, and later read back out.

Examples

Output: 10
Explanation: x is bound to 10 in the frame, then looked up.

Input:  bind_and_read("score", 88)
Output: 88
Explanation: define writes the binding; lookup reads it back.

Input:  bind_and_read("+", "primitive add")
Output: 'primitive add'
Explanation: any value, even a string, can be bound to a symbol.

Constraints

  • symbol is a string; value is any Python value.
  • Use the Frame methods define and lookup — do not touch frame.bindings directly.

Explanation: lookup stops at the local frame.

Output: -4
Explanation: same value in both frames; the local one still answers first.

Constraints

  • symbol is a string; the two values are any Python values.
  • The local frame's parent must be the global frame.

9. Add an if Special Form

Write scheme_eval_if, a copy of scheme_eval that adds the if special form. An if combination has the shape (if test consequent alternative). Evaluate test; if it is truthy, evaluate and return consequent, otherwise evaluate and return alternative. Crucially, only ONE of the two branches is ever evaluated — that is exactly why if cannot be an ordinary call.

Examples

Input:  scheme_eval_if(scheme_list("if", scheme_list("=", 0, 0), 1, 2), make_global())

Explanation: (= 0 0) is True, so the consequent 1 is returned.

Input:  scheme_eval_if(scheme_list("if", scheme_list("=", 3, 0), 1, 2), make_global())
Output: 2
Explanation: (= 3 0) is False, so the alternative 2 is returned.

Input:  scheme_eval_if(scheme_list("if", scheme_list("=", 4, 4), scheme_list("+", 10, 5), scheme_list("*", 10, 5)), make_global())
Output: 15
Explanation: the test is True, so only (+ 10 5) is evaluated; (* 10 5) is skipped.

Constraints

  • The combination is (if test consequent alternative): test is rest.first, consequent is rest.second.first, alternative is rest.second.second.first.
  • Evaluate the test first, then exactly one branch; never both.

Solutions

Setup

These problems all run on the section's interpreter machinery — a Scheme program represented as Pair data, Frame environments with a parent link, and the mutually recursive scheme_eval / scheme_apply engine. Copy this block first, then paste any solution under it:

1. Bind and Read a Name

Approach — define then lookup. A frame is a table of bindings; define writes a binding into the current frame and lookup reads it back, exactly the two Frame operations the section names.

def bind_and_read(symbol, value):
    frame = Frame()
    frame.define(symbol, value)
    return frame.lookup(symbol)

Why it works. define always writes into the current frame, so the binding is present when lookup searches the same frame and finds it immediately.

Complexity. Time O(1), Space O(1). One binding written, one lookup in a single frame.

assert bind_and_read("x", 10) == 10
assert bind_and_read("score", 88) == 88
assert bind_and_read("+", "primitive add") == "primitive add"

2. Number of Elements in a Scheme List

Approach — walk the second links. A Scheme list is a chain of Pairs ending in nil; counting its length means following s.second from pair to pair until nil.

def list_length(s):
    count = 0
    while s is not nil:
        count = count + 1
        s = s.second
    return count

Why it works. Each Pair contributes one element and points to the rest through second; the loop stops exactly when it reaches the nil that ends the list.

Complexity. Time O(n), Space O(1). One pass over the n pairs, with only a counter kept.

assert list_length(scheme_list("+", 2, 3)) == 3
assert list_length(scheme_list(42)) == 1
assert list_length(nil) == 0

3. Classify a Scheme Expression

Approach — the evaluator's first question. scheme_eval sorts an expression into symbol, atom, or combination before choosing a rule; this function answers just that question in the same order.

def classify(expr):
    if isinstance(expr, str):
        return "symbol"
    if not isinstance(expr, Pair):
        return "atom"
    return "combination"

Why it works. Symbols are strings, combinations are Pairs, and everything else (numbers, booleans) is a self-evaluating atom; checking in that order matches the evaluator's own branching.

Complexity. Time O(1), Space O(1). Two type checks and nothing more.

assert classify("square") == "symbol"
assert classify(7) == "atom"
assert classify(scheme_list("+", 1, 2)) == "combination"

4. Local Binding Shadows the Parent

Approach — current frame answers first. Because lookup checks the current frame before following the parent link, a binding in the local frame shadows any binding for the same name in the parent.

def shadowed_value(symbol, global_value, local_value):
    g = Frame()
    g.define(symbol, global_value)
    local = Frame(g)
    local.define(symbol, local_value)
    return local.lookup(symbol)

Why it works. local.lookup(symbol) finds the symbol in local.bindings on its first check and returns immediately, never walking to the parent where the global value lives.

Complexity. Time O(1), Space O(1). Two frames built, a single immediate lookup.

assert shadowed_value("x", 10, 99) == 99
assert shadowed_value("size", 1, 2) == 2
assert shadowed_value("k", -4, -4) == -4

5. Count Frames a Lookup Visits

Approach — count the parent-link walk. A lookup consults the current frame, then its parent, then its parent's parent, until the binding is found; counting those frames measures how far outward the search walks.

def lookup_depth(env, symbol):
    depth = 1
    frame = env
    while symbol not in frame.bindings:
        frame = frame.parent
        depth = depth + 1
    return depth

Why it works. The counter starts at 1 for the starting frame; each miss steps to frame.parent and adds one, so the final count is the number of frames the real lookup would have consulted.

Complexity. Time O(d), Space O(1), where d is the depth at which the symbol is found. One step per frame in the walk.

6. Evaluate a Nested Arithmetic Program

Approach — run the eval/apply engine. A fresh global frame supplies the primitives, and scheme_eval recursively evaluates the operator and operands of every combination before applying, bottoming out at self-evaluating atoms.

def run_program(expr):
    env = make_global()
    return scheme_eval(expr, env)

Why it works. scheme_eval evaluates the inner (* 3 4) to 12 first, then evaluates (+ 2 12) — the operands are turned into values before the procedure is applied, which is the call rule.

Complexity. Time O(n), Space O(h), where n is the number of subexpressions and h the nesting depth. One eval per node; recursion depth equals nesting.

assert run_program(scheme_list("+", 2, scheme_list("*", 3, 4))) == 14
assert run_program(scheme_list("-", scheme_list("*", 5, 5), 1)) == 24
assert run_program(7) == 7

7. Define Then Use a Name in One Session

Approach — one frame, two forms. Evaluating the define form mutates the global frame in place, so the binding it creates is visible when the second expression is evaluated in that same frame.

def define_then_use(name, value_expr, use_expr):
    env = make_global()
    scheme_eval(scheme_list("define", name, value_expr), env)
    return scheme_eval(use_expr, env)

Why it works. The define branch of scheme_eval evaluates value_expr and calls env.define(name, ...); because env is reused, the later scheme_eval(use_expr, env) can lookup that name.

Complexity. Time O(n), Space O(h), in the size and depth of the two expressions. Two evaluations sharing one environment.

assert define_then_use("x", 10, scheme_list("+", "x", 5)) == 15
assert define_then_use("y", scheme_list("*", 6, 7), "y") == 42
assert define_then_use("z", 0, scheme_list("=", "z", 0)) is True

8. Add a quote Special Form

Approach — a new branch that does not evaluate. A special form is recognized by its first element and given its own rule; quote's rule is to hand back its operand as data, skipping the evaluation that an ordinary call would do.

def scheme_eval_q(expr, env):
    if isinstance(expr, str):
        return env.lookup(expr)
    if not isinstance(expr, Pair):
        return expr
    first, rest = expr.first, expr.second
    if first == "quote":
        return rest.first
    if first == "define":
        env.define(rest.first, scheme_eval_q(rest.second.first, env))
        return None
    procedure = scheme_eval_q(first, env)
    args = rest.map(lambda operand: scheme_eval_q(operand, env))
    return scheme_apply(procedure, args, env)

Why it works. The first == "quote" check fires before the call branch, so the quoted datum is returned untouched; all other expressions fall through to the original symbol, atom, define, and call rules.

Complexity. Time O(n), Space O(h), in the size and depth of the evaluated expression. The quote branch is O(1); other forms cost as before.

def quote_eval(datum):
    return scheme_eval_q(scheme_list("quote", datum), make_global())


assert quote_eval(scheme_list("+", 1, 2)).first == "+"
assert quote_eval("hello") == "hello"
assert quote_eval(42) == 42
# Ordinary calls still work through the extended evaluator:
assert scheme_eval_q(scheme_list("+", 2, scheme_list("*", 3, 4)), make_global()) == 14

9. Add an if Special Form

Approach — evaluate the test, then one branch. if is a special form precisely because it must not evaluate both branches; the evaluator checks the test and then evaluates only the matching consequent or alternative.

def scheme_eval_if(expr, env):
    if isinstance(expr, str):
        return env.lookup(expr)
    if not isinstance(expr, Pair):
        return expr
    first, rest = expr.first, expr.second
    if first == "if":
        test = scheme_eval_if(rest.first, env)
        if test:
            return scheme_eval_if(rest.second.first, env)
        else:
            return scheme_eval_if(rest.second.second.first, env)
    if first == "define":
        env.define(rest.first, scheme_eval_if(rest.second.first, env))
        return None
    procedure = scheme_eval_if(first, env)
    args = rest.map(lambda operand: scheme_eval_if(operand, env))
    return scheme_apply(procedure, args, env)

Why it works. The three parts are reached through rest.first, rest.second.first, and rest.second.second.first; evaluating the test first and then exactly one branch is what makes if differ from an ordinary call that would evaluate every operand.

Complexity. Time O(n), Space O(h), in the size and depth of the chosen branch. Only one branch is ever evaluated.

def run_if(expr):
    return scheme_eval_if(expr, make_global())


assert run_if(scheme_list("if", scheme_list("=", 0, 0), 1, 2)) == 1
assert run_if(scheme_list("if", scheme_list("=", 3, 0), 1, 2)) == 2
assert run_if(scheme_list("if", scheme_list("=", 4, 4),
                          scheme_list("+", 10, 5),
                          scheme_list("*", 10, 5))) == 15

10. Build and Apply a One-Argument Procedure

Approach — the environment model in miniature. A LambdaProcedure stores parameters, a body, and a parent frame; applying it makes a new frame whose parent is that saved environment, binds the parameter to the argument, and evaluates the body there.

def apply_lambda(param, body, arg):
    env = make_global()
    proc = LambdaProcedure(scheme_list(param), body, env)
    return scheme_apply(proc, scheme_list(arg), env)

Why it works. scheme_apply recognizes the LambdaProcedure, opens a call_frame extending proc.env, binds the parameter to the argument value, and then scheme_evals the body in that frame — so the body's reference to the parameter resolves to the argument.

Complexity. Time O(n), Space O(h), in the size and depth of the body. One new frame plus evaluating the body.

assert apply_lambda("x", scheme_list("*", "x", "x"), 5) == 25
assert apply_lambda("n", scheme_list("+", "n", 1), 41) == 42
assert apply_lambda("a", scheme_list("-", scheme_list("*", "a", "a"), "a"), 4) == 12

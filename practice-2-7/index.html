2.7 · Object Abstraction — Coding Practice

◷ approx. 12 min read · 31 worked examples

These problems drill the special methods that let your own classes join Python's built-in syntax: __repr__/__str__ for string conversion, __len__, __getitem__, __call__, the comparison and arithmetic dunders, computed @property attributes for multiple representations, and type-tag dispatch for generic operations across kinds.

> Work each problem from the starter signature first. Run your function against the Examples, then open the matching solution below to check your approach and its time and space cost.

Easy

1. Two String Forms

Write a Money class that stores a whole number of cents. Give it two string forms. The programmer-facing __repr__ returns Money(<cents>), the recipe that rebuilds it. The human-facing __str__ returns a dollar amount like $12.50, with the cents always shown as two digits.

Examples

Input:  repr(Money(1250))
Output: 'Money(1250)'
Explanation: __repr__ shows the constructor-style recipe.
Input:  str(Money(1250))
Output: '$12.50'
Explanation: __str__ formats dollars and a two-digit cents remainder.
Input:  str(Money(5))
Output: '$0.05'
Explanation: 5 cents is zero dollars and a padded 05.

Constraints

  • cents is an integer with cents >= 0.
  • Define both __repr__ and __str__; each must return a string.
  • Pad the cents remainder to two digits (e.g. with '{1:02d}').

Starter

class Money:
    def __init__(self, cents):
        self.cents = cents

    def __repr__(self):
        pass

    def __str__(self):
        pass
Hint

cents // 100 is the dollars and cents % 100 is the leftover cents; format the leftover with 02d so single digits get a leading zero.

Medium

5. Comparable Versions

Write a Version class holding a major and minor number. Define __eq__ so two versions are equal when both parts match, and __lt__ so versions order first by major, then by minor. Once __lt__ and __eq__ exist, Python derives >, and built-ins like min and sorted start working too.

Examples

Input:  Version(1, 4) < Version(2, 0)
Output: True
Explanation: a smaller major number comes first.
Input:  Version(2, 1) < Version(2, 3)
Output: True
Explanation: equal majors fall back to comparing minors.
Input:  Version(1, 4) == Version(1, 4)
Output: True
Explanation: both parts are equal.

Constraints

  • major and minor are non-negative integers.
  • Define __eq__ and __lt__; do not define __gt__ (Python reflects a > b into b < a).

Starter

class Version:
    def __init__(self, major, minor):
        self.major = major
        self.minor = minor

    def __repr__(self):
        return 'Version({0}, {1})'.format(self.major, self.minor)

    def __eq__(self, other):
        pass

    def __lt__(self, other):
        pass
Hint

In __lt__, compare major first; only when the majors tie does the minor decide.

6. Adding Durations

Write a Duration class storing a number of seconds. Support + with __add__ so adding two Duration objects gives a new Duration with the summed seconds. Also define __radd__ so that an int on the left, as in 60 + Duration(30), still works: when the left operand is a plain integer, add it as seconds. Use NotImplemented for operands you do not handle so Python can try the reflected method.

Examples

Input:  Duration(30) + Duration(45)
Output: Duration(75)
Explanation: __add__ sums the seconds and wraps the result.
Input:  60 + Duration(30)
Output: Duration(90)
Explanation: int + Duration: the int can't add a Duration, so Duration.__radd__ runs.
Input:  5 + Duration(5)
Output: Duration(10)
Explanation: the reflected add treats the int 5 as 5 seconds.

Constraints

  • seconds is a non-negative integer.
  • Define __add__ (Duration + Duration) and __radd__ (int + Duration).
  • Return NotImplemented for unsupported operand types; use isinstance to check.

Starter

class Duration:
    def __init__(self, seconds):
        self.seconds = seconds

    def __repr__(self):
        return 'Duration({0})'.format(self.seconds)

    def __eq__(self, other):
        return self.seconds == other.seconds

    def __add__(self, other):
        pass

    def __radd__(self, other):
        pass
Hint

Duration + Duration is handled by __add__; int + Duration falls through the int's __add__ and lands in Duration.__radd__, where other is the integer.

7. A Temperature With Two Scales

Build two representations of the same physical temperature. Celsius stores degrees in Celsius and exposes fahrenheit and kelvin as computed @property attributes. Fahrenheit stores degrees in Fahrenheit and exposes celsius and kelvin the same way. Because the properties recompute on every read, changing the stored value updates the derived readings automatically.

Examples

Input:  Celsius(100).fahrenheit
Output: 212.0
Explanation: 100 * 9 / 5 + 32 = 212.0, computed on access.
Input:  Fahrenheit(212).celsius
Output: 100.0
Explanation: (212 - 32) * 5 / 9 = 100.0.
Input:  Celsius(100).kelvin
Output: 373.15
Explanation: 100 + 273.15 = 373.15; both scales agree on this physical temperature.

Constraints

  • degrees is a number.
  • Expose the converted readings with @property so they are read as attributes (no parentheses).
  • Do not store the converted values; compute them on each read.

Starter

class Celsius:
    def __init__(self, degrees):
        self.degrees = degrees

    @property
    def fahrenheit(self):
        pass

    @property
    def kelvin(self):
        pass

class Fahrenheit:
    def __init__(self, degrees):
        self.degrees = degrees

    @property
    def celsius(self):
        pass

    @property
    def kelvin(self):
        pass
Hint

Each property is a one-line formula over the stored self.degrees; Fahrenheit.kelvin can reuse self.celsius and add 273.15.

8. A Shared Interface for Areas

Build a small shape hierarchy that compares by area. A base Shape defines __lt__ and __eq__ purely in terms of an area() method, without knowing which shape it holds. Then Rectangle, Square (a Rectangle with equal sides), and Circle (use area = 3 * radius * radius) each supply their own area(). Because the comparisons speak only to the shared area() interface, you can sort a mixed list of shapes.

Examples

Input:  Square(3) < Rectangle(2, 6)
Output: True
Explanation: Square area 9 is less than Rectangle area 12, compared via __lt__.
Input:  Rectangle(2, 6) == Circle(2)
Output: True
Explanation: both have area 12, so __eq__ (defined on the base) reports equal.
Input:  [shp.area() for shp in sorted([Rectangle(2, 6), Square(3), Circle(2)])]
Output: [9, 12, 12]
Explanation: sorted() uses the base __lt__, ordering the shapes by area.

Constraints

  • Define __lt__ and __eq__ once, on the base Shape, using only area().
  • Square should reuse Rectangle by calling Rectangle.__init__ with equal sides.
  • Use area = 3 * radius * radius for the circle (a deliberately simple stand-in for pi).

Starter

Hint

The base comparisons call self.area() and other.area(); each subclass only needs its own area(), and Square delegates to Rectangle.__init__(self, side, side).

Setup

1. Two String Forms

Approach — the two name tags. This is the repr/str split from 2.7-A: __repr__ returns the canonical, programmer-facing recipe Money(<cents>), and __str__ returns the readable, human-facing dollar amount. Each method must return a string.

class Money:
    def __init__(self, cents):
        self.cents = cents

    def __repr__(self):
        return 'Money({0})'.format(self.cents)

    def __str__(self):
        return '${0}.{1:02d}'.format(self.cents // 100, self.cents % 100)

Why it works. repr(m) calls m.__repr__() and str(m) calls m.__str__(). Integer division gives the dollars and % 100 gives the leftover cents, which 02d pads to two digits. Displaying the object in a list ([Money(1250)]) uses __repr__ on the element, so it shows Money(1250).

Complexity. Time O(1), Space O(1). Both methods do constant formatting work.

m = Money(1250)
assert repr(m) == 'Money(1250)'
assert str(m) == '$12.50'
assert repr(Money(5)) == 'Money(5)'
assert str(Money(5)) == '$0.05'
assert str(Money(307)) == '$3.07'
assert '{0}'.format(m) == '$12.50'
assert repr([Money(1250)]) == '[Money(1250)]'

2. Length of a Playlist

Approach — __len__ and the truthiness fallback. From 2.7-B: len(x) is sugar for x.__len__(), and when a class defines __len__ but no __bool__, Python uses the length as the truth value — length 0 is false, anything else is true.

class Playlist:
    def __init__(self, tracks):
        self.tracks = tracks

    def __len__(self):
        return len(self.tracks)

Why it works. len(playlist) dispatches to __len__, which reports the wrapped list's length. Because no __bool__ is defined, bool(playlist) falls back to that length, so an empty playlist is falsy and a nonempty one is truthy.

Complexity. Time O(1), Space O(1). len on a list is constant time.

empty = Playlist([])
mix = Playlist(['a', 'b', 'c'])
assert len(empty) == 0
assert len(mix) == 3
assert bool(empty) is False
assert bool(mix) is True
if not empty:
    assert True
else:
    assert False
assert len(Playlist(['only'])) == 1

3. Indexing a Diagonal

Approach — __getitem__ as bracket syntax. From 2.7-B, d[i] is translated into d.__getitem__(i). The method needs no stored sequence at all — it computes each element from a rule, returning the value only at the matching index.

class Diagonal:
    def __init__(self, value, size):
        self.value = value
        self.size = size

    def __getitem__(self, i):
        if i == self.size:
            return self.value
        return 0

Why it works. Every bracket access calls __getitem__ with the index. The method compares i to the stored size; only the one matching position returns value, and all others return 0.

Complexity. Time O(1), Space O(1). One comparison per access; nothing is stored per index.

d = Diagonal(7, 2)
assert d[0] == 0
assert d[1] == 0
assert d[2] == 7
assert d[5] == 0
assert d.__getitem__(2) == 7
assert Diagonal(9, 0)[0] == 9

4. A Callable Discount

Approach — the callable object. Following Adder from 2.7-B, defining __call__ makes an instance behave like a function. The remembered percent lives in self, the object's own state, the way a closure would keep it in a parent frame.

class Discount:
    def __init__(self, percent):
        self.percent = percent

    def __call__(self, price):
        return price - price * self.percent / 100

Why it works. Writing parentheses after an instance, half_off(80), triggers half_off.__call__(80). The method reads self.percent and returns the price with that percentage removed.

Complexity. Time O(1), Space O(1). A single arithmetic expression per call.

half_off = Discount(50)
assert half_off(80) == 40.0
assert half_off(10) == 5.0
ten_off = Discount(10)
assert ten_off(100) == 90.0
assert Discount(0)(42) == 42.0
assert Discount(25).__call__(40) == 30.0

5. Comparable Versions

Approach — comparison dunders. From 2.7-B, operators are special-method calls: a == b runs a.__eq__(b) and a < b runs a.__lt__(b). Defining just these two lets Python reflect > into the other operand's <, and lets min/sorted order instances.

class Version:
    def __init__(self, major, minor):
        self.major = major
        self.minor = minor

    def __repr__(self):
        return 'Version({0}, {1})'.format(self.major, self.minor)

    def __eq__(self, other):
        return self.major == other.major and self.minor == other.minor

    def __lt__(self, other):
        if self.major != other.major:
            return self.major < other.major
        return self.minor < other.minor

Why it works. __eq__ is true only when both parts match. __lt__ compares major first and only consults minor on a tie, which is exactly lexicographic order. a > b has no __gt__, so Python evaluates b < a instead, and min uses __lt__ to find the smallest.

Complexity. Time O(1), Space O(1). Each comparison checks at most two fields.

assert Version(1, 4) == Version(1, 4)
assert (Version(1, 4) == Version(1, 5)) is False
assert Version(1, 4) < Version(2, 0)
assert Version(2, 1) < Version(2, 3)
assert (Version(2, 3) < Version(2, 1)) is False
assert (Version(1, 0) < Version(1, 0)) is False
assert Version(3, 0) > Version(2, 9)
assert min(Version(2, 0), Version(1, 9)) == Version(1, 9)

6. Adding Durations

Approach — __add__ with a reflected __radd__. This is the full + protocol from 2.7-B. For x + y, Python tries x.__add__(y) first; if the left operand can't help (here, a plain int), it tries y.__radd__(x). Returning NotImplemented is what lets the protocol fall through to the reflected method.

class Duration:
    def __init__(self, seconds):
        self.seconds = seconds

    def __repr__(self):
        return 'Duration({0})'.format(self.seconds)

    def __eq__(self, other):
        return self.seconds == other.seconds

    def __add__(self, other):
        if isinstance(other, Duration):
            return Duration(self.seconds + other.seconds)
        return NotImplemented

    def __radd__(self, other):
        if isinstance(other, int):
            return Duration(self.seconds + other)
        return NotImplemented

Why it works. Duration + Duration is handled directly by __add__. For 60 + Duration(30), the int's __add__ cannot add a Duration, so Python calls Duration(30).__radd__(60), which treats the int as seconds and returns Duration(90).

Complexity. Time O(1), Space O(1). One type check and one addition per operation.

assert Duration(30) + Duration(45) == Duration(75)
assert 60 + Duration(30) == Duration(90)
assert (Duration(10) + Duration(0)).seconds == 10
assert (5 + Duration(5)).seconds == 10
assert repr(Duration(30) + Duration(45)) == 'Duration(75)'

7. A Temperature With Two Scales

Approach — computed @property attributes. From 2.7-C, @property turns a method into a read-only attribute, so a derived value is read with no parentheses, just like stored data. Each class stores one scale and computes the others fresh on every read, so they never go stale.

class Celsius:
    def __init__(self, degrees):
        self.degrees = degrees

    @property
    def fahrenheit(self):
        return self.degrees * 9 / 5 + 32

    @property
    def kelvin(self):
        return self.degrees + 273.15

class Fahrenheit:
    def __init__(self, degrees):
        self.degrees = degrees

    @property
    def celsius(self):
        return (self.degrees - 32) * 5 / 9

    @property
    def kelvin(self):
        return self.celsius + 273.15

Why it works. Reading c.fahrenheit runs the decorated method against the current self.degrees, so after c.degrees = 0 the next read recomputes to 32.0. Fahrenheit.kelvin reuses the celsius property, and because both classes expose kelvin, the two representations agree on a shared physical temperature.

Complexity. Time O(1), Space O(1). Each property is one arithmetic formula; nothing derived is stored.

boil_c = Celsius(100)
assert approx(boil_c.fahrenheit, 212.0)
assert approx(boil_c.kelvin, 373.15)
boil_c.degrees = 0
assert approx(boil_c.fahrenheit, 32.0)
assert approx(boil_c.kelvin, 273.15)
boil_f = Fahrenheit(212)
assert approx(boil_f.celsius, 100.0)
assert approx(boil_f.kelvin, 373.15)
assert approx(Celsius(100).kelvin, Fahrenheit(212).kelvin)

8. A Shared Interface for Areas

Approach — a shared interface plus message passing. From 2.7-C: write the comparison logic once on a base class, in terms of an area() message, and let each subclass answer that message its own way. Shape.__lt__ and Shape.__eq__ never ask which shape they hold; they only call area().

Why it works. Every comparison routes through the single area() interface, so a Square and a Circle compare just fine even though they store different data. Square reuses Rectangle by calling its __init__ with equal sides, and sorted orders a mixed list because it relies on __lt__.

Complexity. Time O(1) per comparison, Space O(1). Sorting a list of n shapes is O(nlogn) comparisons.

r = Rectangle(2, 6)
s = Square(3)
c = Circle(2)
assert r.area() == 12
assert s.area() == 9
assert c.area() == 12
assert s < r
assert (r < s) is False
assert r == c
assert max(r, s, c, key=lambda shp: shp.area()).area() == 12
ordered = sorted([r, s, c])
assert [shp.area() for shp in ordered] == [9, 12, 12]

9. Money in Two Representations With Type Tags

Approach — generic addition by type tag. This is the dispatching Number from 2.7-D. __add__ first compares the operands' type_tag strings: equal tags take the same-kind add, while differing tags look up a cross-type function in adders, keyed on the ordered pair of tags. Registering both orders keeps + commutative.

Why it works. Cents(50) + Cents(25) sees matching tags and calls Cents.add. Cents(50) + Dollars(1) sees ('cents', 'dollars'), finds add_cents_and_dollars, and converts the dollars to cents before adding. The mirror entry add_dollars_and_cents swaps its arguments and reuses the same function, so order does not matter.

Complexity. Time O(1), Space O(1). A tag comparison and at most one dictionary lookup per addition.

assert repr(Cents(50) + Cents(25)) == 'Cents(75)'
assert repr(Dollars(2) + Dollars(3)) == 'Dollars(5)'
assert repr(Cents(50) + Dollars(1)) == 'Cents(150)'
assert repr(Dollars(1) + Cents(50)) == 'Cents(150)'
assert (Cents(50) + Dollars(1)).cents == 150
assert (Dollars(2) + Cents(150)).cents == 350

10. A Vector With Many Special Methods

Approach — an operator-overloading suite. This gathers the special methods from 2.7-B into one class: length, indexing, equality, addition, scalar multiplication, and a callable interface. Each piece of built-in syntax (len, [], ==, +, *, ()) is a hidden call to the matching dunder.

class Vector:
    def __init__(self, coords):
        self.coords = coords

    def __repr__(self):
        return 'Vector({0})'.format(self.coords)

    def __len__(self):
        return len(self.coords)

    def __getitem__(self, i):
        return self.coords[i]

    def __eq__(self, other):
        return self.coords == other.coords

    def __add__(self, other):
        return Vector([a + b for a, b in zip(self.coords, other.coords)])

    def __mul__(self, scalar):
        return Vector([a * scalar for a in self.coords])

    def __call__(self, other):
        return sum(a * b for a, b in zip(self.coords, other.coords))

Why it works. len(v), v[i], v == w, v + w, v * 2, and v(w) each dispatch to the dunder of the same name. __add__ pairs coordinates with zip and adds them; __mul__ scales each component; __call__ zips and sums the products to give the dot product. __add__ and __mul__ return fresh Vector objects, so they compose.

Complexity. Time O(n) for +, *, and the call (one pass over n coordinates); O(1) for len, indexing, and per-element ==. Space O(n) for the new vectors + and * build.

v = Vector([1, 2, 3])
w = Vector([4, 5, 6])
assert len(v) == 3
assert v[0] == 1
assert v[2] == 3
assert repr(v) == 'Vector([1, 2, 3])'
assert v + w == Vector([5, 7, 9])
assert v * 2 == Vector([2, 4, 6])
assert v(w) == 32
assert Vector([2, 0]) == Vector([2, 0])
assert (Vector([2, 0]) == Vector([2, 1])) is False
assert (v + w) * 1 == Vector([5, 7, 9])
assert (v + w)(v) == 5 * 1 + 7 * 2 + 9 * 3

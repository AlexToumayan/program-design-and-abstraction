Chapter 2/2.8 Efficiency

2.8 Example: Exponentiation

◷ approx. 13 min read · 4 worked examples

By the end of this lesson you will be able to write three different functions that all compute the same thing, bn, and explain why they are not equally good. You will write a linear recursive version, an equivalent iterative version, and a logarithmic version that uses a trick called squaring. More importantly, you will be able to count the work each one does and say, in order-of-growth terms, why one of them leaves the other two far behind as the exponent grows.

The big idea here is not exponentiation itself. It is that the answer and the work to reach the answer are two different things. All three functions return the same number. They do wildly different amounts of work to get there.

A Real-World Analogy

Suppose you need 16 identical copies of a single page and you only have a machine that copies one sheet at a time.

The obvious method is to feed the original through, then make a copy of a copy, then another, one sheet per pass, until you have all 16. That is 15 passes through the machine. The work grows in lockstep with the number of copies you want: 100 copies would take 99 passes.

Now suppose the machine can copy a small stack at once. Make 1 copy from the original, so now you hold 2 sheets. Copy that stack of 2 to get 4. Copy the stack of 4 to get 8. Copy the stack of 8 to get 16. That is only 4 passes, because every pass doubles what you already have.

Both methods end with 16 identical pages. The pile of paper at the end is the same. But one method took 15 passes and the other took 4, because the second one reuses work it already did instead of starting over for each new sheet. That gap between "build it one at a time" and "double what you have" is exactly the gap between the slow exponentiation functions and the fast one in this lesson.

The Recursive Definition

We want a function that takes a base b and a positive integer exponent n and computes bn. Exponentiation is repeated multiplication, and repeated work has a natural recursive description. One copy of b, times the rest:

bn=b⋅bn−1

That rule reduces a bigger problem (bn) to a slightly smaller one of the same shape (bn−1). Like every recursion, it needs a base case so it stops. The base case is the exponent zero:

b0=1

It is worth being clear about why b0 is 1 rather than 0. The exponent counts how many copies of b get multiplied together. When zero copies remain, you want a starting value that does not change the product once the kept copies of b are multiplied back in. That value is 1, because 1 is the identity for multiplication:

1⋅b=b

So 1 is the correct value to return when no copies of b are left to contribute.

This definition translates directly into a recursive function.

Expected output:

32

Reading the Recursive Function

The body of exp is two lines. Read the base-case line one token at a time. This line is a statement, so the final row states the action it performs:

if n == 0:
├─ if .......... begins a conditional: run the body only when the test is true
├─ n ........... the current exponent
├─ == .......... equality test: is the left value equal to the right value
├─ 0 ........... the base-case exponent, zero copies of b remain
├─ : ........... ends the header; the indented body follows
└─ when n is 0, return 1 and stop the recursion

Now read the recursive line. This is the line that does the actual reduction. It is an expression returned from the function, so the final row gives the value it produces:

return b * exp(b, n-1)
├─ return ...... hand this value back to whoever called exp
├─ b ........... one kept copy of the base
├─ * ........... multiply the kept copy by the rest of the power
├─ exp ......... call exp again to compute the smaller power
├─ ( ........... begins the arguments
├─ b ........... same base, unchanged
├─ , ........... separates the arguments
├─ n-1 ......... one smaller exponent, the remaining copies of b
├─ ) ........... ends the arguments
└─ evaluates to b times b raised to the n minus 1

The shape to notice: exp keeps exactly one b for itself and asks a recursive call to produce all the rest. Each level peels off one factor.

Tracing the Recursive Work

A trace makes the cost visible. For exp(2, 5), Python cannot finish the top multiplication until the inner call returns, so the calls stack up before any multiplying happens:

exp(2, 5)
= 2 * exp(2, 4)
= 2 * (2 * exp(2, 3))
= 2 * (2 * (2 * exp(2, 2)))
= 2 * (2 * (2 * (2 * exp(2, 1))))
= 2 * (2 * (2 * (2 * (2 * exp(2, 0)))))
= 2 * (2 * (2 * (2 * (2 * 1))))
= 32

Read top to bottom and you see the chain grow until exp(2, 0) hits the base case and returns 1. Then read bottom to top and the multiplications resolve one at a time on the way back up.

Two costs follow from this picture. The number of multiplications is one per level, so the time grows in proportion to n:

Θ(n)

The space also grows in proportion to n, because all of those calls are still open and waiting at the same time. Each waiting call keeps its own frame on the stack until the base case is reached:

Θ(n)

This is a linear recursive process: linear time and linear space.

An Equivalent Iterative Version

Just as with factorial, we can rewrite this same computation as a linear iteration. It does a similar number of multiplications but uses constant space, because it never builds a stack of waiting calls. Instead it carries the running product forward in a single variable.

Expected output:

32

The loop header deserves a token-by-token read, because the underscore surprises people. It is a statement that sets up repetition, so the final row states the action:

for _ in range(n):
├─ for ......... begins a loop that repeats the body
├─ _ ........... the loop variable; the underscore signals we will not use its value
├─ in .......... draws successive values from the sequence on the right
├─ range ....... the function that produces the numbers 0, 1, ..., n minus 1
├─ ( ........... begins the argument to range
├─ n ........... how many values to produce, so the body runs n times
├─ ) ........... ends the argument
├─ : ........... ends the header; the indented body follows
└─ run the body n times, ignoring the loop value each time

The underscore _ is an ordinary Python name. Programmers use it by convention to say: "this loop needs to repeat a fixed number of times, but I do not care about the counter's value." Here the body only needs to fire n times; the actual numbers range hands out are irrelevant, so they go into _ and are ignored.

The body multiplies the running result by b once per pass. After n passes, result holds bn.

This version still performs n multiplications, so its time is unchanged:

Θ(n)

But it keeps only the single name result (plus the loop machinery) alive at any moment, rather than a growing chain of frames. Its space is constant:

Θ(1)

A Better Idea: Successive Squaring

We can compute exponentials in far fewer steps by using successive squaring. Look at what the recursive method does for b8. It multiplies eight copies, one factor per step:

b⋅(b⋅(b⋅(b⋅(b⋅(b⋅(b⋅b))))))

That is seven multiplications. But there is a shortcut. Instead of building the power one factor at a time, build it by repeatedly squaring a power you already have:

b2=b⋅b
b4=b2⋅b2
b8=b4⋅b4

Three multiplications reach the eighth power. Each line reuses the result of the line above it, so each step doubles the exponent instead of adding one to it.

This is the moment to pause. The answer, b8, has not changed. What changed is the path to it. By squaring, each multiplication doubles the exponent we have reached, the same way each pass of the stack copier doubled the pile of paper.

Handling Any Exponent: Even and Odd

Squaring works perfectly when the exponent is a power of 2, because you can keep halving it down to 1. To handle a general exponent, we need a rule for both even and odd values. The recursive rule is:

bn={(b21​n)2b⋅bn−1​if n is evenif n is odd​

Read it as two cases. When n is even, compute the power of half the exponent and square it, which halves the problem in one step. When n is odd, peel off one copy of b, which leaves n-1, an even exponent, ready for the squaring case on the next step.

A worked example makes the alternation concrete. Watch the exponent for 29:

2^9 = 2 * 2^8       (9 is odd: peel off one 2)
2^8 = (2^4)^2       (8 is even: halve and square)
2^4 = (2^2)^2       (4 is even: halve and square)
2^2 = (2^1)^2       (2 is even: halve and square)
2^1 = 2 * 2^0       (1 is odd: peel off one 2)

The odd steps add one factor; the even steps cut the exponent in half. Halving is what makes this fast.

Fast Exponentiation

The source expresses this method with two functions: a tiny helper, square, and the recursion itself, fast_exp.

Expected output:

1267650600228229401496703205376

The even branch is where the speedup lives. Read it one token at a time. It is an expression returned from the function, so the final row gives its value:

return square(fast_exp(b, n//2))
├─ return ...... hand this value back to the caller
├─ square ...... the helper that multiplies its input by itself
├─ ( ........... begins square's argument
├─ fast_exp .... recursive call to compute the half-power first
├─ ( ........... begins fast_exp's arguments
├─ b ........... same base, unchanged
├─ , ........... separates the arguments
├─ n//2 ........ floor division: half the exponent, kept as an integer
├─ ) ........... ends fast_exp's arguments
├─ ) ........... ends square's argument
└─ evaluates to the square of b raised to n divided by 2

The test that selects this branch also deserves a careful read. It is the parity check that decides even versus odd:

if n % 2 == 0:
├─ if .......... begins a conditional
├─ n ........... the current exponent
├─ % ........... remainder operator: what is left after dividing
├─ 2 ........... divide by two to test parity
├─ == .......... equality test
├─ 0 ........... a remainder of zero means n is even
├─ : ........... ends the header; the indented body follows
└─ when n is even, halve the exponent and square the result

A few details are easy to get wrong. The // operator is floor division, which keeps the exponent an integer; plain / would produce a float like 50.0, which is the wrong type for an exponent here. The expression square(fast_exp(...)) computes the smaller power first and then squares it, not the other way around, because Python must evaluate the argument before it can call square. And the else: pairs with the second if, the parity test, not with the base-case if.

Why the Growth Is Logarithmic

The process evolved by fast_exp grows logarithmically with n in both space and number of steps. Here is the reasoning. Computing b2n with fast_exp requires only one more multiplication than computing bn, because b2n is just bn squared. So each additional multiplication you are allowed roughly doubles the size of the exponent you can reach.

Turn that around: to reach an exponent of n, the number of multiplications grows about as fast as the logarithm of n base 2 (a logarithm base 2 is just the count of how many times you can double from 1 before you reach n). The process has:

Θ(logn)

growth. You can watch the halving directly. For fast_exp(2, 100), the exponent moves like this on the way down to the base case:

100 -> 50 -> 25 -> 24 -> 12 -> 6 -> 3 -> 2 -> 1 -> 0

The even values halve; the odd values drop by one to become even. Either way the exponent shrinks fast.

The difference between Θ(logn) and Θ(n) becomes striking as n grows large. For an exponent of 1000, exp and exp_iter each do about 1000 multiplications, while fast_exp requires only about 14:

exp or exp_iter with n = 1000: about 1000 multiplications
fast_exp with n = 1000: about 14 multiplications

That 14 is not a contradiction of the base-2 reasoning: it is about 10 for the halving steps (you can double from 1 to 1000 roughly 10 times) plus a few more for the odd exponents peeled off along the way, which adds up to roughly 14 in total. The exact count depends on how many odd exponents show up along the way, but the shape of the growth is logarithmic, and that is what matters as the input scales.

A Step-by-Step Trace of fast_exp

To see the machine model, trace fast_exp(2, 10). First the calls descend, each one parking until its inner call returns:

fast_exp(2, 10)   n even -> square(fast_exp(2, 5))
fast_exp(2, 5)    n odd  -> 2 * fast_exp(2, 4)
fast_exp(2, 4)    n even -> square(fast_exp(2, 2))
fast_exp(2, 2)    n even -> square(fast_exp(2, 1))
fast_exp(2, 1)    n odd  -> 2 * fast_exp(2, 0)
fast_exp(2, 0)    base   -> 1

The chain of n values is 10 -> 5 -> 4 -> 2 -> 1 -> 0, only six calls for an exponent of 10. Then the values return upward, each level either squaring or multiplying by b:

fast_exp(2, 0) returns 1
fast_exp(2, 1) returns 2          (2 * 1)
fast_exp(2, 2) returns 4          (square of 2)
fast_exp(2, 4) returns 16         (square of 4)
fast_exp(2, 5) returns 32         (2 * 16)
fast_exp(2, 10) returns 1024      (square of 32)

Compare that with exp(2, 10), which would open eleven frames and do ten multiplications in a straight line. The fast version reaches the same 1024 with far fewer calls because each squaring step skips over an entire doubling of the exponent.

⚠Common Beginner Traps

  • Thinking that fewer recursive calls means a different answer. All three functions return the same value; they differ only in how much time and space they use to compute it.
  • Using / instead of // for the exponent. n / 2 produces a float such as 50.0, while n // 2 keeps the exponent an integer, which is what the recursion needs.
  • Forgetting the odd case. A function that only halves even exponents has no way to make progress on an odd value like 5, and the recursion never reaches the base case.
  • Attaching the else: to the wrong if. In fast_exp the else: pairs with the parity test if n % 2 == 0:, not with the base-case if n == 0:.
  • Reading Θ(logn) as "one step." It does not mean constant work. It means the number of steps grows slowly, because the exponent is repeatedly cut in half rather than reduced by one.

✓Check Your Understanding

  1. Why is b0 defined as 1 in the recursive definition, rather than 0?
  2. Why does exp use Θ(n) space, while exp_iter uses Θ(1) space, even though both do about n multiplications?
  3. Trace the first step of fast_exp(3, 6): which of the three branches runs first, and what exponent does the resulting recursive call receive?
  4. Why must fast_exp use // rather than / when it halves the exponent?
  5. Why does repeatedly halving the exponent lead to Θ(logn) growth?
Show the answers
  1. The exponent counts how many copies of b are multiplied, and 1 is the identity for multiplication, so 1⋅b=b. Returning 1 when zero copies remain lets the kept copies of b multiply back in without changing the product. Returning 0 would make every result 0.
  2. exp builds a chain of about n recursive calls that all stay open at once, waiting to multiply on the way back up, so the call stack grows with n. exp_iter carries the running product in a single name and updates it in a loop, so only a fixed amount of memory is alive at any moment.
  3. Since 6 is not 0 and 6 % 2 == 0, the even branch runs first: return square(fast_exp(3, 6//2)). The recursive call receives the exponent 3.
  4. The exponent passed to the recursive call must stay an integer. // is floor division and yields an integer like 50; / is true division and yields a float like 50.0, which is the wrong kind of value for an exponent here.
  5. Computing b2n costs only one more multiplication than computing bn, so each extra multiplication roughly doubles the exponent you can reach. Reaching an exponent of n therefore takes a number of steps that grows like the logarithm of n, which is exactly Θ(logn).

§Summary

Exponentiation is a small problem that exposes a large idea: the same mathematical result can be reached by processes with very different costs. The recursive exp peels off one factor per call and runs as a linear recursive process, Θ(n) time and Θ(n) space. The iterative exp_iter carries a running product forward, keeping the linear Θ(n) time but cutting space to Θ(1). The fast version, fast_exp, uses successive squaring with an even/odd rule so that each step roughly halves the exponent, giving Θ(logn) growth in both time and space. As the exponent grows, the difference is dramatic: about 1000 multiplications for the linear versions versus about 14 for the logarithmic one at n of 1000. The answer never changed; the path to it did.

▶Step-Through Checkpoint

This block runs fast_exp(2, 10), this lesson's logarithmic capstone. Stepping through it draws the environment diagram that explains the logarithmic growth: as the fast_exp frames stack up their n halves on each even step and drops by one on each odd step, 10 -> 5 -> 4 -> 2 -> 1 -> 0, so the chain stays short, and you see square frames open only on the way back up, never alongside the deepest fast_exp frame. Before you step through it, write down three predictions: how many local frames open over the whole run, counting square's alongside fast_exp's; whether any square frame is open at the moment the stack is deepest; and which function the very last frame to open belongs to.

Check each prediction as the frames appear. At the deepest point six fast_exp frames are open and square has none, because Python must finish evaluating the argument before it can call square. Every squaring happens on the way back up: three square frames, nine local frames in all, and the last frame to open is square's, turning 32 into the 1024 that the final line binds to result.

Chapter 4/4.3 Declarative Programming

4.3.1 Tables and Declarative Queries

◷ approx. 12 min read · 1 worked example

Up to now you have written programs that tell the computer how to do its work, step by step. This lesson introduces a different style, called declarative programming, where you describe what result you want and let a separate program work out the steps.

By the end of it you will be able to do five things: explain what a database and a table are, read a SQL session at the sqlite> prompt, build a one-row table with select, stack rows together with union, give a table a lasting name with create table, and read a whole stored table back with select *. The language you will use to do all of this is SQL.

Asking for the Result Instead of Giving the Recipe

Imagine ordering food at a restaurant. You do not walk into the kitchen and recite a procedure:

walk to refrigerator -> take out tomatoes -> chop tomatoes -> heat pan -> ...

You say one thing:

I would like tomato soup.

That is a declarative request. You state the result you want, and the kitchen decides the process that produces it. You do not care whether the cook chops before heating the pan or the other way around, as long as the soup arrives.

SQL works in exactly this spirit. A SQL query names the table of results you want to see. The database system, not you, decides how to compute it. Your job shifts from writing the recipe to describing the dish. In Chapter 1 a program was a recipe you wrote out step by step; declarative programming flips that — you describe the finished dish and let the system choose the recipe.

What a Database Is

Data values are often kept in large repositories called databases. A database is a store of data values together with an interface for retrieving and transforming those values. Each value stored in a database is called a record. Records that share the same structure are grouped into tables. You retrieve and transform records using queries, which are statements written in a query language. By far the most widely used query language today is Structured Query Language, or SQL, pronounced "sequel."

SQL is a declarative programming language. Its statements do not describe computations directly; they describe the desired result of some computation. It is the job of the database system's query interpreter to design and carry out a process that produces that result. This is a real departure from the procedural style of Python, where you, the programmer, spell out the computational process directly. A declarative language abstracts those procedural details away and lets you focus on the shape of the result.

A practical note before we start: the SQL standard is fixed, but most database systems add their own proprietary extensions. The examples in this lesson use the variant implemented in Sqlite. You can follow along by installing Sqlite or by using an online SQL interpreter.

Reading a SQL Session

Every example in this lesson is a short conversation with the Sqlite interpreter. You type a statement at the sqlite> prompt, end it with a semicolon, and the interpreter prints the result on the next line. When a statement runs past one line, Sqlite shows a ...> prompt and keeps reading until it reaches the closing semicolon.

Two pieces of grammar show up in almost every query. The keyword select means "construct a result table containing the values I describe." The semicolon ; means "this statement is finished." Without the semicolon, Sqlite assumes you have more to type and keeps waiting. With those two rules in hand, the smallest useful query is one that builds a single row, which is exactly where we start below.

What a Table Is

Every result a query hands back is a table, even one that holds a single cell. In general a table, also called a relation, has a fixed number of named and typed columns. Each row of the table is one data record and holds one value for each column.

A spreadsheet is a good mental picture. Suppose you want to record some cities by their map coordinates. You would want a column for how far north or south each city is, a column for how far east or west, and a column for its name:

Latitude Longitude Name
38 122 Berkeley
42 71 Cambridge
45 93 Minneapolis

The vertical categories are the columns. Here latitude and longitude both hold numeric values, and name holds a string. The horizontal records are the rows, and each row represents one city, fixing its location by its latitude and longitude. That is the structure we will build up piece by piece in SQL.

Creating One Row With select

You can construct a table with a single row directly, without storing anything first, using a select statement. The trick is to list the row's values separated by commas, and to attach a column name to each value with the keyword as. Every SQL statement ends in a semicolon.

Start with a warm-up that names a single column, so the as pattern is clear before three columns arrive at once:

select 38 as latitude;

That returns a one-row, one-column table whose column is named latitude and whose value is 38. Now bridge up to two columns, which forces you to use the comma as a separator:

select 38 as latitude, 122 as longitude;

That returns one row with two named columns. With the comma understood, here is the official three-column city row exactly as it appears in the source:

sqlite> select 38 as latitude, 122 as longitude, "Berkeley" as name;
38|122|Berkeley

The second line is the output. Each row is printed on its own line, with the column values separated by a vertical bar. So 38|122|Berkeley means a single row whose three column values are 38, 122, and Berkeley.

Read the official line one token at a time. Notice that as appears three times, once per column, and the commas separate the column descriptions from each other:

select 38 as latitude, 122 as longitude, "Berkeley" as name;
├─ select ......... begin a result table from the values that follow
├─ 38 ............. the value of the first column
├─ as latitude .... name that first column latitude
├─ , .............. separate this column description from the next
├─ 122 ............ the value of the second column
├─ as longitude ... name that second column longitude
├─ , .............. separate this column description from the next
├─ "Berkeley" ..... the text value of the third column
├─ as name ........ name that third column "name" (the column's name happens to be the word name)
├─ ; .............. end the statement
└─ returns        one row: 38 | 122 | Berkeley

Combining Rows With union

One row is not much of a table. To build a table with many rows, you combine several one-row select statements with union, which stacks the rows of two tables into one. The column names come from the table on the left, so only the first select needs to spell out the as names; the later rows just supply values in the same order. Spacing within a line does not affect the result.

As a bridge, here is union joining just two rows, which is enough to see how the pieces line up:

select 38 as latitude, 122 as longitude, "Berkeley" as name union
select 42, 71, "Cambridge";

The first select defines the columns latitude, longitude, and name; the second supplies a second row of values in that same order. Now here is the full official three-row table exactly as it appears in the source, with the values aligned by hand for readability:

sqlite> select 38 as latitude, 122 as longitude, "Berkeley" as name union
   ...> select 42,             71,               "Cambridge"        union
   ...> select 45,             93,               "Minneapolis";
38|122|Berkeley
42|71|Cambridge
45|93|Minneapolis

The ...> prompt is Sqlite telling you it is still reading one statement. The command is not finished until the final semicolon appears, so you can spread it across as many lines as you like. The extra spaces lining up the numbers are purely for human eyes; they change nothing about the result.

Here is the union keyword between the first two rows, read one token at a time. The leftmost select has already fixed the column names, so the second select carries only values:

... name union select 42, 71, "Cambridge" ...
├─ union ......... combine the rows of the table on the left with the table on the right
├─ select ........ begin the next row's values
├─ 42 ............ value for the first column, latitude (name reused from the left)
├─ , ............. separate this value from the next
├─ 71 ............ value for the second column, longitude
├─ , ............. separate this value from the next
├─ "Cambridge" ... value for the third column, name
└─ returns       a table with the Berkeley row stacked above the Cambridge row

Tracing How union Builds the Table

It helps to watch the three-row query assemble its result one step at a time. Each select produces a tiny one-row table, and each union stacks the next row underneath:

step 1: select 38 as latitude, 122 as longitude, "Berkeley" as name
        -> 38 | 122 | Berkeley                 (columns named latitude, longitude, name)

step 2: ... union select 42, 71, "Cambridge"
        -> 38 | 122 | Berkeley
           42 |  71 | Cambridge                (new row added below, same columns)

step 3: ... union select 45, 93, "Minneapolis"
        -> 38 | 122 | Berkeley
           42 |  71 | Cambridge
           45 |  93 | Minneapolis              (final table, three rows)

The column names are decided once, by the first row, and every later row simply slots its values into those same columns.

Naming a Table With create table

The three-row table above is useful, but it vanishes as soon as the query finishes. To keep it around so that later queries can refer to it, give it a name with a create table statement. This statement can also create empty tables, but here we use the form that names an existing table defined by a select.

sqlite> create table cities as
   ...>    select 38 as latitude, 122 as longitude, "Berkeley" as name union
   ...>    select 42,             71,               "Cambridge"        union
   ...>    select 45,             93,               "Minneapolis";

Notice that this statement produces no output rows. Its effect is to store a table, not to display one. The work is the same union query you already traced; the new part is the phrase create table cities as, which names the result.

Read that opening phrase one token at a time:

create table cities as <select ...> ;
├─ create table ... begin a statement that stores a named table
├─ cities ......... the name the stored table will have
├─ as ............. introduce the query whose result becomes that table
├─ <select ...> ... the union query that produces the three rows
├─ ; .............. end the statement
└─ does           evaluates the query and saves its result under the name cities

This is much like assignment in Python, where x = 2 + 2 evaluates the right side and binds the name x to the result. The difference is that the thing being named here is an entire table of rows, not a single value.

Reading a Whole Table With select *

Once a table has a name, that name can appear in a from clause inside a select statement, telling the query where to get its rows. To display every column of every row, use the special form select *, where the star stands for "all columns."

sqlite> select * from cities;
38|122|Berkeley
42|71|Cambridge
45|93|Minneapolis

The output is the full cities table you stored, printed one row per line. Read the line one token at a time:

select * from cities;
├─ select ..... begin a result table
├─ * .......... all columns of the source, in order
├─ from ....... name where the rows come from
├─ cities ..... the stored table to read rows from
├─ ; .......... end the statement
└─ returns    every column of every row of cities

The star here has nothing to do with multiplication. In a select clause, * means "every column." And from cities means "use the stored table named cities as the source of rows." Put together, the statement says: show every column from every row of cities.

The Machine Model of a Table Query

It is worth holding a rough picture of what happens when you press Enter on a query. The database does not read English. It reads characters, recognizes keywords like select and from, finds the table you named, and produces rows:

SQL text -> parser -> query plan -> table result

The important idea is the split you saw at the very start. You wrote the what: "every column from every row of cities." The database built the how: the actual plan that walks the stored rows and prints them. In this lesson you are learning the language from the user's side, describing results. Later sections build the interpreter that turns those descriptions into procedures.

⚠Common Beginner Traps

  • Forgetting the semicolon. Without ;, Sqlite shows the ...> prompt and keeps waiting, because it thinks the statement is unfinished.
  • Reading * as multiplication. In a select clause, * means all columns, not "times."
  • Thinking union can change the column names. The leftmost select fixes the column names; every later row only supplies values in that same order.
  • Repeating the as names on every union row. Only the first row needs them. The later rows are bare values.
  • Treating SQL as Python. SQL has its own grammar and keywords. You describe the result you want; you do not write the loop that builds it.

✓Check Your Understanding

  1. What is the difference between describing how to compute a result and describing what result you want?
  2. In select 38 as latitude, 122 as longitude, "Berkeley" as name;, which part gives the column latitude its name, and what does the comma do?
  3. When you combine rows with union, which select decides the column names of the final table?
  4. What does create table cities as ... accomplish that a plain select does not?
  5. In select * from cities;, what does * mean and what does from cities tell Sqlite?
  6. Write down the two output lines that Sqlite prints for this query, in order: select 40 as latitude, 105 as longitude, "Denver" as name union select 47, 122, "Seattle";
Show the answers
  1. Describing how gives step-by-step instructions for the machine to follow. Describing what states the desired result and lets the database's query interpreter choose the execution steps. SQL is declarative, so you describe the result.
  2. The phrase 38 as latitude names the value 38 as the column latitude; as attaches a column name to the value. The comma separates one column description from the next.
  3. The leftmost (first) select decides the column names. Later rows joined by union only supply values, reusing those same column names in order.
  4. create table cities as ... evaluates the query and stores its result under the name cities, so later queries can refer to it by name. A plain select only displays its rows once and stores nothing.
  5. The * means all columns, in order. The clause from cities tells Sqlite to use the stored table named cities as the source of rows.
  6. The query prints two rows, one per line, with column values separated by vertical bars:
   40|105|Denver
   47|122|Seattle

The first select names the columns latitude, longitude, name, and union stacks the Seattle row beneath the Denver row.

§Summary

Declarative programming describes the result you want and lets a separate interpreter find the steps. SQL is the declarative query language used with databases, where records of the same structure are grouped into tables, also called relations. A table has named, typed columns and rows of records. In this lesson you built tables from nothing and read them back: select with as names the columns of a single row, union stacks rows into a larger table, create table ... as gives that table a lasting name, and select * from reads every column of every row of a stored table. Each statement ends in a semicolon, and each describes a result rather than a procedure. Later sections turn these descriptions into the queries and interpreters that make a database useful.

▶Step-Through Checkpoint

SQL tables are not Python lists internally, but modeling rows as ordinary Python data first makes the rows-and-columns idea concrete before you return to SQL's more compact notation. The next block represents each row as a tuple of (latitude, longitude, name) and prints them with a comprehension, the same way select * from cities; prints the whole table. Stepping through it draws the environment diagram behind that comprehension: the cities list and its three (latitude, longitude, name) tuples sit on the heap, and as the comprehension runs you watch latitude, longitude, and name get rebound for each tuple while the new rows list fills with one formatted string per record. Before you step through it, write down three predictions: the exact string the comprehension builds from the first tuple, what name holds when the comprehension reaches the Cambridge tuple, and what row holds on the loop's final pass.

Here f"{latitude}|{longitude}|{name}" is a formatted string: each name in the braces is replaced by that variable's value, so the | bars sit between the three column values.

Check each prediction as you step. The list cities fills with three tuples, each one a record, and unpacking (latitude, longitude, name) is the Python echo of naming the three columns with as. The comprehension builds one formatted string per row, and the three printed lines, separated by vertical bars, are exactly what select * from cities; produces.
